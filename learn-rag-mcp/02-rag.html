Part 02 · Retrieval

RAG: Retrieval-Augmented Generation

The single most important technique for making LLMs useful with your data. By the end of this page you'll understand every moving part — embeddings, vector search, chunking, and the retrieve-then-generate loop.

⏱ ~18 min read Difficulty · Beginner Prereq · 01 · LLMs

1 · The problem RAG solves

From Part 01: an LLM's knowledge is frozen at training time and generic. Ask it "What's our refund policy?" or "Summarize the Q3 report we just published" and it has no idea — that information was never in its training data. Worse, it may hallucinate a confident, wrong answer.

⚠️
The naive fix doesn't scale

"Just paste the documents into the prompt!" works for one page — but you can't paste a 10,000-page knowledge base into every request (context window limits + cost + the model loses focus). We need to send only the relevant snippets.

2 · What RAG is (plain English)

Retrieval-Augmented Generation = Retrieval (find the relevant info) + Augmented (add it to the prompt) + Generation (let the LLM write the answer using it).

💡
The open-book exam analogy

A closed-book exam (plain LLM) forces you to answer from memory — you'll guess when unsure. RAG turns it into an open-book exam: before answering, you look up the relevant pages, then write your answer based on what you found. Same brain, far more accurate, with citations.

The trick is the "looking up" step. You can't keyword-match reliably — a user asking about "time off" should find a doc titled "vacation and leave policy" even though no words overlap. We need search by meaning, not by keywords. That's what embeddings give us.

3 · Embeddings — turning meaning into numbers

An embedding is a list of numbers (a vector) that represents the meaning of a piece of text. A special model — an embedding model — converts text → vector. Texts with similar meaning get vectors that are close together in space.

Similar meanings land near each other. Real embeddings have hundreds or thousands of dimensions; here we show 2 for intuition.

So if I embed the question "How do I reset my password?" and embed every paragraph of my docs, the paragraph about password resets will have the closest vector — even if it says "credential recovery." We measure closeness with cosine similarity (1.0 = identical meaning, 0 = unrelated).

In code, embedding is one call:

python
# OpenAI example
from openai import OpenAI
client = OpenAI()

vec = client.embeddings.create(
    model="text-embedding-3-small",
    input="How do I reset my password?"
).data[0].embedding

print(len(vec))   # 1536  → a list of 1536 floats
print(vec[:3])  # [0.021, -0.044, 0.013, ...]
🔑
Embedding model ≠ chat model

They're different models with different jobs. The chat model (e.g. gpt-4o, claude-opus-4-8) writes answers; the embedding model (e.g. text-embedding-3-small) only turns text into vectors. RAG uses both.

4 · Vector databases & similarity search

Once your documents are vectors, you need somewhere to store them and a fast way to ask "which stored vectors are closest to this query vector?" That's a vector database.

Vector DBBest forNotes
ChromaLearning, local prototypesPip-installable, runs in your script. We'll use this.

Pinecone / Weaviate / QdrantProduction, scaleManaged/hosted, handle millions of vectors.
pgvectorAlready using PostgresAdds vector search to your existing DB.

The core operation is always the same: db.query(query_vector, top_k=3) → returns the 3 most similar stored chunks. You don't implement the math; the DB does it efficiently.

5 · Chunking — splitting documents

You don't embed a whole 50-page PDF as one vector — that would blur all its meaning into a single point and you couldn't retrieve specific facts. Instead you split it into chunks (e.g. ~500–1000 characters each), and embed each chunk separately.

Each chunk becomes its own searchable vector. Retrieval returns whole chunks, so chunk size affects answer quality.
📐
Chunking rules of thumb
  • Size: ~500–1000 characters (or ~100–250 tokens) is a solid default.
  • Overlap: let chunks overlap by ~10–20% so a sentence split across the boundary isn't lost.
  • Respect structure: split on paragraphs/headings when possible, not mid-word.
  • Too big = retrieval is imprecise & wastes context. Too small = chunks lose meaning. Experiment.

6 · The full RAG pipeline, step by step

RAG has two phases. The indexing phase happens once (or whenever docs change). The query phase happens on every user question.

Top: build the index once. Bottom: for each question, embed it, retrieve the closest chunks, stuff them into the prompt, and let the LLM answer.

7 · What the final prompt actually looks like

This demystifies the whole thing. After retrieval, you literally build a string like this and send it to the LLM:

prompt
You are a helpful assistant. Answer the question using ONLY the
context below. If the answer isn't in the context, say
"I don't know based on the provided documents."

--- CONTEXT (retrieved chunks) ---
[Chunk 1] Employees accrue 1.5 vacation days per month, up to
a maximum of 25 days per year...

[Chunk 2] Unused vacation days roll over to the next year but
expire after 18 months...

--- QUESTION ---
How many vacation days can I earn in a year?
✅
This is the "aha" moment

RAG isn't magic — it's smart search + a well-built prompt. The LLM never touched your database. You retrieved the right text and handed it over. The instruction "use ONLY the context" is what dramatically reduces hallucination.

8 · RAG vs. fine-tuning vs. long context

ApproachWhat it doesUse when
RAGRetrieves facts at query time and injects themKnowledge changes often, needs citations, large/private corpora. Best default for facts.
Fine-tuningRe-trains the model to change its behavior/styleYou need a specific tone, format, or skill — not for injecting facts.
Long contextJust paste everything into a huge context windowSmall, one-off docs. Simple but costly & doesn't scale to big corpora.
💡
Common myth

"To make the AI know my docs, I should fine-tune it." Usually wrong. Fine-tuning teaches how to behave, not what facts to know. For "know my documents," reach for RAG first — it's cheaper, updatable instantly, and gives you sources.
