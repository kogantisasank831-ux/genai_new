Part 04 · Protocol

🔌 MCP: Model Context Protocol

Often called "USB-C for AI." MCP is an open standard that lets any AI app connect to any tool or data source the same way — no custom glue per integration. Here's what it is, why it matters, and how to build one.

⏱ ~18 min read Difficulty · Intermediate Prereq · Part 03 · Agents

1 · What MCP is (the USB-C analogy)

Before USB-C, every device had its own charger and cable. Chaos. USB-C created one connector that works across phones, laptops, monitors, and drives. MCP does the same for AI.

It's an open protocol (created by Anthropic, now broadly adopted) that defines a standard way for an AI application to discover and use external tools, data, and prompts. Build a tool as an MCP server once, and any MCP-compatible app — Claude Desktop, IDEs, custom agents — can use it instantly.

💡
In one line

MCP standardizes the "tool" concept from Part 03 so tools become reusable, shareable plugins instead of bespoke code locked inside one app.

2 · The N×M → N+M shift

Recall the spaghetti from Part 03: N apps each needing custom code for M tools = N×M integrations. MCP collapses this. Each tool implements MCP once (M servers); each app speaks MCP once (N clients). Total work = N+M.

Everyone speaks one protocol through a common standard. Write a tool once; every app can use it.

Without a standard, every app needs hand-written glue for every tool — and that work multiplies. With MCP it merely adds:

Without MCP · custom glueN apps × M tools = N×M bespoke integrations to write and maintain
With MCP · one standardM servers + N clients = N+M — each side implements the protocol once

3 · The architecture: hosts, clients, servers

🖥️

Host

The AI app the user interacts with — Claude Desktop, an IDE, your custom agent. It contains the LLM.

🔗

Client

Lives inside the host. One client per server connection; it speaks the MCP protocol.

⚙️

Server

A small program you (or a vendor) write that exposes tools/data. E.g. a "filesystem server" or "GitHub server."

The host's LLM uses its MCP client to talk to one or more servers. Each server is an independent, reusable program.
🔁
Transport

Servers run either locally (the host launches them and talks over stdio — standard input/output) or remotely (over HTTP). Under the hood the messages are JSON-RPC. The SDK hides all of this — you just write functions.

4 · The 3 things a server exposes ("primitives")

PrimitiveWhat it isExample
ToolsFunctions the model can call to take action (the main one)create_issue(), send_email(), search_docs()
ResourcesRead-only data the model can load as contextA file's contents, a DB schema, a config
PromptsReusable prompt templates the user can invoke"Summarize this PR", "Draft a reply"
🔑
Start with Tools

90% of MCP servers are mostly tools. Resources and prompts are nice extras you can add later.

5 · Simplest possible example: a "hello" tool

The official Python SDK (mcp, with FastMCP) makes a server astonishingly short. A tool is just a decorated function:

python
# pip install "mcp[cli]"
# file: hello_server.py
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("hello-server")

@mcp.tool()
def greet(name: str) -> str:
    """Return a friendly greeting for the given name."""
    return f"Hello, {name}! 👋"

if __name__ == "__main__":
    mcp.run()   # runs over stdio by default
🧩
That's a complete MCP server

Notice three things the SDK does for you: (1) the function name becomes the tool name, (2) the docstring becomes the description the LLM reads, (3) the type hints (name: str) auto-generate the input schema. You write a normal Python function; MCP handles the protocol.

6 · Build a real MCP server (full code)

Let's make something useful: a server with multiple tools and a resource. This one does math and exposes a tiny knowledge base.

python
# file: util_server.py    —    run with:  python util_server.py
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("utilities")

# ---- TOOL 1: reliable math (LLMs are bad at arithmetic) ----
@mcp.tool()
def multiply(a: float, b: float) -> float:
    """Multiply two numbers and return the exact result."""
    return a * b

# ---- TOOL 2: word count ----
@mcp.tool()
def word_count(text: str) -> int:
    """Count the number of words in a piece of text."""
    return len(text.split())

# ---- TOOL 3: a (fake) database lookup ----
EMPLOYEES = {"alice": "Engineering", "bob": "Sales"}

@mcp.tool()
def find_department(name: str) -> str:
    """Look up which department an employee works in."""
    return EMPLOYEES.get(name.lower(), "Not found")

# ---- RESOURCE: read-only data the model can load as context ----
@mcp.resource("info://company")
def company_info() -> str:
    """Basic facts about the company."""
    return "Acme Inc. Founded 2010. HQ in Berlin. 200 employees."

if __name__ == "__main__":
    mcp.run()
✅
Test it without an LLM

Run the MCP Inspector — a visual tool to poke at your server:
npx @modelcontextprotocol/inspector python util_server.py
It opens a browser UI where you can list and call your tools manually. Great for debugging before wiring up a model.

7 · Using your server in a client (Claude Desktop)

To let Claude Desktop use your server, add it to the config file and restart the app.

Config location: claude_desktop_config.json (macOS: ~/Library/Application Support/Claude/; Windows: %APPDATA%\Claude\).

json
{
  "mcpServers": {
    "utilities": {
      "command": "python",
      "args": ["C:\\path\\to\\util_server.py"]
    }
  }
}

Restart Claude Desktop. You'll see a 🔌/tools icon. Now ask: "What department does Alice work in?" and Claude will call your find_department tool automatically. You just extended an AI app with zero changes to the app itself — that's the power of the standard.

🧩
Custom agents too

Not just Claude Desktop. Any program can be an MCP host using the client SDK (mcp.client) — your own Python agent can launch a server, list its tools, and feed them to the LLM exactly like the Part 03 agent loop. The server is reusable across all of them.

9 · Why MCP is significant

  • Write once, use everywhere. A tool built as an MCP server works across every compatible host — no rewrites.
  • An ecosystem, not silos. Hundreds of community servers already exist; you plug them in like browser extensions.
  • Separation of concerns. Tool authors and app authors work independently against a shared contract.
  • Future-proofing. Swap the underlying LLM or app; your servers keep working.
  • Security boundaries. Servers declare exactly what they expose; hosts control what's allowed and ask the user to approve.
⚠️
A note on safety

MCP servers can take real actions (delete files, run queries, post messages). Only install servers you trust, scope their permissions tightly (e.g. a single folder, read-only DB user), and keep a human in the loop for destructive actions. The protocol supports user approval prompts — use them.
