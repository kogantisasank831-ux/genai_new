Part 03 · Agents

Agents & Tool Use

An LLM that can only talk is limited. An LLM that can decide to use tools and act on the world becomes an agent. This is the conceptual bridge to MCP.

⏱ ~15 min read Difficulty · Intermediate Prereq · 01 · LLMs, 02 · RAG

1 · From chatbot to agent

A plain LLM is like a brilliant advisor locked in a room with no phone: it can tell you what to do, but can't do anything itself. An agent hands that advisor a set of tools — a calculator, a search box, a database connection — and lets it choose which to use, use them, see the results, and keep going until the task is done.

Plain LLM (chatbot)Agent
OutputText onlyText and actions
"What's 8,234 × 591?"Guesses (often wrong)Calls a calculator tool → exact
"What's the weather now?""I can't access live data"Calls a weather API → real answer
Control flowOne turn in, one turn outLoops: act, observe, decide, repeat

2 · What a "tool" is to an LLM

A tool (a.k.a. "function") is just a function you describe to the LLM: its name, what it does, and what inputs it takes. The model can't run code — but it can output a structured request saying "please run get_weather with city="Paris"." Your code runs it and feeds the result back.

json
# A tool definition you give to the model (Anthropic/OpenAI both use this shape)
{
  "name": "get_weather",
  "description": "Get the current weather for a city.",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {"type": "string", "description": "e.g. Paris"}
    },
    "required": ["city"]
  }
}
💡
The model never executes anything

It only decides to call a tool and produces the arguments. You (your program) actually run the function and return the output. This separation is exactly what MCP standardizes.

3 · The agent loop (think → act → observe)

An agent isn't one API call — it's a loop. The model thinks, optionally calls a tool, you run it, you give it the result, and it continues until it has a final answer.

Perceive / inputthe user goal lands in the prompt
🧠 Reason"Do I need a tool? Which one?"
🔧 Act · call a toolmodel emits a structured tool request
👀 Observe resultyour code runs it, feeds output back ↺
✅ Final answerno tool needed → loop ends
The loop repeats — call a tool, observe the result, decide again — possibly many times, until the model produces a final answer.

4 · A real agent in code

Here's a complete, runnable mini-agent with one tool. Read the comments — this is the heart of every agent framework, stripped to essentials.

python
# pip install anthropic
import anthropic

client = anthropic.Anthropic()

# 1) The actual Python function the tool maps to
def get_weather(city):
    fake = {"Paris": "18°C, cloudy", "Tokyo": "27°C, sunny"}
    return fake.get(city, "Unknown city")

# 2) Describe the tool to the model
tools = [{
    "name": "get_weather",
    "description": "Get current weather for a city.",
    "input_schema": {"type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"]}
}]

messages = [{"role": "user", "content": "What's the weather in Tokyo?"}]

# 3) THE AGENT LOOP
while True:
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=500,
        tools=tools, messages=messages)

    if resp.stop_reason != "tool_use":        # model gave a final answer
        print(resp.content[0].text)
        break

    messages.append({"role": "assistant", "content": resp.content})

    for block in resp.content:           # run each requested tool
        if block.type == "tool_use":
            result = get_weather(**block.input)
            messages.append({"role": "user", "content": [{
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": result}]})
    # loop again so the model can use the result
✅
Trace what happens

(1) you ask about Tokyo → (2) model replies "I want to call get_weather(city=Tokyo)" → (3) your loop runs the function → (4) you feed back "27°C, sunny" → (5) model writes the final sentence "It's 27°C and sunny in Tokyo." That round-trip is agency.

5 · RAG is just a tool: "agentic RAG"

Here's a beautiful connection. Remember RAG's retrieval step? You can expose it as a tool called search_documents. Now the agent decides for itself when it needs to look something up — and can search multiple times, refine queries, or combine it with other tools.

Classic RAGAgentic RAG
When to retrieveAlways, every questionModel decides if & how many times
QueriesJust the user's questionModel rewrites/splits the query
Other toolsNoneCan combine search + calculator + APIs
🧩
Takeaway

"Retrieval" and "tool use" are the same idea. A document search is one tool; a weather API is another; a database query is another. Which raises an obvious question: if every capability is a tool, isn't there a standard way to define and connect them? Yes — that's MCP.

6 · The pain point that MCP fixes

Imagine you've built agents and want to connect them to: GitHub, Google Drive, a Postgres DB, Slack, and your internal API. Without a standard, you write custom tool code for every (agent × tool) pair. Five tools across three apps = lots of bespoke, duplicated glue.

Every app needs custom code for every tool. This is the "N×M problem." MCP turns it into N+M.
