1 · Step 1 — Run it with no key (watch the loop)

Lesson 1 was all reading. Now you make an agent actually run. The good news: it works immediately, with no API key, because it ships with a fake "offline brain" so you can see the loop turn before dealing with any setup. Open a terminal and run:

bash
cd teach-agents/code
python agent.py

If Python is installed you'll see output within a second. If you get python: command not found, install Python 3 first (any 3.9+), then re-run. You don't need to install anything else yet — the offline brain uses only the standard library.

2 · Reading the output, line by line

You'll see something like this. Every block is one turn of the agent loop from Lesson 1. Let's decode it.

output
[brain] No GEMINI_API_KEY found -> using OFFLINE fallback brain.

--- STEP 1 ---
  REASON  -> {'action': 'list_tags'}
  ACT     -> called list_tags()
  OBSERVE -> ["T101_reactor_temp", "P201_pump_pressure", ...]

--- STEP 2 ---

... (more steps) ...

=== AGENT REPORT ===
Anomalies detected:
- V401_vibration: 3 anomaly point(s); worst z=4.4 at 2026-06-23T10:40
  • REASON is the Reason stage of the loop — the brain deciding the next action. Here, at step 1, it decides it must first discover what sensors exist (list_tags).
  • ACT is the Act stage — the tool actually being called.
  • OBSERVE is the Observe stage — the tool's result, which is added to the history and fed into the next REASON.
  • The cycle repeats: having listed the tags, it inspects each one with detect_anomalies, and when every tag has been checked it emits a finish action with a written report.
REASONbrain picks the next action (JSON)
ACTthe chosen tool is called
OBSERVEresult appended to history ↩ back to REASON
🔑
What you're looking at

This printout is the agent loop made visible. There is no magic underneath — just look → think → do → look again, repeated, with the result of each tool call informing the next decision.

🧭
A deliberate imperfection (we'll fix it in Lesson 3)

Notice the agent flags the vibration spike but says nothing about the reactor temperature — even though, in the data, the temperature is slowly climbing toward danger. That's not a bug in the agent; it's a limitation of the tool it used (a z-score). Hold that thought — it's the entire subject of Lesson 3.

3 · A full walkthrough of agent.py

Now we read the whole program. It's about 130 lines and divides into four parts: tools, the brain, the loop, and a tiny entry point. Open code/agent.py alongside this.

3a · The tools — what the agent can do

python
DATA_FILE = "sensor_data.csv"

def load_data(path=DATA_FILE):
    with open(path, newline="") as f:
        return list(csv.DictReader(f))

def list_tags():
    """TOOL: return the sensor tag names available."""
    rows = load_data()
    return [c for c in rows[0].keys() if c != "timestamp"]
  1. load_data is a helper (not a tool the agent calls directly) — it reads the CSV into a list of dictionaries, one per row. Because the path is relative, the program must be run from inside code/ — that's why the instructions always say cd teach-agents/code first.
  2. list_tags is the agent's first real tool. It returns the column names (the sensor "tags") minus the timestamp column. The agent calls this when it needs to discover what sensors exist — exactly what it did at STEP 1 in the output.
python
def detect_anomalies(tag, z_threshold=3.0):
    rows = load_data()
    vals = [float(r[tag]) for r in rows]
    mean = statistics.mean(vals)
    sd = statistics.pstdev(vals) or 1e-9
    hits = []
    for r in rows:
        v = float(r[tag])
        z = (v - mean) / sd
        if abs(z) >= z_threshold:
            hits.append({"timestamp": r["timestamp"], "value": v, "z": round(z, 2)})
    return {"tag": tag, "mean": ..., "anomaly_count": len(hits), "anomalies": hits}
  1. This is the second tool — the one that does the actual anomaly math. Note that it is ordinary Python statistics, not AI. This is the Lesson-1 nuance in the flesh: the "intelligence" of detection is classical code; the model never does the math.
  2. It computes the mean and standard deviation (sd) of one tag's values. The or 1e-9 guards against dividing by zero if a tag never varies.
  3. For each reading it computes a z-score: how many standard deviations the value sits from the mean. A z of 3 means "3σ from normal" — unusually far.
  4. Any reading with |z| ≥ z_threshold (default 3.0) is recorded as a "hit". The tool returns a structured result: the stats plus the list of anomalous points. Structured output matters — the brain will read these fields to decide what to do next.
python
TOOLS = {
    "list_tags": list_tags,
    "detect_anomalies": detect_anomalies,
}
  1. The tool registry: a dictionary mapping a tool's name (a string the brain can emit) to the actual Python function. The loop uses this to turn the brain's decision ("call list_tags") into a real function call. The agent can do exactly these two things and nothing else — capability = the sum of the registry. (Lesson 1, Section 5, made concrete.)

3b · The brain — what decides the next action

python
SYSTEM_PROMPT = """You are a plant reliability agent. Goal: find anomalies ...
You can call tools. Respond ONLY with JSON:
either  {"action":"list_tags"}  or  {"action":"detect_anomalies","tag":"<tag>"}
or, when finished,  {"action":"finish","report":"<plain-english findings>"}."""
  1. The system prompt is the agent's job description (Lesson 1 glossary). It tells the model its role, its goal, the exact tools available, and — crucially — the format it must reply in (JSON with an action field). Constraining the output format is what lets our code reliably parse the model's decision.
python
def llm_brain(history):
    import google.generativeai as genai
    genai.configure(api_key=os.environ["GEMINI_API_KEY"])
    model = genai.GenerativeModel("gemini-1.5-flash")
    convo = SYSTEM_PROMPT + "\n\nHistory so far:\n" + json.dumps(history, indent=2)
    convo += "\n\nWhat is your next action? Respond with one JSON object only."
    resp = model.generate_content(convo)
    text = resp.text.strip().strip("`")
    if text.startswith("json"): text = text[4:]
    return json.loads(text)
  1. The real brain. It builds a prompt that is the system prompt + the entire history so far (everything the agent has done and observed). This is the Perceive stage: the model literally "sees" the goal and all prior steps.
  2. It sends that to Gemini and gets text back. Because the model sometimes wraps JSON in markdown fences (```json), the next two lines strip those off.
  3. json.loads(text) parses the model's reply into a Python dictionary like {"action": "detect_anomalies", "tag": "T101..."} — the decision the loop will act on. This is the Reason stage, performed by a real LLM.
python
def offline_brain(history):
    actions_taken = [h["action"] for h in history if "action" in h]
    if "list_tags" not in actions_taken:
        return {"action": "list_tags"}
    tags = next((h["result"] for h in history if h.get("action") == "list_tags"), [])
    inspected = [h.get("tag") for h in history if h.get("action") == "detect_anomalies"]
    for t in tags:
        if t not in inspected:
            return {"action": "detect_anomalies", "tag": t}
    # ... summarize findings into a finish action ...
  1. The offline brain is a hard-coded substitute that follows a fixed policy: first list the tags, then inspect each one not yet inspected, then finish with a summary. It is deliberately "dumb" — but it produces the same shape of decision the real brain does, so the loop can't tell the difference. That's what lets you watch the loop with zero setup.
python
def get_brain():
    if os.environ.get("GEMINI_API_KEY"):
        return llm_brain      # real LLM decides each step
    return offline_brain      # hard-coded fallback policy
  1. The switch. If a key is present in the environment, use the real LLM; otherwise fall back. This one function is the entire difference between "toy demo" and "real agent." We'll come back to it in Section 5 — it's the most important idea in the file.

3c · The agent loop — the heart of everything

python
def run(max_steps=12):
    brain = get_brain()
    history = []                                  # the agent's memory
    for step in range(1, max_steps + 1):
        decision = brain(history)                 # REASON
        action = decision["action"]
        if action == "finish":
            print(decision["report"]); return decision["report"]
        if action == "list_tags":                 # ACT
            result = TOOLS["list_tags"]()
        elif action == "detect_anomalies":
            result = TOOLS["detect_anomalies"](decision["tag"])
        history.append({"action": action, "result": result, ...})   # OBSERVE
  1. history = [] is the agent's memory — the running record of every action and its result. Each loop iteration appends to it, so the brain's view grows richer every step.
  2. for step in range(1, max_steps + 1) is the loop itself, with a built-in step limit (12). The limit is a safety net: if the brain never decides to finish, the agent stops anyway instead of running forever (and, with a paid model, running up a bill).
  3. decision = brain(history) is Reason — ask the brain what to do next, given the whole history.
  4. The if/elif block is Act — it looks the chosen tool up in the registry and calls it. If the brain ever asks for a tool that doesn't exist, the loop stops safely.
  5. history.append(...) is Observe — the result is stored, becoming part of what the brain perceives next iteration. Then the loop repeats. These ~12 lines are the entire definition of an agent from Lesson 1, in runnable form.
✅
In plain English

run() is a for loop that, each time around, asks the brain "what now?", does it, remembers the result, and goes again — stopping when the brain says "done" or after 12 tries.

4 · Step 2 — Get a free LLM key and go live

The offline brain is fine for seeing the loop, but it can't reason — it just follows a script. To make the agent genuinely intelligent, plug in a real model. Google's Gemini has a generous free tier with no credit card required. Follow these steps:

Create a free API key

Go to aistudio.google.com/apikey, sign in with a Google account, and click "Create API key." Copy the key somewhere safe.

Install the SDK

bash
pip install google-generativeai pandas

Set the key as an environment variable

In PowerShell (Windows):

powershell
$env:GEMINI_API_KEY = "paste-your-key-here"

This only lasts for the current terminal window — that's fine for learning. (To make it permanent later, add it to your system environment variables.)

Run the agent again

In the same terminal window so it can see the key:

bash
python agent.py

The top line now reads [brain] Using Google Gemini (live LLM). A real model is now choosing which tags to inspect and writing the report in its own words. Run it a couple of times — you may notice the phrasing of the report changes, and it might inspect tags in a different order. That variability is the model reasoning, not a script executing.

⚠️
Keep your key private

Never paste an API key into a file you might share, commit to git, or send in chat. Setting it as an environment variable (as above) keeps it out of your source code. If a key ever leaks, revoke it in AI Studio and make a new one.

5 · Step 3 — How the brain swap works (the key idea)

Re-read get_brain() from Section 3b. When you added the key, not a single line of the loop changed. The loop calls brain(history); whether brain is the dumb offline policy or live Gemini is decided once, up front. Everything downstream is identical.

The agent loopfixed · never changes
calls brain(history)one clean seam
offline_brain · Gemini · local model · Claudeswap freely behind the seam
🔑
The lesson inside the lesson

The loop is the agent; the brain is a replaceable part. This separation is why agent code is so reusable: you can develop and debug the whole loop offline (free, fast, deterministic), then swap in any model — Gemini today, a local model tomorrow, Claude next week — without touching the structure. Good agent architecture isolates the model behind a clean seam exactly like this.

6 · From hand-parsed JSON to real tool calling

Our brain returns JSON that we parse by hand (json.loads). That's perfect for learning because you can see every moving part. But it's fragile — if the model returns slightly malformed JSON, the parse fails.

Production agents avoid this with built-in function calling (also called "tool calling"). Instead of asking the model to "please reply in JSON," you declare your tools to the SDK with their names, descriptions, and input types. The model then emits a structured, validated tool request, and the SDK routes it for you — no hand-parsing, no malformed JSON.

🔧
Function calling / tool calling

A model feature where you register tools with the SDK and the model returns a structured request to call one (with arguments the SDK validates), rather than free-form text you must parse yourself. Same agent loop; sturdier plumbing.

The reference is Gemini function calling. We deliberately did not start there — the hand-parsed version makes the loop visible. Once the loop is second nature (after Lesson 4), upgrading to real function calling is a small, safe change.
