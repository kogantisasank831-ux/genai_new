1 · Why one agent isn't enough

The very first line of your EnM spreadsheet states the thesis of this entire lesson:

"For this area, individual agents may not work — need orchestration of multiple agents to achieve an outcome."

Why not just build one big "reliability agent" with all the tools? Three reasons, all of which you'll feel the moment you try:

A fuzzy goal makes a confused agent

"Handle reliability" is too broad. An agent reasons best when its goal is sharp: "find anomalies," "find the root cause," "recommend a fix." Each is a different mode of thinking.

Too many tools dilute the reasoning

The more tools an agent has, the more ways it can pick the wrong one. Specialists with a handful of relevant tools choose better than a generalist with thirty.

Specialists are reusable and testable

A "root cause" agent can be improved, tested, and swapped independently — exactly the clean-seam principle from Lessons 2 and 3, now applied at the agent level.

So we split the outcome into specialists — each a small Lesson-1 agent with one sharp goal — and have them hand work to each other:

Three specialists in a chain. Each is an agent; the chain is the orchestration.

2 · The two new ideas: handoff & orchestrator

💡
Handoff

One agent's output becomes the next agent's input. The Anomaly Agent's list of flagged tags is handed off to the RCA Agent; the RCA Agent's ranked diagnoses are handed off to the Remediation Agent. Data flows down the chain.

💡
Orchestrator

The "conductor" that runs each agent in order and passes the results along. It holds the workflow; the agents hold the expertise. The orchestrator itself does no domain work — it coordinates.

💡
Orchestration

The overall pattern of coordinating multiple specialized agents so their combined work produces one business outcome. This is the central concept of the EnM document.

✅
In plain English

Think of an orchestra: each musician (agent) is a specialist who plays one part; the conductor (orchestrator) decides who plays when and keeps them in sync. No single musician plays the whole symphony.

5 · Walkthrough of orchestrator.py

Open code/orchestrator.py. The structure is: shared data (limits + knowledge base), then three agent functions, then a tiny orchestrator that strings them together.

5a · The shared knowledge — limits and the knowledge base

python
LIMITS = {"T101_reactor_temp": 420, "V401_vibration": 4.0, ...}

KNOWLEDGE_BASE = {
    "T101_reactor_temp": {
        "failure_mode": "Thermal runaway / catalyst degradation (FMEA: FM-07)",
        "likely_causes": ["Feed composition shift", "Coolant flow loss", ...],
        "remediation": ["Verify coolant loop F-CW flow", "Reduce feed rate 10%", ...],
    },
    ...
}
  1. LIMITS are the safe operating bounds per tag — the kind of thing a "Constraint Agent" (EnM scenario #8) would own. The Anomaly Agent forecasts against these.
  2. KNOWLEDGE_BASE is a tiny stand-in for EnM's Contextualization Agent — in a real system this is a knowledge graph built from SOPs and best practices. Here it's a dictionary mapping each tag to its failure mode, likely causes, and remediation steps. The quality of the whole system is capped by the quality of this knowledge base — a key insight for EnM #2 and #4.

5b · Agent 1 — the Anomaly Agent

python
from forecast_tool import forecast_breach   # reuse Lesson 3's tool

def anomaly_agent():
    findings = []
    for tag, limit in LIMITS.items():
        spikes = z_anomalies(tag)
        forecast = forecast_breach(tag, limit)
        if spikes or forecast["steps_until_breach"] is not None:
            findings.append({"tag": tag, "spikes": len(spikes),
                             "forecast": forecast["verdict"],
                             "steps_until_breach": forecast["steps_until_breach"]})
    return findings    # HANDOFF to RCA Agent
  1. It imports and reuses Lesson 3's forecast_breach tool. This is composition in action — the specialist built in one lesson becomes a tool for an agent in the next. Nothing was rewritten.
  2. For every tag it runs both checks: z_anomalies (spikes, backward-looking) and forecast_breach (trends, forward-looking). Using both is exactly the Lesson-3 conclusion that the two are complementary.
  3. A tag is flagged if it has spikes or a forecast breach. The list of flagged findings is returned — that return value is the handoff to the next agent.

5c · Agent 2 — the RCA Agent

python
def rca_agent(findings):
    diagnoses = []
    for f in findings:
        kb = KNOWLEDGE_BASE.get(f["tag"], {})
        urgency = 0
        if f["steps_until_breach"] is not None:
            urgency += max(1, 30 - f["steps_until_breach"])   # sooner = more urgent
        urgency += f["spikes"] * 5                            # spikes add severity
        diagnoses.append({"tag": f["tag"], "failure_mode": kb.get("failure_mode", ...),
                          "likely_causes": kb.get("likely_causes", ...), "rpn": urgency})
    diagnoses.sort(key=lambda d: d["rpn"], reverse=True)
    return diagnoses    # HANDOFF to Remediation Agent
  1. It receives the findings (the handoff input) and, for each, looks up the failure mode and likely causes from the knowledge base — this is the FMEA mapping the EnM doc describes.
  2. It computes a simple RPN-style urgency score (the EnM "Risk Priority Number"): a breach that's sooner scores higher (30 - steps), and spikes add severity. This is a deliberately crude formula you can change — prioritization is a design decision, not a fixed truth.
  3. It sorts by RPN so the most urgent issue is first, then hands the ranked diagnoses off.

5d · Agent 3 and the orchestrator

python
def remediation_agent(diagnoses):
    plan = []
    for d in diagnoses:
        kb = KNOWLEDGE_BASE.get(d["tag"], {})
        steps = kb.get("remediation", ["No SOP found — escalate to LT/LD engineer"])
        plan.append({"tag": d["tag"], "rpn": d["rpn"], "steps": steps})
    return plan

def orchestrate():
    findings  = anomaly_agent()           # step 1
    if not findings: return               # nothing to hand off
    diagnoses = rca_agent(findings)       # step 2 — fed step 1's output
    plan      = remediation_agent(diagnoses)  # step 3 — fed step 2's output
    # ... print the final plan ...
  1. The Remediation Agent looks up the recommended SOP steps for each diagnosed tag. If a tag isn't in the knowledge base, it falls back to "escalate to a senior engineer" — a sensible default that mirrors real operations.
  2. orchestrate() is the orchestrator, and notice how small it is. It does no domain work at all. It just calls the three agents in order and threads each one's output into the next — findings → rca_agent → diagnoses → remediation_agent. Those arrows are the handoffs. This tiny function is the whole "orchestration."
  3. The early return when there are no findings is a small but important touch: if the first agent finds nothing, there's nothing to hand off, so the chain stops cleanly.
🎯
The big realization

An orchestrated multi-agent system is mostly ordinary function composition: specialist functions whose outputs feed the next. The "agents" hold the expertise; the orchestrator is a short script that wires them together. Once you see this, the intimidating EnM diagrams become readable.

The contrast is worth seeing side by side. One overloaded generalist tries to do everything in a single loop; the orchestrated pipeline splits the work into focused specialists whose outputs hand off down the chain:

6 · Workflow vs. autonomous orchestration

Our orchestrator runs a fixed order: Anomaly → RCA → Remediation, every time. That makes it a workflow — predictable, easy to test, easy to trust. An autonomous orchestrator would instead let an LLM decide, at runtime, which agent to invoke next. Both are valid; which one fits a given EnM scenario is a real engineering decision — and it's the entire subject of Lesson 5.

Recap
  • EnM outcomes need orchestration: many specialist agents, not one generalist.
  • Handoff = one agent's output becomes the next's input. Orchestrator = the conductor that wires them in order.
  • Our chain is EnM #2: Anomaly → RCA → Remediation, with a knowledge base standing in for Contextualization.
  • The Anomaly Agent reuses Lesson 3's forecaster — specialists compose.
  • An orchestrated system is largely function composition; the orchestrator itself is tiny.
