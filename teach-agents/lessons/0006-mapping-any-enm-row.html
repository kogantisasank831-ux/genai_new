A method to map any EnM row to an implementation

The payoff of the whole course: a repeatable checklist that converts any vague "we want an agent for X" into a buildable, defensible design — and the one call that matters most, LLM reasoning vs. classical computation.

🎯
By the end of this lesson you will be able to
  • Apply a repeatable 5-step method to turn any EnM row into a concrete agent design.

1 · The 5-step method

This is the payoff of the whole course: a checklist that converts any vague "we want an agent for X" into a buildable, defensible design. Run every EnM row through these five steps.

The decomposition method: one business outcome breaks into sub-goals; each sub-goal gets the reasoning-vs-computation call plus a goal, tools and loop; then handoffs wire them into one orchestrated shape.

Name the business outcome

One sentence, taken from the xlsx "Purpose" column. If you can't state it in one sentence, the scope is too fuzzy to design for.

List the sub-tasks

What distinct things must happen to get from raw inputs to that outcome? Aim for 3–6. These become your candidate agents.

For each sub-task, decide: LLM or classical?

Simulation, solver, statistics, ML → classical tool. Reasoning, contextualizing, explaining, routing → LLM agent. (This is the crucial step — Section 2 is entirely about it.)

Define the handoffs

What does each agent output, and which agent consumes it? Draw the arrows. (Lesson 4's chain is the template.)

Choose the shape

Workflow or autonomous (Lesson 5's decision rule), and mark where the guardrails / constraint agents go.

2 · Why Step 3 is the whole game

🔑
Reasoning or computation?

Beginners draw every box as "an AI agent." Experts ask one question of each box: "Is this reasoning (something a thoughtful person does with language and judgment) or computation (something a formula, solver, or model does with numbers)?" In EnM, most boxes are computation wrapped as tools; only a few are genuine LLM reasoning. Getting this split right is the difference between a buildable design and a fantasy — and it's the single skill this whole course was built to give you.

A quick test for any sub-task: "Could I, in principle, write an exact formula or algorithm for this?" If yes, it's classical — wrap it as a tool. "Does it require weighing fuzzy context, explaining, or deciding what to do next?" If yes, that's where the LLM belongs.

Sub-task sounds like…Usually…
"optimize", "simulate", "predict the value", "calculate"Classical (solver / sim / model as tool)
"detect anomalies", "forecast the trend", "score / rank"Classical (statistics / ML as tool)
"explain to the operator", "summarize", "decide which tool", "answer the question"LLM reasoning
"build a knowledge graph from SOPs", "retrieve relevant docs"Mixed: retrieval (classical) + LLM reasoning over results

3 · Worked example — Scenario #1: the "Digital Console Engineer"

Step 1 — Outcome (from xlsx): real-time optimization of furnaces, reactors, distillation, cracking, hydrotreating, and polymer units.

Step 2 — Sub-tasks: the xlsx already names three, and we'll surface a fourth that's implied. Step 3 — LLM or classical? applied to each:

Sub-task (from xlsx)LLM or classical?Why
First-Principle Agent — invoke kinetic simulation, generate yield vectors on demandClassical (simulation as tool)Runs physics/chemistry kinetic equations. Zero language involved. The agent only calls it with parameters.
Evolution Agent — differential evolution to find global-maximum yieldClassical (solver as tool)A deterministic optimization algorithm. The "intelligence" is the search math, not a model. You could write its exact algorithm — so it's classical.
Physical-Properties Agent — predict product slates via exponential smoothing between yield stepsClassical (interpolation as tool)Numerical interpolation/smoothing — again a formula, wrapped as a tool.
(implied) Console reasoning layer — explain recommended setpoints to the operator, flag trade-offsLLM reasoningThe genuinely linguistic part: turn cold numbers into operator guidance, weigh context, answer "why."
⚠️
The punchline for #1

Almost the entire "Digital Console Engineer" is classical optimization and simulation. The LLM's role is thin: orchestrate the tools, respect constraints, and translate results into human guidance. If a vendor pitches this as "an LLM that optimizes the plant," that's a red flag — the LLM does not optimize; the solver does. (The same pattern, scaled plant-wide, is scenario #8 Integrated Margin Management.)

💡
Hint for #4 (Knowledge Capture)

This one flips the usual balance — it's mostly LLM + retrieval: a Contextualization Agent (build a knowledge graph from SOPs and answer FAQs — this is RAG) hands off to a Gap Analysis Agent (compare "ideal state" vs "current state" and recommend steps to close the gap). Very different from #1, where the LLM was thin and the solvers did the work. That contrast is the lesson: the right architecture depends entirely on whether the work is reasoning or computation.

6 · Where to go from here (turning knowledge into wisdom)

You now have the full arc: concept (L1) → first agent (L2) → forecasting (L3) → orchestration (L4) → choosing the shape (L5) → mapping any row (L6). The last ingredient is wisdom — testing these designs against reality:

  • Inside your org: walk the EnM team through a row you mapped and let them poke holes. That's the tightest, highest-value feedback loop you have — real constraints, real domain experts.
  • r/AI_Agents — post a decomposition and ask "is this a sane design?" Practitioners will tell you where the autonomy is misplaced.
  • Revisit Building Effective Agents now that you have the full mental model — it reads very differently the second time.
🎯
Recap — the method in one breath
  • 1. Outcome (one sentence) → 2. Sub-tasks (3–6) → 3. LLM or classical? (the crucial call) → 4. Handoffs → 5. Shape (workflow vs. autonomous + guardrails).
  • Step 3 is the whole game: ask "reasoning or computation?" of every box.
  • In EnM, most boxes are computation wrapped as tools; the LLM conducts and explains.
  • You can now decompose any row — and, crucially, defend why each piece is shaped the way it is.
