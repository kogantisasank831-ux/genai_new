Lesson 1 · Agent Literacy

What is an agent, really?

⏱ ~15 min read Type · Concept (no code yet) Prereq · none Next · Lesson 2 (hands-on)
🎯
By the end of this lesson you will be able to
  • Define an "AI agent" in one precise sentence — and say what it is not.
  • Draw the agent loop from memory and explain each of its five stages.
  • Explain why tools are the source of an agent's power.
  • Point at any row of EnM Agents.xlsx and see the agent loop hiding inside it.
  • Avoid the single most common beginner mistake: confusing "agent" with "an LLM call".

1 · Why this matters before anything else

"Agent" is one of the most over-used words in technology right now. Vendors attach it to everything; demos make it look like magic. If you start by learning a specific framework, you'll learn that framework's vocabulary but not the underlying idea — and you'll be unable to tell a real agent design from marketing.

So we start somewhere deliberately un-magical. By the end of this single lesson you will have a mental model so simple you can hold all of it in your head at once — and yet it is the same model that underlies the most sophisticated multi-agent systems in your EnM Agents.xlsx. Everything else in this course is a variation on what you learn here.

2 · The one-sentence definition (unpacked)

🔑
The whole idea

An agent is a language model that's given a goal and a set of tools, then left to work toward that goal on its own — deciding what to do, doing it, and adjusting based on what happens, step after step.

That sentence has four load-bearing words. Let's take each one slowly, because the whole rest of the course hangs off them.

🧠
Language model

The "brain". A program (like Gemini, GPT, or Claude) that, given some text, predicts useful text in response. On its own it can only produce text. It cannot read a sensor, run a simulation, or change anything in the world. Remember this limitation — it's exactly why the other three words are needed.

🎯
Goal

What we want accomplished, stated in plain language: "find anomalies in this sensor data and explain them," or "optimize the reactor yield within safety limits." The goal is what the agent keeps measuring its progress against.

🔧
Tools

Functions we hand the model so it can actually do things: read a file, run a query, call an API, execute a simulation. A tool is the bridge between "a model that produces text" and "a system that affects the world". (Section 5 is entirely about this.)

🔁
On its own, step after step

The model is not asked once and done. It is placed in a loop: it acts, sees the result of its action, and decides what to do next based on that result. This is the difference between answering a question and pursuing a goal.

✅
In plain English

An agent is a chatbot that has been (a) given a job, (b) given some buttons it can press to get things done, and (c) allowed to keep pressing buttons and reacting until the job is finished.

3 · Agent vs. chatbot: the two missing pieces

You already know what a chatbot is: you type something, it replies once, and it stops. To turn that chatbot into an agent, you add exactly two things — no more.

ChatbotAgent
How many times does it respond?Once, then stopsRepeatedly, until the goal is met
Can it take actions in the world?No — only text outYes — via tools
Does it react to results?NoYes — each result shapes the next step
What's added?—a loop + tools

That's it. A chatbot answers once and stops. An agent keeps going — it takes an action, sees what happened, and uses that to decide its next move. What makes this possible is the loop (it runs again and again instead of replying once) and the tools (it can actually do things, not just talk). Nothing more mystical than that.

⚠️
A common misconception, cleared up now

An agent is not "an AI that thinks for itself" or "a smarter model". The model inside an agent is the same model you'd chat with. What's different is the harness around it — the loop and the tools. Two teams using the identical model can build a brilliant agent or a useless one, purely by how they design that harness. Agent quality is mostly engineering, not model magic.

4 · The agent loop, stage by stage

Every agent — from a 30-line toy to the full EnM orchestration — runs some version of this cycle. This diagram is worth memorizing; we will return to it in every lesson.

① Perceivetake in the goal + history so far
② Reasondecide the single best next action
③ Actcall a tool
④ Observefeed the result back ↩ to ② — until done ⑤

The agent loop. Stages ②–④ repeat until the agent decides it's done (stage ⑤).

Now each stage in detail, with a concrete example: an agent whose goal is "find out which sensor is misbehaving."

① Perceive — take in the current situation

The model is given everything it can currently "see": the goal, plus the history so far (what it has already done and what those actions returned). At the very first step, this is just the goal and the starting data. Example: "Goal: find the misbehaving sensor. You have not done anything yet."

② Reason — decide the next action

The model thinks: "given the goal and what I know, what is the single most useful thing to do next?" Example: "I don't even know which sensors exist yet. I should list them first." The output of this stage is a decision, usually expressed as "call tool X with these inputs."

③ Act — call a tool

The decision is carried out. Your code runs the chosen tool. Example: it calls list_tags(), which runs ordinary Python and returns ["temp", "pressure", "flow", "vibration"]. The model did not produce that list — a tool did. The model only chose to ask for it.

④ Observe — feed the result back in

The tool's output is added to the history. Now the agent "knows" something it didn't before. Example: it now knows the four sensor names. This new knowledge becomes part of the next Perceive.

⑤ Loop or finish

Back to Reason with the enriched history. The agent inspects each sensor, one tool call at a time, until it concludes "vibration is the misbehaving one" — at which point it stops. A safety step limit (e.g. "max 12 steps") prevents an agent from looping forever if it gets confused.

✅
In plain English

The loop is just: look → think → do → look again → repeat. The only reason it feels powerful is that the "do" step can call real tools, and the "look again" step lets the model learn from what just happened — so each step can be smarter than the last.

5 · Tools: where the power actually comes from

Here is the most important practical insight in the whole lesson. A language model, by itself, can only generate text. It cannot read your live plant data, run a kinetic simulation, query a database, or send an email. Everything an agent can actually accomplish, it accomplishes through its tools.

🔧
Tool

A function you write (or expose) that the agent is allowed to call. Each tool has a name, a description of what it does, and inputs it accepts. The agent's total capability is exactly the union of all its tools — nothing more, nothing less.

This has a liberating consequence: to make an agent more capable, you give it better tools — you don't need a better model. A "weather agent" is just an LLM + a get_weather(city) tool + a loop. A "plant reliability agent" is an LLM + tools like read_sensor(), forecast_trend(), and lookup_failure_mode() + a loop. The interesting engineering is almost always in the tools, not the model.

🏭
Why this framing pays off for EnM

When you evaluate any EnM proposal, your first question should be: "What tools does each agent have, and what can those tools actually do?" If someone says "an AI agent that optimizes the plant" but can't tell you what optimization tool it calls, they haven't designed an agent — they've described a wish.

7 · The most important EnM nuance — read this twice

⚠️
Not every "agent" is an LLM

Several of the boxes above are not language models at all. The forecasting inside Anomaly Prediction is a classical statistical/ML model. The "First Principle Agent" in scenario #1 runs a real kinetic simulation (physics and chemistry equations). The "Evolution Agent" runs differential evolution — a deterministic optimization algorithm. None of these "think" in language.

So what is the language model doing in these systems? It is the orchestrator and reasoner: it decides which tool to call, in what order, interprets the numbers that come back, respects the constraints, and translates the result into guidance a human engineer can act on. The heavy numerical lifting is done by simulations, solvers, and ML models — each wrapped as a tool the agent can call.

Confusing "agent" with "LLM call" is the number-one beginner mistake. It is also exactly the mistake the EnM document is warning against when its very first line says: "individual agents may not work — need orchestration of multiple agents to achieve an outcome." Translated: no single LLM is going to run your plant; you need a coordinated system of specialists, most of which are classical tools, conducted by a reasoning layer.

🎯
Recap — the whole lesson in five bullets
  • An agent = a language model + a goal + tools + a loop.
  • The agent loop is: Perceive → Reason → Act → Observe → repeat, until done (with a step limit for safety).
  • A model alone only produces text; tools are what let it affect the world. Capability = the sum of its tools.
  • EnM scenarios are this same loop, orchestrated across multiple specialist agents.
  • Many EnM "agents" are simulations/solvers/ML wrapped as tools, not LLMs. The LLM conducts; the tools compute.
