1 - HTTP request vs WebSocket connection

HTTPWebSocket
Request/response; client initiates each request.Long-lived full-duplex connection; either side can send frames after upgrade.
Natural stateless scaling and caching semantics.Per-connection state, disconnect handling, and capacity limits matter.
Use for CRUD, commands, polling, and ordinary APIs.Use for interactive bidirectional real-time communication.

2 - Handshake and lifecycle

  1. Client sends an HTTP upgrade request.
  2. Server authenticates/validates and accepts or rejects.
  3. The protocol switches to WebSocket frames.
  4. Both sides send text/binary/control frames.
  5. Either side closes; application cleans up connection state.
websocket_app.py
from fastapi import FastAPI, WebSocket, WebSocketDisconnect
from pydantic import BaseModel, ValidationError

app = FastAPI()

class ClientEvent(BaseModel):
    type: str
    payload: dict

@app.websocket("/ws")
async def websocket_endpoint(ws: WebSocket) -> None:
    user = await authenticate_ws(ws)
    if user is None:
        await ws.close(code=1008)
        return

    await ws.accept()
    try:
        while True:
            raw = await ws.receive_json()
            try:
                event = ClientEvent.model_validate(raw)
            except ValidationError:
                await ws.send_json({"type": "error", "code": "invalid_event"})
                continue
            reply = await handle_event(user, event)
            await ws.send_json(reply)
    except WebSocketDisconnect:
        await connection_registry.remove(user.id, ws)

3 - Authentication and authorization

Authenticate during the handshake using a secure cookie, short-lived token, or supported subprotocol mechanism; do not put long-lived secrets into logged URLs. Authorization is per message/action and may change during a long connection. Close denied connections with an appropriate policy code and avoid leaking details.

4 - Reliability concerns

ConcernDesign response
Dead connectionsPing/pong or application heartbeat; idle timeout; cleanup on disconnect.
Slow clientBounded outbound queue; drop/coalesce noncritical events or disconnect; never let memory grow unbounded.
ReconnectClient backoff with jitter; session/resume token; last acknowledged sequence.
Duplicate eventsMessage ID and idempotent server handling.
OrderingPer-session/channel sequence number; define whether global ordering is required.
CancellationStop producer tasks when the socket closes; structured task ownership.
Large messagesMaximum frame/message size, schema limits, compression caution.

5 - Horizontal scaling

A single process can keep an in-memory connection registry, but multiple workers need shared routing information and a broker such as Redis Pub/Sub, NATS, or Kafka depending durability and ordering needs. Load balancers must support WebSocket upgrades and long-lived timeouts. Sticky sessions can simplify routing but do not replace shared state or failover design.

LayerQuestion to answer
Load balancerUpgrade support, idle timeout, connection draining.
Gateway workersMaximum connections, file descriptors, memory per connection, event-loop lag.
BrokerFan-out, durability, ordering, replay and backpressure.
Application stateEphemeral presence vs durable messages and acknowledgements.
ClientReconnect, deduplication, resume and offline UX.

7 - Testing a FastAPI WebSocket

test_websocket.py
from fastapi.testclient import TestClient


def test_websocket_message(client: TestClient) -> None:
    with client.websocket_connect("/ws", cookies={"session": "test"}) as ws:
        ws.send_json({"type": "ping", "payload": {}})
        assert ws.receive_json()["type"] == "pong"


def test_invalid_event(client: TestClient) -> None:
    with client.websocket_connect("/ws", cookies={"session": "test"}) as ws:
        ws.send_json({"unexpected": True})
        assert ws.receive_json()["code"] == "invalid_event"

8 - High-signal interview questions

Q When would you choose WebSockets over HTTP? ›
When the product requires a long-lived bidirectional low-latency channel. For ordinary commands or one-way updates, HTTP or SSE may be simpler.
Q How do you handle slow consumers? ›
Use bounded per-connection queues and a policy to coalesce, drop noncritical updates, or disconnect. Backpressure must prevent unbounded memory growth.
Q How do you scale across workers? ›
Keep connections on gateway workers and distribute events through a broker/shared routing layer; define presence, ordering, durability, and reconnect behavior.
Q How do you authenticate a WebSocket? ›
Validate identity before accepting using a secure supported credential, then authorize every message/action and handle credential expiry or revocation.
Q What happens when the client disconnects during a model stream? ›
Cancellation should propagate to producer tasks and downstream requests where possible; clean up queues/state and record whether work should resume or terminate.

9 - Knowledge check

1 - What is the first defense against a slow client?
Backpressure protects memory and service health.

SSE can be simpler when only server-to-client streaming is required.
