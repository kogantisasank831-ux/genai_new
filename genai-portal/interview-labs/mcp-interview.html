MCP for Interviews

Interview outcome

Explain MCP architecture and lifecycle, choose tools vs resources vs prompts, design a secure server, and compare MCP with REST APIs and in-process tool functions.

1 - The architecture model

MCP is a client-host-server protocol. The host coordinates the AI application and permissions, creates a client connection for each server, and the server exposes capabilities. The data layer uses JSON-RPC messages; the transport layer carries them over mechanisms such as stdio or Streamable HTTP.

ComponentResponsibilityInterview trap
HostOwns user experience, model integration, permissions, and context aggregation.Calling the host just another MCP server.
ClientMaintains one session to one server; negotiates capabilities and routes protocol messages.Assuming one global client directly connects every server.
ServerExposes bounded tools, resources, and prompts around one capability domain.Putting all enterprise systems into one over-privileged server.
TransportMoves framed protocol messages.Confusing transport security with application authorization.

2 - Tools, resources, and prompts

PrimitiveUse it forControl orientation
ToolActions or computations the model may invoke.Model-controlled; host should mediate consequential calls.
ResourceRead-only context identified by a URI.Application/client-controlled context loading.
PromptReusable parameterized interaction templates.User/application initiated.

from pydantic import Field
from typing import Annotated

mcp = FastMCP("orders")

@mcp.tool()
async def lookup_order(
    order_id: Annotated[str, Field(pattern=r"^ORD-[0-9]{6}$")]
) -> dict:
    """Return status and items for one authorized order."""
    return await order_service.lookup(order_id)

3 - Lifecycle and capability negotiation

A strong answer mentions initialization, protocol-version/capability negotiation, normal operation, notifications where supported, and graceful shutdown. Do not describe MCP as only a tool registry.

4 - Transport decisions

TransportGood fitOperational concern
stdioLocal process launched by a trusted host; simple desktop/IDE integration.Never log to stdout because it corrupts protocol messages; use stderr.
Streamable HTTPRemote/shared servers, web deployment, standard network controls.Authentication, authorization, origin/security policy, session and scaling design.

5 - Security model

ThreatDefense
Over-broad toolsNarrow operations, allow-lists, bounded schemas and outputs.
Unauthorized accessAuthenticate transport caller and authorize every domain resource server-side.
Prompt injection in resources/tool outputTreat all returned content as untrusted; separate instructions from data and restrict follow-on authority.
Destructive side effectsDry-run, idempotency, confirmation, human approval, audit log.
Secret exposureKeep secrets in server environment/secret manager, never in tool descriptions or model-visible output.
Supply-chain server riskExplicit server trust, least-privilege runtime, version review, network/file isolation.

6 - MCP vs REST vs function calling

OptionStrengthUse case
REST/gRPC APIStable service-to-service contract, broad client ecosystem.Normal application and backend integrations.
In-process tool functionLowest overhead and simplest for one application.Private helper owned by one codebase.
MCP serverRuntime discovery and reusable AI-facing capability contract across hosts.The same bounded capability is shared by multiple AI clients.
🧠
Senior design pattern

Keep business logic in ordinary services/functions, then expose it through REST, MCP, jobs, or CLI adapters. MCP should not become the domain layer.

7 - High-signal interview questions

Q What problem does MCP solve? ›
It standardizes how AI hosts discover and invoke external capabilities and context, reducing bespoke integrations. It does not automatically solve authorization or trust.
Q Tool vs resource? ›
A tool performs an action or computation selected by the model. A resource is read-only context addressed by URI and loaded by the application/client.
Q How would you secure a database MCP server? ›
Expose constrained read operations, enforce identity and row/tenant authorization server-side, cap queries/results, audit calls, and avoid arbitrary SQL.
Q Why not expose every REST endpoint as a tool? ›
Service APIs are often too granular or powerful. Tools should be task-oriented, safe, clearly described, and designed for model selection.
Q How do you scale remote MCP? ›
Stateless or externally persisted session data where possible, authenticated Streamable HTTP, load balancing, bounded concurrency, timeouts, observability, and idempotent operations.

8 - Knowledge check

1 - Which primitive should expose a read-only policy document by URI?
Resources provide read-only context identified by URI.
2 - What is dangerous for a stdio MCP server?
stdout carries protocol frames and must not be polluted by logs.

9 - Focused practice

1
Explain

Draw host, two clients, and two servers. Narrate initialization, discovery, one tool call, and shutdown.

2
Implement

Build a read-only order server with one tool and one resource. Add validation, authorization stub, bounded output, and tests.

3
Threat model

Threat-model a remote enterprise MCP server that can create support tickets. Cover identity, scopes, injection, idempotency, approval, audit, and incident revocation.
