RAG for Interviews

Learn the exact RAG concepts you need to explain, debug, implement, and design in engineering interviews without wandering through unrelated theory.

Study + timed coding · 5–7 hDifficulty - IntermediateRAG - Retrieval - System Design

Interview outcome

Design and debug a RAG system from ingestion to evaluation, separate retrieval failures from generation failures, and explain scale, security, latency, and quality trade-offs.

1 - The pipeline you must draw from memory

Offline ingestionOnline serving
Connect sources, parse/OCR, normalize, chunk, enrich metadata, embed, index, version, handle updates/deletes.Authenticate, rewrite if needed, enforce ACL filters, retrieve, rerank, assemble context, generate, cite, evaluate/log.
🧠
Core interview habit

For every RAG answer, name the failure metric at each stage. Do not treat RAG as "vector DB plus prompt."

2 - Retrieval fundamentals

DecisionTrade-off to explain
Chunk size and overlapPrecision vs sufficient context; document structure matters more than arbitrary characters.
Dense vs sparseSemantic paraphrases vs exact identifiers and rare terms.
Hybrid fusionCombines complementary rankings; rank fusion avoids incomparable raw scores.
Top-kHigher recall but more latency, tokens, and distractors.
RerankingBetter precision on a small candidate set at extra latency/cost.
Metadata filtersFreshness, tenant, language, document type, and access control before retrieval.

3 - Evaluation model

LayerUseful measuresQuestion answered
IngestionParse coverage, OCR quality, freshness lag, delete correctness.Is the right content present and current?

def diagnose(case):
    assert case.query and case.expected_evidence
    retrieved = retriever.search(case.query, filters=case.acl)
    retrieval_ok = any(doc.id in case.expected_evidence for doc in retrieved)
    if not retrieval_ok:
        return "fix ingestion/chunking/filter/ranking"
    answer = generator.answer(case.query, retrieved)
    if not is_faithful(answer, retrieved):
        return "fix context assembly/prompt/model/verification"
    return "inspect product expectation or gold label"

4 - Security and permissions

Authorization must be applied before evidence reaches the model. Carry tenant and user scope into retrieval filters, storage paths, caches, traces, and citations. Treat retrieved documents as untrusted input because they can contain prompt injection.

5 - Production design discussion

ConcernStrong design anchor
FreshnessEvent/scheduled ingestion, source version, staleness SLO, idempotent upserts and deletes.
ScaleSeparate ingestion and serving; partition indexes; cache only permission-safe results.
LatencyParallel safe retrieval, candidate limits, rerank budget, streaming generation.
ReliabilityTimeouts, circuit breakers, fallback retrieval, no-answer path.
ObservabilityLog query, filters, chunk IDs, ranks, model/prompt versions, answer citations.
CostEmbedding only changed content, route query complexity, cap context and output.

7 - High-signal interview questions

Q How do you debug a bad RAG answer? ›
First inspect whether the correct evidence was retrieved. If not, investigate ingestion, chunks, filters, embedding, lexical retrieval, and ranking. If evidence is present, inspect context order, prompt, model, and faithfulness.
Q RAG vs fine-tuning? ›
RAG supplies changing/private facts at inference time. Fine-tuning changes behavior or task adaptation. They solve different problems and can be combined.
Q How do you choose chunk size? ›
Start from semantic units and expected queries, then evaluate retrieval and answer quality across a labeled set. There is no universal token count.
Q How do you prevent data leakage? ›
Authenticate the caller, derive ACL filters server-side, pre-filter retrieval, scope caches and traces, and test cross-tenant access.
Q When is long context better? ›
For a bounded coherent document set that fits comfortably and requires broad synthesis. RAG is better for large, changing, permissioned corpora.

8 - Knowledge check

1 - Retrieval recall is poor. What should you change first?
Retrieval must be evaluated independently before generation tuning.
2 - Why pre-filter ACLs instead of post-filtering results?
Security filtering must happen before evidence exposure.

9 - Focused practice

1
Whiteboard

Draw ingestion and serving paths in under three minutes. Add one metric and failure mode to every box.

2
Debug

Create ten queries with known relevant chunks. Measure recall@k, inspect misses, and make one controlled improvement.

3
System design

Design a permission-aware enterprise knowledge assistant for 100,000 users. Cover freshness, deletion, ACLs, hybrid retrieval, reranking, evaluation, latency, and incident debugging.
