LangGraph & Its Components

LangGraph is a library for building stateful, multi-step LLM applications as a graph: you define the steps (nodes), how control flows between them (edges), and a shared state object that every step reads and writes. Where a plain chain runs start-to-finish once, a LangGraph app can loop, branch, pause for a human, and resume — which is exactly what agents need. This page takes it apart one component at a time.

1 · Why a graph — chains vs graphs

A LangChain chain is a straight line: A → B → C, once, top to bottom. That's fine for "retrieve then answer," but it can't express the things agents do:

What a chain can't do

  • Loop — "call a tool, look at the result, decide whether to call another."
  • Branch — take a different path depending on the LLM's decision.
  • Pause & resume — stop for human approval, continue later.
  • Remember — keep state across turns and across restarts.

What a graph adds

LangGraph models the app as nodes connected by edges over a shared state. Edges can be conditional (branch) and can point backward (loop). A built-in checkpointer saves state so a run can pause and resume. This is the control flow agents require.

🔑
The one-sentence model

A LangGraph app = a shared state object + nodes that transform it + edges that decide what runs next, compiled into a runnable graph with optional persistence.

2 · The components at a glance

State

The shared data every node reads & writes. The single source of truth.

Node

A unit of work — a Python function that takes state and returns an update.

Edge

Wiring that says "after node X, go to node Y."

Conditional edge

An edge whose target is chosen at runtime by a router function — branching & looping.

StateGraph

The builder you add nodes/edges to, then .compile() into a runnable app.

Reducer

Rule for how a state field is updated (overwrite vs append). Key for message history.

Checkpointer

Persistence — saves state per step so a graph can pause, resume, and remember across sessions.

ToolNode

A ready-made node that runs whatever tool the LLM asked for.

Human-in-the-loop

interrupt — pause the graph for human input/approval, then continue.

We'll build them up in order; by the end they assemble into a small working agent (§10).

3 · State — the shared object

Everything in a LangGraph run flows through one state object. You define its shape as a typed dictionary; each node receives the current state and returns a partial update, which LangGraph merges in.

python
from typing import TypedDict

class State(TypedDict):
    question: str
    answer: str
    steps: int
💡
Mental model

Think of state as a whiteboard the whole graph shares. A node reads what it needs, writes its contribution, and hands control on. Nodes don't call each other directly — they communicate only through state.

4 · Reducers — how state fields update

By default, returning a field from a node overwrites it. But often you want to append — most importantly for a running list of chat messages. A reducer is a function attached to a field that says how updates combine. The canonical one is add_messages, which appends to (and de-duplicates) a message list.

python
from typing import Annotated, TypedDict
from langgraph.graph.message import add_messages

class State(TypedDict):
    # Annotated[type, reducer] — updates to `messages` are APPENDED, not replaced
    messages: Annotated[list, add_messages]
    steps: int                       # no reducer → each update OVERWRITES
🧩
Why this matters

Without a reducer, each node that "added a message" would wipe the conversation. Annotated[list, add_messages] is what makes message history accumulate correctly across nodes and turns. You can write custom reducers for any append/merge semantics you need.

5 · Nodes — units of work

A node is just a function: state → partial update. It can do anything — call an LLM, run a tool, transform data. It returns only the fields it changes.

python
from langchain_openai import ChatOpenAI
llm = ChatOpenAI(model="gpt-4o-mini")     # Ollama: ChatOllama(model="llama3.1")

def answer_node(state: State) -> dict:
    reply = llm.invoke(state["messages"])
    return {"messages": [reply],          # appended (add_messages reducer)
            "steps": state["steps"] + 1}  # overwritten
📝
Rules of a node

Return a dict of changes, not the whole state. Keep nodes single-purpose (one LLM call, one tool, one transform) — small nodes make the graph readable and the control flow easy to reason about.

6 · Edges — normal & conditional

Edges define control flow. There are two kinds, plus two special endpoints.

6.1 Normal edges — fixed wiring

"After node A, always go to node B." START marks where the graph begins; END marks where it stops.

python
from langgraph.graph import StateGraph, START, END

g = StateGraph(State)
g.add_node("answer", answer_node)
g.add_edge(START, "answer")     # entry point
g.add_edge("answer", END)       # done after answering

6.2 Conditional edges — branching & looping

This is what makes it a graph and not a chain. A router function reads the state and returns the name of the next node — so the path is decided at runtime. Pointing an edge back to an earlier node creates a loop.

python
def route(state: State) -> str:
    last = state["messages"][-1]
    # if the LLM asked to call a tool, go run it; otherwise finish
    return "tools" if getattr(last, "tool_calls", None) else END

g.add_conditional_edges("agent", route, {"tools": "tools", END: END})
g.add_edge("tools", "agent")     # ← loop back: tool result returns to the agent
START │ ▼ agent (LLM decides: answer or call a tool) │ route(state) ├─"tools"─▶ tools ──┐ │ │ (tool result appended to state) │ ◀──────────────┘ loop back to agent └─END─▶ STOP
A conditional edge + a back-edge form the classic agent loop: think → act → observe → think, until done.

7 · StateGraph — build & compile

The StateGraph is the builder. You add nodes and edges to it, then call .compile() to get a runnable app you can invoke or stream.

python
builder = StateGraph(State)
builder.add_node("agent", agent_node)
builder.add_node("tools", tool_node)
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route, {"tools": "tools", END: END})
builder.add_edge("tools", "agent")

app = builder.compile()                       # → runnable graph
result = app.invoke({"messages": [("user", "What's 23 * 47?")], "steps": 0})
🌊
invoke vs stream

app.invoke(...) runs to completion and returns the final state. app.stream(...) yields the state after each node — use it to show step-by-step progress (great for a UI, and for debugging which node did what).

8 · Checkpointer — persistence & memory

Pass a checkpointer at compile time and LangGraph saves the state after every step. This unlocks three things at once: memory across turns, pause/resume, and recovery after a crash. Each conversation is keyed by a thread_id.

python
from langgraph.checkpoint.memory import MemorySaver   # in-memory (dev)
# production: SqliteSaver / PostgresSaver — same interface, durable storage

app = builder.compile(checkpointer=MemorySaver())

cfg = {"configurable": {"thread_id": "chat_42"}}      # identifies the conversation
app.invoke({"messages": [("user", "My name is Alice.")]}, cfg)
app.invoke({"messages": [("user", "What's my name?")]}, cfg)   # → "Alice" (state persisted)
🧠
This is short-term memory, done for you

The checkpointer is the conversation-buffer / working-memory pattern from the Memory tab §5 — LangGraph just manages it automatically and persists it by thread_id. Swap MemorySaver for SqliteSaver/PostgresSaver and the same app remembers across restarts.

9 · Tools & ToolNode

An agent acts by calling tools. You bind tools to the LLM so it can request them, and add a ToolNode that executes whatever the LLM asked for. The conditional edge from §6.2 routes between the two.

python
from langchain_core.tools import tool
from langgraph.prebuilt import ToolNode

@tool
def multiply(a: int, b: int) -> int:
    "Multiply two integers."
    return a * b

tools = [multiply]
llm_with_tools = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

def agent_node(state: State) -> dict:
    return {"messages": [llm_with_tools.invoke(state["messages"])]}

tool_node = ToolNode(tools)            # runs whichever tool the LLM requested
📎
The agent loop, assembled

agent node (LLM, may emit tool calls) → conditional edge → ToolNode (runs the tool) → back-edge to agent (sees the result) → repeats until the LLM answers with no tool call. That's the loop diagrammed in §6.2. The concept is from 08 · Agentic AI →; LangGraph gives it durable structure.

10 · Human-in-the-loop

Because the checkpointer can pause and resume a run, LangGraph can stop and wait for a human — for approval before a risky action, or to collect missing input. The interrupt primitive pauses inside a node; you resume by invoking again with the human's decision.

python
from langgraph.types import interrupt, Command

def approve_node(state: State) -> dict:
    decision = interrupt({"action": "run SQL DELETE", "sql": state["sql"]})  # ⏸ pause here
    if decision != "approve":
        return {"answer": "Cancelled by the operator."}
    return {"answer": run_sql(state["sql"])}                                 # ▶ resume

# first call pauses at the interrupt; the graph state is checkpointed
app.invoke({...}, cfg)
# later: resume with the human's answer
app.invoke(Command(resume="approve"), cfg)
🛑
Why this is a guardrail

Human-in-the-loop is the structural way to gate irreversible or sensitive actions (deletes, sends, payments) behind explicit approval — the "always ask before a destructive action" rule from the Guardrails tab, enforced by the graph rather than by hoping the model behaves.

11 · Putting it together — a tool-using agent

All the components in one minimal but complete graph: an agent that can multiply numbers via a tool, looping until it has the final answer, with memory via a checkpointer.

agent_graph.py
from typing import Annotated, TypedDict
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode
from langgraph.checkpoint.memory import MemorySaver

# 1 · State (with a message reducer)
class State(TypedDict):
    messages: Annotated[list, add_messages]

# 2 · Tools + an LLM that can request them
@tool
def multiply(a: int, b: int) -> int:
    "Multiply two integers."
    return a * b

llm = ChatOpenAI(model="gpt-4o-mini").bind_tools([multiply])

# 3 · Nodes
def agent(state: State) -> dict:
    return {"messages": [llm.invoke(state["messages"])]}

# 4 · Router (conditional edge)
def route(state: State):
    last = state["messages"][-1]
    return "tools" if getattr(last, "tool_calls", None) else END

# 5 · Build the graph
builder = StateGraph(State)
builder.add_node("agent", agent)
builder.add_node("tools", ToolNode([multiply]))
builder.add_edge(START, "agent")
builder.add_conditional_edges("agent", route, {"tools": "tools", END: END})
builder.add_edge("tools", "agent")          # loop back with the tool result

# 6 · Compile with persistence
app = builder.compile(checkpointer=MemorySaver())

# 7 · Run
cfg = {"configurable": {"thread_id": "demo"}}
out = app.invoke({"messages": [("user", "What is 23 * 47, then add 1?")]}, cfg)
print(out["messages"][-1].content)          # → 1082
▶️
What happens at run time

agent sees the question and emits a multiply(23, 47) tool call → router sends it to tools → ToolNode returns 1081 → back to agent, which adds 1 and answers 1082, no more tool calls → router returns END. The whole trajectory is checkpointed under thread_id="demo".

13 · Best practices & pitfalls

  • Keep nodes small and single-purpose — one LLM call or one tool per node makes the graph readable and debuggable.
  • Use the right reducer — add_messages for chat history; without a reducer a field is overwritten (a common "where did my messages go?" bug).
  • Always set a recursion/step limit — a bad router can loop forever; cap iterations so a runaway agent stops.
  • Stream for visibility — app.stream() shows which node ran and what it changed; pair with Langfuse to trace it in production.
  • Pick a durable checkpointer for prod — MemorySaver is dev-only; use Sqlite/Postgres savers so state survives restarts.
  • Gate risky actions with human-in-the-loop — don't let the graph run a delete/send/payment without an interrupt approval step.
  • Keep state lean — store what nodes need, not everything; large state is copied each step and costs tokens when fed to the LLM.
