System Map: Agents, Layers & Stack

Before writing prompts, learn where intelligence belongs—and where it absolutely does not belong—in a production recruitment system.

⏱ Study + design exercise · 3–5 hFoundationBased on the uploaded ATS app

1 · The mental model

Think of each ATS agent as a typed intelligence function. The service gives it a narrow problem, the model returns a JSON-shaped suggestion, and normal application code decides what to store or act on.

prepared business context＋role-specific instructions→validated JSON suggestion
💡
Why this is better than “let the agent run the app”

Recruitment decisions carry fairness, privacy and audit requirements. Narrow agents reduce the blast radius: they cannot directly update candidates, schedule meetings or change pipeline stages. Services remain the authority.

2 · Request flow and layer boundaries

The service layer is the orchestration boundary. It assembles trusted context, invokes the agent, validates the result, persists selected fields and writes audit records.
LayerWhat it ownsWhat it must not own
RouterHTTP shape, authentication, role gatePrompt construction or SQL

AgentSystem prompt, user prompt, output contractDomain writes or workflow transitions
RepositoryDatabase access and row scopingHiring decisions

service_boundary.py
# The key boundary: agents do not write business tables.
# They transform prepared inputs into validated structured output.

async def screen_application(db, user, application_id):
    application = await application_repo.get_by_id(db, application_id)
    prompt_input = build_screening_input(application)

    result = await resume_screening.run(**prompt_input)

    # The service decides what may be persisted and audited.
    application.ai_screen_score = result["match_score"]
    application.ai_screen_result = result
    await audit_service.record(db, action="SCREEN", performed_by=user.user_id)
    await db.commit()
    return result

3 · Tech stack used — and why

Python 3.11 + FastAPI

Clear async functions, strong typing and fast API development. Agent calls and database calls are I/O-bound, so async keeps the server responsive while waiting.

Pydantic v2

Turns model output into a contract. Scores can be bounded, required fields checked and malformed output rejected before persistence.

SQLAlchemy 2 + Alembic

Separates domain persistence from agent logic and gives repeatable schema migrations.

PostgreSQL + JSONB

Relational truth for candidates and workflows; JSONB for flexible agent result snapshots and run metadata.

Anthropic async SDK

A single configurable model client with token usage and API error classes available to the wrapper.

React + TanStack Query

Server-state caching, mutation feedback and easy invalidation after agent-backed actions.

MinIO

CV and offer files stay outside the database. The browser never speaks directly to storage without a server-authorized URL.

Docker Compose

Packages frontend and backend consistently while databases, object storage and the model API remain environment-configured.

5 · Why bounded task agents fit this app

🧪

Testable

One input contract and one output contract per task. Regression datasets are straightforward.

🔐

Safer

No broad tool permissions. The LLM cannot directly mutate the recruitment workflow.

🔎

Auditable

Every call records model, token counts, latency, status and entity context.

6 · Knowledge check

Which layer should decide whether an agent result is stored on an application?
The service owns business rules and transactions. The agent only returns a validated suggestion.
Are these six components open-ended autonomous agents?
They are deliberately narrow. That makes the ATS easier to reason about and safer to operate.
