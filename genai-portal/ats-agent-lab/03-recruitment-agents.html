Lesson 03 · Core recruitment intelligence

Build the Recruiting Agents

Build the three agents that turn a requisition and a set of CVs into structured, reviewable recruiting support.

⏱ Study + implementation · 4–7 hIntermediateBased on the uploaded ATS app

1 · Three agents, one recruiting story

The JD becomes shared context. Screening explains one candidate; matching compares a small pool. Neither makes the shortlist decision.

2 · Agent A — JD creation

Problem: Requisition forms are structured but not candidate-friendly. The agent converts known facts into professional markdown without inventing salary, benefits or company claims.

Inputs
  • Position and project facts
  • Essential and desired skills
  • Experience, location, work mode
Outputs
  • JD markdown
  • SEO title
  • Keywords
Safety
  • Allowed facts only
  • Saved as a new version
  • Human edit path remains
agents/jd_creation_simplified.py
class JdOutput(BaseModel):
    jd_markdown: str = Field(min_length=1)
    seo_title: str = ""
    keywords: list[str] = []

async def create_jd(rrf: dict, skills: list[dict]) -> dict:
    facts = pick_jd_safe_fields(rrf)  # do not dump the entire ORM object
    user = f"""requisition: {json.dumps(facts)}
essential_skills: {essential(skills)}
desired_skills: {desired(skills)}
Return: {{"jd_markdown": str, "seo_title": str, "keywords": [str]}}"""
    return await call_llm_json(system=JD_SYSTEM, user=user,
                               output_model=JdOutput)
📝
Why version instead of overwrite?

Generated text is a draft, not truth. Version history preserves who created each revision and lets a hiring manager compare or roll back.

3 · Agent B — resume screening

Problem: Recruiters need a consistent first pass across many CVs. The agent scores against explicit criteria and must provide evidence from the CV.

agents/resume_screening_simplified.py
async def screen_candidate(*, application_id, job, candidate):
    user = SCREEN_TEMPLATE.format(
        title=job.title,
        min_exp=job.min_experience_years,
        essential=", ".join(job.essential_skills),
        desired=", ".join(job.desired_skills),
        jd=job.jd_markdown[:4_000],
        cand_exp=candidate.experience_years or "unknown",
        cv_text=(candidate.cv_text or "")[:12_000],
    )
    return await call_llm_json(
        agent_name="resume_screening",
        system=SCREEN_SYSTEM,
        user=user,
        output_model=ResumeScreeningResult,
        entity_id=str(application_id),
        max_tokens=1_500,
    )

Essential skills dominate

Missing must-have skills are penalized more heavily than missing desired skills.

Evidence beats adjectives

Each coverage item includes a CV-grounded evidence string.

Recommendation is advisory

The result supports a recruiter; it does not transition the candidate automatically.

4 · Agent C — candidate matching

Problem: A score for one candidate is not enough when a recruiter needs relative ranking. This agent sees a small pool and a canonical skill vocabulary, including aliases.

services/candidate_ranking.py
async def rank_candidates(rrf, candidate_rows):
    pool = []
    for row in candidate_rows[:10]:
        evidence = row.parsed_cv.get("skills") if row.parsed_cv else None
        pool.append({
            "candidate_id": str(row.id),
            "experience_years": row.experience_years,
            "skills_or_excerpt": evidence or row.cv_text[:2_000],
        })

    return await candidate_matching.run(
        canonical_skills=canonical_skill_payload(rrf.skills),
        min_experience_years=rrf.min_experience_years,
        candidates=pool,
    )
🧩
Why canonical skills matter

“Postgres,” “PostgreSQL” and “PG” should not become three unrelated skills. Normalizing against a skill master makes rankings more comparable and explainable.

5 · Screening vs matching

Resume screeningCandidate matching
QuestionHow well does this person fit?Who fits best in this pool?
ScopeOne applicationUp to ten active candidates
PersistenceScore + full result cached on applicationRanking is advisory; run trace persists
Primary outputEvidence, risks, recommendationComparable order and gap chips

6 · Optimizations already present

  • CV input is capped at 12,000 characters.
  • JD input is capped at 4,000 characters.
  • Matching is limited to ten active candidates.
  • If parsed skills exist, the agent receives them instead of a long CV excerpt.
  • Otherwise only a 2,000-character excerpt is sent for ranking.
  • Full screening output is cached, avoiding a repeat call every time the detail page opens.

8 · Knowledge check

Why is candidate matching limited to a small pool?
Batching is useful, but an unbounded pool would increase cost and make ranking less stable.
