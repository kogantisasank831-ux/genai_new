Optimization, Evaluation & Operations

Move from “the demo looked good” to a repeatable engineering loop for quality, cost, latency and safety.

⏱ Study + evaluation work · 5–8 hAdvanced · LLMOpsBased on the uploaded ATS app

1 · Optimize the whole system, not only the prompt

Production optimization has four goals: quality, latency, cost and reliability. A shorter prompt that loses essential evidence is not an optimization. A larger model that doubles quality for a rare task may be worthwhile.

Left: remove irrelevant context before paying for inference. Right: evaluation is a continuous loop, not a one-time demo.

3 · Cache by business meaning

services/question_cache.py
async def get_or_generate_questions(interview):
    if interview.ai_interview_questions:
        return interview.ai_interview_questions

    # Optional: acquire a short distributed lock keyed by interview ID
    # so two clicks do not create duplicate model calls.
    result = await generate_questions(interview)
    interview.ai_interview_questions = result
    await db.commit()
    return result

A useful cache key should include the inputs that change meaning: CV version, JD version, agent prompt version and model version. Caching only by candidate ID can serve stale results after a new CV or JD.

4 · Route tasks to the smallest model that passes evals

Low-complexity

Formatting a JD or summarizing already structured ratings may work on a faster model.

Higher-complexity

Comparative ranking and nuanced evidence extraction may need a stronger model.

Deterministic work

Do not use a model at all for overlap checks, role gates or state transitions.

Model routing should be driven by your evaluation data, not by prestige or habit.

5 · Build an evaluation set before tuning

evals/screening_eval.py
from dataclasses import dataclass

@dataclass
class ScreeningCase:
    name: str
    job: dict
    candidate: dict
    expected_recommendation: str
    required_missing_skills: set[str]

async def evaluate_case(case: ScreeningCase):
    result = await screen_candidate(job=case.job, candidate=case.candidate)
    return {
        "recommendation_ok": result["recommendation"] == case.expected_recommendation,
        "missing_skill_recall": len(
            case.required_missing_skills & set(result["missing_essential_skills"])
        ) / max(1, len(case.required_missing_skills)),
        "score": result["match_score"],
    }

# Run every case several times; report mean and variance, not one lucky output.

6 · Operational metrics

Latency

p50, p95 and timeout rate per agent and model.

Cost

Input/output tokens, repair-call rate and cost per successful task.

Reliability

Success, degraded, failure and manual-fallback usage.

Quality

Eval pass rate and human correction rate by prompt version.

Drift

Score distribution and output-field changes after model upgrades.

Safety

Injection test pass rate, policy violations and sensitive-output incidents.

7 · Testing pyramid for agents

Unit tests — many

Prompt builders, fence stripping, overlap logic, Pydantic bounds, privacy transforms.

Contract tests — moderate

Mock provider responses: valid JSON, fenced JSON, malformed JSON, 429, timeout, schema drift.

Evaluation tests — regular

Human-labeled cases across normal, edge and adversarial inputs.

Live smoke tests — few

A tiny real API call verifies credentials, model access and durable run logging.
