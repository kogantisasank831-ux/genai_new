RAG, End-to-End

RAG (Retrieval-Augmented Generation) means: instead of asking an LLM to answer purely from its internal memory, we first retrieve relevant knowledge from external sources, then hand that context to the LLM so it can generate a grounded, accurate answer. This page walks every stage of a real production pipeline — from raw files to a cited final response.

⏱ Deep study + implementation · 12–18 h Reference · breadth-first Enterprise / plant & process examples

🔑
The one idea to hold onto

RAG quality is bounded by your weakest stage. A great LLM cannot rescue bad chunks; perfect retrieval cannot rescue a sloppy prompt. Optimize the chain, not one link.

2 · Data sources

These are the knowledge bases RAG retrieves from. The quality of RAG depends heavily on the quality of the source.

Common sources

PDFs · Word · Excel/CSV · database tables · web pages · SharePoint/Confluence · emails · logs · API docs · source code · images (via OCR) · knowledge graphs.

Typical enterprise sources

SOP documents · plant manuals · historical tickets · maintenance reports · QC templates · procurement RFQs · database metadata · API documentation · training logs.

💡
Match the source to the retrieval method

Structured tables (limits, KPIs, master data) are best answered with SQL retrieval; narrative documents with vector search; relationships with a knowledge graph. See §9 Retrieval.

3 · Data ingestion

Ingestion brings raw documents into the pipeline: read files, extract text and tables, parse structure, detect headings, split sections, and keep a reference back to the original source.

From a single PDF you might capture: document name, page number, section title, paragraph text, table text, image captions, created date, author, department.

Metadata is the secret weapon

Rich metadata makes filtering possible, which is often the single biggest precision win in enterprise RAG.

5 · Chunking — the unit of retrieval

Chunking splits documents into smaller pieces that can be embedded and retrieved. The chunk is the atomic unit of retrieval, so its design dominates retrieval quality. A good chunk is small enough to fit the context, large enough to preserve meaning, traceable to its source, and metadata-rich.

5.1 · Fixed-size chunking

Split by a fixed number of tokens/characters, usually with overlap (e.g. size 500, overlap 50 → chunks 1–500, 451–950, 901–1400).

Pros

Simple, fast, easy to implement; fine for generic text.

Cons

Cuts sentences/tables mid-stream; separates headings from content; loses meaning.

Best for: large plain-text docs, logs, generic reports, quick prototypes.

5.2 · Recursive character chunking

Splits using a hierarchy of separators — try paragraphs (\n\n), then lines (\n), then sentences (.), then words ( ), then characters. Falls back only when a piece is still too large.

Pros: preserves readability, better than pure fixed-size. Cons: still no real understanding of meaning; struggles with complex technical sections.

Best for: general RAG, documentation, policy docs, manuals, FAQs.

5.3 · Sentence-based chunking

Chunks are made of complete sentences, so embeddings get clean semantic units. Pros: never cuts a sentence, good embedding quality. Cons: inconsistent chunk sizes; may lose section-level context.

Best for: natural-language docs, policies, SOPs, knowledge articles.

5.4 · Paragraph-based chunking

Each paragraph (or small group) becomes a chunk. Pros: preserves human-written structure, interpretable. Cons: paragraphs vary wildly in length; tables/bullets get awkward.

Best for: reports, business & legal documents, technical descriptions.

5.5 · Heading / section-based chunking practical favorite

Split on document headings (1. Introduction, 2. Scope, 3. Input Tags…). Each section becomes one or more chunks. Pros: preserves hierarchy, great traceability, ideal for enterprise docs. Cons: depends on reliable heading detection; long sections still need sub-splitting.

Best for: SOPs, technical manuals, API docs, project & procurement & regulatory documents. One of the most practical strategies for enterprise RAG.

5.6 · Semantic chunking

Splits on meaning, not length. Embed each sentence, compare adjacent similarity, and start a new chunk when the topic shifts (similarity drops below a threshold). Sentences about model inputs land in one chunk; API endpoint details in another.

Pros: meaning-preserving, higher retrieval accuracy, avoids mixing topics. Cons: more expensive/complex, needs embeddings during ingestion, threshold tuning, slower.

Best for: knowledge-heavy docs, research papers, technical manuals, mixed-topic content.

5.7 · Sliding-window (overlapping) chunking

Overlapping windows (e.g. size 800, overlap 200) so information spanning a boundary isn't lost. Pros: reduces boundary context loss. Cons: more storage, duplicate context, can add retrieval noise (pair with MMR).

Best for: long technical docs, manuals, legal contracts, large reports.

5.8 · Table-aware chunking

Tables must not be blindly split row-by-row or page-by-page. Convert them to a form the LLM reads well:

table → text
The following table defines tag limits:
- PDI1043 has LSL 0 and USL 10 kg/cm2.
- TI1354  has LSL 20 and USL 200 °C.

Methods: Markdown conversion · row-wise text · a table-summary chunk · parent-child table chunking. Best for: Excel, procurement CBAs, QC templates, tag-limit tables, financial/KPI tables.

5.9 · Code-aware chunking

Chunk by code structure — function, class, method, API endpoint, module, config block — so a whole def predict(): … stays together. Best for: Python APIs, FastAPI code, SQL scripts, config-driven frameworks.

5.10 · Parent-child chunking advanced

Store small, precise child chunks for retrieval, but return the larger parent section for generation. Search hits the precise child; the LLM receives the richer surrounding context.

Pros: high retrieval precision + good answer context. Cons: more complex indexing, needs a child→parent map. Best for: enterprise RAG, long manuals, regulatory & project documentation.

5.11 · Contextual Retrieval trending · Anthropic

Before embedding, prepend each chunk with a short LLM-generated blurb situating it in the whole document (“This chunk is from the Q3 maintenance report, section on TI1354 tube leakage…”). The standalone chunk “the limit was raised to 200°C” becomes self-contained instead of ambiguous, so it embeds and matches far better. Index the contextualized text with both vectors and BM25.

Pros: large drop in retrieval-failure rate; fixes the classic “orphan chunk loses its context” problem; pairs especially well with hybrid + rerank. Cons: one LLM call per chunk at ingestion (mitigate with prompt caching), higher build cost & storage. Best for: long manuals, SOPs, regulatory docs, anything where chunks lose meaning out of context.

6 · Embeddings

📎
This has a dedicated module

Embeddings (what a vector is, cosine similarity, choosing a model) are covered in depth in 04 · Embeddings →. Summary below; the cosine/distance math used to score them lives in §10 Retrieval scoring.

An embedding converts text into a numeric vector that captures meaning: "Pump A leakage prediction model" → [0.12, -0.45, 0.88, …]. Similar concepts get similar vectors, so “pump leakage”, “tube leakage” and “seal failure” sit close together.

Model families: OpenAI, Sentence-Transformers, BGE, E5, Cohere, Instructor, multilingual and code embeddings. Choose based on language, domain vocabulary, latency, cost, accuracy, deployment and security constraints.

⚠️
Domain vocabulary breaks generic models

Tokens like PDI1043.PV, FIC1002.PV, LSL, USL, “trip tag”, “loop/subloop”, “factor/KPI mapping” may not be understood by a generic embedding model. Design your chunk text + metadata to spell these out, and lean on keyword retrieval for exact identifiers.

7 · Indexing & storage

🗄️
Goes deeper

Embeddings are stored in a vector database. Common choices: FAISS, Chroma, Pinecone, Weaviate, Milvus, Qdrant, Elasticsearch / OpenSearch, PostgreSQL + pgvector, Azure AI Search, Databricks Vector Search. Each stored chunk typically carries its id, text, vector, source, page, section, metadata, timestamps and access permissions.

8 · Query processing

We don't always send the raw query to retrieval. A vague question like “what are the inputs for this model?” can first be turned into “List the input tags used for the PDI1043 forecasting model.”

8.1 Query rewriting

The LLM rewrites a short/vague query into a clear search query. “what all tags are used?” → “input tags used in PDI1043 forecasting model”.

8.2 Query expansion

Add related terms for recall: “tube leakage” → tube leakage OR exchanger leakage OR heat-exchanger tube failure. Great when teams use different vocabulary.

8.3 HyDE

Hypothetical Document Embeddings: the LLM drafts a hypothetical answer, then embeds that instead of the short query. Helps vague queries; costs an extra LLM call and can inject assumptions.

8.4 Multi-query retrieval

Generate several phrasings of the query, retrieve for each, then merge → dedupe → rerank. Better recall, handles vocabulary mismatch, more retrieval calls.

8.5 Query decomposition

Break a complex question into sub-questions. “Explain the API, model inputs, training logic, database tables, and deployment” becomes five focused retrievals — ideal for complex enterprise questions, and a natural fit for agentic RAG.

9 · Retrieval methodologies

📎
Builds on Module 06 & 07

The retrieve-then-generate loop is introduced in 06 · RAG Basics →; hybrid search, reranking and Graph/Agentic RAG are expanded in 07 · Advanced RAG →. This section is the compact field guide.

9.1 Dense (vector) retrieval

Embed the query, compare against chunk vectors, return nearest neighbours. Understands meaning and synonyms — “pump failure reason” can surface “seal damage”, “bearing issue”, “cavitation”. Weakness: exact IDs, tag names, codes and numbers (e.g. PDI1043.PV) may slip through.

9.2 Sparse / keyword retrieval

Exact or near-exact word matching: BM25, TF-IDF, Elasticsearch/OpenSearch lexical, SQL LIKE. Excellent for exact tag names, part/document IDs, error codes, invoice/KPI ids, API endpoint names. Fast and explainable; poor at synonyms/meaning.

9.3 Hybrid retrieval recommended default

Combine dense + sparse. For “PDI1043 pressure forecasting input tags”, BM25 nails PDI1043 and "input tags" while vector search captures "forecasting model features / pressure prediction variables". Best for technical enterprise data; needs score normalization (see fusion).

9.4 Metadata filtering

Filter the candidate set before/with retrieval (plant=Ammonia · loop=Loop10 · equipment=Pump A). Improves precision, shrinks the search space and enforces access control.

9.5 Knowledge-graph retrieval

Store relationships — PDI1043.PV is_input_to ForecastingModel, Pump A belongs_to Subloop2, KPI_X depends_on PDI1043.PV. Excellent for relationship questions, asset hierarchy and root-cause/dependency tracing. Requires building & maintaining the graph. (See Graph RAG, §16.7.)

9.6 SQL / structured retrieval

If the answer lives in a database, query it directly rather than vectorizing it.

sql
SELECT * FROM tag_master WHERE TAG_NAME = 'PDI1043.PV';

Best for live values, master data, KPI mappings, config tables, inventory, transactions, procurement. A strong enterprise design blends vector (docs) + SQL (structured) + graph (relationships).

9.7 Agentic retrieval

The LLM acts as a router, choosing the tool per question: tag limits → SQL; “explain the methodology” → vector search; “which equipment connects to this KPI?” → graph; “summarize the latest RFQ” → document retrieval.

10 · Retrieval scoring methods

Once candidates are retrieved we score relevance. Different similarity/distance measures suit different cases.

MethodIdeaDirectionBest for
Cosine similarityAngle between vectors, A·B / (‖A‖‖B‖), range −1…1higher = closerText embeddings; ignores magnitude — the common default
Euclidean (L2)Straight-line distance √Σ(Aᵢ−Bᵢ)²lower = closerSpaces where actual distance matters; clustering
Dot productΣ AᵢBᵢhigher = closerNormalized embeddings (≈ cosine); high-perf search
Manhattan (L1)Σ|Aᵢ−Bᵢ|lower = closerSparse/grid-like spaces; less common for text
BM25Term freq × inverse doc freq, length-normalizedhigher = closerExact words & technical identifiers
TF-IDFFrequent-here-but-rare-overall wordshigher = closerSimpler keyword scoring
↕️
Mind the direction

For cosine / dot / BM25, higher is better. For Euclidean / Manhattan, lower is better. Mixing them up silently inverts your ranking.

10.7 Maximal Marginal Relevance (MMR)

Balances relevance to the query against diversity vs. already-selected chunks — conceptually MMR = relevance − similarity_to_selected. Stops the top-5 from being five near-duplicates. Pairs naturally with overlapping chunks.

10.8 Combining rankings — RRF & weighted fusion

Reciprocal Rank Fusion (RRF)

Merge rankings from multiple retrievers by rank position. BM25 → A,B,C and vectors → B,D,A fuse to B,A,D,C. Simple, robust, needs no score calibration — great for hybrid search.

Weighted score fusion

final = α·vector + β·keyword + γ·metadata, e.g. 0.6·cosine + 0.3·bm25 + 0.1·meta. Tune by query type: exact tag → BM25 high; conceptual → vector high; scoped → metadata high.

11 · Reranking

Initial retrieval may return 20–100 chunks; reranking picks the best 5–10. The pattern: retrieve top 50 (fast, approximate) → rerank with a stronger model → keep top 5–10 (slow, accurate).

Rerankers: cross-encoder, LLM-based, Cohere Rerank, BGE reranker, ColBERT, domain-specific. Where embedding retrieval is a quick approximate filter, a reranker deeply compares the query against each chunk.

🎯
Why it helps

For “input tags for PDI1043 forecasting?”, retrieval might return a general description, the input-tag list, an output explanation and alarm logic. A good reranker pushes the input-tag list to the top.

12 · Context selection & compression

The LLM can't take unlimited chunks, so we decide what to pass.

  • Top-K — pass the best K (e.g. 5). Simple but may admit weak chunks.
  • Score threshold — only pass chunks above a relevance score (e.g. cosine > 0.75) to avoid weak context.
  • Diversity / metadata-prioritized — favour varied or high-priority sources (see MMR).
  • Parent-child expansion — retrieve precise children, send their parents (see §5.10).
  • Context compression — summarize/extract only the query-relevant parts before sending.
⚠️
Compression is a double-edged sword

It cuts tokens and sharpens focus and lets you fit more sources — but it can drop an important detail or even hallucinate during the summarization step. Validate compressed context against the originals for high-stakes answers.

13 · Prompt construction

The final LLM input typically stitches together a system instruction, the user question, the retrieved context, and explicit answer/citation/format rules.

prompt template
You are an assistant answering using ONLY the provided context.

Context:
[Chunk 1] ...
[Chunk 2] ...

Question:
What are the input tags for PDI1043 forecasting?

Instructions:
- Answer only from the context.
- If context is insufficient, say so.
- Cite source file and page number.

Good prompt design tells the model to: don't hallucinate · use only retrieved context · cite sources · admit missing information · keep answers structured · resolve conflicts carefully (see §18).

14 · LLM generation

📎
Generation controls in depth

How decoding, temperature and the context window actually work is covered in 01 · Foundations of LLMs →.

The model receives question + context + instructions and generates the answer. Key controls: temperature, max tokens, system prompt, context window, citation format, tool access, response schema.

🌡️
Temperature rule of thumb

Factual RAG → temperature 0–0.3. Creative summarization → slightly higher is acceptable.

15 · Output scoring & evaluation

After generation, score the answer — essential in production. The main dimensions:

Faithfulness / groundedness

Is every claim supported by the retrieved context? If context says “PDI1043.PV is an input” but the answer calls it the “final output KPI”, faithfulness is low. Methods: LLM-as-judge, NLI, claim verification, citation matching.

Answer relevance

Does the answer actually address the question? “The model is deployed using FastAPI” may be true but irrelevant to “what are the input tags?”.

Context precision

relevant retrieved / total retrieved. 3 useful of 10 retrieved → low precision.

Context recall

Did retrieval find everything needed? Answer needs 9 tags but context has 5 → low recall.

Citation accuracy

Does the cited source actually support the sentence? Cites p3 but the fact is on p8 → fail.

Completeness

Covers all required parts? Asked for inputs, outputs, error handling & deployment but only answered the first two → low.

Conciseness

Is it needlessly long? Matters for chatbot UX.

Safety / compliance

Exposes confidential data, credentials, PII, or violates access permissions? (see §17).

⚖️
LLM-as-judge

An LLM scores the answer against a rubric (1 = unsupported … 5 = fully correct, complete, well-cited) across faithfulness, relevance, completeness, citation quality, clarity and safety. Powerful, but the judge can also err — calibrate it against human labels.

17 · RAG architectures

17.1 Naive RAG

Chunk → embed → top-K → LLM. Simple, fast to build, good for a PoC. No reranking, no eval; weak on complex queries and prone to hallucination.

17.2 Advanced RAG production

Adds better chunking, hybrid retrieval, metadata filters, reranking, query rewriting, context compression, citations and evaluation.

17.3 Modular RAG

Every component (chunker, embedder, retriever, reranker, generator, evaluator, guardrail) is swappable — tune each independently. Great for enterprise.

17.4 Agentic RAG

The LLM dynamically chooses tools — vector search, SQL, graph, web search, calculator, code interpreter, API call. See 08 · Agentic AI →.

17.5 Corrective RAG

Checks whether retrieval is good enough; if not, rewrites the query, retries, uses an alternate retriever, asks for clarification, or returns “insufficient context”. Cuts wrong answers.

17.6 Self-RAG

The model reflects mid-generation — “need more retrieval”, “context insufficient”, “claim unsupported”, “weak citation”. Useful but more expensive.

17.7 Graph RAG

Knowledge graph + documents. Traces “which KPIs break if PDI1043.PV is faulty?” via PDI1043.PV → model input → KPI → loop → equipment. See §9.5.

17.8 Multi-modal RAG

Text + tables + images + charts + scans, via OCR, table extraction, image captioning, chart understanding, multi-modal embeddings. Best for engineering drawings, P&IDs, scanned PDFs, dashboards, QC templates.

17.9 RAPTOR trending

Recursively cluster chunks and summarize each cluster, building a tree of summaries (leaves = raw chunks, higher nodes = broader summaries). Retrieve across all levels, so a query can grab a single fact or a document-wide synthesis. Strong for “summarize everything about X” and multi-document questions where flat top-K misses the big picture.

🔀
Agentic-RAG worked example

“Compare this month's production KPI with the documented target and explain the deviation.” → (1) SQL for current KPI, (2) retrieve the target definition from docs, (3) retrieve the deviation SOP, (4) compute the difference, (5) generate the explanation.

18 · Guardrails

📎
Production concern

Observability, cost controls, rate limiting and security/testing sit in 14 · Production GenAI →. Here we focus on RAG-specific guardrails.

Guardrails run before and after generation: access control · PII masking · confidential-data filtering · prompt-injection detection · source-trust filtering · toxicity filtering · hallucination detection · citation enforcement · SQL-safety checks · output-schema validation.

🔒
Access control is non-negotiable

A user in Department A must not retrieve Department B's confidential documents. Enforce permissions during metadata filtering, not just at the UI.

19 · Handling conflicting sources

Documents disagree — old SOP says limit = 10, new SOP says 12. A good RAG system should prefer the latest approved source, surface both when they conflict, cite dates, flag uncertainty, and never silently pick one. The metadata that makes this possible: approved_status · effective_date · version · owner · modified_date · document_type.

20 · Hallucination control

Bad RAG behaviour

No relevant context retrieved, yet the LLM answers confidently anyway.

Good RAG behaviour

“The retrieved context does not contain this information.”

Levers: high-quality retrieval · reranking · strict prompts · citations · low temperature · answer validation · an explicit “I don't know from the provided context” path · source-grounded claim checking.

21 · A production pipeline, end-to-end

  1. Ingest from SharePoint, PDFs, Excel, DB, APIs.
  2. Extract text, tables, metadata and permissions.
  3. Clean and normalize content.
  4. Chunk using section-aware + semantic strategies.
  5. Generate embeddings.
  6. Store in a vector DB with metadata.
  7. Store a keyword index (BM25) too.
  8. User asks a question.
  9. Rewrite & expand the query.
  10. Apply metadata & permission filters.
  11. Run hybrid retrieval.
  12. Rerank the top ~50.
  13. Select the top ~8 chunks.
  14. Compress context if needed.
  15. Generate the answer with citations.
  16. Score faithfulness & relevance.
  17. Return the answer — or say context is insufficient.
  18. Log feedback and improve retrieval.

23 · A strong default enterprise design

🏗️
Put together, a robust stack looks like this
  • Chunking: section-aware + recursive + semantic refinement
  • Retrieval: hybrid = vector + BM25
  • Scoring: cosine (dense) + BM25 (keyword), combined with RRF or weighted fusion
  • Reranking: cross-encoder or LLM reranker
  • Context: top-K + threshold + parent-child expansion
  • Generation: low-temperature LLM with a strict source-grounded prompt
  • Evaluation: faithfulness + answer relevance + context precision + recall
  • Guardrails: access control + citation check + hallucination check

Far better than basic vector search alone.

24 · A simple end-to-end example

User asks: “What are the input tags used for PDI1043 forecasting?”

1 · Query preprocessing

Detect entity = PDI1043; intent = input-tag lookup.

2 · Retrieval

BM25 matches exact “PDI1043”; vectors match “forecasting model input tags”; metadata filter plant=Ammonia, model=PDI1043.

3 · Scoring

Cosine scores semantic matches, BM25 scores exact matches, hybrid fusion combines them.

4 · Reranking

Reranker selects the chunks that actually list input tags.

5 · Context selection

Top 3 chunks go to the LLM.

6 · Generation

Answer lists: FI1023.PV, FIC1002.PV, TI1354.PV, PDI1012.PV, FFI1001.CPV, TI1081.PV, TI1407.PV, PDI1163.PV, PDI1043.PV.

7 · Output evaluation

Check the answer is supported by the chunks, all tags are included, and no extra tags were hallucinated.

25 · Best practices

  • Don't rely only on vector search — use hybrid retrieval for technical enterprise data.
  • Keep chunks meaningful, not just fixed-size.
  • Preserve metadata aggressively.
  • Treat tables separately.
  • Use reranking for precision.
  • Always cite sources.
  • Evaluate retrieval and the final answer separately.
  • Use guardrails for security.
  • Log failed queries and improve chunking/retrieval over time.
🏭
For plant APIs, tag mappings, QC & procurement, database automation

The best-fit design is usually: section-aware + table-aware chunking · hybrid retrieval · metadata filtering · SQL retrieval for structured tables · reranking · faithfulness evaluation · citation-based final answers.
