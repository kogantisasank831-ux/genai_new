How LLMs remember — and how they don't

Memory in LLMs

A large language model is stateless: between two API calls it remembers nothing. Every appearance of "memory" — a chatbot recalling your name, an agent picking up where it left off — is something the application builds around the model by deciding what to put back into its context window each turn. This page explains how that's done, from the basics to production patterns.

⏱ ~30 min read Foundations → intermediate Buffers · summaries · vector memory · agents
🔗
Where this fits

Memory builds on how the context window works (01 · Foundations of LLMs →) and uses the same embeddings + vector-store machinery as retrieval (04 · Embeddings, 05 · Vector Databases). Long-term memory and RAG are close cousins — §7 draws the line. Code uses OpenAI + LangChain; the patterns are provider-agnostic.

1 · The starting point — the model is stateless

Each call to a chat model is independent. The model sees only the messages you send in this request and produces a reply. It has no hidden memory of previous requests.

python · the model forgets
from openai import OpenAI
oai = OpenAI()

oai.chat.completions.create(model="gpt-4o-mini",
    messages=[{"role": "user", "content": "My name is Alice."}])
# → "Nice to meet you, Alice!"

oai.chat.completions.create(model="gpt-4o-mini",
    messages=[{"role": "user", "content": "What's my name?"}])
# → "I don't have that information."   ← no memory of the previous call
🔑
The one idea

"Memory" in an LLM app = deciding what to put into the context window on the next call. The model doesn't remember; your code re-supplies the relevant past every turn. Everything below is a strategy for choosing what to re-supply and how to store it.

2 · The context window — the model's only "RAM"

The context window is the maximum number of tokens the model can attend to in a single request — system prompt + conversation history + retrieved context + the question + room for the answer, all of it. It is the model's working memory, and it is finite.

📏

It's a budget

Everything competes for the same token budget. More history means less room for retrieved context or the answer.

💸

It costs

You pay for every token in the window on every call. Naively replaying a long history gets expensive fast.

🪟

It's transient

When the window fills, something must be dropped or compressed. Nothing persists after the call returns.

So application memory has two jobs: (1) keep the context window within budget, and (2) persist anything that must outlive a single request — both of which the model can't do itself.

3 · Short-term vs long-term memory

The standard split, by analogy to human memory:

🧠
Short-term (working) memory

The current conversation. Lives in the context window for the duration of a session. Implemented by replaying recent messages each turn. Lost when the session ends unless persisted.

🗄️
Long-term memory

Facts that should survive across sessions — user preferences, past decisions, learned context. Stored outside the model (a database or vector store) and selectively retrieved back into the window when relevant.

Short-termLong-term
ScopeOne conversation/sessionAcross sessions, indefinitely
Stored whereIn-memory message list → context windowExternal store (DB / vector DB / files)
How it's usedReplayed verbatim or summarizedRetrieved by relevance, injected as context
Survives restart?No (unless persisted)Yes

4 · Static vs runtime memory — a second axis

Lifespan (§3) is one way to classify memory. There's a second, independent axis that matters just as much: who writes it, and when. Some memory is authored by a human ahead of time; some is written by the app or the model while it runs.

📄
Static (authored) memory

Written by a person before the run and loaded into context every session. It doesn't change as the agent works. CLAUDE.md is the canonical example — also system prompts, a project-instructions file, or a hand-maintained "user preferences" doc.

⚙️
Runtime (dynamic) memory

Written by the application or the model during execution; it grows and changes as work happens. The conversation buffer (§5), the vector memory store (§6), and the model's own scratchpad via the memory tool (§8) are all runtime memory.

Static / authoredRuntime / dynamic
Written byA human, ahead of timeThe app or the model, while running
Changes during a run?No — read-only contextYes — appended/updated as it goes
ExamplesCLAUDE.md, system prompt, project rules, preferences fileConversation buffer, vector store, scratchpad / memory-tool files
Curated howYou edit the fileAutomatically, or by a memory agent
🔑
The two axes are independent

A piece of memory has both a lifespan (short- vs long-term) and an origin (static vs runtime). CLAUDE.md is static + long-term; the conversation buffer is runtime + short-term; a vector memory store is runtime + long-term. The four CoALA agent-memory types in §9 map onto this grid: semantic (facts/rules, e.g. CLAUDE.md) and procedural (skills) are typically static + long-term; working memory is runtime + short-term; episodic memory is runtime + long-term.

💡
Why the distinction is practical

Static memory is the cheapest, most reliable way to give an agent durable knowledge — if a fact is always true (coding conventions, who the user is, house style), put it in a CLAUDE.md-style file rather than hoping the model writes it to runtime memory and recalls it later. Reserve runtime memory for what can't be known ahead of time.

5 · Short-term memory patterns

Three ways to manage the running conversation, trading completeness against token cost.

5.1 Full buffer — replay everything

Keep the entire message list and send it all every turn. Simplest, perfectly faithful — until the conversation outgrows the context window or the cost.

python
history = []                      # short-term memory = this list

def chat(user_msg: str) -> str:
    history.append({"role": "user", "content": user_msg})
    resp = oai.chat.completions.create(model="gpt-4o-mini", messages=history)
    reply = resp.choices[0].message.content
    history.append({"role": "assistant", "content": reply})   # remember the reply too
    return reply

chat("My name is Alice.")
chat("What's my name?")           # → "Alice" — the full history was re-sent

5.2 Sliding window — keep the last N turns

Send only the most recent messages. Bounds cost and stays in-window, but the model forgets anything older than the window.

python
WINDOW = 10                       # keep last 10 messages

def chat(user_msg: str) -> str:
    history.append({"role": "user", "content": user_msg})
    recent = history[-WINDOW:]            # only the tail goes into the window
    resp = oai.chat.completions.create(model="gpt-4o-mini", messages=recent)
    ...

5.3 Summary memory — compress the old, keep the new

Periodically summarize older turns into a compact "story so far," then send summary + recent messages. Retains the gist of a long conversation at a fraction of the tokens. This is the idea behind compaction in long-running agents.

python
summary = ""                      # rolling summary of older turns
recent  = []                      # last few verbatim messages

def summarize(old_summary: str, messages: list) -> str:
    r = oai.chat.completions.create(model="gpt-4o-mini", temperature=0,
        messages=[{"role": "system",
                   "content": "Update the running summary with the new messages. "
                              "Keep names, decisions and facts; be concise."},
                  {"role": "user",
                   "content": f"Summary so far:\n{old_summary}\n\nNew:\n{messages}"}])
    return r.choices[0].message.content

def chat(user_msg: str) -> str:
    global summary, recent
    recent.append({"role": "user", "content": user_msg})
    sys = [{"role": "system", "content": f"Conversation summary:\n{summary}"}]
    resp = oai.chat.completions.create(model="gpt-4o-mini", messages=sys + recent)
    reply = resp.choices[0].message.content
    recent.append({"role": "assistant", "content": reply})
    if len(recent) > 8:           # fold the oldest into the summary
        summary = summarize(summary, recent[:4]); recent = recent[4:]
    return reply

Full bufferGrows unboundedPerfectShort chats, prototypes
Sliding windowBoundedForgets older turnsLong chats where only recent context matters
SummaryLow, near-constantGist preserved, detail lostLong conversations needing overall continuity
🛠️
Frameworks do this for you

LangChain/LangGraph and similar expose these as ready-made memory types (buffer, windowed buffer, summary). Some APIs also offer server-side compaction that summarizes earlier context automatically as you near the window limit — you append the returned compaction block to the next request. The mechanics are exactly §5.3; the framework just hides the bookkeeping.

6 · Long-term memory — persist across sessions

To remember a user between today and next week, store facts outside the model and retrieve them when relevant. The dominant pattern is vector memory: embed each memory, store the vector, and on a new request fetch the most semantically similar memories into the context window.

New user message
Embed & search memory store top-k similar past facts
Inject retrieved memories into the prompt
LLM answers with the recalled context
Write new facts back to the store
Vector long-term memory: retrieve relevant past on the way in, write new memories on the way out.
python · minimal vector memory
import numpy as np
from openai import OpenAI
oai = OpenAI()

MEMORIES: list[dict] = []         # each: {"text": ..., "vec": np.ndarray}

def embed(text: str) -> np.ndarray:
    v = oai.embeddings.create(model="text-embedding-3-small", input=text).data[0].embedding
    return np.array(v)

def remember(text: str):          # write a long-term memory
    MEMORIES.append({"text": text, "vec": embed(text)})

def recall(query: str, k: int = 3) -> list[str]:   # read relevant memories
    if not MEMORIES: return []
    q = embed(query)
    scored = sorted(MEMORIES, key=lambda m: -float(q @ m["vec"]))   # cosine ~ dot (normalized)
    return [m["text"] for m in scored[:k]]

def chat(user_msg: str) -> str:
    recalled = recall(user_msg)
    sys = "Relevant things you remember about the user:\n- " + "\n- ".join(recalled) \
          if recalled else "You have no stored memories yet."
    resp = oai.chat.completions.create(model="gpt-4o-mini",
        messages=[{"role": "system", "content": sys},
                  {"role": "user", "content": user_msg}])
    return resp.choices[0].message.content

remember("The user's name is Alice and she manages the Ammonia plant.")
remember("Alice prefers answers in metric units.")
chat("What units should you use with me?")   # → recalls the metric preference
✍️
What to write to memory

Don't store every message. Store durable facts: stable preferences, decisions, identities, learned corrections. A common pattern is an LLM step that extracts "memory-worthy" facts from a conversation and writes only those — keeping the store small and high-signal.

7 · Memory vs RAG — same machinery, different purpose

Long-term memory and RAG both embed text, store vectors, and retrieve by similarity — so they look identical under the hood. The difference is what is stored and why:

RAG (retrieval)Long-term memory
StoresA knowledge corpus (docs, manuals, tickets)Facts about this user/agent and past interactions
Written byAn ingestion pipeline, ahead of timeThe app, during conversations
ScopeShared across all usersUsually per-user / per-agent
PurposeGround answers in external knowledgePersonalize & maintain continuity
📎
Reuse the deep dive

Chunking, embedding models, vector DBs, similarity metrics and retrieval scoring are covered thoroughly in RAG, End-to-End →. Memory uses the same toolkit — you don't need to relearn it, just point it at per-user facts instead of a document corpus.

8 · The memory tool — letting the model manage its own notes

A newer pattern gives the model a memory tool: a set of file-like operations (view, create, edit, delete) over a dedicated memory directory. The model decides when to jot something down and when to read it back, and your application implements the storage backend.

How it works

You declare a memory tool; the model emits tool calls like "write /memories/preferences.md"; your handler persists the file. On later sessions the model reads its own notes back.

Why it's powerful

The model curates its own long-term memory in natural language — no separate extraction pipeline. Especially useful for long-horizon agents that learn as they work.

🔒
Security — two hard rules

(1) Never let the model store secrets or full PII in memory files — and be careful with personal data under GDPR/CCPA (see Guardrails §5). (2) Confine every file operation to the memory directory: validate model-supplied paths and reject traversal (.., symlinks, absolute paths) — the path comes from the model and is untrusted input. In multi-user systems, give each user their own memory directory with access control.

9 · Memory in agents — the four CoALA types

Everything so far has been a single technique — a buffer, a vector store, a memory file. An agent that runs many steps over many sessions needs several of them at once, each doing a different job. The standard way to organize them is the CoALA framework (Cognitive Architectures for Language Agents), which names four foundational memory types. They're worth learning by name — they're how practitioners talk about agent memory, and each maps onto a technique you've already seen.

🧠
The big idea

These four memory systems are what separate a standard chatbot from a real agent: they let responses be shaped by persistent knowledge and accumulated experience, not just the immediate conversation.

9.1 Working memory — the scratch pad

The agent's context window (§2) — its "RAM." Fast and immediately accessible, holding the current task's transcript and intermediate results. But it is volatile (erased when the session ends) and limited in size, so it's managed with the short-term patterns from §5 (windowing, summarizing/compaction).

  • Lifespan/origin: short-term · runtime.
  • Implemented by: the running message list → context window (§5).
  • Analogy: RAM — what you're actively holding in mind right now.

9.2 Semantic memory — the knowledge base

The agent's store of facts, rules and conventions — durable knowledge that's true regardless of the current task. Crucially, this is often implemented as simple Markdown files — CLAUDE.md is the canonical example. Giving the agent persistent semantic memory is how it stops repeating the same mistakes: write the convention down once, and every future session reads it.

  • Lifespan/origin: long-term · usually static/authored, but can be runtime (a vector store of learned facts, §6).
  • Implemented by: Markdown knowledge files (CLAUDE.md), or a vector memory store (§6) for facts the agent learns on its own.
  • Analogy: a reference manual the agent always has open.

9.3 Procedural memory — how to do things (skills)

Where semantic memory is what the agent knows, procedural memory is how it knows to do things — its defined skills: reusable, step-by-step procedures for specific tasks. The key engineering technique here is progressive disclosure: the agent sees only a lightweight index of available skills by default, and loads the full step-by-step instructions for a skill only when a task actually needs it. This keeps the context window small while giving access to a large library of procedures.

📂
Progressive disclosure, concretely

Each skill is a folder with a short description the agent always sees, plus a full instructions file it reads on demand. So the agent "knows" it has 50 skills available (50 one-line descriptions in context), but only pays the token cost of the one or two it's actually using right now. This is exactly how Claude Code's Skills and the agent-tool ecosystems work.

  • Lifespan/origin: long-term · usually static/authored (you write the skills).
  • Implemented by: a skills directory with an index + per-skill instruction files, loaded on demand.
  • Analogy: a shelf of how-to guides — you grab the right one only when the job calls for it.

9.4 Episodic memory — distilled experience

The agent's record of past interactions and decisions — but, done well, not raw transcripts. Instead the agent compresses experiences into useful notes ("last time this deploy failed because of X; the fix was Y"), so it can learn and improve over time. This is the type that turns a stateless tool into something that gets better with use.

⚠️
The hardest type to engineer

Episodic memory is the trickiest of the four, because the system must also know when to forget. An episodic store that only ever grows fills with obsolete, contradictory notes and degrades recall. Deciding what to distill, what to keep, and what to expire is exactly the job of the memory agent in §9.5.

  • Lifespan/origin: long-term · runtime (the agent writes it as it works).
  • Implemented by: distilled notes in a vector store or memory files (§6, §8), curated by a memory agent.
  • Analogy: a lessons-learned journal, not a security-camera tape.

9.5 How the four map onto this page

WorkingScratch pad / "RAM"Runtime · short-termContext window + short-term patterns (§5)

🎚️
Not every agent needs all four

Match the memory to the job. A simple reflex agent (answer the current question, no continuity) needs only working memory. An advanced coding agent needs all four: working memory for the current task, semantic memory (CLAUDE.md) for project conventions, procedural memory (skills) for repeatable workflows, and episodic memory to learn from past runs. Add types as the agent's job demands them — not before.

🔗
The agent-loop view

Over a long run an agent continually decides what to keep in working memory (compact), what to write to semantic/episodic memory for later, which procedural skill to load, and what to recall when relevant. Good memory management is what keeps an agent coherent across an overnight run. See 08 · Agentic AI → and 12 · LangGraph → for the orchestration side.

9.6 The "memory agent" pattern

So far the main agent manages its own memory inline. As systems grow, that becomes a distraction — every turn spent deciding "should I remember this?" is a turn not spent on the task, and naïve writing fills the store with duplicates and stale facts. The fix is to split memory management into its own role: a memory agent.

A memory agent is a dedicated sub-agent (or background job) whose only job is curating the memory store on behalf of the main agent. The main agent just does the work and emits raw events; the memory agent — usually running asynchronously, off the critical path — turns those into clean, durable memories.

Main agent does the task · emits raw events
Memory agent (async, background)
↳ extract durable facts · consolidate & dedupe · resolve conflicts · prune/expire stale
Memory store small, high-signal, current
The memory agent sits between the main agent and the store, keeping memory clean so recall stays accurate.

What a memory agent typically does:

  • Extract — read the conversation/task and pull out only memory-worthy facts (preferences, decisions, identities, corrections), discarding chatter.
  • Consolidate & dedupe — merge new facts with existing ones instead of appending near-duplicates ("prefers metric" written ten times becomes one memory).
  • Resolve conflicts — when a new fact contradicts an old one, update or supersede it (today's decision overrides last week's) — the same conflict logic RAG uses for documents, see RAG §19.
  • Prune & expire — drop stale or low-value memories so the store stays small and recall stays sharp.
python · a minimal memory agent (run after each session)
import json
from openai import OpenAI
oai = OpenAI()

def memory_agent(conversation: str, existing: list[str]) -> list[str]:
    """Curate long-term memories from a finished conversation."""
    prompt = (
        "You manage an agent's long-term memory. Given the conversation and the "
        "existing memories, return the UPDATED memory list as JSON "
        '{"memories": ["...", ...]}. Rules:\n'
        "- Keep only durable, reusable facts (preferences, decisions, identities).\n"
        "- Merge duplicates; do not repeat an existing memory.\n"
        "- If a new fact contradicts an old one, replace the old one.\n"
        "- Drop anything stale or trivial. Keep the list small and high-signal."
    )
    r = oai.chat.completions.create(
        model="gpt-4o-mini", temperature=0,
        response_format={"type": "json_object"},
        messages=[{"role": "system", "content": prompt},
                  {"role": "user",
                   "content": f"EXISTING MEMORIES:\n{existing}\n\nCONVERSATION:\n{conversation}"}])
    return json.loads(r.choices[0].message.content)["memories"]

# after a session ends (off the user's critical path):
memories = memory_agent(transcript, memories)   # → curated, deduped, current
🧩
You usually don't build this from scratch

Dedicated memory-layer tools for agents implement this pattern for you — extraction, consolidation, conflict resolution, decay and recall — behind a simple add() / search() API. Common ones: Mem0, Letta (formerly MemGPT, the self-editing-memory approach), Zep, and LangMem (LangGraph's memory primitives). Reach for one when hand-rolled vector memory (§6) starts accumulating duplicates and stale facts.

11 · Best practices & pitfalls

  • Always budget the context window — system prompt + memory + retrieval + question + answer must all fit, with headroom.
  • Store durable facts, not chatter — high-signal memories keep recall accurate and the store small.
  • Summarize before you overflow, not after — don't wait for a hard context-limit error.
  • Scope memory per user and enforce access control — one user's memory must never leak into another's context.
  • Never persist secrets or full PII in memory (buffers, summaries, vector stores or memory files) — mask first.
  • Validate memory-tool paths — treat model-supplied file paths as untrusted; confine to the memory directory.
  • Decay or expire stale memories — old preferences and outdated facts should age out or be overwritten. This is the hard part of episodic memory: knowing when to forget.
  • Prefer static for what's always true — durable facts and conventions belong in semantic memory (a CLAUDE.md-style file), not in runtime memory the agent has to re-learn and re-recall.
  • Use progressive disclosure for skills — keep procedural memory as a lightweight index and load full instructions on demand, so a big skill library doesn't bloat the context window.
  • Don't over-provision — add CoALA memory types only as the agent's job demands them (a reflex agent needs just working memory). Each extra store is more to curate and more to get wrong.
  • Make memory observable — log what was recalled and written (see Langfuse →) so you can debug "why did it forget / why did it recall that?".
