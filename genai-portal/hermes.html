Hermes (Nous Research)

Hermes is a family of open-weight, instruction-tuned LLMs from Nous Research, fine-tuned on top of open base models (mostly Llama, also Mistral/Qwen). Their claim to fame for builders: strong, structured function-calling and steerability, in a model you can download and run on your own hardware — making Hermes a popular "brain" for local, private, free agents.

⏱ ~20 min read Beginner → intermediate Open weights · Ollama · local
🔗
Where this fits

Hermes is one concrete open model you can drop into the patterns taught elsewhere in this hub. Running models locally is covered in 03 · Local LLMs & Ollama →; using a model as an agent brain is the Claude Agent and LangGraph tabs (Hermes slots in where Claude/GPT would). Where those use a hosted model, Hermes is the local, no-API-key alternative.

1 · What Hermes is

Hermes models are fine-tunes: Nous Research takes a strong open base model and post-trains it for instruction-following, conversation, and — crucially — reliable tool/function calling and adherence to a system prompt.

🔓

Open weights

Downloadable and self-hostable. No API key, no per-token cost, your data never leaves your machine.

🛠️

Function-calling first

Trained to emit structured tool calls reliably — the capability that makes a model usable as an agent.

🎛️

Steerable

Strong system-prompt adherence and "neutral" alignment — it follows your instructions rather than refusing freely.

🔑
The one-line summary

Hermes = an open, locally-runnable model tuned for tool calling. When you want an agent's brain to be free, private and offline, Hermes is one of the go-to choices.

📦
Sizes & quantization

Smaller variants (e.g. 8B) run on a single consumer GPU or a modern laptop, especially quantized (compressed to 4-bit, etc.). Bigger variants (70B+) need more VRAM but reason better. The quantization/VRAM trade-off is exactly the topic of 03 · Local LLMs & Ollama.

3 · Running Hermes locally with Ollama

The fastest way to try Hermes is Ollama, which downloads and serves the model with one command and exposes an OpenAI-compatible API.

bash
# pull a Hermes model (name depends on the tag available in the registry)
ollama pull hermes3

# chat with it directly
ollama run hermes3 "Explain function calling in one sentence."

# or serve the OpenAI-compatible endpoint (default: http://localhost:11434/v1)
ollama serve

Because Ollama speaks the OpenAI API, you can talk to Hermes with the standard OpenAI client — just point base_url at Ollama and use any key string:

python
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")  # key unused locally

resp = client.chat.completions.create(
    model="hermes3",
    messages=[{"role": "user", "content": "Give me three uses for a vector database."}],
)
print(resp.choices[0].message.content)

4 · Function calling — Hermes's signature feature

What makes Hermes useful for agents is reliable function calling: given a list of tools, it emits a structured call you can parse and execute. Two ways to use it:

4.1 Via the OpenAI tools API (easiest)

Through Ollama's OpenAI-compatible endpoint, you can pass tools just like you would to GPT, and read back tool_calls. The agent loop is then identical to any other OpenAI-style model.

python
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "parameters": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"],
        },
    },
}]

resp = client.chat.completions.create(
    model="hermes3",
    messages=[{"role": "user", "content": "What's the weather in Paris?"}],
    tools=tools,
)
call = resp.choices[0].message.tool_calls[0]
print(call.function.name, call.function.arguments)   # → get_weather {"city": "Paris"}

4.2 The native Hermes tool-call format

Under the hood, Hermes was trained to express tool calls with explicit tags in its output — typically a system prompt listing the tools, and the model replying inside <tool_call>…</tool_call> tags containing JSON. Knowing this helps when you use Hermes without the OpenAI wrapper (e.g. raw transformers):

hermes tool-call style (illustrative)
System: You are a function-calling AI. Available tools:
<tools>[{"name":"get_weather","arguments":{"city":"string"}}]</tools>
Respond with a tool call as: <tool_call>{"name":..,"arguments":..}</tool_call>

User: What's the weather in Paris?

Assistant: <tool_call>{"name": "get_weather", "arguments": {"city": "Paris"}}</tool_call>

Your code parses the <tool_call> JSON, runs the tool, and feeds the result back (often inside a <tool_response> tag) for the next turn — the same loop, just with tag-parsing instead of an SDK field.

🧭
Which to use

Prefer §4.1 (OpenAI tools API via Ollama) — it hides the tag format and lets you reuse the exact agent loop from the Claude Agent tab and frameworks like LangGraph. Use the native format (§4.2) only when you run the raw weights yourself and need full control over the prompt template.

5 · Hermes as an agent brain

Once Hermes can call functions, it slots into the agent loop from the Claude Agent tab — the only change is the model and the client. Here's the loop with Hermes via Ollama:

local_agent.py
import json
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

tools = [{
    "type": "function",
    "function": {
        "name": "multiply",
        "description": "Multiply two numbers.",
        "parameters": {"type": "object",
            "properties": {"a": {"type": "number"}, "b": {"type": "number"}},
            "required": ["a", "b"]},
    },
}]

def run_tool(name, args):
    if name == "multiply":
        return str(args["a"] * args["b"])
    return "unknown tool"

messages = [{"role": "user", "content": "What is 23 * 47?"}]

while True:
    resp = client.chat.completions.create(model="hermes3",
                                          messages=messages, tools=tools)
    msg = resp.choices[0].message
    if not msg.tool_calls:                      # Hermes answered → done
        break
    messages.append(msg)                        # keep the assistant turn
    for call in msg.tool_calls:                 # run each requested tool
        args = json.loads(call.function.arguments)
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": run_tool(call.function.name, args),
        })

print(messages[-1].content if messages[-1].get("role") == "tool" else msg.content)
🔁
Same loop, different brain

This is the identical gather → act → observe loop — only the model string (hermes3) and the local base_url differ. That's the whole point: agent architecture is model-agnostic, so you can swap a hosted model for a local Hermes when privacy or cost demands it.

6 · When to choose Hermes (vs a hosted model)

Choose Hermes (local) when…Choose a hosted model (Claude/GPT) when…
Data must stay on-prem / privateYou want the strongest reasoning available
You want zero per-token cost at scaleYou don't want to manage GPUs / infra
You need to run fully offlineYou need very long context or the latest capabilities
You want to fine-tune / fully control the modelYou want managed safety, tools and SLAs
⚖️
Honest trade-off

An 8B–70B Hermes won't match a frontier hosted model on the hardest reasoning or longest-horizon agentic tasks. But for scoped, well-defined tool-using agents — especially where privacy, cost or offline operation matter — it's an excellent, capable choice. Many teams use both: Hermes for routine/private steps, a frontier model for the hard ones.

7 · Best practices & pitfalls

  • Use the OpenAI-compatible endpoint (via Ollama) so your agent code is portable across Hermes, GPT and others — don't hand-parse tags unless you must.
  • Match model size to hardware — start with a quantized 8B to validate the workflow, scale up only if reasoning quality requires it (see Local LLMs).
  • Validate tool-call JSON — local models can occasionally emit malformed arguments; parse defensively and retry on failure.
  • Keep the tool set small & descriptions sharp — smaller models choose tools less reliably than frontier models, so clear, few tools help a lot.
  • Add your own guardrails — open models have lighter built-in safety, so the Guardrails tab rules (scope, secrets, PII, injection) matter more, not less.
  • Cap iterations — as with any agent loop, bound the steps so a confused local model can't spin forever.
  • Trace it — point Langfuse at the OpenAI-compatible calls to see latency, tokens and tool usage, just like a hosted model.
