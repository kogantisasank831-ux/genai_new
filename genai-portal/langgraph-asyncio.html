AsyncIO for LangGraph

Learn the Python concurrency concepts that make production LangGraph systems responsive and safe. This lab moves from the event-loop mental model to async nodes, structured concurrency, cancellation, timeouts, backpressure, streaming, debugging, and interview practice.

🎯
Interview outcome

After this lab you should be able to explain the event loop, write an async LangGraph node, choose between sequential awaits, gather, and TaskGroup, enforce timeouts and concurrency limits, and diagnose why “async code” can still block a production service.

1 · Why AsyncIO matters in LangGraph

Agent graphs spend most of their time waiting: model APIs, vector databases, HTTP tools, SQL drivers, object storage, and checkpoints. AsyncIO lets one Python process make progress on other work while one operation is waiting for I/O. It improves concurrency and throughput; it does not make CPU-heavy Python calculations faster.

Good async work

HTTP calls, async database clients, LLM requests, streaming, sockets, queues.

Bad assumption

async def alone gives concurrency. It does not; code must reach an await on non-blocking I/O.

LangGraph connection

Use async nodes plus graph.ainvoke() or graph.astream().

2 · The mental model: coroutine, task, event loop

TermInterview-ready definitionCommon trap
Coroutine functionA function declared with async def.Calling it does not run it; it creates a coroutine object.
Coroutine objectAn awaitable representing suspended work.Forgetting to await it causes “coroutine was never awaited”.
TaskA coroutine scheduled on the event loop so it can progress concurrently.Creating untracked fire-and-forget tasks loses errors and lifecycle control.
Event loopThe scheduler that resumes ready tasks and waits for I/O readiness.Blocking calls freeze every task on that loop.
async_basics.py
import asyncio

async def fetch(name: str, delay: float) -> str:
    await asyncio.sleep(delay)   # yields control to the event loop
    return f"{name} finished"

async def main() -> None:
    # Sequential: roughly 3 seconds total
    first = await fetch("docs", 1)
    second = await fetch("web", 2)
    print(first, second)

asyncio.run(main())
🧠
Core interview distinction

Concurrency means tasks overlap while waiting. Parallelism means work executes simultaneously, often on multiple CPU cores. AsyncIO normally provides single-threaded concurrency, not CPU parallelism.

3 · Sequential await vs gather vs TaskGroup

TechniqueUse it whenFailure behaviour
await a(); await b()B depends on A, or ordering is required.Simple propagation; no overlap.
asyncio.gather(...)You want results in input order and understand its exception semantics.By default the first exception propagates, while other awaitables are not automatically cancelled.
asyncio.TaskGroupRelated child tasks should succeed or fail as one structured unit.On a child failure, remaining children are cancelled; errors are grouped.
structured_concurrency.py
import asyncio

async def retrieve(source: str) -> dict[str, str]:
    await asyncio.sleep(0.2)
    return {"source": source, "text": f"evidence from {source}"}

async def retrieve_all() -> list[dict[str, str]]:
    async with asyncio.TaskGroup() as group:
        tasks = [
            group.create_task(retrieve("docs")),
            group.create_task(retrieve("web")),
            group.create_task(retrieve("database")),
        ]
    return [task.result() for task in tasks]

asyncio.run(retrieve_all())

Google-style follow-up: Why prefer structured concurrency? Because child tasks have an explicit parent scope, errors are not silently orphaned, and cleanup/cancellation becomes predictable.

graph = builder.compile()

async for event in graph.astream(
    {"question": "Explain backpressure"},
    stream_mode="updates",
):

✅
Design boundary

Use LangGraph edges for workflow-level parallel branches. Use AsyncIO inside a node when that node itself must perform several independent I/O calls. Do not create an extra concurrency layer without a reason; it complicates limits, retries, tracing, and cancellation.

5 · Production patterns: timeout, cancellation, limits

bounded_async_node.py
import asyncio
from typing_extensions import TypedDict

class State(TypedDict):
    urls: list[str]
    documents: list[str]

MAX_IN_FLIGHT = 8
semaphore = asyncio.Semaphore(MAX_IN_FLIGHT)

async def fetch_document(client, url: str) -> str:
    async with semaphore:                 # backpressure
        async with asyncio.timeout(3.0):  # end-to-end deadline
            response = await client.get(url)
            response.raise_for_status()
            return response.text

async def load_documents(state: State) -> dict[str, list[str]]:
    try:
        async with asyncio.TaskGroup() as group:
            tasks = [group.create_task(fetch_document(client, url))
                     for url in state["urls"]]
        return {"documents": [task.result() for task in tasks]}
    except asyncio.CancelledError:
        # cleanup belongs in finally/async context managers; do not swallow cancellation
        raise

Timeout

Bounds latency and resource usage. Decide whether the timeout applies per request, per node, or to the whole graph run.

Semaphore

Caps in-flight work so a large state cannot create thousands of sockets or trigger provider rate limits.

Cancellation

Cooperative. Await points receive cancellation; cleanup resources and re-raise CancelledError.

Retry

Retry only transient, idempotent operations with bounded attempts, jitter, and a total deadline.

6 · Blocking code inside async applications

blocking_fix.py
import asyncio
import requests

# WRONG: requests.get blocks the event-loop thread.
async def wrong(url: str) -> str:
    return requests.get(url, timeout=5).text

# Transitional fix for blocking I/O libraries.
async def acceptable_bridge(url: str) -> str:
    response = await asyncio.to_thread(requests.get, url, timeout=5)
    return response.text

# Preferred: use a native async HTTP client.
async def preferred(client, url: str) -> str:
    response = await client.get(url)
    return response.text

For CPU-bound parsing, embeddings implemented in Python, or heavy numerical work, to_thread may keep the loop responsive but does not guarantee CPU scaling because of the GIL. Use vectorized/native code, a process pool, or a separate worker service when CPU parallelism is required.

7 · Debugging checklist

SymptomLikely causeFix
“coroutine was never awaited”Called an async function without awaiting/scheduling it.await fn() or create a managed task.
Async service freezes during one callBlocking library or CPU loop on the event-loop thread.Native async client, to_thread, process/worker.
Provider returns 429 under loadUnbounded fan-out.Semaphore, queue, backoff, provider-aware rate limiter.
Tasks continue after request disconnectCancellation swallowed or detached tasks.Structured concurrency and cancellation propagation.
Race-like state corruptionShared mutable object changed by concurrent tasks.Return isolated results and merge deterministically via reducers.

8 · Interview questions and strong answer anchors

Q What happens when an async function is called? ›
It returns a coroutine object. The body runs only when the coroutine is awaited or scheduled as a task by an event loop.
Q Does AsyncIO use multiple threads? ›
The normal event-loop model is one thread executing one Python task at a time, switching at await points. Threads can be introduced deliberately, for example with asyncio.to_thread.
Q Why can a single blocking call be disastrous? ›
Because the event loop cannot schedule any other task while the loop thread is blocked, so the latency of unrelated requests rises too.
Q TaskGroup vs gather? ›
Both run awaitables concurrently. TaskGroup gives structured lifecycle and cancels sibling tasks when one fails, making it the safer default for related child work. Gather is useful when its ordering and exception behaviour are exactly what you need.
Q Where should concurrency limits live in a LangGraph system? ›
At every scarce boundary: graph/service admission, per-provider/API semaphore or rate limiter, database pool, and worker queue. The smallest local semaphore alone is not a complete capacity strategy.
Q How would you make an async tool call reliable? ›
Use a total deadline, bounded retries for transient errors, idempotency where side effects exist, cancellation propagation, concurrency/rate limits, structured logs/traces, and explicit error mapping into graph state.

9 · Knowledge check

1 · Which workload benefits most directly from AsyncIO?
AsyncIO is strongest for I/O-bound concurrency. CPU-heavy work needs a different execution strategy.
2 · What is the safest default for a set of related child tasks that should fail together?
TaskGroup provides structured concurrency and manages sibling cancellation on failure.
3 · An async LangGraph node calls requests.get(). What is the main risk?
Synchronous network I/O inside async code blocks the entire event loop.
