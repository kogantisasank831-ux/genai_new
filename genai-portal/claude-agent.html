How a Claude Agent Works

A Claude agent isn't one big prompt — it's a loop. Claude is given a goal and a set of tools, then it repeatedly decides what to do next, your code runs the tool it asked for, the result goes back to Claude, and it decides again — until the task is done. This page walks through how that loop runs, the tools that power it, and how the Claude Agent SDK manages it for you.

1 · What "a Claude agent" means

A plain Claude API call is one-shot: you send messages, you get one reply. An agent adds three things on top:

🎯

A goal

A task to accomplish, not just a question to answer — "fix this failing test," "research X and write a summary."

🛠️

Tools

Actions Claude can take in the real world — read a file, run a command, search the web, call your API.

🔁

A loop

Claude runs many turns, choosing tools and reacting to results, until the goal is met — not a single response.

🔑
The one-sentence model

A Claude agent = Claude in a loop with tools. The model decides what to do; your code (or the SDK) executes the tool and feeds the result back; repeat until Claude stops asking for tools.

2 · The agent loop — step by step

This loop is the heart of every Claude agent. Anthropic frames it as three repeating phases: gather context → take action → verify work.

1 · Gather contextread files · search · recall memory
2 · Claude decidesthink → emit a tool call (or finish)
3 · Take actionyour code runs the requested tool
4 · Observe / verifytool result returns to Claude
stop_reason?tool_use → loop back · end_turn → done
The agent loop. While Claude keeps asking for tools (stop_reason == "tool_use") it loops back to step 1; when it answers (end_turn) the loop ends. (Text version below.)
┌───────────────────────────────────────────────┐ │ │ ▼ │ 1. GATHER CONTEXT read files, search, recall memory │ │ │ ▼ │ 2. CLAUDE DECIDES think → emit a tool call (or finish) │ │ │ ▼ │ 3. TAKE ACTION your code runs the tool │ │ │ ▼ │ 4. OBSERVE / VERIFY tool result returns to Claude │ │ │ └────────── stop_reason == "tool_use"? ─────────┘ │ no (end_turn) ▼ DONE — final answer
Each turn, Claude either asks for a tool (loop continues) or finishes (loop ends). The stop_reason field tells you which.

Concretely, in one turn:

  1. You send the conversation (system prompt + history + tool definitions) to Claude.
  2. Claude responds. If it wants to act, the response contains a tool_use block and stop_reason == "tool_use".
  3. Your code executes that tool and sends the result back as a tool_result.
  4. Claude sees the result and either calls another tool or answers (stop_reason == "end_turn"), ending the loop.

3 · Tools — how Claude takes action

Tools are the agent's hands. You describe each tool (name, what it does, its input schema); Claude reads those descriptions to decide when to call one. There are two kinds:

Your tools (client-side)

Functions you implement — query a DB, hit your API, edit a file. Claude emits a tool_use; your code runs it and returns the result.

Anthropic tools (server-side)

Built-in tools Claude runs on Anthropic's side — web search, code execution. You just declare them; no execution loop on your end.

python · defining a tool
tools = [{
    "name": "get_weather",
    "description": "Get the current weather for a location.",
    "input_schema": {
        "type": "object",
        "properties": {"location": {"type": "string", "description": "City name"}},
        "required": ["location"],
    },
}]
✍️
Good tool descriptions are most of the work

Claude decides when to call a tool from its description — be prescriptive about when to use it, not just what it does ("Call this when the user asks about current prices or recent events"). On recent Claude models, which reach for tools deliberately, a clear trigger condition in the description noticeably improves how reliably the tool gets used.

4 · The loop in raw code (Claude API)

Before reaching for the SDK, it's worth seeing the loop by hand — it's only a few lines, and it makes clear what the SDK automates. You keep calling the Messages API, executing any tool Claude asks for, until stop_reason is no longer tool_use.

manual_agent.py
import anthropic

client = anthropic.Anthropic()                 # reads ANTHROPIC_API_KEY from env
messages = [{"role": "user", "content": "What's the weather in Paris?"}]

def run_tool(name, args):                       # your real implementations
    if name == "get_weather":
        return f"18°C and clear in {args['location']}"
    return "unknown tool"

while True:
    resp = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=1024,
        thinking={"type": "adaptive"},          # let Claude decide how much to reason
        tools=tools,
        messages=messages,
    )

    if resp.stop_reason != "tool_use":          # Claude is done
        break

    messages.append({"role": "assistant", "content": resp.content})  # keep the turn

    tool_results = []
    for block in resp.content:
        if block.type == "tool_use":
            result = run_tool(block.name, block.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,        # must match the tool_use id
                "content": result,
            })
    messages.append({"role": "user", "content": tool_results})        # feed results back

print(next(b.text for b in resp.content if b.type == "text"))
🧠
Two details that matter

(1) Always append the full resp.content back as the assistant turn — it carries the tool_use blocks Claude needs to see. (2) Each tool_result must carry the matching tool_use_id, and all results for one turn go back in a single user message.

5 · The tool runner — let the SDK drive the loop

Writing that loop by hand gets repetitive. The SDK's tool runner automates it: you define tools as typed functions and the SDK calls the API, runs your tools, feeds results back, and repeats until Claude is done.

python · @beta_tool + tool_runner
import anthropic
from anthropic import beta_tool

client = anthropic.Anthropic()

@beta_tool
def get_weather(location: str) -> str:
    """Get the current weather for a location.

    Args:
        location: City name, e.g. "Paris".
    """
    return f"18°C and clear in {location}"

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[get_weather],
    messages=[{"role": "user", "content": "What's the weather in Paris?"}],
)

for message in runner:        # the SDK runs the whole loop; each step yields a message
    print(message)
⚖️
Tool runner vs manual loop

Use the tool runner for the common case — it handles the loop, schema generation and result-feeding. Drop to the manual loop when you need fine-grained control: custom logging, conditional execution, or a human approval gate before each tool runs (the human-in-the-loop pattern from Guardrails and LangGraph).

6 · Thinking — how Claude plans inside the loop

Modern Claude models can think before acting — an internal reasoning step where the model plans its next move. For agents, you enable adaptive thinking and let Claude decide how much to reason per step.

python
resp = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=4096,
    thinking={"type": "adaptive"},          # Claude decides when/how much to think
    output_config={"effort": "high"},       # how hard to work: low | medium | high | xhigh | max
    tools=tools,
    messages=messages,
)
🎚️
effort is the main dial for agents

effort controls how thoroughly Claude reasons and acts. For agentic/coding work, high (or xhigh) is the sweet spot; drop to medium/low for routine, latency-sensitive steps. Higher effort up front often reduces total turns on hard tasks. Pair adaptive thinking with a generous max_tokens so the model has room to plan and act.

7 · Context & sessions — staying coherent over a long run

The Messages API is stateless: you resend the full conversation each turn (see the Memory tab). Over a long agent run the history can outgrow the context window, so Claude agents manage context with a few built-in mechanisms:

MechanismWhat it doesWhen
Prompt cachingCaches a stable prefix (system prompt + tools) so repeated turns are cheap and fastAlways — big win for agents that resend a large fixed prefix
Context editingAutomatically clears stale old tool results from the historyLong tool-heavy runs where old results are no longer needed
CompactionAutomatically summarizes earlier context as you near the window limitVery long conversations that would otherwise overflow
Memory toolLets Claude write notes to a file store it reads back in later sessionsCross-session memory / learning over time
🧩
This is the Memory tab, applied

Prompt caching + context editing + compaction are how a Claude agent keeps its working memory within budget; the memory tool is its long-term memory. The concepts and trade-offs are in the Memory tab — here they're the specific levers a Claude agent uses.

8 · Connecting tools at scale — MCP

Defining every tool inline doesn't scale. MCP (Model Context Protocol) is the standard way to plug a whole catalog of external tools and data sources into a Claude agent — GitHub, Slack, a database, your internal services — without hand-writing each schema. The agent connects to an MCP server, which advertises its tools, and Claude can call them like any other tool.

🔌
Goes deeper

MCP has its own treatment in 09 · Model Context Protocol → and the RAG · MCP · Agents · LLMs tab. For a Claude agent, the takeaway is: MCP is how you give it tools you didn't write yourself, in a standard, swappable way.

9 · The Claude Agent SDK — what it gives you

You can build an agent on the raw API (you saw the loop in §4). The Claude Agent SDK packages the patterns this page describes so you don't reassemble them every time:

  • The loop, managed — the tool runner drives gather → act → verify for you.
  • Tools made easy — typed functions become tool schemas automatically; built-in Anthropic tools (web search, code execution, file/bash, memory) are one declaration.
  • Context management — prompt caching, context editing and compaction wired in.
  • MCP support — connect external tool servers.
  • Subagents — spin off a focused sub-agent (often a cheaper/faster model) for a sub-task without losing the main thread.
🏗️
When to use what

Raw API + manual loop → maximum control, custom harness. Agent SDK / tool runner → build a custom agent fast on your own infrastructure. Managed Agents (Anthropic-hosted) → when you want Anthropic to run the loop and host the sandbox where tools execute. Start simple; reach for more structure only when the task needs it.

10 · A worked example — a two-tool agent

An agent with two tools that loops until the task is done: look up a value, then do arithmetic on it. Uses the manual loop so every step is visible.

two_tool_agent.py
import anthropic
client = anthropic.Anthropic()

tools = [
    {"name": "get_price", "description": "Get the unit price of a product in USD.",
     "input_schema": {"type": "object",
        "properties": {"sku": {"type": "string"}}, "required": ["sku"]}},
    {"name": "multiply", "description": "Multiply two numbers.",
     "input_schema": {"type": "object",
        "properties": {"a": {"type": "number"}, "b": {"type": "number"}},
        "required": ["a", "b"]}},
]

PRICES = {"PUMP-A": 1250.0}
def run_tool(name, args):
    if name == "get_price":  return str(PRICES.get(args["sku"], 0))
    if name == "multiply":   return str(args["a"] * args["b"])
    return "unknown tool"

messages = [{"role": "user",
             "content": "What's the total cost of 8 units of PUMP-A?"}]

while True:
    resp = client.messages.create(
        model="claude-opus-4-8", max_tokens=1024,
        thinking={"type": "adaptive"}, tools=tools, messages=messages)
    if resp.stop_reason != "tool_use":
        break
    messages.append({"role": "assistant", "content": resp.content})
    results = [{"type": "tool_result", "tool_use_id": b.id,
                "content": run_tool(b.name, b.input)}
               for b in resp.content if b.type == "tool_use"]
    messages.append({"role": "user", "content": results})

print(next(b.text for b in resp.content if b.type == "text"))
# → "8 units of PUMP-A at $1,250 each is $10,000."
▶️
The trajectory

Claude calls get_price("PUMP-A") → gets 1250 → calls multiply(8, 1250) → gets 10000 → answers, no more tool calls (end_turn), loop ends. Two tools, two turns of the loop, automatic — that's an agent.

11 · Safety & control

An agent that can act needs guardrails on those actions — this is not optional once tools can change real state.

  • Gate risky tools — require human approval before a tool that deletes, sends, or pays. In the manual loop you simply pause before run_tool and ask; with Managed Agents this is the always_ask permission policy.
  • Refusals — Claude may decline an unsafe request (stop_reason == "refusal"). Check for it before reading the answer, and handle it gracefully.
  • Cap the loop — set a max-iterations limit so a confused agent can't loop forever.
  • Don't leak secrets / enforce scope — the full rulebook is the Guardrails tab; it applies directly to agent tools and outputs.
  • Trace it — log every step, tool call and cost with Langfuse so you can see why the agent did what it did.

12 · Best practices & pitfalls

  • Start simple — a single API call or a short workflow beats an agent when the task is well-defined. Reach for the loop only when steps can't be specified up front.
  • Write trigger-condition tool descriptions — tell Claude when to call each tool, not just what it does.
  • Keep the tool set focused — too many tools dilutes tool selection; use MCP + tool search if you have a large catalog.
  • Append the full assistant turn and match every tool_use_id — the two most common manual-loop bugs.
  • Use adaptive thinking + high effort for hard agentic work; give it room with a generous max_tokens.
  • Cache the stable prefix (system prompt + tools) — agents resend it every turn, so caching is a big cost/latency win.
  • Always cap iterations and gate destructive actions — an autonomous loop with real tools must fail safe.
  • Give a clear goal up front — well-specified tasks in the first turn let Claude work more autonomously and efficiently than dribbling context across many turns.
