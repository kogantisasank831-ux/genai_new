Security, Prompt Injection & Responsible AI

Security answers should rely on identity, policy and validation—not on asking the model to behave.

10 high-signal questionsEssential · every production roleAnswer aloud, then open
No prompt can be the security boundary. Identity, data authorization, tool policy and output validation must surround the model.
🎯
How to practise this topic

Use the phrase “the model is not the security boundary,” then explain which deterministic control enforces each rule.

10 questions
01What is prompt injection?prompt-injectionsecurityowasp＋
Strong answer

Prompt injection occurs when untrusted text attempts to change the model’s instructions or make it reveal data or misuse tools. It can come directly from a user or indirectly from retrieved webpages, documents or emails. Because the model cannot perfectly distinguish instructions from data, the solution is defense in depth rather than a single detection prompt.

30-second version

Prompt injection is untrusted content trying to control model behavior. Treat all external text as data and limit what the model can access or do.

Likely follow-up: What is indirect prompt injection?
02How do you defend against prompt injection in a chatbot?prompt-injectiondefense-in-depthtools＋
Strong answer

Authenticate the user, authorize every retrieved document and tool call server-side, separate and label untrusted content, minimize available context, use least-privilege tools, validate arguments and outputs, and require confirmation for consequential actions. Add detectors and red-team tests, but assume some attacks will bypass them and make the blast radius small.

  • Never place secrets in prompts.
  • Do not let retrieved text define tool policy.
  • Audit sensitive retrieval and actions.
30-second version

Use identity, retrieval ACLs, least-privilege tools, validation and human approval. Detection is helpful, but containment is the real control.

Likely follow-up: Why is a stronger system prompt not enough?
03How do you implement RBAC in a RAG system?ragrbacmulti-tenancy＋
Strong answer

Authorization must be enforced before retrieval or at query time using tenant and document ACL filters derived from the authenticated principal. The model should receive only authorized chunks. I also protect caches and indexes with the same scope, test cross-tenant queries and log document IDs returned to each request.

30-second version

Filter by user and tenant before evidence reaches the model. Asking the model not to mention unauthorized data is not access control.

Likely follow-up: What if the vector database has weak filtering support?
04How do you prevent SQL injection in a normal Python API?sql-injectiondatabasepython＋
Strong answer

Never concatenate user input into SQL. Use parameterized queries or an ORM, validate identifiers against allowlists, use least-privilege database accounts, set statement timeouts and log suspicious patterns. Input escaping alone is brittle because the database driver should handle value binding.

query.py
from sqlalchemy import select

ALLOWED_SORT = {"name": Candidate.name, "created": Candidate.created_at}

def build_query(tenant_id: str, sort: str):
    column = ALLOWED_SORT.get(sort, Candidate.created_at)
    return (select(Candidate)
            .where(Candidate.tenant_id == tenant_id)
            .order_by(column.desc()))
30-second version

Parameterized queries plus least privilege are the core controls. User input never becomes executable SQL text.

Likely follow-up: How do you safely support dynamic sort columns?
05How do you secure an LLM-based text-to-SQL feature?text-to-sqlsql-injectionguardrails＋
Strong answer

I avoid executing arbitrary generated SQL. The model should produce a constrained query plan or select from known metrics, then deterministic code compiles parameterized SQL against allowlisted read-only views. I enforce tenant filters, row and time limits, block DDL/DML, estimate cost, validate the AST and audit every query.

30-second version

Generate a constrained plan, compile it safely, run through a read-only account and enforce tenant, row and time limits.

Likely follow-up: Would you ever let the model query production directly?
06How do you handle PII in a GenAI application?piiprivacyobservability＋
Strong answer

Collect only what the task needs, classify fields, redact or tokenize before external model calls where possible, encrypt in transit and at rest, restrict access by purpose and tenant, and define retention and deletion. Traces need special care because prompts and outputs can duplicate PII into observability systems.

30-second version

Minimize, redact, encrypt, authorize and expire PII. Apply the same rules to prompts, caches and traces.

Likely follow-up: How would you debug without storing raw sensitive prompts?
07Can the system prompt or hidden instructions be kept secret?system-promptsecretssecurity＋
Strong answer

They should not be treated as secrets because users may infer or extract them. I avoid placing credentials, private data or security decisions in prompts. The application enforces authorization and tool policy outside the model, and disclosure of prompt wording should not create a security incident.

30-second version

Assume prompts can leak. Keep secrets and enforcement in normal security controls, not hidden text.

Likely follow-up: How do you protect API keys used by tools?
08How do you secure tool use by an agent?toolsauthorizationagent＋
Strong answer

The model proposes a typed action; a policy layer authenticates the caller, checks role, tenant, resource, parameters and risk, then executes with a narrowly scoped credential. Tools have timeouts, quotas and idempotency, return minimal data, and high-impact actions require preview or approval.

30-second version

The model never owns credentials or authorization. A server-side policy gate approves each typed, least-privilege tool call.

Likely follow-up: What is a confused-deputy risk in an agent system?
09How would you address bias in an AI-assisted hiring workflow?responsible-aibiashiring＋
Strong answer

I keep the model advisory, remove protected attributes and proxies where legally appropriate, evaluate error and recommendation rates across relevant groups, document data and prompt limitations, and require human review with evidence. I preserve original inputs and decisions for audit and provide an appeal or correction path.

30-second version

Use the model as bounded decision support, measure disparate impact, show evidence, keep humans accountable and maintain an audit trail.

Likely follow-up: Can removing protected attributes eliminate bias?
10What does responsible AI mean in engineering terms?responsible-aigovernancenist＋
Strong answer

It becomes concrete requirements: intended use and prohibited use, risk classification, data governance, evaluation thresholds, human oversight, transparency, monitoring, incident response and decommissioning. Controls should be proportional to impact and reviewed when the model, data or usage changes.

30-second version

Responsible AI is a lifecycle of documented purpose, measured risk, human control, monitoring and response—not a disclaimer.

Likely follow-up: Who owns the risk after launch?
