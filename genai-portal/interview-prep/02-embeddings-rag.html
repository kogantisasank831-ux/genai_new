Embeddings, Vector Search & RAG

Interviewers increasingly test whether you can diagnose retrieval quality—not merely describe a vector database.

10 high-signal questionsHigh frequency · engineering rolesAnswer aloud, then open
A strong RAG answer is a pipeline, not a single prompt. Interviewers usually probe each stage and its failure modes.
🎯
How to practise this topic

Draw the retrieval pipeline while answering. For every component, name the metric you would inspect and one failure it can introduce.

10 questions
01What is an embedding?embeddingevaluation＋
Strong answer

An embedding is a dense numerical representation learned so that items with related meaning are near one another in vector space. We use it for semantic retrieval, clustering, deduplication and recommendation. The embedding model, input formatting and domain language must match the task, and quality should be tested on real query-document pairs.

30-second version

An embedding converts meaning into coordinates so related items can be found by distance. Its usefulness is task- and domain-dependent.

Likely follow-up: How would you evaluate two embedding models for an enterprise corpus?
02Cosine similarity, dot product or Euclidean distance—which should you use?similarityvector-search＋
Strong answer

The correct metric is the one the embedding model was trained for. Cosine compares direction and ignores magnitude; dot product retains magnitude and is efficient, especially with normalized vectors; Euclidean measures geometric distance. I follow model guidance and verify ranking quality empirically rather than choosing by habit.

30-second version

Match the index metric to the embedding model, then verify recall and ranking on representative queries.

Likely follow-up: What changes if vectors are normalized?
03How do you choose a chunking strategy?chunkingdocumentstrade-off＋
Strong answer

I start from how users ask questions and how much context is needed to answer them. I preserve semantic boundaries such as headings, clauses or table rows, include useful metadata, and use controlled overlap only when relationships cross boundaries. I then compare retrieval recall, answer quality and token cost across chunk sizes.

30-second version

Chunk for retrieval units, not arbitrary character counts. Preserve meaning, attach metadata and tune with end-to-end evaluation.

Likely follow-up: How would you chunk tables, code or scanned PDFs?
04When would you choose a dedicated vector database versus PostgreSQL with pgvector?vector-dbpgvectorarchitecture＋
Strong answer

Pgvector is attractive when the corpus and traffic fit PostgreSQL, metadata joins matter, and operational simplicity is valuable. A dedicated vector database can offer easier horizontal scaling, advanced filtering, hybrid features and specialized index operations. I decide from scale, query patterns, consistency requirements and team operations—not benchmark headlines alone.

30-second version

Use pgvector for simplicity and relational integration; choose a dedicated engine when vector scale or search features justify another system.

Likely follow-up: What measurements would trigger a migration?
05How does HNSW work at a high level?hnswannindex＋
Strong answer

HNSW builds a multi-layer proximity graph. Search starts in sparse upper layers to move quickly toward the target region, then explores denser lower layers for accurate neighbors. Parameters trade memory, build time, query latency and recall, so they must be tuned with the real corpus and filters.

30-second version

HNSW is a navigable graph: coarse layers find the region, dense layers refine the nearest neighbors. Faster search costs memory and approximate recall.

Likely follow-up: Compare HNSW with an inverted-file index.
06What is hybrid search and why is it useful?hybrid-searchbm25retrieval＋
Strong answer

Hybrid search combines lexical retrieval, which is strong for exact terms, identifiers and rare names, with dense retrieval, which captures semantic similarity. Scores can be fused using rank-based methods such as reciprocal rank fusion, then reranked. It is especially useful in enterprise data containing acronyms, product codes and natural-language concepts.

30-second version

Combine keyword precision with semantic recall, then fuse and rerank. Enterprise corpora usually need both.

Likely follow-up: How do you combine scores from different retrievers?
07What is reranking, and where does it fit?rerankinglatencyquality＋
Strong answer

A first-stage retriever cheaply returns a broad candidate set. A cross-encoder or specialized reranker then jointly examines the query and each candidate to produce a better order. Because it is more expensive, I apply it to a small top-k and measure whether the quality gain justifies added latency.

30-second version

Retrieve broadly and cheaply; rerank a small candidate set more accurately. It improves precision at a latency cost.

Likely follow-up: How do you choose retrieval k and reranking k?
08A RAG system is returning poor documents. How do you debug it?debuggingretrievalobservability＋
Strong answer

I first separate retrieval failure from generation failure. I inspect the exact query, filters, access scope, chunk text, retrieved ranks and scores; compare lexical and dense results; and check ingestion, OCR, metadata and embedding-version drift. Then I run a labeled retrieval set using recall@k or MRR before changing the prompt.

  • Create a small set of queries with known relevant chunks.
  • Inspect misses by category instead of averaging them away.
  • Version corpus, chunks and embeddings together.
30-second version

Log the query and retrieved evidence, test retrieval independently, and fix ingestion, chunking, filters or ranking before touching generation.

Likely follow-up: What if recall is good but answers are still wrong?
09When is long-context prompting better than RAG?long-contextragtrade-off＋
Strong answer

It can be better for a small number of bounded documents when cross-document reasoning matters, the entire input fits comfortably, freshness is handled elsewhere and latency or cost remains acceptable. RAG is better for large, frequently changing or permissioned corpora. A hybrid can retrieve a document set and then use a larger context for synthesis.

30-second version

Use long context for bounded, coherent material; use RAG for scale, freshness, permissions and targeted evidence.

Likely follow-up: What is the “lost in the middle” concern?
10How do you evaluate a RAG system end to end?evaluationragllm-as-judge＋
Strong answer

I evaluate retrieval separately using recall@k, MRR or nDCG; generation using groundedness, answer correctness, completeness and citation accuracy; and product behavior using task success, latency and cost. I use human-reviewed test cases, deterministic checks where possible and carefully calibrated LLM judges, then monitor real-query slices in production.

30-second version

Measure retrieval, generation and product outcome separately so you know what to fix. Add latency, cost and safety—not only answer quality.

Likely follow-up: How do you prevent an LLM judge from hiding regressions?
