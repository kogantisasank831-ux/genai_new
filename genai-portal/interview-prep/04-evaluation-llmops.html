Evaluation, Observability & LLMOps

A production candidate should explain how quality is measured before release and diagnosed after release.

10 high-signal questionsHigh signal · production rolesAnswer aloud, then open
LLMOps is a feedback loop: test before release, observe after release, then turn failures into new evaluation cases.
🎯
How to practise this topic

Answer with a measurement hierarchy: deterministic checks first, domain labels next, calibrated model judges where appropriate, then online product metrics.

10 questions
01What is a golden evaluation dataset?evaluationdatasetregression＋
Strong answer

It is a versioned set of representative inputs with expected outputs, labels, evidence or scoring criteria. It should include common tasks, difficult slices, past incidents and adversarial cases, with provenance and reviewer agreement. I keep a stable holdout for release comparison and continuously add production failures to a separate regression set.

30-second version

A golden set is the repeatable test contract for model behavior. It represents real use, edge cases and known failures, and is versioned like code.

Likely follow-up: How large should it be?
02How do you evaluate open-ended LLM output?evaluationrubricquality＋
Strong answer

I decompose quality into dimensions such as correctness, groundedness, completeness, style and policy compliance. Deterministic validators handle schemas and citations; human rubrics establish trusted labels; semantic or LLM-based graders scale selected dimensions. I report slice-level results and inspect disagreements instead of hiding them in one score.

30-second version

Break “quality” into measurable dimensions and use the cheapest reliable evaluator for each one. Keep human calibration.

Likely follow-up: Why is exact match usually insufficient?
03What are the risks of LLM-as-a-judge?llm-as-judgebiascalibration＋
Strong answer

A judge can prefer verbose answers, share biases with the model being tested, be sensitive to ordering or prompts, and drift when its version changes. I blind or randomize comparisons, calibrate against human labels, use multiple checks for high-stakes decisions and version the judge prompt and model.

30-second version

LLM judges scale evaluation but are not ground truth. Calibrate, version and monitor them against human decisions.

Likely follow-up: How would you measure judge agreement?
04What should be captured in a GenAI trace?observabilitytracingprivacy＋
Strong answer

At minimum: request and trace IDs, user/tenant context without unnecessary PII, model and prompt versions, retrieval queries and document IDs, tool calls, token usage, latency by span, validation outcomes, errors, safety decisions and user feedback. Sensitive content should be redacted or access-controlled with retention limits.

30-second version

A trace should reconstruct what the system saw, chose, called, cost and returned—without turning observability into a data leak.

Likely follow-up: How would you debug a bad answer from a trace?
05How do you version a GenAI application?versioningreleasereproducibility＋
Strong answer

I version the model, decoding parameters, system prompt, examples, output schema, retriever configuration, embedding model, corpus snapshot, tools and policy. A release manifest ties these together so an evaluation result and production trace are reproducible. Components can then be rolled back independently.

30-second version

Version the whole behavior bundle, not only the prompt. A release manifest makes evaluation and rollback meaningful.

Likely follow-up: What happens when an embedding model changes?
06Which online metrics matter?metricsproductionslo＋
Strong answer

I combine task success or resolution, user correction and escalation rates, groundedness or citation acceptance, tool success, safety events, p50/p95 latency, availability and cost per successful task. Metrics must be sliced by language, tenant, task and model route because averages can conceal failures.

30-second version

Track product success, quality, safety, reliability and cost—and slice them by the users and tasks that can fail differently.

Likely follow-up: Which metric would you put on an executive dashboard?
07How do you test a GenAI pipeline in CI?testingcievaluation＋
Strong answer

Unit-test prompt builders, chunkers, policies and tool adapters without a live model. Add contract tests for schemas, mocked provider failures and deterministic fixtures. Run a small stable eval set on every change and a larger gated suite before release, with thresholds and a human review path for ambiguous regressions.

30-second version

Use normal software tests around the model and evaluation gates for model behavior. CI should catch both code and quality regressions.

Likely follow-up: How do you keep CI cost and flakiness under control?
08Users report hallucinations after a release. What do you inspect first?incidentdebuggingobservability＋
Strong answer

I compare release manifests and traces: model or prompt change, retrieval result changes, corpus ingestion, filters, tool errors, truncation and validation. I reproduce affected queries, determine whether the issue is retrieval or generation, roll back if the impact is material, then add the incident to regression tests.

30-second version

Reproduce from traces, isolate the failing stage, mitigate quickly, then convert the incident into a permanent test.

Likely follow-up: What if only one tenant is affected?
09What should block a GenAI release?release-gaterisktrade-off＋
Strong answer

A release should fail if it violates safety or privacy policies, breaks output contracts, materially regresses critical task slices, exceeds latency or cost budgets, or lacks rollback and traceability. Thresholds should reflect risk: a creative assistant can tolerate different errors than a hiring or financial workflow.

30-second version

Use explicit quality, safety, operational and cost gates proportional to the product risk.

Likely follow-up: How do you handle a quality gain with a large latency regression?
10What does drift mean in a GenAI system?driftmonitoringdata＋
Strong answer

Drift can occur in user queries, source documents, retrieval distributions, tool behavior, model versions or product expectations. I monitor input and retrieval characteristics, task outcomes and evaluator scores by slice. When a shift appears, I sample and relabel data before deciding whether to update prompts, indexes, routing or models.

30-second version

GenAI drift is broader than model drift: users, knowledge, retrieval, tools and requirements can all move. Monitor outcomes and distributions together.

Likely follow-up: How do you distinguish drift from a one-off incident?
