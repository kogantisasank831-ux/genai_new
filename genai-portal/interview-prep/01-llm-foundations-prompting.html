LLM Foundations & Prompting

Explain the model clearly, then connect each concept to a product decision, failure mode or control.

Interview answers become stronger when they connect model behavior to context, controls and measurable product outcomes.

How to practise this topic

Answer each question without jargon first. Then add one concrete production implication, such as latency, reliability, evaluation or cost.

10 questions
01How is generative AI different from traditional predictive machine learning?foundationtrade-off＋
Strong answer

Traditional predictive ML usually maps a fixed feature vector to a bounded label or number. Generative models learn a probability distribution over sequences and produce new text, code, images or audio token by token. In production, that means outputs are open-ended and probabilistic, so we need grounding, validation, evaluation and fallbacks rather than only a threshold.

30-second version

Predictive ML selects from a predefined output space; GenAI creates a sequence. The flexibility is valuable, but it adds non-determinism and requires stronger controls.

Likely follow-up: When would you still prefer a classical ML model over an LLM?
02Explain a transformer to a non-specialist.transformerattention＋
Strong answer

A transformer reads a sequence by letting every token decide which other tokens matter for its current representation. Self-attention creates those relationships in parallel, while feed-forward layers transform each position. Repeating these blocks gives the model a contextual representation that supports next-token prediction.

30-second version

A transformer repeatedly asks: “Which earlier words matter to understanding this word?” Attention builds those links, then the model predicts what should come next.

Likely follow-up: Why did transformers replace recurrent networks for large language models?
03What are tokens, and why do they matter in system design?tokenscostlatency＋
Strong answer

Tokens are the units the model reads and generates; they may be words, word pieces or punctuation. They determine context-window usage, latency and cost. Tokenization also affects multilingual performance and whether long documents need chunking, summarization or retrieval.

30-second version

Tokens are the model’s units of work. More input or output tokens usually means more latency and cost, and the context limit determines how much evidence can be supplied at once.

Likely follow-up: How would you estimate token cost before launch?
04What is a context window? Does a larger one remove the need for RAG?contextragtrade-off＋
Strong answer

The context window is the maximum number of input and output tokens handled in one request. A larger window helps with long conversations and documents, but it does not guarantee that the model will find the right evidence, respect access controls or stay cost-efficient. RAG still provides targeted retrieval, freshness, citations and measurable retrieval quality.

30-second version

A large context window is capacity, not guaranteed attention. RAG remains useful for fresh, permissioned and precisely retrieved evidence.

Likely follow-up: When would you choose long-context prompting instead of RAG?
05What do temperature and top-p control?decodingtemperature＋
Strong answer

They shape sampling from the model’s next-token probability distribution. Lower temperature makes high-probability choices dominate; top-p restricts sampling to the smallest token set whose cumulative probability reaches p. For extraction and tool calls I use low variability, while creative tasks can tolerate more diversity.

30-second version

Both control randomness. Use conservative decoding for structured, factual or repeatable tasks and more diversity only when creativity is the goal.

Likely follow-up: Why can the same prompt still produce different answers at temperature zero?
06Why do LLMs hallucinate?hallucinationgroundingdebugging＋
Strong answer

The training objective rewards plausible continuation, not fact verification. Hallucinations increase when the prompt lacks evidence, the question is outside the model’s knowledge, retrieval fails, or instructions push the model to answer anyway. I reduce them with grounding, abstention, citations, constrained outputs and task-specific evaluation—not by claiming they can be eliminated.

30-second version

The model is optimized to continue text plausibly, not to check truth. Ground it, validate it and allow “I do not know.”

Likely follow-up: How would you debug a sudden increase in hallucinations?
07What makes a production prompt different from an ad-hoc prompt?promptingproductionversioning＋
Strong answer

A production prompt is a versioned interface. It defines the role, task, trusted context, boundaries, output schema, examples and failure behavior. It is paired with tests, telemetry and a rollback path, and it never relies on the prompt itself as the authorization or security layer.

  • Separate instructions from untrusted data.
  • Specify what to do when evidence is missing.
  • Keep output shape machine-validated.
30-second version

Treat the prompt like code: clear contract, version control, tests, observability and rollback.

Likely follow-up: What would you include in a prompt regression suite?
08When would you use zero-shot, few-shot or explicit reasoning guidance?promptingfew-shotreasoning＋
Strong answer

Zero-shot is simplest when the task and output contract are clear. Few-shot examples help when labels, tone or edge cases are hard to describe. For complex reasoning, I ask for structured intermediate artifacts or a plan that can be validated, but I do not depend on exposing private chain-of-thought; I evaluate the final decisions and evidence.

30-second version

Start zero-shot, add examples for ambiguous behavior, and use verifiable intermediate structures for complex tasks.

Likely follow-up: How do examples create bias or prompt overfitting?
09How do structured outputs and function calling improve reliability?structured-outputtoolsvalidation＋
Strong answer

They constrain the model to a schema or named tool contract instead of free-form prose. The server still validates types, ranges and business rules, rejects unknown fields and controls whether a tool may run. Structured output improves parseability, but it does not make the content correct or safe by itself.

schema.py
from pydantic import BaseModel, Field, ValidationError

class CandidateScore(BaseModel):
    score: int = Field(ge=0, le=100)
    evidence: list[str]

def validate(raw: dict) -> CandidateScore:
    return CandidateScore.model_validate(raw)
30-second version

A schema makes output machine-readable; server-side validation makes it usable. Tool execution remains an application decision.

Likely follow-up: What should happen when schema validation fails?
10How do you choose between prompting, RAG and fine-tuning?ragfine-tuningdecision＋
Strong answer

I use prompting to define behavior, RAG to supply private or frequently changing knowledge, and fine-tuning to change consistent behavior, style or task performance at scale. They can be combined. I first fix data quality and evaluation, because fine-tuning cannot repair missing or unauthorized knowledge and RAG cannot fix every behavioral weakness.

30-second version

Prompt for instructions, retrieve for knowledge, fine-tune for repeatable behavior. Choose from the failure you are trying to solve.

Likely follow-up: Give an example where fine-tuning is the wrong answer.
