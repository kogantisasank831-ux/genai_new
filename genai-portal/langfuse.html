Observability · from basics to a working app

Langfuse, End-to-End

Langfuse is an open-source observability and evaluation platform for LLM applications. Once your app sends a prompt to an LLM, Langfuse is how you answer: What did the model see? What did it return? How long did it take, what did it cost, was the answer any good — and where did it go wrong? This page takes you from the core mental model to a fully instrumented RAG Q&A app.

⏱ Deep study + practice · 6–10 h Beginner → intermediate Python · OpenAI · LangChain · FastAPI
🔗
Where this fits

Observability is part of running GenAI in production — the wider topic (cost, guardrails, rate limiting, testing) lives in 14 · Production GenAI →. The RAG app we instrument here builds on 06 · RAG Basics →. Code examples use OpenAI + LangChain; swapping to a local Ollama model or to Claude is a one-line change, noted where relevant.

1 · Why observability — the problem Langfuse solves

A traditional web app is mostly deterministic: same input → same output, and a stack trace tells you what broke. LLM apps are different on every axis:

Non-deterministic

The same prompt can return different answers. "It worked when I tried it" proves nothing.

Multi-step

One user question may trigger retrieval, reranking, several LLM calls and tool calls. A failure could be in any link.

Costly & slow

Tokens cost money and latency is user-visible. You need per-call cost and timing, not just an aggregate bill.

Quality is fuzzy

There is no exit code for "the answer was wrong." You need explicit scores — from users, heuristics, or an LLM judge.

🔑
The one-line definition

Langfuse records a trace of everything your LLM app did for one request — every step, its inputs/outputs, timing, tokens, cost and quality scores — so you can debug, measure and improve it.

2 · The core data model

Almost everything in Langfuse is one of five objects. Learn these and the rest is detail.

ObjectWhat it representsExample
TraceOne end-to-end request through your app. The top-level container."User asked: What are the input tags for PDI1043?"
SpanA unit of work inside a trace — anything with a start/end. Spans nest.retrieve, rerank, build_prompt
GenerationA special span for an LLM call. Captures model, prompt, completion, tokens, cost.The gpt-4o-mini call that wrote the answer
ScoreA quality measurement attached to a trace/generation. Numeric, boolean or categorical.faithfulness = 0.9, thumbs_up = true
Session / UserGroups traces — a multi-turn conversation (session) or all of one person's traces (user).session_id="chat_42", user_id="alice"

A trace is a tree. Here is what one RAG request looks like once instrumented:

▼ TRACE rag-qa · user="alice" · session="chat_42" · 1.84s · $0.0007 ▼ SPAN retrieve 320ms · k=3 ▼ SPAN rerank 110ms · 3 of 12 kept ▼ SPAN build_prompt 2ms ▼ GENERATION answer gpt-4o-mini · in 812 / out 96 tok · $0.0007 · 1.4s ● SCORE faithfulness=0.92 (llm-judge) ● SCORE user_feedback=1 (👍)
One trace, nested spans, one generation, two scores. This is the unit you inspect in the Langfuse UI.

3 · Getting set up

You have two ways to run Langfuse; both expose the same SDK and UI.

☁️ Langfuse Cloud

Sign up at cloud.langfuse.com, create a project, copy the public + secret keys. Fastest start; free tier for learning.

🐳 Self-hosted (Docker)

Run it on your own infra — required for sensitive/on-prem data. Clone the repo and docker compose up.

3.1 Install & configure

bash
pip install langfuse openai langchain langchain-openai

Configure with environment variables — never hard-code keys (see Guardrails §secrets):

.env
LANGFUSE_PUBLIC_KEY=pk-lf-...
LANGFUSE_SECRET_KEY=sk-lf-...
LANGFUSE_HOST=https://cloud.langfuse.com   # or your self-hosted URL
OPENAI_API_KEY=sk-...

3.2 Verify the connection

python
from langfuse import get_client

langfuse = get_client()          # reads keys from env
assert langfuse.auth_check()     # True if keys + host are valid
print("Langfuse connected ✓")
💡
Async by design

The SDK batches events and ships them in the background, so instrumentation adds ~no latency to your app. Call langfuse.flush() before a short-lived script exits, or events can be lost.

4 · Your first trace — three ways

There are three levels of integration. Start with whichever matches your stack.

4.1 The @observe decorator (most ergonomic)

Decorate any function and it becomes a span; the outermost decorated call becomes the trace. Nesting is automatic.

python
from langfuse import observe, get_client
from openai import OpenAI

oai = OpenAI()
langfuse = get_client()

@observe()                       # ← this call becomes a SPAN
def retrieve(question: str) -> list[str]:
    # ... your vector search ...
    return ["PDI1043.PV is an input tag for the forecasting model."]

@observe()                       # ← outermost @observe = the TRACE
def answer_question(question: str) -> str:
    context = retrieve(question)
    resp = oai.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": "Answer only from the context."},
            {"role": "user", "content": f"Context:\n{context}\n\nQ: {question}"},
        ],
    )
    return resp.choices[0].message.content

print(answer_question("What is PDI1043.PV?"))
langfuse.flush()

4.2 The OpenAI drop-in wrapper (auto-captures LLM calls)

Swap your import and every OpenAI call is automatically logged as a generation — model, tokens and cost captured for you, no extra code.

python · before → after
# before
from openai import OpenAI
# after — same API, now auto-traced
from langfuse.openai import OpenAI

oai = OpenAI()
oai.chat.completions.create(model="gpt-4o-mini", messages=[...])  # logged ✓
🔁
Using a different model?

Ollama (local, free): point the OpenAI client at base_url="http://localhost:11434/v1" — the wrapper still traces it. Claude: use the Anthropic SDK with the LangChain callback (§4.3) or the @observe decorator; for Claude calls prefer model="claude-opus-4-8" with thinking={"type":"adaptive"} rather than passing a token budget.

4.3 The LangChain callback (instrument a whole chain)

Pass one callback handler and every step of a LangChain pipeline — retriever, prompt, LLM — becomes spans/generations automatically.

python
from langfuse.langchain import CallbackHandler
from langchain_openai import ChatOpenAI

handler = CallbackHandler()
llm = ChatOpenAI(model="gpt-4o-mini")

llm.invoke("Explain RAG in one sentence.",
           config={"callbacks": [handler]})   # ← whole call traced

5 · Sessions, users & metadata

Traces become useful when you can group and filter them. Attach a user_id to see everything one person did, a session_id to stitch a multi-turn conversation, and tags/metadata to slice by anything (plant, model version, feature flag).

python
from langfuse import observe, get_client

@observe()
def answer_question(question: str) -> str:
    get_client().update_current_trace(
        user_id="alice",
        session_id="chat_42",
        tags=["rag", "ammonia-plant"],
        metadata={"plant": "Ammonia", "loop": "Loop10", "app_version": "1.4.0"},
    )
    ...
🧭
Why this matters

In the UI you can then ask: "show me all low-faithfulness traces for plant=Ammonia on app_version 1.4.0" — which is exactly how you find a regression introduced by a deploy.

6 · Scoring — measuring quality

A trace tells you what happened; a score tells you whether it was any good. Scores are the bridge from observability to evaluation. Three sources:

👍

User feedback

Thumbs up/down, star rating, "was this helpful?". The ground truth that matters most.

📐

Heuristic / programmatic

Cheap deterministic checks: did it cite a source? is it valid JSON? is it under N tokens?

⚖️

LLM-as-judge

A second LLM scores faithfulness/relevance against a rubric. Scales without humans (but can err).

6.1 Record a user-feedback score

python
from langfuse import get_client
langfuse = get_client()

# inside a trace, capture its id so the UI can attach feedback later
with langfuse.start_as_current_span(name="answer") as span:
    trace_id = langfuse.get_current_trace_id()
    ...

# when the user clicks 👍 / 👎
langfuse.create_score(trace_id=trace_id, name="user_feedback",
                      value=1, data_type="NUMERIC")   # 1 = up, 0 = down

6.2 An LLM-as-judge faithfulness score

This is the same idea as the RAG-evaluation faithfulness metric — see RAG, End-to-End §15 for the conceptual depth; here we wire it into Langfuse.

python
import json
from openai import OpenAI
from langfuse import get_client

oai, langfuse = OpenAI(), get_client()

def judge_faithfulness(context: str, answer: str) -> float:
    rubric = ("Score 0-1 how fully the ANSWER is supported by the CONTEXT. "
              "Reply with JSON: {\"score\": <0..1>, \"reason\": \"...\"}.")
    r = oai.chat.completions.create(
        model="gpt-4o-mini", temperature=0,
        messages=[{"role": "system", "content": rubric},
                  {"role": "user",
                   "content": f"CONTEXT:\n{context}\n\nANSWER:\n{answer}"}],
        response_format={"type": "json_object"},
    )
    verdict = json.loads(r.choices[0].message.content)
    return float(verdict["score"])

score = judge_faithfulness(context, answer)
langfuse.create_score(trace_id=trace_id, name="faithfulness",
                      value=score, data_type="NUMERIC")

7 · Prompt management

Hard-coding prompts in source means every wording tweak is a code deploy, and you can't tell which prompt version produced which trace. Langfuse stores prompts as versioned, named objects you fetch at runtime.

python
from langfuse import get_client
langfuse = get_client()

# create/update once (or via the UI)
langfuse.create_prompt(
    name="rag-answer",
    prompt="Answer ONLY from the context. Cite source + page. "
           "If insufficient, say so.\n\nContext:\n{{context}}\n\nQ: {{question}}",
    labels=["production"],            # promote a version to 'production'
)

# fetch at runtime — pinned to the 'production' label
prompt = langfuse.get_prompt("rag-answer", label="production")
text = prompt.compile(context=context, question=question)
🎯
The payoff

Linking a generation to a prompt version lets you compare quality scores across prompt versions — turning "I think the new prompt is better" into a measured A/B result. Edit prompts in the UI without redeploying code.

8 · Datasets & offline evaluation

Scores on live traffic tell you how production is doing. Datasets let you test before you ship: a fixed set of inputs (and optional expected outputs) you run your app against on every change.

  1. Create a dataset of representative questions (plus gold answers if you have them).
  2. On each candidate (new prompt, new model, new chunking), run every item and produce a trace + scores.
  3. Compare the average scores between runs — that's your regression test for quality.
python
dataset = langfuse.get_dataset("rag-eval-set")

for item in dataset.items:
    with item.run(run_name="prompt-v3") as root:      # links trace to the run
        answer = answer_question(item.input["question"])
        langfuse.create_score(trace_id=root.trace_id, name="faithfulness",
                              value=judge_faithfulness(item.input["context"], answer))
langfuse.flush()

9 · Sample application — a traced RAG Q&A API

Everything above, assembled into one small but realistic app: a FastAPI /ask endpoint that retrieves, generates, self-scores, and returns the answer plus its Langfuse trace URL. A second endpoint records user feedback as a score.

📦
What it demonstrates

Nested spans (retrieve → generate), a generation with token/cost capture, an automatic faithfulness score, session/user grouping, prompt management, and a feedback loop — the whole page in ~70 lines.

app.py
import json
from fastapi import FastAPI
from pydantic import BaseModel
from langfuse import get_client, observe
from langfuse.openai import OpenAI          # auto-traced OpenAI client

app = FastAPI()
oai = OpenAI()
langfuse = get_client()

# --- tiny in-memory "vector store" stands in for a real retriever ---
DOCS = {
    "PDI1043": "PDI1043.PV is an input tag for the Loop10 forecasting model. "
               "Other inputs: FI1023.PV, FIC1002.PV, TI1354.PV.",
    "deploy":  "The API is deployed via a Kubernetes Deployment and Service YAML.",
}

@observe()                                  # → SPAN: retrieve
def retrieve(question: str, k: int = 2) -> str:
    hits = [v for key, v in DOCS.items() if key.lower() in question.lower()]
    return "\n".join(hits[:k]) or "No relevant context found."

@observe()                                  # → GENERATION (auto via wrapper)
def generate(question: str, context: str) -> str:
    prompt = langfuse.get_prompt("rag-answer", label="production")
    messages = [
        {"role": "system", "content": "Answer only from the context. Cite if possible."},
        {"role": "user",   "content": prompt.compile(context=context, question=question)},
    ]
    resp = oai.chat.completions.create(model="gpt-4o-mini", temperature=0,
                                       messages=messages)
    return resp.choices[0].message.content

def judge_faithfulness(context: str, answer: str) -> float:
    r = oai.chat.completions.create(
        model="gpt-4o-mini", temperature=0,
        response_format={"type": "json_object"},
        messages=[{"role": "system",
                   "content": 'Rate 0-1 how supported the answer is by the context. '
                              'JSON: {"score": <0..1>}'},
                  {"role": "user",
                   "content": f"CONTEXT:\n{context}\n\nANSWER:\n{answer}"}])
    return float(json.loads(r.choices[0].message.content)["score"])

class Ask(BaseModel):
    question: str
    user_id: str = "anonymous"
    session_id: str | None = None

@app.post("/ask")
@observe()                                  # → TRACE (outermost @observe)
def ask(body: Ask):
    langfuse.update_current_trace(
        user_id=body.user_id, session_id=body.session_id, tags=["rag-api"])

    context = retrieve(body.question)
    answer  = generate(body.question, context)

    score = judge_faithfulness(context, answer)
    trace_id = langfuse.get_current_trace_id()
    langfuse.create_score(trace_id=trace_id, name="faithfulness",
                          value=score, data_type="NUMERIC")

    return {"answer": answer, "faithfulness": round(score, 2),
            "trace_id": trace_id,
            "trace_url": langfuse.get_trace_url()}

class Feedback(BaseModel):
    trace_id: str
    helpful: bool

@app.post("/feedback")
def feedback(body: Feedback):
    langfuse.create_score(trace_id=body.trace_id, name="user_feedback",
                          value=1 if body.helpful else 0, data_type="NUMERIC")
    return {"ok": True}

9.1 Run it

bash
uvicorn app:app --reload

# ask a question
curl -s localhost:8000/ask -H 'content-type: application/json' \
  -d '{"question":"What is PDI1043.PV?","user_id":"alice","session_id":"chat_42"}'
# → {"answer":"PDI1043.PV is an input tag...","faithfulness":0.95,
#    "trace_id":"...","trace_url":"https://cloud.langfuse.com/trace/..."}

curl -s localhost:8000/feedback -H 'content-type: application/json' \
  -d '{"trace_id":"","helpful":true}'
🔍
What you'll see in the Langfuse UI

One trace per /ask, with retrieve and generate as nested spans, the LLM call as a generation (model, 812+96 tokens, ~$0.0007, latency), a faithfulness score, and — once you POST feedback — a user_feedback score. Filter by user_id=alice or session_id=chat_42 to follow the conversation.

10 · Dashboards & what to watch

With traffic flowing, Langfuse's dashboards turn raw traces into the numbers you actually run a service on:

SignalWhy you watch it
Traces / volume over timeUsage trends, spikes, dead features
Latency (p50 / p95 / p99)The tail is what users feel — p95 matters more than the average
Cost (per trace, per user, per day)Catch a runaway prompt or an expensive model before the bill does
Token usageInput vs output split; ballooning context is the usual culprit
Score trends (faithfulness, feedback)Quality regressions — especially right after a deploy
Error rateFailed generations, timeouts, guardrail blocks

11 · Best practices

  • Instrument the whole chain, not just the LLM call — retrieval and reranking are where most RAG bugs hide.
  • Always attach user_id and session_id — debugging without them is guesswork.
  • Score in production, not just offline — combine cheap heuristics on every trace with sampled LLM-judge scoring.
  • Manage prompts in Langfuse so you can A/B versions and trace which one ran.
  • Keep a dataset and re-run it on every prompt/model/chunking change — it's your quality regression test.
  • Never log secrets or full PII into trace inputs/outputs — mask first (see Guardrails).
  • Call flush() in scripts and serverless handlers, or you'll lose the tail of your events.
  • Sample expensive evals — you don't need an LLM judge on 100% of traffic; 5–10% often suffices to spot trends.
🔒
Data sensitivity

Trace inputs/outputs can contain user data. For regulated or on-prem workloads, self-host Langfuse and mask secrets/PII before they reach the trace. This is the same boundary discussed in the Guardrails tab.
