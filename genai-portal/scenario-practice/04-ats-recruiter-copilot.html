Scenario 04 · Architecture practice

ATS Recruiter Copilot

Extend the ATS Agent Build Lab into an interview-ready architecture with fairness, audit and human control.

System design practiceIntermediateTarget: 20 min
Interview prompt

Design a recruiter copilot for an ATS. It should draft job descriptions, screen resumes against a role, suggest matching candidates, generate interview questions, propose schedules and summarize panel feedback. Recruiters remain accountable for decisions.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which actions are advisory versus allowed to update the ATS?
  • What candidate volume, file formats and languages?
  • Which protected attributes and regional regulations apply?
  • How should recruiters correct or override AI output?
  • What evidence and audit detail is required?

Assumptions for this answer

  • 10,000 applications per month and 200 recruiters.
  • Six bounded AI capabilities; no autonomous rejection.
  • Candidate data is sensitive and retained under policy.
  • Interactive tasks target p95 under 8 seconds; batch screening is queued.
  • All recommendations show evidence and can be edited.

2 · Requirements and success criteria

Must do
  • Draft editable JDs from approved role facts.
  • Extract resume evidence and score against explicit criteria.
  • Rank a bounded candidate pool with skill gaps.
  • Generate round-specific questions and summarize feedback.
Must protect
  • Human approval for consequential workflow changes.
  • No hidden use of protected attributes.
  • Traceable inputs, model versions, evidence and overrides.
  • Deterministic conflict checking for schedules.

3 · Proposed architecture

Agents produce structured suggestions. ATS services own business rules, persistence and workflow state, with recruiters reviewing consequential output.
🤖
Where agents fit

Use bounded task agents rather than an open-ended autonomous recruiter. Each capability gets a narrow prompt and schema. Deterministic services assemble trusted context, verify schedules, enforce authority and decide what is stored.

5 · Latency, cost and load handling

Fast path

  • Parse and normalize resumes asynchronously at upload, not during every screening.
  • Precompute canonical skills and embeddings; limit candidate matching to an authorized shortlist.
  • Use smaller models for extraction and stronger models only for nuanced synthesis.
  • Cache stable role context while keeping candidate-specific output isolated.

Scale and reliability

  • Queue batch screening with per-role priorities and concurrency caps.
  • Separate interactive recruiter requests from bulk jobs.
  • Use idempotent run records so retries do not duplicate scores or notifications.
  • Degrade gracefully by showing parsed facts and allowing manual workflow if the model fails.

6 · Security design

Prompt-injection boundary

  • Treat resume and JD text as untrusted; ignore embedded instructions or hidden text.
  • The model receives no database credentials and cannot move candidates or send offers.
  • Validate scores, evidence references and recommendation enums.
  • Strip unsupported external links and detect suspicious document content for review.

Application and data security

  • Enforce recruiter/manager permissions in services and scope every candidate query.
  • Encrypt candidate files, minimize traces and apply retention/deletion policy.
  • Audit model, prompt, input versions, output, reviewer edits and final decision separately.
  • Monitor fairness slices and never infer protected attributes for decisioning.

Important trade-off

Automating rejection may save time but creates high fairness and compliance risk. Keep recommendations advisory until evidence, calibration and governance justify any expanded authority.

class ScreeningResult(BaseModel):
    match_score: int = Field(ge=0, le=100)
    matched_requirements: list[str]
    missing_requirements: list[str]
    evidence: list[str]
    recommendation: Literal["review", "possible_match", "insufficient_evidence"]

async def screen(application_id: str, recruiter):
    app = await repo.get_authorized(application_id, recruiter)
    result = await screening_agent.run(
        role=app.role.criteria, resume=app.resume.normalized_text
    )
    await audit.record("AI_SCREEN_SUGGESTION", app.id, result.model_dump())
    return result  # No automatic rejection or stage change.

9 · A crisp two-minute answer

  1. Define six narrow capabilities and keep recruiter decisions and ATS writes in services.
  2. Preprocess documents and use typed outputs with evidence, not opaque scores.
  3. Queue batch screening, reserve interactive capacity and make runs idempotent.
  4. Protect candidate data, neutralize document injection and audit edits plus final decisions.
  5. Evaluate evidence quality, recruiter usefulness, downstream outcomes, fairness slices, latency and cost.
