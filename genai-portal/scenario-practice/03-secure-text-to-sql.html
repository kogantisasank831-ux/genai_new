Scenario 03 · Architecture practice

Secure Text-to-SQL Analytics Assistant

Design natural-language analytics without giving a language model unrestricted database access.

System design practiceAdvancedTarget: 25 min
Interview prompt

Design an analytics assistant for business users who ask questions such as “Which products lost revenue in South India last quarter?” It must query a multi-tenant warehouse safely, explain its answer and return a chart within 10 seconds.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which metrics and dimensions are officially defined?
  • Is arbitrary ad-hoc SQL required, or can we use a semantic layer?
  • What tenant, row and column restrictions apply?
  • What warehouse and peak query volume?
  • How should ambiguous business terms be resolved?

Assumptions for this answer

  • A governed semantic layer defines revenue, product, geography and time.
  • 100 tenants, 30 peak QPS and p95 under 10 seconds.
  • Queries are read-only and results are capped.
  • Users may see only their tenant and role-approved measures.
  • Every result must expose the interpreted query and source freshness.

2 · Requirements and success criteria

Must do
  • Interpret question into approved measures, dimensions and filters.
  • Ask clarification for ambiguity.
  • Execute a safe analytical query and produce explanation plus chart.
  • Save useful queries as governed templates.
Must protect
  • No DDL, DML, unrestricted joins or raw sensitive columns.
  • Tenant and row policies cannot be bypassed by a prompt.
  • Warehouse cost and execution time are bounded.
  • Results are reproducible and auditable.

3 · Proposed architecture

The model creates a typed semantic query plan. Deterministic code validates and compiles it; the model never submits arbitrary SQL.
🤖
Where agents fit

A full agent is optional. Use a fixed loop—interpret, clarify if needed, validate, execute, explain. An agent may help choose among metadata, glossary and chart tools, but it should never gain a general SQL execution tool.

5 · Latency, cost and load handling

Fast path

  • Cache semantic metadata and repeated result sets by tenant, role, plan and data version.
  • Run metadata lookup before the model and send only relevant schema, not the entire warehouse.
  • Use warehouse result caching and pre-aggregations for common measures.
  • Return query interpretation early, then chart after execution.

Scale and reliability

  • Apply per-tenant query quotas and separate interactive from scheduled workloads.
  • Estimate cost before execution; reject or require confirmation for large scans.
  • Set warehouse statement timeouts, row caps and dedicated resource groups.
  • Queue expensive reports and notify users asynchronously.

6 · Security design

Prompt-injection boundary

  • Treat column descriptions and sample values as untrusted metadata.
  • Constrain model output to a semantic plan schema; reject unknown measures or operators.
  • Do not expose a raw SQL execution tool or system catalog credentials.
  • Use canary prompts and red-team tests for instruction injection through data values.

Application and data security

  • Compile parameterized SQL and allowlist every identifier; values are bound by the driver.
  • Enforce tenant predicates in the semantic/compiler layer and again through warehouse row policies.
  • Use read-only views, no sensitive raw columns, row limits and statement timeouts.
  • Log plan, compiled SQL hash, user, role, cost and result size.

Important trade-off

Arbitrary text-to-SQL offers flexibility but expands security and correctness risk. A governed semantic layer covers fewer questions but produces more trustworthy, explainable and cost-controlled analytics.

8 · Python implementation sketch

service.py
from typing import Literal
from pydantic import BaseModel, Field

class Filter(BaseModel):
    field: Literal["region", "product", "quarter"]
    op: Literal["eq", "in"]
    value: str | list[str]

class QueryPlan(BaseModel):
    measure: Literal["revenue", "orders", "margin"]
    dimensions: list[Literal["product", "region", "quarter"]]
    filters: list[Filter] = Field(default_factory=list)
    limit: int = Field(default=100, le=500)

async def execute(plan: QueryPlan, principal):
    sql, params = compiler.compile(plan, tenant_id=principal.tenant_id)
    return await readonly_db.fetch(sql, params, timeout=5)

9 · A crisp two-minute answer

  1. Clarify governed metrics, permissions, ambiguity, load and warehouse limits.
  2. Have the model produce a typed semantic query plan instead of arbitrary SQL.
  3. Validate identifiers and policies, then compile parameterized SQL against read-only views.
  4. Use pre-aggregations, result caching, cost estimates and workload isolation for latency and load.
  5. Measure semantic correctness, numerical accuracy, query cost and cross-tenant security.
