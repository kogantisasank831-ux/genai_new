The C-D-S-S-M Answer Framework

A calm, repeatable way to turn any GenAI scenario into a strong architecture answer.

C · D · S · S · MReusable in every interview10–20 minute answer
C

Clarify

Users, volume, data sensitivity, freshness, actions, success metric.

D

Design

Data flow, model/RAG/agent boundary, APIs, storage and fallbacks.

S

Scale

Latency budget, caching, queues, concurrency, rate limits and cost.

S

Secure

Identity, tenant ACLs, prompt injection, tool policy, PII and audit.

M

Measure

Offline evals, online SLOs, traces, human feedback and rollback.

1 · Clarify the problem

Ask only the questions that change the architecture. A good opening is: “I’ll confirm users, scale, data sensitivity, actions and success criteria, then propose the design.”

Users and jobs

Who uses it? What decision or task becomes faster? Is the output advisory or allowed to trigger actions?

Data and freshness

Which sources, formats and permissions? How quickly must updates appear? Is data tenant-specific or regulated?

Traffic and SLO

Peak QPS, concurrent sessions, response-time target, availability, cost ceiling and acceptable degradation.

Quality

What does “correct” mean? Which errors are unacceptable? Who reviews low-confidence or consequential outputs?

2 · Design from deterministic outside to probabilistic inside

trusted application code＋bounded model capability→validated product behavior
  • Keep identity, authorization, money, workflow state and database writes in normal code.
  • Use RAG when answers need private or fresh evidence; use tools when the system must act or fetch live structured data.
  • Use an agent only when the next step depends on observations. Prefer a fixed workflow when the sequence is known.
  • Require schemas, timeouts, budgets, stop conditions and a fallback for every model-dependent step.

3 · Build a latency budget

StageQuestions to answerCommon optimization
GatewayAuthentication, rate limit, payload sizeEdge cache, request limits
Retrieval/toolsHow many calls? Can they run in parallel?Hybrid indexes, parallel I/O, result cache
ModelPrompt tokens, output tokens, model tierRouting, smaller model, streaming, prefix cache
Post-processingValidation, citations, policy checksDeterministic fast checks, async audit

4 · Treat model input and output as untrusted

No prompt can be the security boundary. Identity, data authorization, tool policy and output validation must surround the model.
  • Authorize retrieval before sending text to the model; never retrieve broadly and ask the model to hide unauthorized data.
  • Label retrieved text as data, not instructions. Detect suspicious patterns, but rely on least-privilege tools and server-side policy.
  • For SQL, generate or select from a restricted query plan, parameterize values, use a read-only account, allowlisted views, row limits and timeouts.
  • Redact or tokenize PII where possible; encrypt data, maintain retention rules and log every consequential action.

5 · Finish with measurement

Name both quality and operational metrics. Explain how production failures become new regression cases and how you roll back model, prompt, retriever or tool versions independently.

Qualitytask success, groundedness, precision/recall
Safetypolicy violations, data leakage, unsafe actions
Operationsp50/p95 latency, error rate, availability
Economicstokens, tool calls and cost per successful task
