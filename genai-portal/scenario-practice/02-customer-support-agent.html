Scenario 02 · Architecture practice

Customer Support Resolution Agent

Design a support agent that can answer, inspect an account and perform a small set of reversible actions safely.

System design practiceAdvancedTarget: 25 min
Interview prompt

Design a customer-support assistant for a consumer SaaS product. It should answer product questions, inspect a customer account, issue eligible credits, update a subscription and hand off complex cases. Peak traffic is 150 requests per second during incidents.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which actions are allowed and what monetary limits apply?
  • What authentication level is required before account access?
  • Which regions, languages and support channels?
  • How fresh is the product knowledge base?
  • What is the target containment rate without reducing customer trust?

Assumptions for this answer

  • Authenticated web chat with 150 peak request QPS.
  • Credits under a policy limit are reversible; cancellation requires confirmation.
  • Account and billing systems already expose APIs.
  • p95 first token under 2 seconds and simple resolutions under 12 seconds.
  • Human agents use an existing ticketing platform.

2 · Requirements and success criteria

Must do
  • Answer from approved product knowledge.
  • Look up account and billing state after authentication.
  • Propose or execute allowlisted actions under policy.
  • Summarize and transfer context to a human agent.
Must protect
  • No cross-account data access.
  • Exactly-once behavior for financial side effects.
  • Clear user confirmation and action receipts.
  • Survive incident traffic without cascading failures.

3 · Proposed architecture

The agent may choose among tools, but a policy service authorizes every call and side-effecting actions use confirmation plus idempotency.
🤖
Where agents fit

An agent fits because the next step depends on what the user says and what tools return. Keep the graph bounded: classify, retrieve or inspect, propose an action, pass policy, obtain confirmation, execute, verify and stop. A deterministic policy service owns eligibility and limits.

5 · Latency, cost and load handling

Fast path

  • Route FAQ-only requests to a small model and skip agent loops.
  • Parallelize knowledge retrieval and safe read-only account lookup after intent classification.
  • Stream conversational text, but never stream an unapproved action as completed.
  • Summarize history and cap loop iterations, tool calls and generated tokens.

Scale and reliability

  • Use admission control and priority queues so authenticated action requests are not starved by anonymous traffic.
  • Bulkhead account, billing and model dependencies with separate concurrency pools.
  • Circuit-break unhealthy tools and fall back to ticket creation with status messaging.
  • Autoscale on concurrent conversations, queue depth and provider headroom.

6 · Security design

Prompt-injection boundary

  • Keep tool descriptions narrow and never expose credentials to the model.
  • Treat knowledge and customer messages as untrusted; retrieved instructions cannot alter policy.
  • Validate tool name, typed arguments, account ownership and action risk server-side.
  • Require a fresh confirmation token for cancellation, credit or plan changes.

Application and data security

  • Reauthenticate or step up MFA for sensitive account changes.
  • Use idempotency keys, append-only audit records and reconciliation for financial actions.
  • Redact payment data and minimize support transcripts.
  • Rate-limit account enumeration and protect ticket handoff payloads.

7 · How we know it works

Resolutionverified containment and repeat-contact rate
Actionstool success, reversal and unauthorized-attempt rate
ExperienceCSAT, handoff quality and p95 response time
Economicscost per resolved conversation and human minutes saved
⚖️
Important trade-off

Maximizing autonomous containment can harm trust. Optimize for verified resolution and safe handoff, not the percentage of chats that avoid a human.

8 · Python implementation sketch

service.py
from pydantic import BaseModel

class CreditRequest(BaseModel):
    account_id: str
    amount: int
    reason: str

async def issue_credit(req: CreditRequest, user, idempotency_key: str):
    assert req.account_id in user.allowed_accounts
    decision = await policy.check_credit(user=user, request=req)
    if not decision.allowed:
        raise PermissionError(decision.reason)
    return await billing.credit(
        req, idempotency_key=idempotency_key
    )

# The agent can propose this tool call; only this wrapper may execute it.

9 · A crisp two-minute answer

  1. Separate FAQ answering, read-only account inspection and side-effecting actions by risk.
  2. Use a bounded agent graph because routing depends on conversation and tool observations.
  3. Put eligibility, identity, confirmation and idempotency in deterministic services.
  4. Protect incident traffic with quotas, bulkheads, queues and graceful ticket handoff.
  5. Evaluate verified resolution, action safety, repeat contacts, latency and cost.
