Scenario 06 · Architecture practice

Invoice & Document Processing Workflow

Design a high-throughput document pipeline where deterministic validation and human review surround model extraction.

System design practiceIntermediateTarget: 20 min
Interview prompt

Design a system that processes 500,000 supplier invoices per month from email and uploads. It should extract fields and line items, match purchase orders, detect duplicates, route exceptions and prepare approved records for ERP posting.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which invoice formats, languages and handwritten content?
  • What accuracy is required for totals, tax and bank details?
  • Which ERP and purchase-order systems are authoritative?
  • Can the system post automatically or only prepare a draft?
  • What is the SLA and peak upload pattern?

Assumptions for this answer

  • PDFs and images, mostly English and Indian invoice formats.
  • Peak 10 documents per second and processing SLA under 5 minutes.
  • High-risk fields require exact validation and confidence thresholds.
  • ERP posting remains human-approved initially.
  • Documents and extracted data require seven-year audit retention.

2 · Requirements and success criteria

Must do
  • Ingest, virus-scan, classify and OCR documents.
  • Extract header fields and line items into a schema.
  • Match PO, supplier and tax rules; detect duplicates.
  • Route exceptions to a review queue and export approved records.
Must protect
  • No duplicate posting after retries.
  • Field-level provenance and correction history.
  • Scalable asynchronous processing.
  • Secure document storage and supplier isolation.

3 · Proposed architecture

The model extracts candidates; deterministic validators reconcile totals, supplier identity and purchase orders before any ERP action.
🤖
Where agents fit

This is primarily a workflow, not an open-ended agent. A bounded model handles classification and extraction. An agent can assist exception investigation by choosing read-only supplier or PO tools, but posting authority remains deterministic and human-controlled.

5 · Latency, cost and load handling

Fast path

  • Use fast template or OCR rules for known suppliers, and route uncertain documents to the stronger model.
  • Process pages in parallel but reconcile line items and totals after aggregation.
  • Cache supplier profiles, tax rules and PO summaries.
  • Do not place human review on the synchronous upload path; show progress asynchronously.

Scale and reliability

  • Partition queues by document type and priority; autoscale OCR/extraction workers independently.
  • Use content hashes plus supplier/invoice keys to detect duplicate submissions.
  • Persist each stage result so retries resume rather than reprocess the whole document.
  • Use dead-letter queues and replay tooling for parser or provider incidents.

6 · Security design

Prompt-injection boundary

  • Treat document text, QR codes and hidden layers as untrusted input.
  • The extraction model has no ERP or email-sending tool.
  • Validate URLs and ignore embedded instructions unrelated to the extraction schema.
  • Compare OCR text, visual evidence and business records for suspicious changes.

Application and data security

  • Encrypt originals and use short-lived signed URLs.
  • Restrict supplier and business-unit access; redact bank details in normal traces.
  • Parameterize all database operations and use a scoped ERP service account.
  • Record original value, model value, reviewer correction and final posted value.

7 · How we know it works

Extractionfield precision/recall and line-item accuracy
Automationstraight-through rate with post-review error
Operationsqueue age, processing SLA and cost per invoice
Riskduplicate blocks, fraud flags and incorrect-posting rate
⚖️
Important trade-off

A high straight-through rate is not valuable if financial errors rise. Tune automation thresholds by field risk and supplier history, not one global confidence score.

8 · Python implementation sketch

service.py
class Invoice(BaseModel):
    supplier_tax_id: str
    invoice_number: str
    currency: str
    subtotal: Decimal
    tax: Decimal
    total: Decimal
    line_items: list[LineItem]

def validate_invoice(inv: Invoice, po: PurchaseOrder) -> list[str]:
    errors = []
    if inv.subtotal + inv.tax != inv.total:
        errors.append("totals_do_not_reconcile")
    if inv.supplier_tax_id != po.supplier_tax_id:
        errors.append("supplier_mismatch")
    if inv.total > po.remaining_value:
        errors.append("po_limit_exceeded")
    return errors

9 · A crisp two-minute answer

  1. Frame the problem as an asynchronous document workflow, not a free-running agent.
  2. Preserve original files and provenance, then OCR, extract to a schema and validate deterministically.
  3. Use queues, stage checkpoints, hashes and idempotency for scale and retries.
  4. Route risky or low-confidence fields to humans; keep ERP posting gated.
  5. Measure field accuracy, straight-through rate after audit, queue SLA, duplicate prevention and financial errors.
