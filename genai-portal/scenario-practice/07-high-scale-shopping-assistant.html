Scenario 07 · Architecture practice

High-Scale Shopping Assistant

Design a consumer assistant that combines conversational intent with real-time catalog, inventory and policy data.

System design practiceAdvancedTarget: 25 min
Interview prompt

Design a shopping assistant for a large marketplace serving millions of users. It should help users discover and compare products, answer policy questions, check live inventory and add selected items to a cart. Peak traffic is 3,000 requests per second during sales.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which categories and languages are in scope?
  • Is personalization allowed, and what consent exists?
  • What inventory freshness is required?
  • May the assistant add to cart automatically or only after confirmation?
  • What peak concurrency and cost target per session?

Assumptions for this answer

  • 3,000 request QPS and hundreds of thousands of concurrent sessions.
  • Catalog has structured attributes, lexical index and embeddings.
  • Inventory and price must be live within seconds.
  • Cart mutation needs explicit user confirmation.
  • p95 first token under 1.5 seconds for simple discovery.

2 · Requirements and success criteria

Must do
  • Understand constraints and search the catalog.
  • Compare a small set with evidence from structured attributes.
  • Check current price, availability and delivery estimate.
  • Add the confirmed SKU and quantity to cart.
Must protect
  • No invented product, price or availability.
  • Stable service during sale spikes and provider throttling.
  • Privacy-respecting personalization and cache isolation.
  • Accessible explanations and clear sponsored-result handling.

3 · Proposed architecture

Discovery can use cached catalog retrieval, while price, stock and cart are live typed tools. The model never invents or writes these fields.
🤖
Where agents fit

Use a shallow agent for multi-turn constraint gathering and tool choice. Keep simple searches on a fast deterministic route. Product ranking can combine search and recommender scores; live inventory and cart mutation remain typed services with confirmation.

5 · Latency, cost and load handling

Fast path

  • Route exact SKU, policy and simple filtered search without a large model.
  • Retrieve catalog and recommendations in parallel; fetch live data only for shortlisted SKUs.
  • Use small models for intent/attribute extraction and a stronger model only for nuanced comparison.
  • Stream explanation after authoritative product cards are ready.

Scale and reliability

  • Precompute embeddings, facets and popular query results; use edge caching for public policy content.
  • Protect live price/inventory services with bulkheads, request coalescing and short TTL caches.
  • Use per-session and global token budgets, load shedding and fallback to standard search UI.
  • Autoscale on active sessions, search latency and model concurrency headroom.

6 · Security design

Prompt-injection boundary

  • Catalog descriptions and reviews are untrusted; they cannot redefine tool rules.
  • Only allowlisted product IDs from search may be sent to live tools or cart.
  • Validate quantity, price version, inventory and user confirmation before mutation.
  • Do not expose seller or internal ranking tools that users are not entitled to invoke.

Application and data security

  • Bind cart actions to authenticated session and use anti-CSRF plus idempotency.
  • Parameterize search/database filters and enforce category/region restrictions.
  • Separate personal features from public caches and respect consent/deletion.
  • Audit sponsored content and prevent the model from presenting ads as organic recommendations.

Important trade-off

Personalization may improve conversion but can reduce diversity and increase privacy risk. Use user controls, explainable constraints and a non-personalized path rather than optimizing only short-term clicks.

8 · Python implementation sketch

service.py
async def compare_products(query: str, user):
    catalog_task = search.find(query, filters=user.allowed_market)
    recs_task = recommender.candidates(user.profile) if user.consent else noop([])
    catalog, recs = await asyncio.gather(catalog_task, recs_task)

    candidates = rank_and_dedupe(catalog, recs)[:6]
    live = await inventory.batch_get([p.sku for p in candidates])
    facts = merge_authoritative(candidates, live)
    return await model.compare_json(query=query, products=facts)

async def add_to_cart(sku: str, qty: int, confirmation, user):
    require_confirmed(confirmation, user, sku, qty)
    return await cart.add(user.id, sku, qty, idempotency_key=confirmation.id)

9 · A crisp two-minute answer

  1. Separate product discovery from live facts and cart mutation.
  2. Use fast routes for simple intent and a shallow agent only for dynamic refinement.
  3. Parallelize catalog and recommendation retrieval; fetch live facts for the shortlist.
  4. Protect sale traffic with caches, bulkheads, budgets and fallback to normal search.
  5. Measure truthfulness, discovery success, latency, conversion and cost per assisted order.
