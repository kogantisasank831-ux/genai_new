Scenario 01 · Architecture practice

Enterprise Knowledge Assistant

Design an internal assistant that answers across company documents without leaking data or inventing policy.

System design practiceIntermediateTarget: 20 min
Interview prompt

Design a GenAI assistant for 20,000 employees. It should answer questions over policies, wikis and project documents, cite sources, respect existing permissions and respond interactively. Documents change throughout the day.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which repositories and file types are in scope?
  • Must permissions match source systems in near real time?
  • What languages and daily active users are expected?
  • Is the assistant advisory, or may it trigger actions?
  • What latency and citation-accuracy targets matter?

Assumptions for this answer

  • 5,000 daily active users and 50 peak QPS.
  • Documents come from SharePoint, Confluence and object storage.
  • Answers target p95 under 5 seconds with visible streaming.
  • No write actions in version one.
  • Every answer must cite accessible evidence or abstain.

2 · Requirements and success criteria

Must do
  • Ingest and update documents incrementally.
  • Retrieve only authorized passages.
  • Answer with source links and quoted evidence.
  • Support follow-up questions without replaying all history.
Must protect
  • Strict tenant and document ACL enforcement.
  • p95 first token under 2 seconds.
  • Auditable retrieval and answer traces.
  • Graceful fallback when retrieval or model is unavailable.

3 · Proposed architecture

Authorization is applied during retrieval. The model never sees chunks the employee is not allowed to read.
🤖
Where agents fit

Start with a deterministic RAG workflow: classify intent, retrieve, rerank, generate and verify citations. Add a bounded agent only if users need dynamic tool choice across several systems. The model must not decide document access.

5 · Latency, cost and load handling

Fast path

  • Run query rewrite and conversation-summary lookup in parallel when possible.
  • Use hybrid retrieval with metadata prefilters and rerank only the top 20–40 chunks.
  • Keep context to the best evidence; stream after validation of citations or buffer the first sentence.
  • Cache embeddings and permission-aware retrieval for stable popular queries.

Scale and reliability

  • Incremental ingestion through a queue; isolate OCR and embedding workers.
  • Autoscale API on active requests and ingestion workers on queue depth.
  • Apply per-user and per-tenant quotas plus a global model concurrency budget.
  • Degrade to lexical search with source links if the model is unavailable.

6 · Security design

Prompt-injection boundary

  • Treat document text as untrusted data and wrap it in explicit evidence blocks.
  • Ignore instructions found inside documents; tools are not exposed in version one.
  • Run injection detectors for telemetry, but rely on ACLs and absence of dangerous tools.
  • Require citations to retrieved chunk IDs and reject unsupported references.

Application and data security

  • Resolve user groups server-side and filter the search query before retrieval.
  • Include identity and ACL version in all cache keys.
  • Encrypt content and traces; redact unnecessary PII and enforce retention.
  • Test cross-group and recently revoked access as security regression cases.

7 · How we know it works

Retrievalrecall@k and permission-filter accuracy
Answergroundedness, completeness and citation precision
Producttask success, abstention and search fallback rate
Operationsp95 TTFT, total latency and cost per accepted answer
⚖️
Important trade-off

A larger context can improve synthesis but increases cost, latency and the chance that irrelevant or malicious text distracts the model. Prefer better retrieval and reranking before expanding the prompt.

8 · Python implementation sketch

service.py
from dataclasses import dataclass

@dataclass
class Principal:
    user_id: str
    group_ids: list[str]

async def answer(question: str, principal: Principal):
    # Authorization is part of retrieval, never delegated to the LLM.
    hits = await search.hybrid(
        query=question,
        filters={"allowed_groups": {"$in": principal.group_ids}},
        limit=40,
    )
    evidence = await reranker.top(question, hits, k=8)
    result = await model.answer_json(
        question=question, evidence=evidence, allow_abstain=True
    )
    validate_citations(result, allowed_ids={x.id for x in evidence})
    return result

9 · A crisp two-minute answer

  1. Clarify repositories, permissions, scale, freshness and acceptable abstention.
  2. Propose incremental ingestion plus ACL-aware hybrid retrieval and reranking.
  3. Keep the first version as a deterministic RAG workflow; identity and authorization stay outside the model.
  4. Budget latency by retrieval, model first token and generation; cache only with permission-aware keys.
  5. Measure retrieval and groundedness separately, and degrade to source search when generation fails.
