Scenario Design Studio / Multilingual Voice Service Assistant
Scenario 05 · Architecture practice

Multilingual Voice Service Assistant

Design a natural voice experience while controlling latency, interruptions, language quality and sensitive actions.

System design practiceAdvancedTarget: 25 min
Interview prompt

Design a multilingual voice assistant for a telecom provider. It should identify the caller, answer plan and billing questions, troubleshoot common issues, open a ticket and transfer to a human. It must support Hindi, English and code-switching, with natural turn-taking.

20:00

1 · Clarify before designing

Start by asking a few questions. It shows that you design for constraints rather than drawing a fashionable architecture.

Questions to ask

  • Which channels and telephony provider are used?
  • What identity verification is required for account information?
  • What p95 response gap feels acceptable?
  • How often do users code-switch, and are transcripts retained?
  • Which actions can be completed without a human?

Assumptions for this answer

  • 10,000 concurrent calls at peak across India.
  • Hindi, English and Hinglish are required.
  • Target end-of-utterance to first audio under 1.5 seconds.
  • Ticket creation is allowed; plan changes require human or stronger confirmation.
  • Calls can transfer with transcript and structured summary.

2 · Requirements and success criteria

Must do
  • Stream speech recognition with endpointing and barge-in.
  • Answer from approved knowledge and inspect an authenticated account.
  • Run bounded troubleshooting and create a ticket.
  • Transfer with context, language and attempted steps.
Must protect
  • Low conversational latency and graceful jitter handling.
  • Accurate language detection without losing named entities.
  • PII protection for audio, transcripts and traces.
  • Fallback to keypad or human when speech quality is poor.

3 · Proposed architecture

Audio is processed incrementally. Read-only retrieval and account lookup can run while the user is speaking, but sensitive actions wait for explicit confirmation.
🤖
Where agents fit

A bounded conversational agent fits because it must react to partial speech, tool results and interruptions. Keep troubleshooting paths as explicit graph nodes, cap turns, and make transfer a normal outcome—not a failure.

5 · Latency, cost and load handling

Fast path

  • Use streaming ASR, endpointing and speculative intent detection before the utterance fully ends.
  • Start retrieval in parallel with final transcript stabilization, then cancel if the intent changes.
  • Route simple intents to a smaller model and use cached TTS for fixed confirmations.
  • Stream TTS sentence by sentence and support immediate barge-in cancellation.

Scale and reliability

  • Partition sessions by call ID and keep media workers sticky but stateless beyond Redis.
  • Autoscale on concurrent streams and audio-frame lag, not HTTP request count.
  • Use regional speech endpoints, provider quotas and per-stage circuit breakers.
  • If the model or RAG is slow, play a brief progress cue and transfer rather than leaving silence.

6 · Security design

Prompt-injection boundary

  • Do not allow spoken or retrieved instructions to alter tool policy.
  • Expose only typed account and ticket tools; no arbitrary browsing or shell access.
  • Require identity state and fresh confirmation in the tool wrapper.
  • Sanitize transcript-derived fields before placing them in downstream tickets.

Application and data security

  • Use step-up verification before account details; bind verified identity to the call session.
  • Encrypt audio and transcripts, minimize retention and redact account numbers from traces.
  • Use scoped service credentials and audit every lookup and ticket action.
  • Detect abuse and rate-limit repeated account or OTP attempts.

7 · How we know it works

⚖️
Important trade-off

Aggressive endpointing makes the assistant feel fast but can cut off slower speakers or code-switched phrases. Tune by language and user behavior, and optimize perceived latency with acknowledgements rather than sacrificing understanding.

8 · Python implementation sketch

service.py
async def on_final_transcript(session, text: str):
    intent_task = asyncio.create_task(router.classify(text, session.language))
    retrieval_task = asyncio.create_task(knowledge.search(text, session.acl))

    intent = await intent_task
    evidence = await retrieval_task if intent.needs_knowledge else []
    reply = await voice_agent.next_turn(
        state=session.state, intent=intent, evidence=evidence
    )
    async for audio_chunk in tts.stream(reply.text, language=session.language):
        if session.user_is_speaking:  # barge-in
            break
        await session.send_audio(audio_chunk)

9 · A crisp two-minute answer

  1. Clarify languages, identity, permitted actions, concurrency and conversational latency.
  2. Build a streaming media pipeline with explicit turn state, barge-in and language-aware ASR/TTS.
  3. Use a bounded agent for dynamic conversation, but deterministic policy for identity and tools.
  4. Parallelize ASR-adjacent work, route simple intents and degrade quickly to a human.
  5. Measure speech quality by language, end-of-turn latency, completion, transfer and safety.
