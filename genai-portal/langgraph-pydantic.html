Pydantic for LangGraph

Learn to make agent inputs, tool calls, structured outputs, and graph boundaries explicit and safe. This lab covers Pydantic v2 fundamentals, validators, strict contracts, unions, LangGraph state choices, current limitations, and system-design interview trade-offs.

⏱ Study + coding · 6–10 hDifficulty · IntermediatePydantic v2 · LangGraph · APIs
🎯
Interview outcome

After this lab you should be able to design strict request and tool schemas, explain Pydantic v2 validation and serialization, choose between TypedDict, dataclass, and BaseModel, and describe the important limitations of Pydantic state inside LangGraph.

1 · Where Pydantic belongs in a LangGraph system

Pydantic validates untrusted or loosely typed data at runtime. In an agent service, the strongest value is at boundaries: HTTP requests, tool arguments, model structured output, environment/settings, messages from queues, and persisted records. It can also define LangGraph state, but that choice has trade-offs.

Ingress

Reject malformed API payloads before graph execution.

Tool contracts

Constrain what the model or caller may ask a tool to execute.

Graph state

Add input validation and nested typed objects, accepting runtime cost and current LangGraph limitations.

2 · Pydantic v2 essentials

models_basics.py
from datetime import datetime
from pydantic import BaseModel, ConfigDict, Field, ValidationError

class SearchRequest(BaseModel):
    model_config = ConfigDict(extra="forbid", str_strip_whitespace=True)

    query: str = Field(min_length=3, max_length=500)
    top_k: int = Field(default=5, ge=1, le=50)
    requested_at: datetime | None = None

raw = {"query": "  async python  ", "top_k": "10"}
request = SearchRequest.model_validate(raw)

print(request.query)                    # "async python"
print(request.top_k, type(request.top_k))  # 10, int
print(request.model_dump())
print(request.model_dump_json())

try:
    SearchRequest.model_validate({"query": "x", "unknown": True})
except ValidationError as exc:
    print(exc.errors())
APIWhat to say in an interview
model_validate(data)Parses and validates Python input into a model.
model_dump()Serializes a model recursively into Python data.
model_dump_json()Produces JSON using Pydantic serialization rules.
ConfigDict(extra="forbid")Fails closed on unexpected fields; useful for security-sensitive contracts.
Field(...)Declares defaults, bounds, descriptions, aliases, and schema metadata.
⚖️
Coercion vs strictness

Pydantic commonly coerces compatible inputs, such as "10" into 10. That is convenient for APIs but can hide upstream defects. Use strict types/configuration where implicit conversion is unsafe.

3 · Field and model validators

validators.py
from typing_extensions import Self
from pydantic import BaseModel, ConfigDict, Field, field_validator, model_validator

class ToolCall(BaseModel):
    model_config = ConfigDict(extra="forbid")

    tool_name: str
    arguments: dict[str, object]
    requires_approval: bool = False
    risk_score: int = Field(ge=0, le=100)

    @field_validator("tool_name")
    @classmethod
    def normalize_tool_name(cls, value: str) -> str:
        value = value.strip().lower()
        if not value:
            raise ValueError("tool_name cannot be empty")
        return value

    @model_validator(mode="after")
    def enforce_approval_policy(self) -> Self:
        if self.risk_score >= 70 and not self.requires_approval:
            raise ValueError("high-risk tools require approval")
        return self

Interview rule: use field validators for one field’s invariant. Use a model validator when the rule depends on several fields. Keep validators deterministic and side-effect free; database calls and network checks belong in service logic, not schema construction.

4 · Nested models, unions, and tool schemas

tool_contracts.py
from typing import Annotated, Literal
from pydantic import BaseModel, Field

class SqlQuery(BaseModel):
    kind: Literal["sql"]
    statement: str = Field(min_length=1)
    read_only: bool = True

class WebSearch(BaseModel):
    kind: Literal["web"]
    query: str = Field(min_length=3)
    max_results: int = Field(default=5, ge=1, le=10)

ToolRequest = Annotated[SqlQuery | WebSearch, Field(discriminator="kind")]

class AgentInput(BaseModel):
    request_id: str
    tool: ToolRequest

A discriminated union selects one schema using a stable tag such as kind. This is clearer and usually faster than trying each union member. In interviews, connect this to safe tool routing: each tool gets a precise contract instead of a single unstructured dictionary.

6 · Pydantic state in LangGraph

pydantic_state.py
from pydantic import BaseModel, ConfigDict, Field
from langgraph.graph import StateGraph, START, END

class ResearchState(BaseModel):
    model_config = ConfigDict(extra="forbid")

    question: str = Field(min_length=3)
    attempt: int = Field(default=0, ge=0, le=3)
    answer: str | None = None

async def answer_node(state: ResearchState) -> dict[str, object]:
    response = await model.ainvoke(state.question)
    return {
        "answer": response.content,
        "attempt": state.attempt + 1,
    }

builder = StateGraph(ResearchState)
builder.add_node("answer", answer_node)
builder.add_edge(START, "answer")
builder.add_edge("answer", END)
graph = builder.compile()

result = await graph.ainvoke({"question": "How does cancellation work?"})
# result is graph output data; validate again if a Pydantic object is required.
validated_output = ResearchState.model_validate(result)
⚠️
Current LangGraph limitations to know
  • Graph output is not automatically returned as a Pydantic model instance.
  • Runtime validation is documented as occurring on input to the first node, not on every later update/output.
  • Validation errors do not necessarily identify the originating node cleanly.
  • Recursive Pydantic validation is slower than TypedDict/dataclass state.
  • The higher-level create_agent factory does not support Pydantic state schemas.

Production implication: do not assume that choosing a Pydantic state makes every node update safe. Validate critical node outputs explicitly, or place validated models at boundaries and keep internal state lightweight.

7 · A stronger architecture: validate at boundaries

validated_boundaries.py
from pydantic import BaseModel, ConfigDict, Field, ValidationError
from typing_extensions import TypedDict

class GraphRequest(BaseModel):
    model_config = ConfigDict(extra="forbid")
    question: str = Field(min_length=3, max_length=1000)
    tenant_id: str

class GraphResponse(BaseModel):
    answer: str
    citations: list[str]

class InternalState(TypedDict):
    question: str
    tenant_id: str
    evidence: list[str]
    answer: str

async def handle(payload: dict) -> GraphResponse:
    request = GraphRequest.model_validate(payload)        # ingress boundary
    result = await graph.ainvoke(request.model_dump())
    return GraphResponse.model_validate(result)           # egress boundary

This pattern is often easier to defend in a system-design interview: strict public contracts, efficient internal state, explicit output validation, and well-defined ownership of failures.

8 · Error handling and API behaviour

Client validation error

Return a structured 4xx response with field locations and safe messages. Do not expose secrets or raw internal traces.

Node/domain error

Represent retryable vs terminal failures explicitly; Pydantic validation is not a substitute for domain error design.

Schema evolution

Add backward-compatible defaults, version external contracts, and test old persisted/checkpoint data.

Observability

Log error type, request/thread identifiers, schema version, and redacted field paths—not sensitive payloads.

10 · Interview questions and strong answer anchors

Q Pydantic vs dataclass? ›
A dataclass primarily generates ordinary class boilerplate and is lightweight. Pydantic builds validated/parsed runtime models, JSON schema, and rich serialization, at extra runtime cost.
Q Validation vs serialization? ›
Validation transforms/checks input into a trusted model. Serialization converts the trusted model into Python/JSON output according to rules and field inclusion settings.
Q Why forbid extra fields? ›
It detects typos and unexpected input early and is valuable for commands/security-sensitive payloads. For forward-compatible public APIs you may deliberately choose a different policy, but it should be explicit.
Q When would you not use Pydantic for LangGraph state? ›
When state is internal/trusted, high-volume, or frequently updated and recursive validation cost is not justified; also when you require every intermediate update to be validated automatically, because current documented behaviour does not provide that guarantee.
Q How do you validate LLM structured output safely? ›
Define a constrained schema, parse/validate the model output, reject or repair only under a bounded policy, distinguish validation failures from provider failures, and never execute side effects until the validated object passes authorization/domain checks.
Q What is a discriminated union? ›
A union whose member is selected by a known tag field. It produces clearer errors and deterministic tool/event routing compared with ambiguous untagged unions.

11 · Knowledge check

1 · What does extra="forbid" do?
It enforces a closed schema for unexpected fields.
2 · Which validator is appropriate for a rule involving risk score and approval flag together?
Cross-field invariants belong at model level.
3 · What is an important limitation of Pydantic LangGraph state?
Current documentation describes runtime validation at the first node input, not every intermediate update.
