Module 07 · Retrieval track

Advanced RAG

Naive RAG (Module 06) demos beautifully and then disappoints in production: it misses exact terms, retrieves near-misses, fumbles vague or multi-part questions, and you have no number telling you how good it is. This module is the toolbox that turns a demo into a system — hybrid search, reranking, query transformation, smarter chunk retrieval, Graph & Agentic RAG, and the evaluation discipline that lets you prove any of it helped.

⏱ ~75 min read Difficulty · Intermediate→Advanced Prereq · Module 06

1 · Executive summary

Advanced RAG is a set of upgrades to each stage of the pipeline, plus the measurement loop that justifies them. You improve recall (find more of the right chunks) with hybrid search and query expansion; you improve precision (rank the right chunks first) with reranking and compression; you improve context quality with parent-document and Graph RAG; and you handle hard questions with Agentic RAG. Crucially, you measure every change with a RAG evaluation harness so improvements are real, not vibes.

🎯

Better recall

Hybrid (dense+sparse) search, query expansion/rewriting, multi-query — find chunks naive vector search misses.

🔬

Better precision

Cross-encoder reranking and context compression put the right evidence first and strip the noise.

🧠

Better reasoning

Parent-document, Graph RAG, and Agentic RAG handle context, relationships, and multi-hop questions.

🔑
The one idea to remember

Retrieve broadly, then rank precisely, and measure everything. The winning pattern is two-stage: a cheap, high-recall first pass (hybrid) feeds an expensive, high-precision reranker — all validated by an eval set. Don't add a technique you can't prove helped.

2 · First-principles breakdown

Why — recall and precision are different problems

From classic ML you know the precision/recall tension. RAG has the same split: recall = did we retrieve the chunk that contains the answer at all? precision = is the right chunk near the top, with little noise around it? These need different tools. Hybrid search and query expansion raise recall; reranking and compression raise precision. Treating them as one knob is why naive tuning plateaus.

🪜
The two-stage mental model

Stage 1 (retrieve, cheap, recall-first): cast a wide net — fetch top-50 with hybrid search. Stage 2 (rerank, expensive, precision-first): a cross-encoder re-scores those 50 and keeps the best 5. You only run the expensive model on a small candidate set, getting precision without paying to score the whole corpus.

4 · Deep technical explanation

Hybrid search & Reciprocal Rank Fusion

Dense finds meaning; sparse (BM25) finds exact terms. Hybrid runs both and fuses results. RRF is the robust default — it merges by rank, not raw scores (which live on incomparable scales): score(d) = Σ 1/(k + rank_i(d)) over each retriever, with k≈60.

Query transformation

  • Rewriting — clean a messy/conversational query into a search-friendly one ("what about the other one?" → resolve the reference).
  • Expansion / multi-query — generate several paraphrases, retrieve for each, union the results → higher recall for vague queries.
  • HyDE (Hypothetical Document Embeddings) — ask the LLM to draft a fake answer, embed that, and retrieve with it; the hypothetical answer often sits closer to real passages than the bare question.
  • Decomposition — split a multi-part question into sub-questions, retrieve for each (bridges to Agentic RAG).

Context compression

Retrieved chunks contain irrelevant sentences that waste tokens and dilute attention (lost-in-the-middle, Module 02). Compression trims each chunk to only the parts relevant to the query — via an extractive LLM filter or a small relevance model — shrinking cost and sharpening the prompt.

Parent-document (small-to-big) retrieval

The chunking dilemma: small chunks retrieve precisely but lack context; large chunks have context but match poorly. Parent-document resolves it: index small chunks for retrieval, but when one hits, return its parent (the surrounding section/page) to the LLM. You get precise matching and full context.

Figure 3 · Match on small chunks, generate from their larger parents — precision + context.

Graph RAG

For questions about relationships ("how are X and Y connected?", "summarize everything about entity Z"), vector search alone is weak. Graph RAG extracts entities and relationships into a knowledge graph, then retrieves connected subgraphs (and/or community summaries) as context. It shines for multi-hop and global "sense-making" queries that span many documents.

Agentic RAG

The most powerful pattern: let an agent (Module 08) decide how to retrieve. It can rewrite the query, choose among multiple indexes/tools, retrieve iteratively, judge whether it has enough evidence, and retry — turning retrieval from a single static call into an adaptive loop. This is where Retrieval and Agents tracks merge.

Hallucination mitigation in RAG

Before generation

  • Score threshold → "I don't know" (Module 06)
  • Rerank so true evidence is on top
  • Compress to remove distractors

After generation

  • Require citations; verify each claim maps to a chunk
  • Faithfulness check (LLM-as-judge): is every statement supported?
  • Self-correction loop if unsupported claims found

📊

You cannot improve what you don't measure. Build a golden eval set (question, gold-answer, gold-source) before adding techniques. Frameworks like RAGAS compute these metrics. Every change (chunk size, hybrid, rerank) is an A/B against the eval set — keep it only if the numbers move. This connects directly to Module 14.

6 · Practical Python

Minimal — add a reranker to your RAG

rerank_minimal.py
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query, candidates, top_n=5):
    # candidates: list[str]. Cross-encoder scores (query, doc) pairs.
    pairs = [(query, c) for c in candidates]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(candidates, scores), key=lambda x: -x[1])
    return [c for c, _ in ranked[:top_n]]

# Retrieve 50 cheaply with vectors, then rerank to the best 5.

Intermediate — hybrid search with Reciprocal Rank Fusion

hybrid_rrf.py
from rank_bm25 import BM25Okapi
import numpy as np

def reciprocal_rank_fusion(ranked_lists, k=60):
    """Merge multiple ranked lists of doc ids by RRF."""
    scores = {}
    for lst in ranked_lists:
        for rank, doc_id in enumerate(lst):
            scores[doc_id] = scores.get(doc_id, 0.0) + 1.0 / (k + rank)
    return sorted(scores, key=scores.get, reverse=True)

def hybrid_search(query, corpus, doc_vecs, embed, bm25, k=50):
    # Dense ranking
    qv = embed(query)
    dense = np.argsort(-(doc_vecs @ qv))[:k].tolist()
    # Sparse ranking
    sparse = np.argsort(-bm25.get_scores(query.split()))[:k].tolist()
    # Fuse — robust to incomparable score scales
    return reciprocal_rank_fusion([dense, sparse])

bm25 = BM25Okapi([d.split() for d in corpus])

Production — an advanced retriever (query rewrite → hybrid → rerank → compress)

advanced_retriever.py
class AdvancedRetriever:
    """Two-stage retrieval with query transform, hybrid fuse, rerank, compress."""
    def __init__(self, embed, bm25, store, reranker, llm):
        self.embed, self.bm25, self.store = embed, bm25, store
        self.reranker, self.llm = reranker, llm

    def retrieve(self, query, tenant, k_first=50, k_final=5):
        # 1 · query transform: rewrite + 2 paraphrases (multi-query)
        queries = self._expand(query)
        # 2 · hybrid retrieve for every query variant, union candidates
        cand_ids = set()
        for q in queries:
            cand_ids.update(self._hybrid(q, tenant, k_first))
        cands = self.store.fetch(cand_ids)
        # 3 · rerank with cross-encoder (precision)
        scored = self.reranker.predict([(query, c.text) for c in cands])
        top = [c for c, _ in
               sorted(zip(cands, scored), key=lambda x: -x[1])[:k_final]]
        # 4 · context compression: keep only query-relevant sentences
        return [self._compress(query, c) for c in top]

    def _expand(self, query):
        out = self.llm.complete(
            f"Rewrite this search query and give 2 paraphrases, one per line:\n{query}",
            temperature=0.3)
        return [query] + [l.strip() for l in out.splitlines() if l.strip()][:2]

    def _compress(self, query, chunk):
        chunk.text = self.llm.complete(
            f"Extract ONLY sentences relevant to: {query}\n\n{chunk.text}",
            temperature=0)
        return chunk
⚖️

Every stage adds latency and cost. Query expansion and compression each cost an LLM call; reranking adds a model pass. Add them one at a time and keep only what your eval set rewards. A great reranker often beats five other tricks combined.

Evaluating it (RAGAS-style)

eval_rag.py
# Golden set drives every decision. Pseudocode for the core loop.
def evaluate(pipeline, golden):     # golden: [{q, gold_answer, gold_source_id}]
    rec = faith = rel = 0
    for ex in golden:
        out = pipeline.answer(ex["q"])
        retrieved_ids = {s["id"] for s in out["sources"]}
        rec   += ex["gold_source_id"] in retrieved_ids          # context recall
        faith += judge_supported(out["answer"], out["sources"])  # LLM-as-judge
        rel   += judge_relevant(out["answer"], ex["q"])
    n = len(golden)
    return {"recall": rec/n, "faithfulness": faith/n, "answer_relevance": rel/n}

7 · Ollama implementation

All advanced techniques run locally — reranking, query expansion, compression, and the LLM-judge can all use local models.

local_advanced.py
import ollama

# Query expansion with a local model
def expand_local(query):
    out = ollama.chat(model="qwen2.5", messages=[{"role": "user",
        "content": f"Give 3 search paraphrases (one per line) for: {query}"}])
    return [l.strip() for l in out["message"]["content"].splitlines() if l.strip()]

# Local faithfulness judge — keeps the whole eval loop offline
def judge_supported(answer, context):
    verdict = ollama.chat(model="qwen2.5", messages=[{"role": "user",
        "content": f"Is every claim in the ANSWER supported by CONTEXT? "
                   f"Reply YES or NO.\nCONTEXT:{context}\nANSWER:{answer}"}])
    return "YES" in verdict["message"]["content"].upper()
🔒

The cross-encoder reranker (sentence-transformers) and local Ollama models give you an entirely offline advanced-RAG + eval stack — no data or queries ever leave the machine. This is the on-prem capstone backbone.

8 · Industry perspective

🚀 Startups

Add techniques reactively: ship naive RAG, watch failure logs, add hybrid when exact-term misses appear, add a reranker when "right answer ranked low" complaints come in. An eval set early saves months of guessing.

🏢 Enterprises

Often run hybrid + rerank by default, with Graph RAG for connected-entity domains (compliance, investigations, knowledge bases) and rigorous eval gates in CI. Retrieval quality is a tracked SLO with dashboards.

The pragmatic upgrade order (by ROI)

1 · Build an eval set

Before anything else. You can't tune blind.

2 · Fix chunking

Highest leverage, lowest cost. Often the single biggest win.

3 · Add a reranker

Usually the best precision-per-effort upgrade.

4 · Add hybrid search

Fixes exact-term/code recall misses.

5 · Query expansion / parent-doc / compression

Targeted fixes for specific failure modes you've measured.

6 · Graph / Agentic RAG

Only when multi-hop/relationship queries are core — they add real complexity.

Common production challenges: reranker latency budget, query-expansion cost blowup, Graph RAG ingestion complexity, eval-set drift as the corpus grows, and "improvements" that help anecdotes but hurt the aggregate metric.

9 · Top mistakes engineers make

❌ Adding techniques without an eval set

You can't tell if hybrid/rerank helped. Build the golden set first; every change is an A/B.

❌ Fusing raw scores

Dense and BM25 scores live on different scales. Fuse by rank (RRF), not by raw score.

❌ Reranking everything

Cross-encoders are slow. Rerank a small candidate set (top-50), not the whole corpus.

❌ Over-aggressive compression

Strip too much and you delete the answer. Compress conservatively and measure faithfulness.

❌ Reaching for Graph RAG first

It's heavy. Most apps win more from chunking + rerank. Use Graph RAG only for genuine relationship queries.

❌ Stacking every trick at once

You won't know what helped, and latency explodes. Add one, measure, keep or drop.

10 · Interview preparation

Beginner  What is hybrid search and why use it? ›

Running both dense (semantic) and sparse (lexical/BM25) retrieval and fusing the results. Dense captures meaning/paraphrase; sparse captures exact terms, IDs, and codes that embeddings blur. Together they recover relevant chunks that either alone would miss.

Beginner  What does a reranker do? ›

It re-scores a candidate set with a cross-encoder that processes the query and document together, capturing their interaction far better than separate embeddings. You retrieve broadly (cheap bi-encoder) then rerank the top candidates (accurate cross-encoder) to push the best evidence to the top.

Intermediate  Bi-encoder vs cross-encoder — why use both? ›

Bi-encoders embed query and docs independently, so doc vectors are precomputed and retrieval is fast — but the comparison is coarse. Cross-encoders feed query+doc through the model together for an accurate relevance score, but can't be precomputed (too slow over a whole corpus). So: bi-encoder retrieves many candidates fast; cross-encoder reranks the small set precisely. Best of both.

Intermediate  Explain parent-document retrieval and the problem it solves. ›

Small chunks match precisely but lack surrounding context; large chunks have context but match poorly (diluted embeddings). Parent-document indexes small child chunks for retrieval but returns the larger parent (section/page) to the LLM — precise matching plus full context for generation.

Senior  How do you reduce hallucination in a RAG system? ›

Layered defenses. Retrieval: raise recall (hybrid, expansion) and precision (rerank) so the right evidence is present and on top; score threshold + "I don't know" when nothing qualifies. Prompt: instruct grounding only, require citations. Post-generation: a faithfulness judge verifies every claim maps to context, with a self-correction retry on unsupported claims. Measurement: track faithfulness on an eval set so regressions are caught. No single trick — defense in depth.

Senior  When is Graph RAG worth its complexity? ›

When questions are about relationships or require global synthesis across many documents — multi-hop ("how is A connected to C through B?"), entity-centric summaries ("everything about supplier X"), or "themes across the whole corpus." Vector search retrieves locally similar chunks but can't traverse relationships. Graph RAG extracts an entity/relationship graph and retrieves connected subgraphs or community summaries. For typical fact-lookup Q&A it's overkill — chunking + hybrid + rerank wins on ROI.

Staff · design  Design an evaluation-driven RAG improvement program. ›

Establish a golden eval set (representative questions with gold answers + gold source ids), refreshed as the corpus evolves. Define metrics: context recall & precision (retrieval), faithfulness & answer relevance (generation). Build an offline harness (RAGAS-style) that runs on every change and gates deploys (CI). Treat each technique — chunk size, hybrid, rerank, expansion — as an experiment with a hypothesis and an A/B against the baseline; keep only statistically meaningful wins, watching the latency/cost budget. Add online signals (thumbs, citation-click-through, "I don't know" rate) and sample real failures back into the eval set. Pair with observability/tracing (Module 14) so you can attribute regressions to a stage. The deliverable isn't a clever pipeline — it's a measurement loop that compounds.

11 · Knowledge check

1 · Why fuse hybrid results with Reciprocal Rank Fusion instead of adding raw scores?
Cosine similarities and BM25 scores live in different ranges, so summing them is meaningless. RRF uses rank position (1/(k+rank)), which is scale-free and robust.
2 · A reranker uses a cross-encoder because it…
Cross-encoders see query+doc jointly, so they model relevance far better than independently-embedded vectors — at the cost of speed, which is why they only rerank a small candidate set.
3 · Your users search by exact error codes and vector search keeps missing them. Best fix?
Dense embeddings blur exact tokens like "E-4042". Sparse/BM25 matches them literally; hybrid (dense+sparse, fused) recovers both meaning and exact terms.
4 · What must exist before you start adding advanced RAG techniques?
Without an eval set (recall, precision, faithfulness, relevance), you're tuning blind. Every technique should be an A/B you keep only if the metrics improve.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q State the two-stage retrieve-then-rerank pattern and why it's efficient. ›
Cheap high-recall retrieval (hybrid bi-encoder) fetches ~50 candidates; an expensive high-precision cross-encoder reranks just those to the best 5. You only run the slow model on a small set, getting precision without scoring the whole corpus.
Q Write the RRF scoring formula. ›
score(d) = Σ over retrievers of 1 / (k + rank_i(d)), with k ≈ 60. Higher (better) rank in any list raises the fused score.
Q What problem does parent-document retrieval solve? ›
The small-vs-large chunk dilemma: index small chunks for precise matching but return their larger parent section to the LLM for full context.
Q Name the four core RAG evaluation metrics. ›
Context recall, context precision (retrieval); faithfulness, answer relevance (generation).
Q When does Agentic RAG beat single-shot retrieval? ›
When the question needs adaptive/iterative retrieval — rewriting the query, choosing among tools/indexes, retrieving in multiple hops, and judging whether enough evidence has been gathered before answering.
