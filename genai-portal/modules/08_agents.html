Agentic AI

So far the LLM has been a one-shot function: prompt in, answer out. An agent breaks that frame. It runs the model in a loop, lets it use tools (search, code, APIs, your RAG pipeline), observe the results, and decide the next step — until the task is done. This is how an LLM goes from "answers questions" to "gets things done," and it's the conceptual core of MCP, LangGraph, CrewAI, and every capstone agent.

⏱ ~65 min read Difficulty · Intermediate→Advanced Prereq · Modules 01, 06–07

1 · Executive summary

An agent is an LLM placed in a control loop where it can call tools and act on their results. Instead of producing a final answer in one pass, the model reasons about what to do, takes an action (calls a tool), observes the outcome, and repeats. Give it a goal and a toolbox, and it figures out the steps. The dominant pattern, ReAct (Reason + Act), interleaves thinking and acting; planning, reflection, and memory make it more capable and reliable.

❓

Why it exists

Many tasks need actions and fresh data, not just text — query a DB, run code, call an API, retrieve docs, then synthesize. A single forward pass can't do that.

🔧

Problem it solves

Multi-step, tool-using, dynamic tasks where the right next step depends on what previous steps returned.

🌐

Where it fits

The behavioral core. MCP (09) standardizes tools; LangGraph (12) orchestrates the loop; CrewAI (13) coordinates many agents.

🔑
The one idea to remember

An agent is a while-loop around an LLM with tools. Reason → act (call tool) → observe → repeat until done. Everything else — planning, reflection, memory — is structure layered on that loop to make it smarter and safer.

2 · First-principles breakdown

What — an LLM that can do, not just say

Picture a smart but blindfolded analyst in a locked room: brilliant reasoning, but no access to your data and no way to act. That's a bare LLM. An agent hands that analyst a phone (call APIs), a calculator (run code), a filing cabinet (retrieval), and a notepad (memory) — and lets them work step by step. The model decides which tool to use; your code executes it and feeds back the result.

Why — the world isn't in the weights

LLMs are frozen, stateless, and can't compute reliably or fetch live data (Modules 01–02). Tools fill exactly those gaps: a calculator for arithmetic, search for current facts, a database for your data, code execution for logic. The loop lets the model use a tool's output to decide its next action — which is what "agency" means in practice.

🔄
Why a loop, not one shot?

Because the model can't know in advance what a tool will return. "Search for X" might yield a result that demands a follow-up query, or reveal the question was ambiguous. The loop lets the agent adapt to observations — exactly like a person debugging: try something, see what happens, adjust.

How — the agent loop, step by step

1 · Receive goal + tools

The agent gets the user's objective and a list of available tools (name, description, input schema).

2 · Reason (Thought)

The LLM decides what to do next given the goal and history so far.

3 · Act (Action)

It emits a structured tool call (which tool, what arguments).

4 · Observe

Your runtime executes the tool and feeds the result back into the context.

5 · Loop or finish

Repeat from step 2 until the model decides it has enough to answer — then it emits the final response.

4 · Deep technical explanation

Tool calling — how the model "uses" a tool

The model can't run code; it emits a structured request to. Modern LLMs are trained for native tool/function calling: you pass tool schemas (name, description, JSON parameter spec); the model returns a JSON object naming a tool and its arguments; your runtime executes it and returns the result as a new message. The description and schema quality directly determine whether the model picks the right tool — tool design is prompt engineering.

ReAct vs plan-and-execute

PatternHowBest for
ReActInterleave reason → act → observe, one step at a timeExploratory, dynamic tasks where steps depend on results
Plan-and-executeMake a full plan up front, then execute steps (optionally re-plan)Predictable multi-step tasks; cheaper (less re-reasoning)
ReflectionAfter acting, critique own output and retry/improveQuality-critical tasks (code, writing) where self-review helps

Planning & decomposition

For complex goals, the agent first decomposes the task into sub-goals (a plan), then executes them — sometimes spawning sub-agents (Module 13). Planning reduces flailing: an explicit plan keeps the agent on track instead of greedily reacting step by step and losing the thread.

Reflection — the self-correction loop

Reflection adds a critique step: after producing an action or answer, the agent (or a separate critic) evaluates it — "did this actually work? is the code correct? did I answer the question?" — and revises if not. This is the agentic analogue of the faithfulness check in Module 07 and dramatically improves reliability on hard tasks.

Memory in depth

  • Working memory — the current context window; the live scratchpad of thoughts/observations for this task.
  • Short-term memory — the conversation/episode history. As it overflows the window, summarize older turns (compression, Module 07) to preserve gist.
  • Long-term memory — persistent facts/preferences stored in a vector DB and retrieved when relevant. This is literally RAG over the agent's own past. It's how an assistant "remembers you" across sessions.

Agent failure modes (know these cold)

🔁 Loops & flailing

Agent repeats the same failing action. Fix: max-step limit, loop detection, force a re-plan or escalate.

🛠️ Wrong-tool / bad-args

Picks the wrong tool or malformed arguments. Fix: better tool descriptions, schema validation, retry with the error message.

💸 Cost / latency blowup

Each step is an LLM call; long loops get slow and expensive. Fix: step budgets, caching, cheaper models for sub-steps.

🎭 Hallucinated tool results

Model "imagines" a tool output instead of waiting for the real one. Fix: enforce that observations come only from real execution; never let the model fabricate them.

🧭 Goal drift

Loses sight of the original objective mid-loop. Fix: re-inject the goal each step; planning; reflection checkpoints.

⚠️ Unsafe actions

Destructive tool calls (delete, send, pay). Fix: human-in-the-loop approval (Module 12), allow-lists, dry-run, and least-privilege tools.

Agent evaluation

Agents are harder to evaluate than RAG because they're multi-step and non-deterministic. Measure:

DimensionWhat to measure
Task successDid it achieve the goal? (final-state check / golden outcome)
Trajectory qualityWere the tool choices sensible? Any wasted/redundant steps?
Tool-call correctnessRight tool, valid args, correct interpretation of results
EfficiencySteps, tokens, cost, latency to completion
SafetyDid it avoid unsafe/destructive actions and respect guardrails?

6 · Practical Python

Minimal — a hand-rolled ReAct loop (no framework)

react_minimal.py
import json
from openai import OpenAI

client = OpenAI()

def calculator(expr): return str(eval(expr))   # demo only — never eval untrusted input!
def get_weather(city): return f"{city}: 22°C, clear"
TOOLS = {"calculator": calculator, "get_weather": get_weather}

schema = [
  {"type": "function", "function": {"name": "calculator",
   "description": "Evaluate a math expression",
   "parameters": {"type": "object", "properties": {"expr": {"type": "string"}}, "required": ["expr"]}}},
  {"type": "function", "function": {"name": "get_weather",
   "description": "Current weather for a city",
   "parameters": {"type": "object", "properties": {"city": {"type": "string"}}, "required": ["city"]}}},
]

def agent(goal, max_steps=6):
    msgs = [{"role": "user", "content": goal}]
    for _ in range(max_steps):                       # step budget = loop guard
        r = client.chat.completions.create(model="gpt-4o-mini",
                messages=msgs, tools=schema, temperature=0)
        m = r.choices[0].message
        if not m.tool_calls:                        # no tool → final answer
            return m.content
        msgs.append(m)
        for call in m.tool_calls:                     # execute each tool call
            args = json.loads(call.function.arguments)
            result = TOOLS[call.function.name](**args)   # REAL execution
            msgs.append({"role": "tool", "tool_call_id": call.id, "content": result})
    return "Stopped: step budget exhausted."     # failure-mode guard

print(agent("What is 18% of 2400, and the weather in Mumbai?"))

Intermediate — add reflection & long-term memory

reflective_agent.py
class ReflectiveAgent:
    def __init__(self, llm, tools, memory_store):
        self.llm, self.tools, self.memory = llm, tools, memory_store

    def run(self, goal, max_steps=8):
        # 1 · recall long-term memory relevant to this goal (RAG over past)
        recalled = self.memory.search(goal, k=3)
        context = "\n".join(m.text for m in recalled)
        answer = self._react_loop(goal, context, max_steps)
        # 2 · reflect: critique and optionally redo
        critique = self.llm.complete(
            f"Does this fully answer '{goal}'? If not, say what's missing.\n{answer}")
        if "missing" in critique.lower():
            answer = self._react_loop(f"{goal}\nFix: {critique}", context, max_steps)
        # 3 · write durable facts back to long-term memory
        self.memory.add(f"Q:{goal} A:{answer}")
        return answer

Production — a guarded agent runtime (budgets, validation, safety)

agent_runtime.py
import time, logging
from dataclasses import dataclass, field

log = logging.getLogger("agent")

@dataclass
class Budget:
    max_steps: int = 10
    max_tokens: int = 50_000
    used_tokens: int = 0

class GuardedAgent:
    """Production loop with step/token budgets, arg validation, safe tools, tracing."""
    UNSAFE = {"delete", "drop", "send_payment"}   # require approval

    def __init__(self, llm, tools, validators, approver=None):
        self.llm, self.tools, self.validators, self.approver = llm, tools, validators, approver

    def _exec_tool(self, name, args, trace):
        if name not in self.tools:
            return f"ERROR: unknown tool {name}"          # wrong-tool guard
        try:
            self.validators[name](args)                  # schema/arg validation
        except Exception as e:
            return f"ERROR: invalid args: {e}"            # feed error back → self-correct
        if name in self.UNSAFE and self.approver:
            if not self.approver(name, args):              # human-in-the-loop
                return "DENIED by approver"
        result = self.tools[name](**args)                # real execution only
        trace.append({"tool": name, "args": args, "result": result})
        return result

    def run(self, goal, budget=None):
        budget = budget or Budget()
        msgs, trace = [{"role": "user", "content": goal}], []
        for step in range(budget.max_steps):       # loop/flailing guard
            if budget.used_tokens > budget.max_tokens:
                return {"answer": "Budget exhausted.", "trace": trace}
            decision = self.llm.decide(msgs, list(self.tools))
            budget.used_tokens += decision.tokens
            if decision.is_final:
                log.info("agent.done", extra={"steps": step, "trace_len": len(trace)})
                return {"answer": decision.content, "trace": trace}
            obs = self._exec_tool(decision.tool, decision.args, trace)
            msgs += [decision.message, {"role": "tool", "content": str(obs)}]
        return {"answer": "Max steps reached.", "trace": trace}
🛡️

Every failure mode from §4 maps to a guard here: step budget (loops), token budget (cost), validators (bad args), the UNSAFE set + approver (unsafe actions), and the trace (evaluation + observability, Module 14). Agents without guards are incidents waiting to happen.

7 · Ollama implementation

Local models can run agents too — but pick a model trained for tool calling. Llama 3.x and Qwen 2.5 support native function calling via Ollama's tools API.

local_agent.py
import ollama, json

def get_stock(item): return json.dumps({"item": item, "qty": 42})

tools = [{"type": "function", "function": {
    "name": "get_stock", "description": "Inventory count for an item",
    "parameters": {"type": "object", "properties": {"item": {"type": "string"}}, "required": ["item"]}}}]

msgs = [{"role": "user", "content": "How many pump valves are in stock?"}]
r = ollama.chat(model="qwen2.5", messages=msgs, tools=tools)

for call in r["message"].get("tool_calls", []):
    name = call["function"]["name"]
    args = call["function"]["arguments"]
    obs = get_stock(**args)                      # real execution
    msgs += [r["message"], {"role": "tool", "content": obs}]

final = ollama.chat(model="qwen2.5", messages=msgs)
print(final["message"]["content"])
🧪

Smaller local models are weaker at multi-step tool use than frontier models — they loop and mis-call more. Keep tool sets small, descriptions crisp, and step budgets tight when running agents locally. This is a great place to measure the failure modes from §4.

8 · Industry perspective

🚀 Startups

Build narrow, reliable agents: one job, a handful of well-tested tools, tight budgets. "Agentic" workflows that are mostly deterministic with one LLM decision point ship faster and break less than open-ended autonomy.

🏢 Enterprises

Demand guardrails, human-in-the-loop for consequential actions, full trace logging for audit, least-privilege tools, and rigorous eval before any agent touches production systems. Safety and observability are non-negotiable.

The "how agentic?" spectrum

Fixed workflowcode calls LLM at set points
RouterLLM picks one of N branches
Tool-using agentReAct loop, bounded tools
Autonomous agentopen-ended, self-directed (risky)

The senior instinct: use the least agency that solves the problem. More autonomy = more capability but exponentially more failure surface, cost, and unpredictability. Most production "agents" are closer to the left of this spectrum than the hype implies.

Common production challenges: non-determinism breaking tests, cost/latency of multi-step loops, tool reliability (an agent is only as good as its tools), prompt-injection via tool results, and proving safety before launch.

9 · Top mistakes engineers make

❌ No step/cost budget

Unbounded loops burn money and hang. Always cap steps and tokens, with a graceful stop.

❌ Vague tool descriptions

The model picks tools from their descriptions. Poor descriptions = wrong-tool errors. Tool design is prompt engineering.

❌ Trusting hallucinated observations

Never let the model invent tool outputs. Observations must come only from real execution.

❌ Too much autonomy too soon

Open-ended agents are unpredictable. Start with the least agency that works; add autonomy only when measured.

❌ No human-in-the-loop for risky actions

Delete/send/pay must require approval. Least-privilege tools + an approver gate.

❌ Not logging the trace

Without the full thought/action/observation trace you can't debug, evaluate, or audit. Trace everything.

10 · Interview preparation

Beginner  What is an LLM agent? ›

An LLM run in a control loop where it can call tools and act on their results. Instead of one-shot text output, it reasons about the next step, calls a tool, observes the result, and repeats until the goal is met — letting it perform multi-step, tool-using, dynamic tasks.

Beginner  What is ReAct? ›

Reason + Act: a pattern where the agent interleaves Thought (reason about the next step), Action (call a tool), and Observation (the tool's result), looping until it can give a final answer. The interleaving lets it adapt each step to what tools return.

Intermediate  How does tool/function calling actually work? ›

You pass the model tool schemas (name, description, JSON parameter spec). The model, trained for this, returns a structured request naming a tool and its arguments instead of plain text. Your runtime executes the tool and returns the result as a new message. The model never runs anything itself — it requests; your code executes. Description/schema quality drives correct tool selection.

Intermediate  Explain the three layers of agent memory. ›

Working memory: the current context window — the live scratchpad for this task. Short-term memory: the conversation/episode history, summarized as it grows to fit the window. Long-term memory: persistent facts stored in a vector DB and retrieved when relevant — essentially RAG over the agent's own past, enabling cross-session recall.

Senior  An agent keeps looping on the same failed action. How do you diagnose and fix it? ›

Inspect the trace: it's usually a tool returning an error the model can't act on, a missing capability, or an ambiguous goal. Fixes: enforce a max-step budget with graceful stop; add loop detection (repeated identical actions → break/re-plan); feed tool errors back clearly so the model can self-correct; improve the failing tool's description or add the missing tool; add a reflection/re-plan step; escalate to a human after N failures. Long-term, add this case to the agent eval set.

Senior  How do you evaluate an agent given non-determinism? ›

Don't assert exact outputs. Measure: task success (final-state/golden-outcome check), trajectory quality (sensible tool choices, no wasted steps — often LLM-as-judge over the trace), tool-call correctness (right tool, valid args), efficiency (steps/tokens/cost/latency), and safety (no unsafe actions). Run a suite of representative tasks repeatedly to capture variance, gate deploys on success rate, and sample production failures back into the suite.

Staff · design  Design a safe agent platform for internal employees to automate workflows. ›

Core: a guarded agent runtime (step/token budgets, arg validation, full trace logging). Tools as a governed registry with least-privilege scopes, schemas, and per-tool risk classification; consequential actions (write/delete/send/pay) require human-in-the-loop approval and dry-run. Safety: input/output guardrails, prompt-injection defenses on tool results, allow-lists, sandboxed code execution. Observability: every trace logged for audit + replay; cost/latency per run; alerting on loops and budget breaches. Evaluation: a task suite gating any new tool/agent in CI; staged rollout. Memory: per-user long-term memory with access controls and PII handling. Start agents on the low-autonomy end of the spectrum and increase autonomy only as eval evidence accrues. Governance and traceability are the product, not the model.

11 · Knowledge check

1 · In ReAct, what immediately follows an "Action"?
The loop is Thought → Action → Observation. The runtime executes the tool and feeds the real result back as the observation, which informs the next Thought.
2 · When the model emits a tool call, who runs the tool?
The model only emits a structured request (tool name + args). Your code runs the tool and returns the real output. Letting the model fabricate the result is a classic failure mode.
3 · Long-term agent memory is essentially…
Long-term memory stores facts in a vector DB and retrieves them when relevant — exactly the embeddings + retrieval pattern from Modules 04–06, pointed at the agent's history.
4 · Which is the single most important guard against runaway cost?
Each step is an LLM call. Without a step/token budget, a looping agent runs indefinitely, burning money and time. Bound it and stop gracefully.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q Describe the agent loop in one sentence. ›
An LLM repeatedly reasons about the next step, calls a tool, observes the real result, and loops until it decides it can give a final answer.
Q ReAct vs plan-and-execute — one line each. ›
ReAct interleaves reason/act/observe one step at a time (adaptive). Plan-and-execute makes a full plan up front then executes it (cheaper, predictable).
Q What does a reflection step add? ›
A self-critique: the agent (or a critic) evaluates its own output ("did this work / answer the question?") and revises if not — boosting reliability on hard tasks.
Q Name four agent failure modes and one guard each. ›
Loops→step budget; wrong-tool/bad-args→validation + good descriptions; cost blowup→token budget; hallucinated observations→real execution only; goal drift→re-inject goal/plan; unsafe actions→human approval. (Any four.)
Q Why not assert exact outputs when evaluating agents? ›
Agents are multi-step and non-deterministic. Measure task success, trajectory quality, tool-call correctness, efficiency, and safety instead — over repeated runs to capture variance.
