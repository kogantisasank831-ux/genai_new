Module 10 · Frameworks track

LangChain

You've now built LLM calls, embeddings, RAG, and agents largely from scratch — which is the right way to understand them. LangChain is the framework that packages those patterns into composable, swappable building blocks so you stop rewriting glue. Its modern core is small and elegant: the Runnable interface and the LCEL pipe operator. Learn those two ideas and the rest is library.

⏱ ~55 min read Difficulty · Intermediate Prereq · Modules 01, 04–07

1 · Executive summary

LangChain is an orchestration framework for LLM applications. Its modern foundation is the Runnable — a standard interface (invoke, batch, stream, async variants) implemented by every component (prompts, models, parsers, retrievers). Because they share that interface, you compose them with the pipe operator (|) into a chain: prompt | model | parser. This is LCEL (LangChain Expression Language), and it gives you streaming, batching, async, and retries for free.

❓

Why it exists

Every LLM app re-implements prompting, retries, RAG wiring, memory, streaming. LangChain standardizes these as interchangeable parts.

🔧

Problem it solves

Composition. Swap a model, add a retriever, change a parser — without rewriting the pipeline. Provider-agnostic by design.

🌐

Where it fits

The "glue + components" layer. Pairs with LlamaIndex (11) for data, LangGraph (12) for stateful loops, and your vector DB (05).

🔑
The one idea to remember

Everything is a Runnable; you compose them with |. chain = prompt | model | parser. Once something is a Runnable, it gets streaming, batching, async, and retries automatically. Master LCEL and LangChain stops being magic.

2 · First-principles breakdown

What — Unix pipes for LLM components

In the shell, cat file | grep error | sort | uniq chains small programs that each read input and produce output. LCEL is the same idea for AI: each component takes an input and returns an output, and | feeds one into the next. prompt | model | parser means "format the prompt, send it to the model, parse the result." The data flows left to right.

🔧
The Runnable contract

A Runnable promises five methods: invoke(x) (one input → one output), batch([x]) (many in parallel), stream(x) (yield tokens as they arrive), plus async ainvoke/astream. Because every piece implements this same contract, they're freely interchangeable and composable — that's the entire design.

Why — composition beats monoliths

Without a framework, a RAG pipeline is a tangle of bespoke functions. As requirements change (new model, add reranking, stream the output, run 100 queries in parallel), you rewrite plumbing. LCEL turns each concern into a swappable Runnable. Want streaming? It's already there. Want to swap OpenAI for a local Ollama model? Change one line — the chain is unaffected because the interface is identical.

How — three building blocks

PromptTemplatefills variables → a prompt
ChatModelprompt → model response
OutputParserresponse → str / JSON / object
Result

Add a retriever Runnable in front and you have RAG; add memory and you have a chatbot. Same composition rules throughout.

4 · Deep technical explanation

LCEL composition primitives

ConstructMeaning
a | b (RunnableSequence)Run a, feed its output to b (left→right pipe)
{"x": a, "y": b} (RunnableParallel)Run a and b on the same input concurrently; output a dict
RunnablePassthrough()Pass the input through unchanged (often inside a parallel dict)
RunnableLambda(fn)Wrap any plain function as a Runnable
.with_retry() / .with_fallbacks()Add resilience to any Runnable declaratively

Why LCEL gives you streaming & batching "for free"

Because the composition graph is declarative, LangChain knows the structure of your pipeline. So when you call .stream() on a chain, it streams from the model node and pipes tokens through downstream parsers; .batch() fans inputs out in parallel; async works end-to-end. You didn't write any of that concurrency — the framework derives it from the Runnable graph.

Retrievers — RAG as a Runnable

A retriever is a Runnable that takes a query string and returns documents. Any vector store (Module 05) exposes one via .as_retriever(). Because it's a Runnable, it drops straight into a chain — and you can wrap it with reranking, multi-query, or compression (Module 07) and it's still just a Runnable.

Memory — the modern way

Older LangChain had bespoke "Memory" classes; the modern approach is message history wrapped around a chain with RunnableWithMessageHistory (and for anything stateful/branching, you graduate to LangGraph's checkpointed state, Module 12). The principle is unchanged from Module 08: working/short-term history in context, long-term in a store.

Output parsing & structured output

Parsers turn raw text into usable types: StrOutputParser (plain text), JsonOutputParser, or .with_structured_output(PydanticModel) to force a typed schema (great for extraction). Structured output + temperature 0 (Module 01) is the reliable pattern for machine-consumable results.

Tradeoffs & limitations

Strengths

  • Provider-agnostic, swappable components
  • Free streaming/batching/async/retries
  • Huge integration ecosystem
  • Tight LangSmith tracing (Module 14)

Limits

  • Abstraction can hide what's happening (debug the trace)
  • Fast-moving API; pin versions
  • LCEL is for DAGs — loops/branches need LangGraph
  • Easy to over-abstract a simple call

6 · Practical Python

Minimal — your first LCEL chain

chain_minimal.py
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("Explain {topic} in 2 sentences.")
model  = ChatOpenAI(model="gpt-4o-mini", temperature=0)
parser = StrOutputParser()

chain = prompt | model | parser            # LCEL composition

print(chain.invoke({"topic": "vector databases"}))   # one call
print(chain.batch([{"topic": "RAG"}, {"topic": "MCP"}]))  # parallel
for tok in chain.stream({"topic": "agents"}):       # streaming, free
    print(tok, end="", flush=True)

Intermediate — a full RAG chain with a retriever

rag_chain.py
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import FAISS

# 1 · index documents (Modules 04–05)
store = FAISS.from_texts(
    ["Returns accepted within 30 days.", "Free shipping over $50."],
    embedding=OpenAIEmbeddings())
retriever = store.as_retriever(search_kwargs={"k": 2})   # a Runnable

prompt = ChatPromptTemplate.from_template(
    "Answer ONLY from context, cite, or say you don't know.\n"
    "Context:\n{context}\n\nQuestion: {question}")

def format_docs(docs): return "\n\n".join(d.page_content for d in docs)

# 2 · compose: retriever runs in parallel with the passed-through question
chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4o-mini", temperature=0)
    | StrOutputParser()
)
print(chain.invoke("How long do I have to return something?"))

Intermediate+ — conversational memory & structured output

memory_and_structured.py
from langchain_core.chat_history import InMemoryChatMessageHistory
from langchain_core.runnables.history import RunnableWithMessageHistory
from pydantic import BaseModel, Field

# --- memory: wrap a chain with per-session message history ---
_sessions = {}
def get_history(session_id):
    return _sessions.setdefault(session_id, InMemoryChatMessageHistory())

chat = RunnableWithMessageHistory(chain, get_history)    # remembers per session
chat.invoke({"question": "My name is Sam."},
           config={"configurable": {"session_id": "u1"}})

# --- structured output: force a typed schema ---
class Ticket(BaseModel):
    priority: str = Field(description="low | medium | high")
    summary: str

extractor = ChatOpenAI(temperature=0).with_structured_output(Ticket)
print(extractor.invoke("Server is down for all users!!"))
# Ticket(priority='high', summary='Server outage affecting all users')

Production — a resilient, traced, swappable RAG service

prod_chain.py
import os
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

def build_model():
    # Provider-agnostic: same chain, local or hosted, by config.
    if os.getenv("LLM_BACKEND") == "local":
        from langchain_ollama import ChatOllama
        return ChatOllama(model="qwen2.5", temperature=0)
    from langchain_openai import ChatOpenAI
    return ChatOpenAI(model="gpt-4o-mini", temperature=0)

def build_chain(retriever, prompt):
    model = build_model().with_retry(stop_after_attempt=3)   # resilience
    return (
        {"context": retriever, "question": RunnablePassthrough()}
        | prompt | model | StrOutputParser()
    )

# Tracing: set LANGCHAIN_TRACING_V2=true + LANGCHAIN_API_KEY and every
# invoke is captured in LangSmith — inputs, outputs, latency, token cost.
# Serve with LangServe or wrap chain.invoke() in your FastAPI route.
🔭

The biggest production win is tracing: with LangSmith, every chain run records the exact prompt, retrieved docs, model output, latency, and cost at each step — turning the "abstraction hides things" weakness into full observability. We build on this in Module 14.

7 · Ollama implementation

LangChain's provider-agnostic design means going local is a one-import change — the chain itself never changes.

langchain_ollama.py
# pip install langchain-ollama
from langchain_ollama import ChatOllama, OllamaEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_community.vectorstores import FAISS

# Local model + local embeddings → fully offline LangChain RAG
emb = OllamaEmbeddings(model="nomic-embed-text")
store = FAISS.from_texts(["Ollama serves on port 11434."], embedding=emb)

prompt = ChatPromptTemplate.from_template("Context:{context}\nQ:{question}")
chain = (store.as_retriever() | (lambda d: d[0].page_content)
         | (lambda c: {"context": c, "question": "What port?"})
         | prompt | ChatOllama(model="qwen2.5", temperature=0) | StrOutputParser())
print(chain.invoke("What port?"))
🔁

This is the core promise: the only difference between the cloud and local versions is which integration package you import. The chain structure, streaming, batching, and retries are identical — write once, run anywhere.

8 · Industry perspective

🚀 Startups

Use LangChain to go from idea to RAG/agent demo fast, leaning on its loaders, vector-store integrations, and LangSmith tracing. Watch for over-abstraction — drop to raw SDK calls where the chain adds no value.

🏢 Enterprises

Value provider-agnosticism (avoid lock-in, switch models by config), the integration breadth, and LangSmith for eval + observability + prompt management. Pin versions and wrap chains behind their own service boundary.

LangChain vs LlamaIndex vs LangGraph — the boundary

Use…When you need…
LangChain (LCEL)General composition of prompts/models/retrievers/tools as a DAG; provider-agnostic glue
LlamaIndex (11)Data-centric RAG: ingestion, indexing strategies, advanced retrievers, query engines
LangGraph (12)Stateful, cyclic, branching flows — agents, loops, human-in-the-loop (LCEL is DAG-only)

Common production challenges: breaking changes across versions (pin them), abstraction obscuring the real prompt/tokens (use tracing), over-engineering simple calls, and dependency bloat from langchain-community (import only what you use).

9 · Top mistakes engineers make

❌ Using LangChain for a single LLM call

If you just need one completion, call the SDK. LCEL pays off when you compose, stream, batch, or swap parts.

❌ Not pinning versions

The API moves fast. Pin exact versions and read migration notes before upgrading.

❌ Debugging blind

Abstractions hide the real prompt. Turn on LangSmith tracing (or print intermediate steps) to see what's actually sent.

❌ Forcing loops into LCEL

LCEL is a DAG — no cycles. Agents/retries/branches belong in LangGraph (Module 12).

❌ Ignoring structured output

Parsing free text with regex is fragile. Use with_structured_output for machine-consumable results.

❌ Old "Chain"/"Memory" classes

Legacy LLMChain/ConversationChain are deprecated. Use LCEL + RunnableWithMessageHistory.

10 · Interview preparation

Beginner  What is LangChain and why use it? ›

An orchestration framework for LLM apps that standardizes components (prompts, models, parsers, retrievers) behind a common Runnable interface, so you compose them with the pipe operator and swap parts without rewriting plumbing. It provides provider-agnosticism, free streaming/batching/async/retries, a large integration ecosystem, and tracing via LangSmith.

Beginner  What does prompt | model | parser mean? ›

An LCEL chain: format the prompt with input variables, send it to the model, then parse the model's response (e.g. to a plain string). The | pipes each step's output into the next, like Unix pipes. The whole chain is itself a Runnable you can invoke/batch/stream.

Intermediate  What is the Runnable interface and why does it matter? ›

The common contract — invoke, batch, stream, and async variants — implemented by every LangChain component. Because they all share it, any component is interchangeable and composable, and the framework can derive streaming, batching, and async automatically from the composition graph. It's the foundation that makes LCEL work.

Intermediate  How do you build a RAG chain in LCEL? ›

Run the retriever and the question in parallel into a dict ({"context": retriever | format_docs, "question": RunnablePassthrough()}), pipe that into a prompt template, then into the model, then an output parser. The parallel dict fetches context while passing the question through; the pipes sequence prompt→model→parse. The retriever comes from vectorstore.as_retriever() and is itself a Runnable.

Senior  When would you NOT use LangChain, or drop to raw SDK? ›

For a single, simple completion the abstraction adds dependency weight and indirection with no benefit — call the provider SDK. Drop to raw calls in hot paths where you need precise control over requests/streaming, or when debugging an issue the abstraction obscures. Use LangChain where composition, swappability, integrations, or tracing genuinely pay off; for cyclic/stateful agent flows, use LangGraph instead of bending LCEL. Match the tool to the complexity.

Senior  How do you make a LangChain app production-ready? ›

Pin exact package versions; enable LangSmith tracing for observability and prompt/version management; add .with_retry()/.with_fallbacks() for resilience and a fallback model; use structured output for machine-consumable results; keep prompts in a versioned registry; build an eval set (Module 07/14) gating prompt/chain changes; wrap chains behind your own service boundary (FastAPI/LangServe) so you can swap the framework if needed; and set temperature/timeouts/token budgets explicitly. Treat the chain as one component in your system, not the whole architecture.

Staff · design  Design a provider-agnostic LLM layer for an org using LangChain. ›

A thin internal library exposing chains built from langchain-core Runnables, with the model created by a factory driven by config (so local Ollama, OpenAI, or a gateway are interchangeable). Centralize: a versioned prompt registry; retry/fallback policies (primary → cheaper/local fallback); structured-output schemas; LangSmith tracing for per-team cost/latency and eval; and guardrails wrapping inputs/outputs (Module 14). Expose chains via a service boundary (LangServe/FastAPI) so consumers don't import LangChain directly — that keeps you free to swap or drop the framework. Gate every prompt/chain change on an eval set in CI. The goal: teams get composable, observable, swappable LLM components without lock-in to any single provider or even to LangChain itself.

11 · Knowledge check

1 · What does the pipe operator do in LCEL?
a | b builds a RunnableSequence: run a, pass its output to b. Like Unix pipes, data flows left to right through the composed Runnables.
2 · You call .stream() on a chain and tokens appear live. Who implemented the streaming?
Because every component is a Runnable and the graph is declarative, LangChain provides streaming/batching/async for free — you didn't write the concurrency.
3 · Your app needs an agent loop with retries and branching. Best tool?
LCEL composes acyclic pipelines. Stateful loops, conditional branches, and human-in-the-loop belong in LangGraph (Module 12).
4 · How do you switch a chain from OpenAI to a local Ollama model?
Both models implement the same Runnable interface, so only the model node changes. Provider-agnosticism is a core LangChain value.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q What are the five methods of the Runnable interface? ›
invoke, batch, stream, and their async variants ainvoke/astream (plus abatch). The shared contract is what makes components composable.
Q Write the shape of an LCEL RAG chain. ›
{"context": retriever | format_docs, "question": RunnablePassthrough()} | prompt | model | parser — parallel dict fetches context + passes the question, then sequence prompt→model→parse.
Q What is a retriever, in Runnable terms? ›
A Runnable that takes a query string and returns documents; obtained from a vector store via .as_retriever(), so it drops into any chain.
Q LangChain vs LangGraph — when each? ›
LangChain/LCEL for acyclic composition (prompt|model|retriever|parser). LangGraph for stateful, cyclic, branching flows — agents, loops, human-in-the-loop.
Q How do you add resilience to a chain declaratively? ›
.with_retry() for automatic retries and .with_fallbacks([...]) to fall back to another model/chain on failure — both available on any Runnable.

14 · Mini project — "Composable RAG Toolkit"

  • .with_retry() + a fallback model; LangSmith tracing toggle

Stretch
