Vector Databases

Module 04 gave you vectors. But comparing a query against ten million stored vectors one-by-one is far too slow for a live request. A vector database is purpose-built to answer "which vectors are closest to this one?" in milliseconds — by trading a sliver of accuracy for enormous speed using approximate nearest-neighbor indexes. This module is the difference between a toy and a system that scales.

⏱ ~55 min read Difficulty · Intermediate Prereq · Module 04

1 · Executive summary

A vector database stores embeddings plus metadata and serves k-nearest-neighbor (kNN) queries at scale. Its core trick is the Approximate Nearest Neighbor (ANN) index — most commonly HNSW — which finds almost the true nearest neighbors in logarithmic-ish time instead of scanning everything. You give up a small, tunable amount of recall to gain orders-of-magnitude lower latency.

❓

Why it exists

Brute-force similarity over millions of high-dim vectors is O(N·d) per query — too slow. ANN indexes make it fast enough for production.

🔧

Problem it solves

"Find the top-k closest vectors out of millions, with metadata filters, in single-digit milliseconds, while accepting live updates."

🌐

Where it fits

The storage + retrieval engine of RAG. It sits between embeddings (04) and RAG (06).

🔑
The one idea to remember

Vector search is a recall-vs-latency dial, not a binary. ANN trades exactness for speed; you tune index parameters to hit your recall target at your latency budget. Knowing which knobs to turn is the whole skill.

2 · First-principles breakdown

What — a search engine for geometry

A normal database finds rows by exact match or range on a column. A vector database finds rows by proximity in space: "give me the 5 vectors nearest to this one." Because there's no notion of "exactly equal" for 768 floats, the query is inherently a ranking-by-distance, not a lookup.

Why — brute force doesn't scale

The honest answer ("compare the query to every vector and sort") is called a flat / exact search. It's perfectly accurate but costs O(N × d) per query — for 10M vectors × 768 dims that's ~7.7 billion multiply-adds per request. Fine for thousands of vectors; impossible for millions at low latency. ANN indexes restructure the data so you only examine a tiny, well-chosen fraction.

🗺️
The intuition behind ANN

Think of finding the nearest coffee shop. You don't measure your distance to every shop on Earth — you start in your neighborhood and hop to closer and closer ones. HNSW does exactly this: build a graph of "who's near whom," then greedily walk toward the query. You might miss the true #1 occasionally, but you find great answers fast.

How — index once, search many

3 · Visual architecture

HNSW — the hierarchical small-world graph

HNSW builds a multi-layer graph. Top layers are sparse "express lanes" for big jumps; lower layers are dense for fine-grained search. A query enters at the top, greedily hops toward the target, then descends layer by layer — like zooming in on a map.

4 · Deep technical explanation

Index families

IndexHow it worksBest for
Flat (brute force)Compare to every vector; exact<~50k vectors, or a ground-truth baseline
HNSWMulti-layer proximity graph; greedy descentThe default — great recall/latency, fast queries
IVFCluster vectors; search only nearest clustersVery large sets where memory matters
IVF + PQIVF + product quantization (compress vectors)Billions of vectors on limited RAM

The HNSW knobs that decide everything

ParamControlsHigher =
MEdges per node (graph density)Better recall, more memory, slower build
ef_constructionEffort while building the graphBetter index quality, slower build (one-time)
ef_searchCandidates explored per queryBetter recall, higher latency — your live dial
🎛️

ef_search is your runtime recall/latency lever. Set M and ef_construction at build time for index quality; tune ef_search per query to trade latency for recall. Always measure recall against a flat-index ground truth before shipping.

Metadata filtering — the production reality

Real queries are "nearest neighbors where tenant = X and date > Y." There are two strategies, and the wrong one wrecks results:

  • Post-filtering: ANN first, then drop non-matching results. Fast, but if filters are selective you may get back zero matches (all top-k were filtered out).
  • Pre-filtering: restrict the search to matching vectors. More correct for selective filters; needs DB support (Qdrant, pgvector, ES do this well).

Distance metrics & quantization recap

Pick the metric your embedding model was trained for — usually cosine (or dot product on normalized vectors, Module 04). Many DBs also offer on-disk quantization (scalar/product) to shrink memory at a small recall cost — the storage analogue of model quantization from Module 03.

Tradeoffs & limitations

Strengths

  • Millisecond kNN over millions of vectors
  • Live inserts/updates/deletes
  • Metadata filtering + hybrid search

Limits

  • ANN ≠ exact (tunable miss rate)
  • HNSW is memory-hungry (graph in RAM)
  • Filtering interacts subtly with ANN recall
  • Re-indexing on embedding-model upgrades (Module 04)

6 · Practical Python

Minimal — FAISS in-memory index

faiss_minimal.py
import faiss, numpy as np

d = 384
vectors = np.random.rand(10000, d).astype("float32")
faiss.normalize_L2(vectors)                 # cosine via inner product

index = faiss.IndexFlatIP(d)               # exact baseline (inner product)
index.add(vectors)

q = vectors[:1]                            # query with the first vector
scores, ids = index.search(q, k=5)
print(ids[0], scores[0])              # nearest 5 (itself first)

Intermediate — HNSW with metadata filtering (Qdrant)

qdrant_intermediate.py
from qdrant_client import QdrantClient, models

client = QdrantClient(":memory:")        # or url="http://localhost:6333"
client.recreate_collection(
    "docs",
    vectors_config=models.VectorParams(size=384, distance=models.Distance.COSINE),
    hnsw_config=models.HnswConfigDiff(m=16, ef_construct=200),
)

client.upsert("docs", points=[
    models.PointStruct(id=1, vector=vec1, payload={"tenant": "acme", "year": 2024}),
    models.PointStruct(id=2, vector=vec2, payload={"tenant": "acme", "year": 2023}),
])

# Pre-filtered ANN: nearest neighbors WHERE tenant=acme AND year>=2024
hits = client.search(
    "docs", query_vector=query_vec, limit=5,
    search_params=models.SearchParams(hnsw_ef=128),    # ef_search
    query_filter=models.Filter(must=[
        models.FieldCondition(key="tenant", match=models.MatchValue(value="acme")),
        models.FieldCondition(key="year", range=models.Range(gte=2024)),
    ]),
)
for h in hits: print(h.id, h.score, h.payload)

Production — pgvector with a clean repository layer

pgvector_repo.py
# Postgres + pgvector: vectors live next to your relational data —
# one DB, transactional, easy joins/filters. Great default for many teams.
import psycopg
from pgvector.psycopg import register_vector

DDL = """
CREATE EXTENSION IF NOT EXISTS vector;
CREATE TABLE IF NOT EXISTS chunks (
  id bigserial PRIMARY KEY,
  doc_id text NOT NULL,
  tenant text NOT NULL,
  content text NOT NULL,
  embedding vector(384),
  model_version text NOT NULL          -- never mix models (Module 04)
);
CREATE INDEX IF NOT EXISTS chunks_hnsw
  ON chunks USING hnsw (embedding vector_cosine_ops)
  WITH (m = 16, ef_construction = 200);
"""

class VectorRepo:
    def __init__(self, dsn):
        self.conn = psycopg.connect(dsn, autocommit=True)
        register_vector(self.conn)
        self.conn.execute(DDL)

    def search(self, qvec, tenant, k=5, ef=128):
        self.conn.execute("SET hnsw.ef_search = %s", (ef,))   # runtime recall dial
        return self.conn.execute("""
            SELECT id, doc_id, content, 1 - (embedding <=> %s) AS score
            FROM chunks
            WHERE tenant = %s                    -- pre-filter
            ORDER BY embedding <=> %s             -- cosine distance operator
            LIMIT %s
        """, (qvec, tenant, qvec, k)).fetchall()
🧷

The <=> operator is pgvector's cosine distance. Storing model_version next to each vector enforces the Module 04 rule and makes re-embedding migrations auditable.

7 · Ollama & local stack

Pair local embeddings (Module 04) with a local vector DB for a 100% offline retrieval stack — nothing leaves the box.

local_stack.py
import ollama, numpy as np
from qdrant_client import QdrantClient, models

emb = lambda t: ollama.embeddings(model="nomic-embed-text", prompt=t)["embedding"]

qc = QdrantClient(":memory:")
qc.recreate_collection("local",
    vectors_config=models.VectorParams(size=768, distance=models.Distance.COSINE))

docs = ["Qdrant runs locally", "FAISS is a library not a server", "pgvector lives in Postgres"]
qc.upsert("local", points=[
    models.PointStruct(id=i, vector=emb(d), payload={"text": d}) for i, d in enumerate(docs)])

hits = qc.search("local", query_vector=emb("vector database inside SQL"), limit=2)
for h in hits: print(h.score, h.payload["text"])
# top hit: "pgvector lives in Postgres"
🐳

Run Qdrant locally with docker run -p 6333:6333 qdrant/qdrant, or just use Postgres+pgvector you already operate. Both are the offline retrieval backbone of the private-RAG capstones.

Choosing a vector store

pgvectorPostgres extensionYou already run Postgres; want transactions + joins + one DB to operate

PineconeManaged cloudYou want zero ops and pay for it

🚀 Startups

Default to pgvector (one fewer system) or FAISS for prototypes. Move to a dedicated DB only when scale, filtering, or hybrid needs demand it.

🏢 Enterprises

Often consolidate on ElasticSearch/OpenSearch (lexical + vector + ops maturity) or a dedicated DB for very large multi-tenant workloads. Care about sharding, replication, backups, and tenant isolation.

Common production challenges: recall silently dropping after a careless re-index, post-filtering returning empty results on selective filters, HNSW RAM blowing the budget at scale, hot-shard imbalance in multi-tenant setups, and forgetting to rebuild the index after bulk loads.

9 · Top mistakes engineers make

❌ Never measuring recall

ANN is approximate. Without a flat-index ground truth, you don't know what you're missing. Always measure recall@k.

❌ Post-filtering selective filters

Filtering after ANN can return zero results. Use pre-filtering when filters are selective.

❌ Wrong distance metric

Using L2 with a cosine-trained model (or unnormalized dot product) silently degrades ranking.

❌ HNSW for a tiny dataset

For <~50k vectors, flat/exact is fast and perfectly accurate. Don't over-engineer.

❌ Ignoring RAM cost

HNSW keeps the graph in memory. Plan capacity; use IVF+PQ or quantization for billions of vectors.

❌ Cranking ef_search globally

High ef_search everywhere tanks latency. Tune per-query/per-route to the recall you actually need.

10 · Interview preparation

Beginner  What is a vector database for? ›

To store embeddings and answer "which stored vectors are nearest to this query vector?" at scale and low latency, using approximate nearest-neighbor indexes — plus metadata filtering. It's the storage + retrieval engine of RAG.

Beginner  What does "approximate" mean in ANN? ›

The index may not return the exact true nearest neighbors every time — it returns very-close ones much faster than brute force. The miss rate (1 − recall) is small and tunable; you trade a little accuracy for big speed gains.

Intermediate  Explain HNSW and its key parameters. ›

A hierarchical navigable small-world graph: multiple layers, sparse on top for long hops, dense at the bottom holding all vectors. Search starts at the top and greedily descends toward the query — roughly O(log N). Params: M (edges/node → recall & memory), ef_construction (build-time quality), and ef_search (query-time candidates → the live recall/latency dial).

Intermediate  Pre-filtering vs post-filtering — when does it matter? ›

Post-filtering runs ANN then discards non-matches — fine for permissive filters, but selective filters can leave you with too few or zero results because the top-k were all filtered out. Pre-filtering restricts the search space to matching vectors first, giving correct top-k for selective filters. Use pre-filtering for tenant/permission/date constraints.

Senior  Recall is fine in tests but bad in prod. Diagnose. ›

Suspects: (1) production data distribution differs from test; (2) ef_search lowered for latency; (3) selective metadata filters + post-filtering starving results; (4) index not rebuilt after a bulk load, or stale after deletes; (5) embedding model/version drift between index and query (Module 04). Reproduce with a flat-index ground truth on prod queries, compare recall@k, and isolate the variable. Fix the specific cause — often raise ef_search or switch to pre-filtering.

Staff · design  Design vector search for 1B vectors, multi-tenant, <50ms p99. ›

1B × 768 floats won't fit one node in RAM as raw HNSW — use IVF+PQ or quantized HNSW to compress, and shard by tenant or hash with replicas for HA. Pre-filter by tenant (route to the tenant's shard for isolation + smaller search space). Tune ef_search per route to hit p99 under the recall SLO; add a small re-rank stage (Module 07) on the top candidates to recover precision lost to quantization. Operate: background re-index jobs for model upgrades (versioned vectors), warm caches, monitor recall@k continuously with a sampled ground-truth job, and capacity-plan RAM. Consider a tiered design: quantized index for breadth, exact rescoring on finalists.

11 · Knowledge check

1 · What is the core tradeoff an ANN index makes?
ANN returns almost the true nearest neighbors much faster than brute force. The miss rate is small and tunable via parameters like ef_search.
2 · Which HNSW parameter is your runtime recall/latency dial?
M and ef_construction are set at build time. ef_search controls how many candidates each query explores — raise it for recall, lower it for latency.
3 · A query filters on a rare tenant and returns 0 results despite matching docs existing. Likely cause?
With selective filters, ANN's top-k may contain no matching rows, so post-filtering yields nothing. Pre-filtering restricts the search to matching vectors and fixes this.
4 · You have 20,000 vectors. Best index choice?
At ~20k vectors, brute-force exact search is millisecond-fast and 100% accurate. ANN complexity only pays off at much larger scales.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q Why can't we just brute-force compare to every vector in production? ›
It's O(N·d) per query — billions of operations for millions of high-dim vectors — far too slow for live latency. ANN indexes restructure data so only a small fraction is examined.
Q Describe HNSW search in one sentence. ›
Enter a multi-layer proximity graph at the sparse top, greedily hop toward the query, and descend into denser layers until you reach the nearest neighbors — roughly O(log N).
Q When do you prefer pre-filtering over post-filtering? ›
When filters are selective (e.g. a specific tenant or recent date), so that searching only the matching subset avoids returning too few or zero results.
Q Name two ways to handle a billion vectors that won't fit in RAM as raw HNSW. ›
IVF+PQ (cluster + product quantization to compress), scalar/product quantization on the vectors, sharding across nodes, and/or exact rescoring of top candidates after a compressed first pass.
Q What must you always measure before trusting an ANN setup? ›
Recall@k against a flat/exact ground truth on representative queries — ANN is approximate, so you verify the miss rate is acceptable at your chosen latency.

14 · Mini project — "Vector Search Benchmark"
