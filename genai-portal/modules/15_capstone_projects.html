pdf-qa/
├── ingest/
│   ├── loader.py          # extract text + page numbers

│   ├── embedder.py        # Module 04 (version-tagged)

Data flow: upload → extract+chunk (page metadata) → embed → upsert. Query → embed → retrieve top-k w/ threshold → grounded prompt → answer + "Sources: file p.N".

Implementation plan: ① ingest one PDF → ② index → ③ retrieve+threshold → ④ grounded prompt with "I don't know" → ⑤ source citations in UI → ⑥ eval set.

Deployment: FastAPI + Streamlit (or React) containers; Qdrant/pgvector container; volume for the index.

🏭
Production considerations

Chunking tuned on an eval set; faithfulness output-guardrail; "I don't know" path; re-ingest on file update; per-query cost tracking; OCR fallback for scanned PDFs.

3 · Multi-PDF RAG

Intermediate  Q&A across a corpus of PDFs with per-document scoping and citations. Modules 05, 06, 07, 11

Architecture: same as PDF QA but every chunk carries doc_id + title metadata, enabling metadata pre-filtering (Module 05) so users can ask "across all docs" or "only in the 2024 handbook."

structure (delta vs #2)
multi-pdf-rag/
├── ingest/batch_ingest.py   # walk a folder, tag doc_id/title/page
├── core/
│   ├── store.py             # metadata filters (doc_id, date)
│   ├── router.py            # DocumentSummaryIndex routing (Module 11)
│   └── rag.py               # scoped retrieval + multi-doc citations
└── ui/                      # doc picker + scope toggle

Data flow: batch ingest tags each chunk → user picks scope (all / specific docs) → pre-filtered retrieval → answer cites which document + page.

Component design: a router (DocumentSummaryIndex, Module 11) picks relevant documents first for large corpora; parent-document retrieval (Module 07) preserves context.

Implementation plan: ① batch ingest w/ metadata → ② scoped retrieval (pre-filter) → ③ doc router for scale → ④ multi-doc citation formatting → ⑤ eval across documents.

Deployment: as #2 + a scheduled ingestion job for new/updated PDFs.

🏭
Production considerations

Pre-filter (never post-filter) for scoping; handle doc deletes/updates idempotently; model_version on vectors for re-embedding; router to avoid scanning thousands of docs per query.

├── retrieval/

│   └── ablation.py         # baseline → +hybrid → +rerank table

Data flow: query → dense & sparse retrieval in parallel → RRF fuse → cross-encoder rerank → grounded generation.

Component design: the AdvancedRetriever (Module 07) as a clean two-stage retrieve→rerank pipeline; ES can supply both lexical and dense in one engine.

Implementation plan: ① dense baseline + eval → ② add BM25 + RRF → ③ add reranker → ④ run the ablation to prove each lift → ⑤ keep what the metrics reward.

Deployment: vector store + (optional) ES; reranker model served alongside; eval gate in CI.

🏭
Production considerations

Reranker latency budget; RRF over raw-score fusion; measure recall@k and faithfulness; cache query vectors; never add a technique the eval doesn't reward (Module 07).

5 · Agent-Based EDA System

Intermediate→Adv.  Natural-language exploratory data analysis: ask questions about a dataset, get stats and plots. Modules 08, 12, 14

Figure 2 · A ReAct agent (Module 08) with data/stats/plot tools and a reflection step, orchestrated as a LangGraph (Module 12).
structure
eda-agent/
├── tools/
│   ├── data.py            # load_csv, get_schema, profile
│   ├── code_exec.py       # SANDBOXED pandas exec
│   └── plot.py            # matplotlib → image
├── graph/eda_graph.py     # LangGraph: agent ↔ tools, reflect (Module 12)
├── api/main.py
└── ui/                    # chat + rendered plots/tables

Data flow: question → agent inspects schema → writes/executes pandas in a sandbox → observes result → optionally plots → reflects → returns a narrated finding + artifacts.

Component design: tools return structured results; code_exec runs in a restricted sandbox (no filesystem/network); reflection node retries on errors.

Implementation plan: ① data/schema tools → ② sandboxed exec tool → ③ ReAct graph → ④ plotting → ⑤ reflection/retry → ⑥ step budget + trace.

Deployment: agent service + a locked-down code-exec sandbox (container with no egress); artifact storage for plots.

🛡️
Production considerations

Code execution is the risk — sandbox hard (no FS/network, resource limits, timeouts). Step budgets (Module 08); never eval raw; trace every tool call; verify results before reporting.

Deployment: agent service with a read-replica DB connection (least privilege); Postgres checkpointer for durable approvals.

🛡️
Production considerations

Read-only DB role; SELECT-only validation + row/time limits; human approval for every query (or auto-approve safe patterns); prompt-injection guard on the NL input (Module 14); full audit trace.

7 · ElasticSearch Investigation Agent

Advanced  An agent that investigates logs/metrics in ES through iterative, multi-step queries (Agentic RAG). Modules 05, 07, 08, 12

Architecture: a LangGraph agent loops — list indices → inspect mappings → search → "do I have enough?" conditional → search again / summarize. This is Agentic RAG (Module 07) over ES's hybrid (BM25 + dense_vector) search.

structure
es-investigate/
├── tools/es.py            # list_indices, get_mapping, search (allow-list)
├── graph/investigate.py   # iterative loop: search → assess → re-search
├── core/synthesize.py     # root-cause summary + cited hits
└── api/main.py

Data flow: "why did checkout error spike at 2pm?" → agent lists indices → gets mappings → runs targeted searches → assesses sufficiency (conditional edge, Module 12) → iterates → synthesizes a cited root-cause summary.

Component design: read-only ES tools with an index allow-list; a "sufficiency" router decides loop-vs-finish; results carry hit IDs for citations.

Implementation plan: ① ES read tools (allow-list, bounded size) → ② single-search agent → ③ add iterative loop + sufficiency router → ④ synthesis with citations → ⑤ step budget + trace.

Deployment: agent service + ES; bounded result sizes; recursion_limit on the investigation loop.

🏭
Production considerations

Read-only + allow-listed indices; bounded result sizes (cost); recursion_limit to avoid endless searching; cite hit IDs; trace every query for audit; injection guard on the NL prompt.

🏭
Production considerations

Least privilege (read-only, allow-list, bounded size); auth on HTTP transport; treat returned log content as untrusted (injection); audit every call; versioned tool schemas so client upgrades don't break (Module 09).

Advanced  A supervisor coordinating specialist agents over manuals, telemetry, and maintenance for an industrial plant. Modules 07, 09, 12, 13, 14

├── agents/
│   ├── supervisor.py      # routes + assembles (Module 13)
│   ├── manuals.py         # RAG over equipment manuals (07/11)

├── graph/plant_graph.py   # LangGraph supervisor + workers
├── mcp_clients/           # connect to ES MCP server

Data flow: "pump P-12 is vibrating, what do I do?" → supervisor → manuals agent (spec/limits) + telemetry agent (recent vibration trend via ES MCP) in parallel → maintenance agent drafts an action → human approves → cited recommendation.

Component design: supervisor topology (Module 13); each specialist has scoped tools (manuals=RAG, telemetry=ES MCP from Project 8); write actions gated by human-in-the-loop (Module 12).

Implementation plan: ① build each specialist standalone → ② supervisor routing → ③ structured handoffs (brief, not transcript) → ④ approval on write actions → ⑤ per-agent tracing + eval vs single-agent baseline.

Deployment: supervisor + worker services; shared MCP servers; Postgres checkpointer; durable HITL.

🛡️
Production considerations

Bounded handoffs (loops); human approval for any physical/maintenance action; per-agent attribution for debugging; right-sized models per role; safety guardrails — incorrect plant advice has real-world consequences.

10 · Enterprise GenAI Platform

Staff-level  The capstone of capstones: a shared platform every GenAI app in the org builds on. All modules — esp. 01, 09, 12, 14

Figure 4 · A central platform: gateway, guardrails, observability, prompt/model registry, eval harness, MCP tool registry, and model routing — inherited by every app team.
structure
genai-platform/
├── gateway/               # auth, routing, fallbacks, prompt cache (01,03)
├── guardrails/            # input+output: PII, injection, faithfulness (14)
├── observability/         # tracing, cost, dashboards (14)
├── registry/
│   ├── prompts/           # versioned prompts
│   └── models/            # model configs + routing rules
├── eval/                  # shared harness + golden sets + CI gates (07,14)
├── mcp_registry/          # governed MCP servers (09)
├── ratelimit/             # per-tenant quotas + budgets (14)
└── sdk/                   # client lib so app teams inherit it all

Data flow: every app calls the gateway → auth + rate limit + routing → input guardrails → app logic (RAG/agent) using registry prompts + MCP tools → output guardrails → response; tracing, cost, and eval observe every step.

Component design: the staff-level synthesis of Modules 01 (gateway), 09 (MCP registry), 12 (durable agent runtime), 14 (guardrails/observability/eval/cost). App teams get safety, observability, and cost control by default.

Implementation plan: ① gateway (auth/routing/fallback) → ② tracing + cost → ③ input/output guardrails → ④ prompt/model registry → ⑤ shared eval harness + CI gates → ⑥ MCP tool registry → ⑦ per-tenant rate limits/budgets → ⑧ client SDK.

Deployment: Kubernetes; hosted + local (vLLM, Module 03) models behind the router; Postgres for checkpoints/registry; self-hosted Langfuse; staged rollouts with eval gates.

🏛️
Production considerations

Multi-tenancy + isolation; data governance/DLP; prompt-injection defense org-wide; cost chargeback + anomaly alerts; change-managed prompts/models with rollback; red-teaming; audit logging; the platform team owns the cross-cutting concerns so product teams inherit them.

Building & sequencing advice

🪜 Build in order

1→4 cement RAG; 5→7 cement agents; 8 adds MCP; 9 adds multi-agent; 10 synthesizes everything. Each reuses the last's components.

📏 Wrap every project in Module 14

Tracing, guardrails, cost accounting, and an eval gate are non-negotiable — add them from the start, not after.

🔒 Default to local first

Build with Ollama (Modules 03–07) to iterate free and offline; swap to hosted by config when needed.

🎯 Prove every choice

Eval-driven: don't add hybrid/rerank/multi-agent unless the metrics reward it. Ship the simplest thing that works.

Portfolio & interview framing

Portfolio  How do I present these projects to a hiring manager? ›

Lead with the system design, not the model. For each: state the problem, show the architecture diagram, name the key tradeoffs you made (chunking, hybrid vs dense, single vs multi-agent), and — crucially — show your eval numbers and how production concerns (cost, safety, observability) were handled. A README with a diagram + eval table + "production considerations" section signals senior thinking far more than a working demo alone.

Senior  Which capstone best demonstrates senior GenAI engineering? ›

Projects 8–10. The MCP ES server shows you understand reusable, governed, least-privilege capability infrastructure. The multi-agent assistant shows orchestration, handoffs, and human-in-the-loop. The platform shows you think in terms of cross-cutting concerns (gateway, guardrails, observability, eval, cost) that every app inherits — the defining trait of a Senior/Staff GenAI Engineer. Pair any of them with an eval table and a clear "what I'd do at 100× scale" section.

Staff · design  Walk me through productionizing the PDF QA project for 10,000 internal users. ›

Ingestion becomes a scheduled, idempotent pipeline with ACL metadata and model_version tagging. Retrieval: hybrid + rerank (Project 4) with per-user access pre-filtering (never post-filter security). Serving behind the platform gateway (Project 10): auth, rate limits, input/output guardrails (faithfulness, PII), tracing, and cost attribution. Quality: a golden eval set gating every change in CI, plus online signals (thumbs, citation clicks) sampled back in. Scale: sharded vector store, prompt caching, cheaper models for easy queries. Safety: "I don't know" guardrail, audit logs, data-retention policy. The shift from prototype to product is 20% retrieval quality and 80% the operational discipline of Module 14.
