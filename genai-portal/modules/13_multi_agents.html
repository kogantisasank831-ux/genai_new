Multi-Agent Systems

A single agent (Module 08) with too many tools and too broad a mandate gets confused — it picks wrong tools, loses the thread, and bloats its context. The fix mirrors how humans scale work: a team of specialists. Multi-agent systems split a complex job across focused agents that each do one thing well, coordinated by a supervisor or by structured handoffs. CrewAI and LangGraph are the two ways you'll build them.

⏱ Study + practice · 10–16 h Difficulty · Advanced Prereq · Modules 08, 12

1 · Executive summary

A multi-agent system decomposes a task across several specialized agents, each with its own role, prompt, and tools, coordinated through a topology — most often a supervisor (a router/manager agent delegates to workers) or a network (agents hand off to each other directly). The wins are specialization (focused prompts/tools beat one bloated agent), modularity (test/iterate one agent at a time), and parallelism. The costs are coordination overhead, latency, and new failure modes.

❓

Why it exists

One agent with 30 tools and 5 jobs is unreliable. Splitting into focused specialists keeps each prompt/context small and each role testable.

🔧

Problem it solves

Complex, multi-domain tasks: research + write + review, or triage → specialist → QA. Divide and conquer with coordination.

🌐

Where it fits

The team layer above single agents (08), built on LangGraph (12) or CrewAI; consumes MCP tools (09) and RAG (06–07).

🔑
The one idea to remember

Multi-agent = specialists + a coordination topology. Most production systems use a supervisor: one manager routes work to focused workers and assembles the result. Reach for multiple agents only when one well-scoped agent genuinely can't cope — coordination isn't free.

2 · First-principles breakdown

What — a project team, not a lone genius

Hand one person research, writing, editing, fact-checking, and formatting and they'll do all of it mediocrely while constantly switching context. Give a small team one role each and the work improves. Multi-agent systems apply this: a researcher agent gathers facts, a writer drafts, a critic reviews — each with a tight prompt and only the tools it needs. A coordinator decides who works when and passes information between them.

🧩
Why specialization helps mechanically

Each agent's prompt and tool list stay small, so the LLM makes better tool choices (Module 08) and its context isn't polluted with irrelevant instructions. A focused "SQL agent" with 3 tools beats a "do-everything agent" with 30 — fewer ways to go wrong, easier to evaluate, and each can use a different model sized to its job.

Why — and the crucial caveat

Multi-agent buys specialization, modularity, and parallelism. But every handoff is a place to lose information, every agent is more LLM calls (cost + latency), and coordination adds failure modes a single agent never had. The senior instinct: don't multiply agents prematurely. Start with one agent; split only when a role is clearly distinct, the single agent is measurably struggling, or you need genuine parallelism.

How — coordination topologies

4 · Deep technical explanation

Handoffs — passing control and context

A handoff transfers control from one agent to another, carrying the relevant context (often the shared message history or a summary). The hard part is what to pass: too little and the receiver lacks context; too much and you pollute its prompt and pay for tokens. In practice, handoffs pass a concise task brief + pointers to shared state, not the entire transcript.

Shared state vs message-passing

Shared state (LangGraph)

Agents are nodes reading/writing one state object (Module 12). Coordination is explicit graph edges; the supervisor is a routing node. Best for control & auditability.

Role + task (CrewAI)

You declare Agents (role, goal, backstory, tools) and Tasks; a Crew runs them sequentially or hierarchically. Higher-level and faster to write; less low-level control.

CrewAI's abstractions

  • Agent — role, goal, backstory, tools, (optionally) its own LLM.
  • Task — a unit of work with a description, expected output, and an assigned agent.
  • Crew — the team + a process: sequential (pipeline) or hierarchical (a manager LLM delegates — the supervisor pattern).

Communication patterns & cost

Every inter-agent message is context one agent must read and another produced — i.e. LLM tokens. A chatty network of agents can explode cost and latency. Keep messages terse, summarize before handing off, and prefer a supervisor (star) topology over a fully-connected mesh to bound communication paths.

Multi-agent failure modes (new ones beyond Module 08)

🔁 Handoff loops

A bounces to B bounces back to A forever. Fix: max handoffs, a supervisor with authority to terminate, loop detection.

📉 Context loss at handoff

Receiver lacks what it needs. Fix: pass a structured brief + shared state; don't assume telepathy.

🗣️ Coordination overhead

Agents spend more tokens talking than working. Fix: fewer agents, terser protocols, supervisor topology.

🎯 Diffuse responsibility

No agent owns the final answer; quality slips. Fix: a designated finalizer/critic and clear DoD per task.

🧭 Conflicting actions

Two agents act on the same resource. Fix: single-writer ownership, locks, or supervisor serialization.

💸 Cost/latency multiplication

N agents ≈ N× calls. Fix: parallelize independent work; right-size models per role; cap total steps.

Evaluating multi-agent systems

Beyond single-agent metrics (Module 08): measure end-to-end task success, per-agent contribution (which agent caused failures?), handoff correctness (right agent, right context), communication efficiency (messages/tokens per task), and coordination overhead vs. a single-agent baseline — to prove the extra complexity earned its keep.

6 · Practical Python

Minimal — a CrewAI sequential crew

crew_minimal.py
# pip install crewai
from crewai import Agent, Task, Crew, Process

researcher = Agent(
    role="Research Analyst",
    goal="Find accurate, current facts on the topic",
    backstory="Meticulous; never invents sources.",
    verbose=True)

writer = Agent(
    role="Technical Writer",
    goal="Write a clear, concise report from the research",
    backstory="Explains complex topics simply.")

research = Task(description="Research EV battery trends in 2025.",
                 expected_output="6 bullet-point facts with sources.",
                 agent=researcher)
write = Task(description="Write a 1-page report from the facts.",
              expected_output="Markdown report with citations.",
              agent=writer)

crew = Crew(agents=[researcher, writer], tasks=[research, write],
            process=Process.sequential)        # pipeline topology
print(crew.kickoff())

Intermediate — hierarchical crew (supervisor) with tools

crew_hierarchical.py
from crewai import Agent, Task, Crew, Process
from crewai_tools import SerperDevTool      # a search tool, e.g.

search = SerperDevTool()

researcher = Agent(role="Researcher", goal="Gather facts",
                    backstory="...", tools=[search])          # only the tools it needs
analyst    = Agent(role="Analyst", goal="Find trends in the facts", backstory="...")
writer     = Agent(role="Writer", goal="Produce the final report", backstory="...")

task = Task(description="Produce a cited 1-page report on EV battery trends.",
            expected_output="Markdown report")

crew = Crew(
    agents=[researcher, analyst, writer],
    tasks=[task],
    process=Process.hierarchical,              # a manager LLM delegates = supervisor
    manager_llm="gpt-4o")                       # the supervisor's brain
print(crew.kickoff())

Intermediate+ — a LangGraph supervisor (full control)

langgraph_supervisor.py
from typing import Annotated, TypedDict, Literal
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]
    next: str

WORKERS = ["researcher", "analyst", "writer"]

def supervisor(state: State):
    # LLM decides which worker acts next, or FINISH (structured output)
    decision = manager_llm.invoke(state["messages"],
                  choices=WORKERS + ["FINISH"])
    return {"next": decision}

def make_worker(name):
    def node(state: State):
        out = worker_agents[name].invoke(state["messages"])
        return {"messages": [out]}        # write result to shared state
    return node

g = StateGraph(State)
g.add_node("supervisor", supervisor)
for w in WORKERS:
    g.add_node(w, make_worker(w))
    g.add_edge(w, "supervisor")              # workers report back to supervisor
g.add_edge(START, "supervisor")
g.add_conditional_edges("supervisor",           # route to chosen worker or finish
    lambda s: s["next"], {**{w: w for w in WORKERS}, "FINISH": END})
app = g.compile()

Production — bounded, observable orchestration

prod_orchestration.py
# Production multi-agent guards (extend the LangGraph supervisor):
#   • recursion_limit       → bound total supervisor↔worker hops (handoff loops)
#   • per-worker tool scopes → least privilege; right-sized model per role
#   • structured handoff brief → avoid context loss / token bloat
#   • checkpointer + tracing → audit which agent did what (Modules 12, 14)
#   • a single finalizer node → one owner of the final answer

cfg = {"configurable": {"thread_id": "job-123"}, "recursion_limit": 20}

def handoff_brief(state, to_agent: str) -> dict:
    """Pass a terse task brief + state pointers, NOT the whole transcript."""
    return {"task": state["current_subtask"],
            "facts": state.get("facts", [])[:10],   # bounded
            "for": to_agent}

result = app.invoke({"messages": [("user", task)]}, cfg)
# Every transition is checkpointed + traced → per-agent attribution for eval.
🛡️

The production rule: bound everything and attribute everything. Cap handoffs (recursion_limit), scope each agent's tools (least privilege, Module 09), pass terse structured briefs (not transcripts), and trace per-agent so your eval can pinpoint which specialist failed.

7 · Ollama implementation

Run a whole crew locally — and right-size models per role to save VRAM and latency.

local_crew.py
from crewai import Agent, Task, Crew, Process, LLM

# Different local models per role — cheap small model for simple roles,
# stronger model for the supervisor/analyst (Module 03 sizing).
small = LLM(model="ollama/gemma2", base_url="http://localhost:11434")
strong = LLM(model="ollama/qwen2.5", base_url="http://localhost:11434")

writer   = Agent(role="Writer", goal="Draft the report", backstory="...", llm=small)
analyst  = Agent(role="Analyst", goal="Reason over data", backstory="...", llm=strong)

crew = Crew(agents=[analyst, writer],
            tasks=[Task(description="Summarize plant downtime causes.",
                        expected_output="Ranked list + 1-paragraph summary",
                        agent=analyst)],
            process=Process.sequential)
print(crew.kickoff())   # fully offline multi-agent run
⚖️

Local multi-agent is where Module 08's "small models flail more" bites hardest — more agents × weaker models = more coordination errors. Keep crews small, roles crisp, tools minimal, and watch the failure modes from §4 closely.

8 · Industry perspective

🚀 Startups

Use CrewAI to prototype a team fast (declare roles + tasks, get a working crew). Validate that multi-agent beats a single tuned agent before committing — often it doesn't, and one good agent ships sooner.

🏢 Enterprises

Favor LangGraph supervisors for control, durability, and auditability (which agent did what, when — Module 12). Right-size models per role for cost, scope tools per agent for security, and gate consequential actions with human-in-the-loop.

9 · Top mistakes engineers make

❌ Reaching for multi-agent first

Most tasks need one good agent. Multi-agent adds cost, latency, and failure modes. Earn it with evidence.

❌ Passing the whole transcript on handoff

Bloats context and cost. Pass a terse brief + shared-state pointers.

❌ No bound on handoffs

Agents ping-pong forever. Cap total hops; give the supervisor authority to terminate.

❌ Overlapping roles

Fuzzy responsibilities cause conflicts and duplicated work. Make each role distinct with a clear DoD.

❌ One model for every role

Right-size: a cheap model for simple roles, a strong one for the supervisor/analyst. Saves cost & latency.

❌ No per-agent observability

When the crew fails you won't know who's at fault. Trace and attribute every step (Module 14).

10 · Interview preparation

Beginner  What is a multi-agent system and why use one? ›

Several specialized agents, each with its own role, prompt, and tools, coordinated to solve a complex task. Benefits: specialization (focused agents make better decisions than one bloated agent), modularity (test/iterate per agent), and parallelism. Costs: coordination overhead, latency, more LLM calls, and new failure modes — so use it only when one agent genuinely can't cope.

Beginner  What is the supervisor pattern? ›

A manager agent that receives the task, delegates subtasks to specialized worker agents, collects their results, and decides the next step until done. It's the dominant production topology because control is centralized and easy to reason about — a star shape rather than a chaotic mesh.

Intermediate  Supervisor vs network (handoff) — tradeoffs? ›

Supervisor centralizes routing: predictable, auditable, easy to bound, but the supervisor is a bottleneck and single point of decision. Network lets any agent hand off to any peer: flexible for fluid workflows, but harder to reason about, easier to loop, and trickier to bound and debug. Most production systems pick supervisor for control; use network when the workflow genuinely needs peer-to-peer transfers.

Intermediate  What's the hard part of a handoff? ›

Deciding what context to transfer. Too little and the receiving agent lacks the information to act; too much and you pollute its prompt and pay for excess tokens. The right approach passes a concise structured brief (the subtask + the few relevant facts/pointers to shared state), not the entire conversation transcript.

Senior  How do you decide between one agent and a multi-agent system? ›

Default to one well-scoped agent. Split into multiple agents when: roles are genuinely distinct (different tools/expertise), the single agent is measurably failing (wrong tool choices, context bloat, lost thread on evals), you need true parallelism, or you need different models/permissions per role. Always validate with an eval that the multi-agent version beats the single-agent baseline on success rate and justifies the extra cost/latency. Complexity must earn its place.

Senior  How do you prevent and detect handoff loops? ›

Prevent: prefer a supervisor topology (bounded paths) over a mesh; give the supervisor sole authority to terminate; set a hard cap on total hops (recursion_limit). Detect: track the handoff sequence and flag repeats (A→B→A→B); monitor messages-per-task and alert on outliers; in the trace, watch for no net progress in shared state across hops. On detection, force termination/escalation and feed the case into the eval suite.

Staff · design  Design a multi-agent customer-support system with safety and observability. ›

Topology: a supervisor/triage agent routes to specialists (billing, technical, account) — bounded star, not a mesh. Build on LangGraph for explicit routing, durable checkpoints, and per-agent attribution. Each specialist gets least-privilege tools via MCP (Module 09) and RAG over its domain docs (07/11). Handoffs carry a structured brief + shared-state pointers, capped by recursion_limit with the supervisor able to escalate to a human. Safety: human-in-the-loop (Module 12) before refunds/account changes; guardrails on inputs/outputs (Module 14). Observability: trace every transition, attribute success/failure per agent, track handoff correctness and tokens/task. Eval: golden conversations gating deploys; compare against a single-agent baseline to justify the architecture. A designated finalizer owns the user-facing response. The deliverables that matter: bounded coordination, least privilege, human gates on consequential actions, and per-agent traceability.

11 · Knowledge check

1 · The most common production multi-agent topology is…
The supervisor (star) pattern centralizes routing and control, making the system predictable, boundable, and auditable — the dominant choice in production.
2 · Why does splitting one big agent into specialists often improve reliability?
Smaller, focused prompts and minimal tools reduce the ways an agent can go wrong (Module 08) and keep context clean — though coordination adds its own costs.
3 · What should a handoff typically carry?
Passing the whole transcript bloats context and cost; passing nothing causes context loss. A terse structured brief + shared-state references is the balance.
4 · Before shipping a multi-agent system, you should…
Multi-agent adds cost, latency, and failure modes. An eval comparing it to a well-tuned single agent proves the complexity is warranted.
