LlamaIndex

LangChain (Module 10) is a general composition framework. LlamaIndex is laser-focused on one thing: connecting your data to LLMs. Where LangChain gives you pipes, LlamaIndex gives you a complete, opinionated RAG stack — document loaders, a node abstraction, pluggable indexes, retrievers, and query engines — so you can go from "a folder of PDFs" to "a working Q&A system" in a handful of lines, then tune every stage.

⏱ ~50 min read Difficulty · Intermediate Prereq · Modules 04–07, 10

1 · Executive summary

LlamaIndex is the "data framework for LLM applications." It models your knowledge as Documents → split into Nodes (chunks with metadata + relationships) → organized by an Index → served through a Retriever and a Query Engine that retrieves, then synthesizes an answer. It encodes the RAG pipeline you built by hand in Modules 04–07 as first-class, swappable components — with strong abstractions for ingestion, advanced retrieval, and response synthesis.

❓

Why it exists

RAG has a lot of moving data-plumbing — loading, chunking, indexing, retrieving, synthesizing. LlamaIndex makes each a clean, reusable abstraction.

🔧

Problem it solves

"Connect my documents/database to an LLM, retrieve well, and synthesize a grounded answer" — with sensible defaults and deep customization.

🌐

Where it fits

The data/RAG specialist. Often used with LangChain (10) for orchestration and LangGraph (12) for agent loops.

🔑
The one idea to remember

Document → Node → Index → Retriever → Query Engine. That's the whole mental model. A Query Engine = retriever + response synthesizer: it fetches relevant Nodes and composes them into a grounded answer.

2 · First-principles breakdown

What — a librarian for your data

Think of LlamaIndex as a librarian. First it catalogs your material: it reads documents and breaks them into index cards (Nodes), each tagged with where it came from and how it relates to neighbors. Then, when you ask a question, the librarian finds the relevant cards (retrieval) and writes you a synthesized answer from them (response synthesis) — citing the source cards. The Index is the catalog; the Query Engine is the librarian answering you.

🧱
Document vs Node

A Document is a whole source (a PDF, a web page, a DB row). A Node is a chunk of it — the actual retrievable unit (Module 06's chunk, formalized). Nodes carry metadata and relationships (prev/next/parent), which is what powers advanced retrieval like sentence-window and auto-merging.

Why — RAG is mostly data engineering

As a data scientist you'll recognize this: the model is the easy part; the hard part is the pipeline that gets clean, well-chunked, well-tagged data to it. LlamaIndex treats RAG as the data problem it really is — giving you loaders for 100+ sources, node parsers with relationship tracking, and indexes that organize nodes for different retrieval strategies. You spend your effort on data quality, not glue.

How — the two phases

Indexing phase (offline)load → parse to Nodes → embed → Index
Query phase (online)Query Engine: retrieve Nodes → synthesize
Grounded answer + source nodes

This is the exact offline/online split from Module 06 — LlamaIndex just gives it names and swappable parts at every stage.

4 · Deep technical explanation

The core objects

ObjectRole
DocumentA whole source with text + metadata
NodeA chunk: text, metadata, embedding, and relationships (prev/next/parent/child)
IndexOrganizes Nodes for retrieval (most commonly VectorStoreIndex)
RetrieverGiven a query, returns the most relevant Nodes
Node postprocessorTransforms retrieved nodes — rerank, filter by metadata/similarity, expand windows
Response synthesizerTurns nodes + query into a final answer (the generation step)
Query engineThe end-to-end "ask a question, get an answer" object (retriever + postprocessors + synthesizer)

Index types

IndexHow it retrievesUse for
VectorStoreIndexEmbedding similarity (Modules 04–05)The default — semantic Q&A
SummaryIndexIterates all nodes (no embedding)Summarize a whole small document
DocumentSummaryIndexRetrieve by per-doc summaries, then drill inRouting across many documents
PropertyGraphIndexKnowledge-graph over entities/relationsGraph RAG (Module 07) — multi-hop

Response synthesis modes

Because retrieved context can exceed the window (Module 01), LlamaIndex offers synthesis strategies:

  • compact (default) — stuff as many nodes as fit, then answer; the pragmatic choice.
  • refine — answer from node 1, then iteratively refine using nodes 2..n; good when context overflows.
  • tree_summarize — recursively summarize node groups bottom-up; good for "summarize everything" queries.

Advanced retrieval as node strategies

The advanced techniques from Module 07 appear here as node parsers/postprocessors:

  • SentenceWindowNodeParser — embed single sentences, return them with surrounding sentences (parent-document idea).
  • HierarchicalNodeParser + AutoMergingRetriever — small chunks that auto-merge into their parent when enough siblings match.
  • SentenceTransformerRerank postprocessor — cross-encoder reranking (Module 07) in one line.
  • MetadataFilters — pre-filter retrieval by node metadata (Module 05).

Storage & persistence

LlamaIndex separates the StorageContext (docstore, index store, vector store). For production you plug in a real vector store (Qdrant/pgvector, Module 05) instead of the in-memory default, and persist the index so you don't re-embed on every restart.

Tradeoffs & limitations

Strengths

  • Fastest path from data → RAG
  • Rich loaders (LlamaHub) & node strategies
  • Advanced retrieval built-in
  • Great for data-centric apps

Limits

  • Less general than LangChain for arbitrary orchestration
  • Defaults can hide cost (embeds everything)
  • Fast-moving API; pin versions
  • Agent/loop story overlaps LangGraph

6 · Practical Python

Minimal — RAG in 5 lines

minimal.py
# pip install llama-index
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader

docs = SimpleDirectoryReader("./data").load_data()   # → Documents
index = VectorStoreIndex.from_documents(docs)         # parse→embed→index
engine = index.as_query_engine()                       # retriever + synthesizer

resp = engine.query("What is the refund window?")
print(resp)                                          # grounded answer
for n in resp.source_nodes:                            # built-in citations
    print(n.metadata, round(n.score, 3))

Intermediate — control chunking, retrieval, models & reranking

tuned.py
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader, Settings
from llama_index.core.node_parser import SentenceSplitter
from llama_index.core.postprocessor import SentenceTransformerRerank
from llama_index.embeddings.huggingface import HuggingFaceEmbedding
from llama_index.llms.openai import OpenAI

# Global settings: embedding model, LLM, chunking (Modules 04, 06)
Settings.embed_model = HuggingFaceEmbedding("BAAI/bge-small-en-v1.5")
Settings.llm = OpenAI(model="gpt-4o-mini", temperature=0)
Settings.node_parser = SentenceSplitter(chunk_size=400, chunk_overlap=60)

docs = SimpleDirectoryReader("./data").load_data()
index = VectorStoreIndex.from_documents(docs)

# Two-stage retrieval: fetch 20, rerank to 4 (Module 07)
reranker = SentenceTransformerRerank(
    model="cross-encoder/ms-marco-MiniLM-L-6-v2", top_n=4)
engine = index.as_query_engine(
    similarity_top_k=20,
    node_postprocessors=[reranker],
    response_mode="compact")

print(engine.query("How do I reset my password?"))

Intermediate+ — sentence-window (parent-document) retrieval

sentence_window.py
from llama_index.core.node_parser import SentenceWindowNodeParser
from llama_index.core.postprocessor import MetadataReplacementPostProcessor

# Embed single sentences (precise match) but answer with a window
# of surrounding sentences (full context) — parent-document, Module 07.
parser = SentenceWindowNodeParser.from_defaults(
    window_size=3, window_metadata_key="window",
    original_text_metadata_key="original")
nodes = parser.get_nodes_from_documents(docs)
index = VectorStoreIndex(nodes)

engine = index.as_query_engine(
    similarity_top_k=3,
    # at query time, swap the matched sentence for its window
    node_postprocessors=[MetadataReplacementPostProcessor(target_metadata_key="window")])

Production — real vector store, persistence & a service

prod_index.py
from llama_index.core import VectorStoreIndex, StorageContext, Settings
from llama_index.vector_stores.qdrant import QdrantVectorStore
from qdrant_client import QdrantClient
from fastapi import FastAPI

# Back the index with a real vector DB (Module 05) — survives restarts,
# scales, supports metadata filtering. No re-embedding on boot.
client = QdrantClient(url="http://localhost:6333")
vstore = QdrantVectorStore(client=client, collection_name="docs")
storage = StorageContext.from_defaults(vector_store=vstore)

def get_index():
    # Build once (ingest job); thereafter load from the store.
    return VectorStoreIndex.from_vector_store(vstore)

app = FastAPI()
engine = get_index().as_query_engine(similarity_top_k=5)

@app.get("/ask")
def ask(q: str):
    r = engine.query(q)
    return {"answer": str(r),
            "sources": [{"meta": n.metadata, "score": n.score}
                        for n in r.source_nodes]}
🔗

This is the same Qdrant store from Module 05 and the same FastAPI shape from Module 06 — LlamaIndex just gives the RAG internals clean, swappable names. Use an IngestionPipeline as a separate scheduled job to keep the index fresh (Module 06's re-ingestion concern).

7 · Ollama implementation — fully local LlamaIndex

Set local models in Settings and the whole stack runs offline — confidential documents never leave the machine.

local_llamaindex.py
# pip install llama-index-llms-ollama llama-index-embeddings-ollama
from llama_index.core import VectorStoreIndex, SimpleDirectoryReader, Settings
from llama_index.llms.ollama import Ollama
from llama_index.embeddings.ollama import OllamaEmbedding

Settings.llm = Ollama(model="qwen2.5", request_timeout=120, temperature=0)
Settings.embed_model = OllamaEmbedding(model_name="nomic-embed-text")

docs = SimpleDirectoryReader("./private_docs").load_data()
index = VectorStoreIndex.from_documents(docs)        # local embeddings
engine = index.as_query_engine()                       # local generation

print(engine.query("Summarize the safety procedure."))   # 100% offline
🔒

Just like LangChain (Module 10), going local is a settings change, not a rewrite. Local LLM + local embeddings + a local Qdrant store = an air-gapped RAG system — the on-prem capstone backbone.

8 · Industry perspective

🚀 Startups

Reach for LlamaIndex when the product is "chat with our data." The 5-line start ships a demo today; the node parsers and rerankers tune it tomorrow. LlamaHub loaders save weeks of connector work.

🏢 Enterprises

Use it as the RAG/data layer behind a broader system: real vector store, scheduled ingestion pipelines, metadata-based access control, and advanced retrieval (sentence-window, auto-merging) for quality. Often paired with LangGraph for agentic flows.

LlamaIndex vs LangChain — how to choose & combine

Lean LlamaIndex when…Lean LangChain when…
The app is fundamentally RAG over your dataYou need general orchestration of many components/tools
You want advanced retrieval out of the boxYou want provider-agnostic LCEL composition
Ingestion & indexing strategy is the hard partThe hard part is multi-step logic/agents (→ LangGraph)

They're not mutually exclusive: a common stack is LlamaIndex for retrieval, exposed as a tool to a LangGraph agent. Use the data specialist for data; use the orchestrator for orchestration.

Common production challenges: default settings silently embedding everything (cost), in-memory store losing the index on restart (persist!), version churn, and treating LlamaIndex as a full app framework when it's a data layer.

9 · Top mistakes engineers make

❌ Using the in-memory store in prod

It re-embeds on every restart and doesn't scale. Plug in Qdrant/pgvector and persist.

❌ Ignoring chunking settings

The default chunk size may not fit your data. Tune SentenceSplitter against an eval set (Module 07).

❌ Skipping reranking

A one-line SentenceTransformerRerank postprocessor is often the biggest quality win.

❌ Not reading source_nodes

Citations are free in the response object — use them for trust and evaluation.

❌ Wrong response mode for the task

Use tree_summarize for "summarize everything", compact/refine for Q&A. Defaults aren't always right.

❌ Forcing agent loops into query engines

For multi-step, stateful flows, use LangGraph and expose the index as a tool.

10 · Interview preparation

Beginner  What is LlamaIndex and how is it different from LangChain? ›

LlamaIndex is a data framework specialized for connecting data to LLMs — loading, chunking into Nodes, indexing, retrieving, and synthesizing answers (RAG). LangChain is a general composition framework for any LLM pipeline. LlamaIndex is the data/RAG specialist with stronger ingestion and advanced-retrieval abstractions; they're often combined.

Beginner  What's the difference between a Document and a Node? ›

A Document is a whole source (a file, page, or record). A Node is a chunk of a Document — the actual retrievable unit — carrying text, metadata, an embedding, and relationships to neighboring nodes (prev/next/parent). Node parsers turn Documents into Nodes.

Intermediate  What exactly is a Query Engine made of? ›

A retriever (fetches relevant Nodes from an index), optional node postprocessors (rerank, filter, window-replace), and a response synthesizer (composes Nodes + query into a grounded answer, with a synthesis mode like compact/refine/tree_summarize). It's the end-to-end "ask → answer with sources" object.

Intermediate  Explain sentence-window / auto-merging retrieval. ›

Both solve the small-vs-large chunk dilemma (Module 07's parent-document). Sentence-window embeds single sentences for precise matching but returns each matched sentence with a window of surrounding sentences for context (via a metadata-replacement postprocessor). Auto-merging uses a hierarchy of chunk sizes and merges small matched chunks into their parent when enough siblings hit — giving precision plus coherent context.

Senior  How do you take a LlamaIndex prototype to production? ›

Swap the in-memory store for a real vector DB (Qdrant/pgvector) and persist the index so you don't re-embed on boot. Move ingestion into a scheduled, idempotent IngestionPipeline that handles updates/deletes and tags model_version (Module 04). Tune chunking and add reranking, validated by an eval set (Module 07). Set local/hosted models and temperature via Settings; add metadata-based access control; expose the query engine behind a service boundary (FastAPI). Add tracing/observability and cost controls (Module 14). Treat it as the data layer, with orchestration/agents handled by LangGraph if needed.

Senior  When would you combine LlamaIndex and LangGraph? ›

When the app needs both strong retrieval and multi-step/agentic control. Use LlamaIndex to build a high-quality query engine (advanced retrieval, synthesis), then expose it as a tool to a LangGraph agent that decides when to retrieve, can retrieve iteratively (Agentic RAG, Module 07), branch, and keep state. LlamaIndex handles "how to retrieve well"; LangGraph handles "how to orchestrate the reasoning loop."

Staff · design  Design a multi-source enterprise knowledge engine with LlamaIndex. ›

Ingestion: LlamaHub readers per source (wiki, tickets, docs, DB) → IngestionPipeline (parse to Nodes with rich metadata: source, ACL, freshness, model_version) → embed → persist to a sharded vector store (Module 05). Retrieval: per-source indexes with a router query engine (DocumentSummaryIndex) to pick relevant sources, sentence-window/auto-merging for context, reranking for precision, and metadata pre-filtering for access control (never post-filter security). Synthesis: compact for Q&A, tree_summarize for overviews, always returning source_nodes for citations. Operations: scheduled re-ingestion to meet freshness SLAs, eval gates (recall/faithfulness, Module 07) in CI, observability + cost tracking (Module 14). Expose the engine as a tool to a LangGraph agent for multi-step questions. The deliverable is a governed, fresh, citable retrieval layer reusable across the org.

11 · Knowledge check

1 · In LlamaIndex, what is the actual retrievable unit?
Documents are whole sources; node parsers split them into Nodes, which are the chunks that get embedded, indexed, and retrieved.
2 · A Query Engine is essentially…
It retrieves relevant Nodes, optionally postprocesses them (rerank/filter), and synthesizes a grounded answer — the end-to-end ask→answer object.
3 · You're running LlamaIndex in production and the index disappears on restart. Why?
The default store is in-memory and re-embeds on boot. Plug in a real vector store (Qdrant/pgvector) and persist the index for production.
4 · Which response mode best fits "summarize this entire document"?
tree_summarize recursively summarizes node groups bottom-up, ideal for whole-document/global summaries. compact/refine are for focused Q&A.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q State the LlamaIndex mental model in five words/objects. ›
Document → Node → Index → Retriever → Query Engine.
Q What three parts make up a Query Engine? ›
Retriever, optional node postprocessors (rerank/filter/window), and a response synthesizer (with a synthesis mode).
Q Name the three response synthesis modes and their use. ›
compact (stuff what fits — default Q&A), refine (iteratively refine over nodes — overflow), tree_summarize (recursive bottom-up — whole-doc summaries).
Q How does LlamaIndex implement parent-document retrieval? ›
SentenceWindowNodeParser (embed sentences) + MetadataReplacementPostProcessor (return the surrounding window at query time); or HierarchicalNodeParser + AutoMergingRetriever.
Q One-line: how do you combine LlamaIndex with LangGraph? ›
Build a query engine in LlamaIndex and expose it as a tool to a LangGraph agent — retrieval specialist + orchestration engine.
