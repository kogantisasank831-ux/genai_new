Local LLMs & Ollama

A frontier API is the fastest way to ship, but it isn't always an option: regulated data can't leave the building, costs balloon at scale, and offline/edge deployments have no internet. Running models locally — on your laptop or a single GPU box — solves all three. Ollama makes it as easy as docker run, and the architecture you learned in Module 02 tells you exactly what hardware you'll need.

⏱ ~55 min read Difficulty · Intermediate Prereq · Modules 01–02, Docker basics

1 · Executive summary

A local LLM is an open-weights model (Llama, Qwen, Gemma, Mistral, Phi…) that you download and run on your own compute, with no API call leaving your machine. The enabling trick is quantization: storing weights in 4–8 bits instead of 16/32, which shrinks a model 3–4× so it fits in consumer VRAM. Ollama is the runtime that packages model + weights + config into one pullable artifact and serves an OpenAI-compatible API on localhost:11434.

❓

Why it exists

Privacy, cost control, offline capability, low latency, and zero per-token billing — at the price of owning the hardware and ops.

🔧

Problem it solves

"My data can't go to a third party" and "my inference bill is unsustainable." Local inference removes both constraints.

🌐

Where it fits

Powers private RAG, on-prem agents, edge devices, and dev loops where you iterate without metered API costs.

🔑
The one idea to remember

Local LLM feasibility is a memory problem. If the quantized weights plus the KV cache fit in your VRAM (or RAM), it runs — fast on GPU, slower on CPU. Everything in this module is about making that fit and serving it cleanly.

2 · First-principles breakdown

What — a file of numbers and a program that runs it

A model is just a large file of weights (a few GB) plus an architecture spec. To "run" it you need a program (an inference engine) that loads those weights into memory and executes the transformer forward pass from Module 02 for each token. Ollama bundles the engine (llama.cpp under the hood), the weights, and a config into one named, versioned artifact — much like a container image.

Why — quantization makes big models fit small hardware

A 7-billion-parameter model in 16-bit precision needs ~14 GB just for weights — too big for an 8 GB GPU. Quantization rounds each weight to fewer bits. The headline trade: a tiny accuracy loss for a huge memory and speed win.

QuantBits/weight7B model sizeQualityUse when
FP16 (full)16~14 GBReferenceBig GPU, max fidelity
Q8_08~7.5 GBNear-losslessQuality-sensitive tasks
Q4_K_M~4.5~4.4 GBVery good (sweet spot)Default for most local use
Q3_K_M~3.5~3.5 GBNoticeable degradationVery tight VRAM

How — the VRAM budget you can do on a napkin

Two things consume memory: the weights (fixed) and the KV cache (grows with context length × layers × hidden size, shrunk by GQA from Module 02).

🧮
Rule of thumb

VRAM ≈ quantized weight size × 1.2 (the 20% covers KV cache + activations at modest context). A Q4 7B model (~4.4 GB) wants ~5–6 GB VRAM; an 8 GB GPU runs it comfortably. Long contexts push the KV-cache term up — that's when the 1.2 becomes 1.5+.

4 · Deep technical explanation

GGUF — the container format for local weights

GGUF is a single-file format (used by llama.cpp/Ollama) that stores quantized tensors plus metadata: architecture, context length, tokenizer, and chat template. One file = fully self-describing model. Ollama wraps a GGUF with a Modelfile (defaults, system prompt, params) into a named tag.

GPU offload, CPU fallback, and hybrid

llama.cpp can split layers between GPU and CPU. If the model doesn't fully fit in VRAM, it offloads as many layers as possible to the GPU and runs the rest on CPU — slower, but it still works. Fully on GPU is fastest; fully on CPU is the universal fallback (good for small models / dev machines).

PlacementSpeedWhen
Full GPU offload⚡ Fastest (tens–hundreds tok/s)Model fits VRAM
Hybrid (partial offload)🟡 ModerateModel slightly too big for VRAM
CPU only🐢 Slow (few tok/s)No GPU; small models; max compatibility

Context length & the KV cache, locally

The architecture from Module 02 has a direct cost here. Ollama's num_ctx sets context length; a larger window allocates a bigger KV cache up front. On a memory-constrained box, raising num_ctx from 2k to 32k can be the difference between fitting and OOM. GQA (grouped-query attention) in Llama/Qwen keeps this cache small — the reason these models are local-friendly.

Throughput vocabulary

  • Prompt eval (prefill) — processing your input prompt, done in parallel; fast.
  • Token generation (decode) — autoregressive, one token at a time; the speed you "feel."
  • tok/s — tokens per second; the headline metric. TTFT — time to first token; what users perceive as responsiveness.

Tradeoffs & limitations

Wins

  • Total data privacy
  • No per-token cost
  • Works offline / on edge
  • Full control over versioning

Costs

  • You own GPU + ops + scaling
  • Open models lag frontier on hardest tasks
  • Quantization can dent quality on edge cases
  • Concurrency needs a real server (vLLM) beyond Ollama's single-box scope

6 · Practical Python

Minimal — chat with a local model

minimal_local.py
# Ollama's native client (pip install ollama)
import ollama

resp = ollama.chat(model="llama3.2", messages=[
    {"role": "user", "content": "Name 3 use cases for local LLMs."}
])
print(resp["message"]["content"])

Intermediate — OpenAI-compatible + streaming + params

compat_stream.py
# Reuse the OpenAI SDK so swapping local↔hosted is a one-line change.
from openai import OpenAI

client = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

stream = client.chat.completions.create(
    model="qwen2.5",
    temperature=0.2,
    max_tokens=300,
    extra_body={"options": {"num_ctx": 8192}},  # context window
    messages=[{"role": "user", "content": "Explain quantization simply."}],
    stream=True,
)
for chunk in stream:
    delta = chunk.choices[0].delta.content or ""
    print(delta, end="", flush=True)

Production — a backend-switchable LLM service (FastAPI)

local_service.py
import os
from fastapi import FastAPI
from pydantic import BaseModel
from openai import OpenAI

# One config flag flips between local Ollama and a hosted gateway.
BACKEND = os.getenv("LLM_BACKEND", "local")
CFG = {
    "local":  {"base_url": "http://localhost:11434/v1", "key": "ollama", "model": "qwen2.5"},
    "hosted": {"base_url": os.getenv("OPENAI_BASE_URL"), "key": os.getenv("OPENAI_API_KEY"), "model": "gpt-4o-mini"},
}[BACKEND]

client = OpenAI(base_url=CFG["base_url"], api_key=CFG["key"])
app = FastAPI(title="Local LLM Service")

class Ask(BaseModel):
    prompt: str
    temperature: float = 0.2

@app.get("/health")
def health():
    try:                               # liveness: is the model loaded & serving?
        client.models.list()
        return {"status": "ok", "backend": BACKEND, "model": CFG["model"]}
    except Exception as e:
        return {"status": "degraded", "error": str(e)}

@app.post("/ask")
def ask(q: Ask):
    r = client.chat.completions.create(
        model=CFG["model"], temperature=q.temperature,
        messages=[{"role": "user", "content": q.prompt}],
    )
    return {"answer": r.choices[0].message.content,
            "tokens": r.usage.total_tokens if r.usage else None}
🐳

In production you'd run Ollama as a container (docker run -d --gpus all -v ollama:/root/.ollama -p 11434:11434 ollama/ollama) and your FastAPI service in another, on the same Docker network — exactly the pattern used in the Local ChatGPT Clone capstone.

7 · Ollama in depth

Core commands

ollama_cheatsheet.sh
ollama pull llama3.2           # download a model

The Modelfile — bake in behavior

A Modelfile is to a model what a Dockerfile is to an image: it pins a base model, default params, a system prompt, and the chat template into a reusable named tag.

Modelfile
FROM qwen2.5:7b-instruct-q4_K_M

# sampling defaults baked into the model tag
PARAMETER temperature 0.1
PARAMETER num_ctx 8192
PARAMETER top_p 0.9
PARAMETER stop "</answer>"

# a persistent system persona
SYSTEM """
You are a precise data-engineering assistant. Answer only from
provided context. If unsure, say "I don't know". Be concise.
"""
build.sh
ollama create de-assistant -f ./Modelfile
ollama run de-assistant "What is GGUF?"   # uses your baked params + system prompt

📏

Pick by size first, family second. A well-chosen 7B-Q4 that fits your GPU and runs fast beats a 14B that thrashes between VRAM and RAM. Benchmark your task — leaderboards rarely match your prompt distribution.

8 · Industry perspective

🚀 Startups

Use Ollama for the dev loop (free iteration, no metered calls) and for cost-sensitive features. Many ship a "small local model for easy requests, hosted frontier model for hard ones" router to cut bills.

🏢 Enterprises

Run open models in a VPC/on-prem for regulated data (healthcare, finance, defense). At scale they graduate from Ollama (single box) to vLLM/TGI clusters with paged-KV batching for high concurrency.

Ollama vs. vLLM — know the boundary

OllamavLLM / TGI
GoalEasy local/dev & single-userHigh-throughput multi-user serving
ConcurrencyLimited (sequential-ish)Continuous batching, many requests

Common production challenges: cold-start model load time, VRAM OOM under long context, single-box concurrency limits, keeping model versions consistent across environments, and securing an exposed :11434 port.

9 · Top mistakes engineers make

❌ Downloading FP16 onto an 8 GB GPU

It won't fit. Start at Q4_K_M and only go higher if quality demands it and VRAM allows.

❌ Cranking num_ctx "to be safe"

Big context pre-allocates a big KV cache and can OOM. Set it to what you actually need.

❌ Expecting frontier quality

A 7B local model isn't GPT-4. Match model size to task difficulty; route hard tasks to a bigger/hosted model.

❌ Exposing :11434 to the internet

Ollama has no auth by default. Keep it on localhost or behind an authenticated gateway/reverse proxy.

❌ Benchmarking only tok/s

TTFT and quality matter as much as throughput. Measure all three on your real workload.

❌ Forgetting the chat template

Wrong prompt formatting silently degrades output. Let Ollama's bundled template handle it; don't hand-format unless you know the model's format.

10 · Interview preparation

Beginner  Why run an LLM locally instead of calling an API? ›

Data privacy (nothing leaves your network), no per-token cost, offline/edge capability, predictable latency, and full version control. The trade-off is that you own the hardware, ops, and scaling, and open models may lag frontier quality on the hardest tasks.

Beginner  What is quantization and what does it cost you? ›

Storing model weights at lower precision (e.g. 4-bit instead of 16-bit), shrinking memory ~3–4× and speeding inference. The cost is a small, usually acceptable accuracy loss; aggressive quants (Q3 and below) degrade quality more noticeably. Q4_K_M is the common sweet spot.

Intermediate  Estimate the VRAM to run a 13B model at Q4. ›

~4.5 bits × 13B / 8 ≈ 7.3 GB of weights, ×1.2 for KV cache + activations ≈ ~9 GB at modest context. A 12 GB GPU runs it comfortably; an 8 GB GPU would need a smaller quant, shorter context, or partial CPU offload (slower).

Intermediate  What's the difference between Ollama and vLLM? ›

Ollama optimizes for easy single-box/local use and developer experience. vLLM optimizes for high-throughput multi-user serving via continuous batching and paged KV cache. Use Ollama to prototype and for desktop/edge; graduate to vLLM/TGI for a production API serving many concurrent users.

Senior  A hospital wants private RAG over patient records. Walk through your local deployment. ›

All compute on-prem/in-VPC; nothing leaves the network. Run a local instruct model (e.g. Qwen2.5/Llama3.x, Q4–Q8 by GPU budget) via vLLM for concurrency, plus a local embedding model. Vector DB on-prem (pgvector/Qdrant). Add a gateway for auth, audit logging, and PII guardrails; pin model + prompt versions; build an eval set of clinical Q&A to gate releases. Document hardware sizing from the VRAM math and plan capacity for peak concurrency. Privacy and auditability are the design center, not latency.

Staff · design  Design a hybrid local+hosted model router to cut cost 70% without hurting quality. ›

Classify each request by difficulty (heuristics + a tiny classifier or the local model's self-assessed confidence). Route easy/high-volume tasks (formatting, extraction, simple Q&A) to a local Q4 model; escalate hard/ambiguous tasks (complex reasoning, long context) to a hosted frontier model. Add a confidence/verification gate: if the local output fails a validator (schema, faithfulness, low logprob), retry on hosted. Track per-route quality with evals and per-route cost; tune the threshold to hit the cost target while keeping quality SLO. Cache identical requests. This routing layer lives in the orchestration tier from Module 01's gateway design.

11 · Knowledge check

1 · What is the primary purpose of quantization?
Quantization stores weights in fewer bits (e.g. 4-bit), shrinking the model ~3–4× so it fits consumer VRAM and runs faster, at a small accuracy cost.
2 · Roughly how much VRAM does a 7B model at Q4 need?
~4.5 bits × 7B / 8 ≈ 4.4 GB of weights, ×1.2 for KV cache/activations ≈ 5–6 GB. That's why it runs on an 8 GB GPU.
3 · You need to serve 200 concurrent users with a local model. Which tool?
Ollama targets single-box/dev use. High concurrency needs continuous batching and a paged KV cache — vLLM/TGI's specialty.
4 · What does a Modelfile let you do?
A Modelfile is like a Dockerfile for models: pin a base, set parameters (temperature, num_ctx, stop), and a system prompt, then ollama create a custom tag.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q What two things consume GPU memory when running a local LLM? ›
The quantized model weights (fixed) and the KV cache (grows with context length × layers, shrunk by GQA). Activations add a small amount.
Q What is GGUF? ›
A single-file format for quantized models used by llama.cpp/Ollama, storing tensors plus metadata (architecture, context length, tokenizer, chat template) — a fully self-describing model file.
Q Give the napkin formula for VRAM need. ›
VRAM ≈ (params × bits/weight ÷ 8) × 1.2. The 1.2 covers KV cache + activations at modest context; it rises with longer context.
Q When do you choose Gemma over Qwen/Llama? ›
When you need maximum efficiency on modest/edge hardware (small 2B/9B sizes, fast, low footprint) and the task doesn't require the heaviest reasoning.
Q What's the difference between TTFT and tok/s? ›
TTFT (time to first token) is perceived responsiveness — how long until output starts. tok/s (tokens per second) is generation throughput — how fast the rest streams.
