Foundations of Large Language Models

Before you build RAG pipelines or multi-agent systems, you need an unshakable mental model of what an LLM actually is: a probabilistic next-token predictor wrapped around a transformer. Everything else in this course is a technique for steering, grounding, or orchestrating this one primitive.

⏱ Study + practice · 8–12 h Difficulty · Beginner→Intermediate Prereq · Python, basic ML

1 · Executive summary

A Large Language Model (LLM) is a neural network trained on a vast corpus of text to do one deceptively simple thing: given a sequence of tokens, predict the probability distribution of the next token. That's it. Chat, summarization, code generation, reasoning, tool-calling — all of it emerges from repeatedly sampling the next token.

❓

Why it exists

To capture the statistical structure of human language and knowledge in a single differentiable function you can query in natural language.

🔧

Problem it solves

One model replaces thousands of bespoke NLP pipelines (classification, extraction, generation) with a single promptable interface.

🌐

Where it fits

The LLM is the reasoning core. Embeddings, vector DBs, RAG, and agents exist to feed it the right context and act on its output.

🔑
The one sentence to remember

An LLM is a function f(tokens) → probability over next token. Generation is calling that function in a loop. Every advanced technique in GenAI is about what you put into the tokens and what you do with the output.

2 · First-principles breakdown

What — the autocomplete that learned to reason

Imagine the world's most sophisticated phone autocomplete. You give it "The capital of France is" and it confidently suggests "Paris". Now scale that intuition up by a trillion: train on most of the public internet, books, and code, with a network large enough to compress patterns about grammar, facts, logic, and style. The result isn't a database — it's a compressed, probabilistic model of how text tends to continue.

Crucially, the model has no memory between calls and no concept of truth. It produces the most statistically plausible continuation given its training. When plausible and true diverge, you get a hallucination — a central theme we'll fix with RAG in later modules.

Why — emergence from scale

Why does "predict the next word" produce something that can write code or pass exams? Because to predict the next token well across billions of diverse examples, the network is forced to learn the underlying structure: syntax, semantics, world facts, and reasoning shortcuts. Capabilities like in-context learning and chain-of-thought emerge once the model and data are large enough — they were never explicitly programmed.

How — the generation loop

Text generation is autoregressive: the model predicts one token, appends it to the input, and repeats. The expensive forward pass through the transformer happens once per token.

Your prompt (text)"Explain RAG in one line"
Tokenizertext → token IDs
Transformer forward pass→ logits over vocabulary
Sampling (temperature, top-p)pick the next token
Append token → loop until stoprepeat per token
Detokenize → answer
📐
Feynman check

If you can't explain to a colleague why generating a 500-token answer costs roughly 500 forward passes (and why that makes long outputs slow and expensive), re-read the loop above before moving on.

4 · Deep technical explanation

Tokenization — the model never sees characters

Models operate on tokens, not words or characters. Modern LLMs use Byte-Pair Encoding (BPE) or similar subword schemes: common words become a single token, rare words split into pieces. A rough rule: 1 token ≈ 4 characters ≈ 0.75 words in English.

⚠️
Why this matters in production

Pricing, latency, and context limits are all measured in tokens — not words. Non-English text and code often tokenize less efficiently (more tokens per character), inflating cost. Always estimate with a real tokenizer, never by counting words.

Logits, softmax, and the probability distribution

The final transformer layer outputs a logit (raw score) for every token in the vocabulary (often 100k+ tokens). A softmax turns these into a probability distribution that sums to 1. Decoding is how we pick from that distribution.

Figure 2 · The model outputs a distribution, not a sentence. Decoding selects one next token from that distribution.

Decoding strategies & the temperature dial

StrategyHow it picksUse when
GreedyAlways the highest-probability tokenDeterministic extraction, structured output
TemperatureScales logits before softmax. Low = sharp/safe, high = flat/creative0.0–0.3 for facts & code; 0.7–1.0 for ideation
Top-kSample only from the k most likely tokensCap the tail of nonsense tokens
Top-p (nucleus)Sample from smallest set whose cumulative prob ≥ pThe modern default; adapts to confidence

The context window & the KV cache

The context window is the maximum number of tokens (prompt + generated output) the model can attend to at once — 8k, 128k, 200k, or 1M depending on the model. Two consequences every senior engineer internalizes:

  • Cost & latency grow with context. Attention is roughly quadratic in sequence length; a 100k-token prompt is dramatically more expensive than 2k.
  • The KV cache stores the keys/values of already-processed tokens so each new token only attends to cached state instead of recomputing everything — this is what makes generation feasible, and what consumes GPU memory in local deployment (Module 03).

Training stages — why a model "follows instructions"

1 · Pre-training

Self-supervised next-token prediction on trillions of tokens. Produces a base model that completes text but doesn't reliably follow instructions.

2 · Supervised fine-tuning (SFT)

Train on curated instruction→response pairs so the model learns the "assistant" format.

3 · Preference alignment (RLHF / DPO)

Optimize toward responses humans prefer, improving helpfulness, harmlessness, and honesty.

From language model to production GenAI system

The trained model is only one component. A useful application surrounds it with context, controls and measurable software behavior.

Figure 3 · Production GenAI is a system, not only a model: context, authorization, tools, validation and evaluation determine reliability.

Tradeoffs & limitations

Strengths

  • Zero/few-shot generalization
  • Natural-language interface
  • Strong on synthesis & style

Hard limits

  • Hallucinates plausible falsehoods
  • Knowledge frozen at training cutoff
  • No native access to private/live data → RAG
  • Stateless between calls → memory/agents

6 · Practical Python

Minimal — call an LLM & inspect tokens

minimal.py
# Provider-agnostic call. Swap the client for OpenAI, Anthropic, or local Ollama.
from anthropic import Anthropic

client = Anthropic()  # reads ANTHROPIC_API_KEY from env

resp = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=200,
    temperature=0,            # deterministic — best for facts
    messages=[{"role": "user", "content": "Explain RAG in one sentence."}],
)
print(resp.content[0].text)
print("input tokens:", resp.usage.input_tokens,
      "output tokens:", resp.usage.output_tokens)

Intermediate — count tokens & estimate cost before you call

estimate.py
# tiktoken works for OpenAI-family tokenization; use it as a fast estimator.
import tiktoken

enc = tiktoken.get_encoding("cl100k_base")

def estimate_cost(prompt: str, expected_out: int = 300,
                  in_per_mtok=3.0, out_per_mtok=15.0) -> dict:
    n_in = len(enc.encode(prompt))
    cost = (n_in / 1e6) * in_per_mtok + (expected_out / 1e6) * out_per_mtok
    return {"input_tokens": n_in, "est_usd": round(cost, 5)}

print(estimate_cost("Summarize this 2-page contract..." * 50))
# {'input_tokens': 750, 'est_usd': 0.00675}

Production — a resilient client with retries, streaming & logging

llm_client.py
import time, logging
from anthropic import Anthropic, APIStatusError, APIConnectionError

log = logging.getLogger("llm")

class LLMClient:
    """Thin production wrapper: retries, timeout, token accounting, streaming."""
    def __init__(self, model="claude-sonnet-4-6", max_retries=3):
        self.client = Anthropic(timeout=30)
        self.model, self.max_retries = model, max_retries

    def complete(self, prompt, *, temperature=0, max_tokens=1024):
        for attempt in range(self.max_retries):
            try:
                t0 = time.perf_counter()
                r = self.client.messages.create(
                    model=self.model, max_tokens=max_tokens, temperature=temperature,
                    messages=[{"role": "user", "content": prompt}],
                )
                log.info("llm.ok", extra={
                    "latency_ms": int((time.perf_counter()-t0)*1000),
                    "in_tok": r.usage.input_tokens, "out_tok": r.usage.output_tokens})
                return r.content[0].text
            except (APIStatusError, APIConnectionError) as e:
                wait = 2 ** attempt          # exponential backoff
                log.warning(f"retry {attempt} after {wait}s: {e}")
                time.sleep(wait)
        raise RuntimeError("LLM unavailable after retries")

    def stream(self, prompt, **kw):           # token-by-token for responsive UIs
        with self.client.messages.stream(
            model=self.model, max_tokens=kw.get("max_tokens", 1024),
            messages=[{"role": "user", "content": prompt}]) as s:
            for text in s.text_stream:
                yield text
📌

This LLMClient is the seed of the production patterns in Module 14. Notice the three pillars already present: retries, token accounting, and streaming.

7 · Ollama implementation

You don't need a cloud API to learn. Ollama runs open models locally with a one-line install and an OpenAI-compatible API. We cover it deeply in Module 03; here's the foundational loop.

setup.sh
# 1. Install (macOS/Linux/Windows) from ollama.com, then:
ollama pull llama3.2        # Meta Llama — great general default
ollama pull qwen2.5         # Qwen — strong reasoning & multilingual
ollama pull gemma2          # Google Gemma — small & efficient

# 2. Chat from the terminal
ollama run llama3.2 "Explain next-token prediction in 2 lines"

# 3. Serve an HTTP API on localhost:11434 (auto-started)
ollama serve
local_llm.py
# Ollama exposes an OpenAI-compatible endpoint — reuse familiar tooling.
from openai import OpenAI

local = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

resp = local.chat.completions.create(
    model="qwen2.5",
    temperature=0.2,
    messages=[{"role": "user", "content": "What is a context window?"}],
)
print(resp.choices[0].message.content)

8 · Industry perspective

🚀 Startups

Start with a hosted frontier model (fastest path to product). Optimize cost later via prompt caching, smaller models for easy tasks, and routing. Speed-to-market beats infra ownership early.

🏢 Enterprises

Care about data residency, audit, and SLAs. Often run models in a VPC, behind a gateway that adds auth, logging, PII redaction, and cost controls. Local/open models (Module 03) appear where data can't leave the perimeter.

The canonical production architecture

API gatewayauth · rate limit · logging

Common production challenges: non-determinism breaks classic testing; latency spikes on long contexts; cost scales with usage in surprising ways; and prompt changes silently regress behavior. Every one of these gets a dedicated treatment in Module 14.

9 · Top mistakes engineers make

❌ Treating the LLM as a database

It recalls patterns, not facts. For anything that must be correct or current, retrieve it and put it in context (RAG).

❌ Counting words instead of tokens

Cost, latency and limits are token-based. Always estimate with a real tokenizer.

❌ Using temperature > 0 for extraction

Structured output and classification want determinism. Set temperature to 0.

❌ Ignoring the context-cost curve

Stuffing 100k tokens "to be safe" multiplies cost and latency and can hurt accuracy (lost-in-the-middle). Retrieve precisely.

❌ No retries or timeouts

LLM APIs fail transiently. Production code needs backoff, timeouts, and graceful degradation.

❌ Assuming determinism

Even at temperature 0, hardware/batching can introduce tiny variations. Test behaviors, not exact strings.

10 · Interview preparation

Beginner  What is an LLM, in one sentence? ›

A neural network (transformer) trained to predict the next token in a sequence; by sampling tokens repeatedly it generates coherent text. It models the statistical structure of language, not a database of facts.

Beginner  What is a token and why use tokens instead of words? ›

A token is a subword unit produced by a tokenizer (e.g., BPE). Subwords give a fixed-size vocabulary that handles any word — common words are one token, rare/novel words split into pieces — avoiding both a huge word vocabulary and a too-granular character model.

Intermediate  Explain temperature, top-k, and top-p. ›

Temperature scales logits before softmax — lower sharpens toward the top token, higher flattens for diversity. Top-k restricts sampling to the k highest-probability tokens. Top-p (nucleus) restricts to the smallest set whose cumulative probability ≥ p, adapting to model confidence. Use temperature 0 for deterministic tasks; top-p ≈ 0.9 with moderate temperature for natural generation.

Intermediate  Why is generating long outputs slow and expensive? ›

Generation is autoregressive — one transformer forward pass per output token. Output cost grows linearly with tokens generated, and attention cost grows with total sequence length. The KV cache avoids recomputing past tokens but consumes memory proportional to context length.

Senior  A model gives confidently wrong answers about your internal docs. Diagnose and fix. ›

The model has no knowledge of private data — it's hallucinating plausible text. Fix by grounding: retrieve relevant internal documents and inject them into context (RAG, Modules 6–7), instruct the model to answer only from provided context and cite sources, and add an evaluation harness to measure faithfulness. Lower temperature and add a "say 'I don't know'" instruction to reduce fabrication.

Senior  How do you make LLM features testable in CI given non-determinism? ›

Pin temperature to 0; assert on properties not exact strings (schema validity, presence of required fields, no PII, factuality against references); use an LLM-as-judge or rubric eval for open-ended outputs; snapshot-test prompts and version them; run a golden-dataset eval on every prompt change to catch regressions.

Staff · design  Design a multi-tenant LLM gateway for an enterprise. ›

Single entry point that all teams call. Responsibilities: auth & tenancy (API keys per team, quotas), routing (model selection by task/cost, fallback chains), safety (PII redaction, input/output guardrails), observability (per-tenant token/cost metrics, tracing, prompt logging with retention policy), resilience (retries, circuit breakers, caching of identical/cacheable prompts), and cost control (budgets, alerts). Keep it stateless and horizontally scalable; push prompt templates to a versioned registry. This is the spine of the Enterprise GenAI Platform capstone.

11 · Knowledge check

1 · What does an LLM fundamentally compute on each forward pass?
An LLM outputs logits → softmax → a probability distribution over the vocabulary for the next token. Sampling then selects one token; the loop repeats.
2 · You're extracting structured fields from invoices. Which temperature?
Extraction is a precision task. Temperature 0 (greedy) gives the most consistent, repeatable structured output and minimizes fabrication.
3 · Why might a 120k-token prompt reduce accuracy, not just cost?
Long contexts increase cost and latency, and empirically models retrieve facts in the middle of very long contexts less reliably. Precise retrieval beats stuffing.

12 · Active recall

Answer in your head first, then reveal. Revisit these tomorrow and in a week (spaced repetition).

Q Describe the autoregressive generation loop in one sentence. ›
The model predicts a distribution for the next token, samples one, appends it to the input, and repeats the forward pass until a stop token or max-length is reached.
Q What is the rough token-to-word ratio in English? ›
About 1 token ≈ 0.75 words (≈ 4 characters). Code and non-English text typically use more tokens per character.
Q What does the KV cache store and why? ›
The keys and values of already-processed tokens, so each new token attends to cached state instead of recomputing the whole sequence — trading GPU memory for speed.
Q Name the three training stages that turn raw text prediction into a helpful assistant. ›
Pre-training (next-token prediction) → Supervised fine-tuning (instruction pairs) → Preference alignment (RLHF/DPO).
