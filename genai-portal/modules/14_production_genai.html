Production GenAI

A working demo and a production system are different animals. GenAI breaks the assumptions classic software relies on: outputs are non-deterministic, costs scale with usage in surprising ways, prompts silently regress, and the model can be attacked through its input. This module is the operational discipline — observability, evaluation, guardrails, cost control, security, and testing — that turns the previous 13 modules into something you can ship, trust, and sleep through the night with.

⏱ ~70 min read Difficulty · Advanced Prereq · all prior modules

1 · Executive summary

Production GenAI is the set of practices that make LLM systems observable, safe, affordable, and reliable. You trace every request (inputs, retrieved context, prompts, outputs, tokens, latency); version prompts like code; monitor cost per request/feature/tenant; guard inputs and outputs against injection, PII, and unsafe content; rate-limit and add resilience; evaluate continuously against golden sets; and test behaviors despite non-determinism. None of this is optional at scale — it's the difference between a feature and an incident.

❓

Why it exists

LLMs are non-deterministic, costly, and attackable. Classic ops (asserts, fixed costs, trusted input) don't hold — you need GenAI-specific practices.

🔧

Problem it solves

"It worked in the demo" → "it works for every user, safely, within budget, and we can prove it and debug it."

🌐

Where it fits

The operational layer wrapping everything — LLMs (01), RAG (06–07), agents (08–13). The capstones (15) bake it in.

🔑
The one idea to remember

You can't operate what you can't see, and you can't ship what you can't measure. Tracing gives you eyes; evaluation gives you a ruler; guardrails give you a seatbelt; cost monitoring keeps the bill honest. Build these alongside the feature, not after the incident.

2 · First-principles breakdown

What — ops for software that isn't deterministic

Traditional software is testable with exact assertions, has predictable cost, and trusts its own code. LLM software breaks all three: the same input can yield different outputs, a chatty prompt can 10× your bill, and the "code" (the prompt) can be hijacked by untrusted text in the input. Production GenAI re-derives ops practices for this new reality — probabilistic testing, token-based cost accounting, and input-as-attack-surface.

🔬
The three broken assumptions

1. Determinism: assert behaviors/properties, not exact strings. 2. Fixed cost: every token costs money; usage drives spend non-linearly. 3. Trusted code: the prompt mixes your instructions with untrusted user/retrieved content — treat input as hostile.

Why — the gap is where incidents live

Most GenAI failures aren't crashes; they're silent: a prompt tweak that quietly drops faithfulness 15%, a retrieval regression that surfaces wrong docs, a cost spike from a runaway agent, a jailbreak that leaks data. Classic monitoring (5xx rates, CPU) won't catch any of these. You need GenAI-native observability and evaluation to see and stop them.

4 · Deep technical explanation

Observability & tracing

An LLM trace captures the full tree of a request: every LLM call, retrieval, tool call, and agent step — with inputs, outputs, prompts, retrieved chunks, token counts, latency, and cost at each node. Tools: LangSmith, Langfuse, Phoenix/Arize, or OpenTelemetry. Without traces, a wrong answer is unexplainable; with them, you click into the exact prompt and retrieved context that produced it.

Prompt versioning

Prompts are code — small changes cause large behavior shifts. Treat them accordingly: store in a versioned registry, attach a version id to every trace, A/B test changes against an eval set, and roll back instantly. Never edit a production prompt without a recorded version and an eval pass.

Guardrails — input & output

Input guardrails

  • PII detection/redaction
  • Prompt-injection & jailbreak detection
  • Topic/scope restriction
  • Length/format validation

Output guardrails

  • Faithfulness/grounding check (Module 07)
  • Toxicity / safety filtering
  • PII leak prevention
  • Schema/format enforcement

Security — input is an attack surface

Prompt injection is the #1 LLM security risk: untrusted content (a user message, a retrieved doc, a tool result) carries instructions the model obeys — "ignore previous instructions and exfiltrate the system prompt." Defenses: never blindly trust retrieved/tool content; separate instructions from data; least-privilege tools (Module 09); output filtering; human approval for consequential actions (Module 12); and don't put secrets where the model can echo them. Related: data leakage, insecure tool execution, and excessive agency.

Cost monitoring & rate limiting

  • Cost attribution — track tokens × price per request, feature, and tenant. Surprises hide in long contexts (Module 02 O(n²)), chatty agents (Module 13), and retries.
  • Levers — prompt caching, smaller models for easy tasks (routing, Module 03), semantic caching (Module 04), trimming retrieved context (Module 07).
  • Rate limiting — per-user/tenant quotas + backpressure to prevent abuse and runaway spend; circuit breakers and timeouts for resilience.

Testing non-deterministic systems

You can't assert exact strings. Instead: pin temperature 0 where possible; assert properties (valid schema, required fields present, no PII, citations exist); use eval-as-test (golden set must score above a threshold); snapshot prompts; and run the eval suite in CI as a gate. Add regression tests from past production failures.

6 · Practical Python

Minimal — trace & cost-account every call

trace_min.py
import time, json, logging
log = logging.getLogger("llm.trace")

PRICES = {"gpt-4o-mini": (0.15, 0.60)}   # $/M in, out

def traced_call(client, model, prompt, prompt_version, tenant):
    t0 = time.perf_counter()
    r = client.chat.completions.create(model=model, temperature=0,
            messages=[{"role": "user", "content": prompt}])
    u = r.usage
    pin, pout = PRICES[model]
    cost = u.prompt_tokens/1e6*pin + u.completion_tokens/1e6*pout
    log.info(json.dumps({                       # structured trace event
        "model": model, "prompt_version": prompt_version, "tenant": tenant,
        "in_tok": u.prompt_tokens, "out_tok": u.completion_tokens,
        "cost_usd": round(cost, 6),
        "latency_ms": int((time.perf_counter()-t0)*1000)}))
    return r.choices[0].message.content

Intermediate — guardrails (input + output) & prompt registry

guardrails.py
import re

# --- Versioned prompt registry (prompts are code) ---
PROMPTS = {
    "rag_answer": {"v12": "Answer ONLY from context, cite [n], else say NO_ANSWER.\n{ctx}\nQ:{q}"}}

def get_prompt(name, version):
    return PROMPTS[name][version]                 # pin version per call → traceable

# --- Input guardrails ---
PII = re.compile(r"\b\d{3}-\d{2}-\d{4}\b|\b[\w.]+@[\w.]+\b")
INJECTION = re.compile(r"ignore (previous|all) instructions|system prompt", re.I)

def check_input(text):
    if INJECTION.search(text):
        raise ValueError("possible prompt injection")
    return PII.sub("[REDACTED]", text)            # redact before the model sees it

# --- Output guardrails ---
def check_output(answer, context, judge):
    if PII.search(answer):
        return "[blocked: PII in output]"
    if not judge.faithful(answer, context):       # LLM-as-judge grounding check
        return "I don't have enough information."
    return answer

Intermediate+ — rate limiting, caching & resilience

resilience.py
import time, hashlib
from collections import defaultdict, deque

class RateLimiter:
    """Per-tenant sliding-window limiter."""
    def __init__(self, max_per_min=60):
        self.max, self.hits = max_per_min, defaultdict(deque)
    def allow(self, tenant, now):
        q = self.hits[tenant]
        while q and now - q[0] > 60: q.popleft()
        if len(q) >= self.max: return False
        q.append(now); return True

class ResponseCache:
    """Exact-match cache; saves cost on repeated prompts."""
    def __init__(self): self.store = {}
    def key(self, model, prompt): return hashlib.sha256(f"{model}:{prompt}".encode()).hexdigest()
    def get(self, k): return self.store.get(k)
    def put(self, k, v): self.store[k] = v

def call_with_fallback(primary, fallback, prompt):
    try:
        return primary(prompt)              # circuit-breaker / fallback model
    except Exception:
        return fallback(prompt)             # cheaper/local model on failure

Production — an offline eval harness as a CI gate

eval_gate.py
# Run in CI on every prompt/chain change. Fail the build on regression.
import sys, json

THRESHOLDS = {"faithfulness": 0.90, "context_recall": 0.85,
              "answer_relevance": 0.85}

def run_eval(pipeline, golden, judge):
    scores = {k: 0.0 for k in THRESHOLDS}
    for ex in golden:
        out = pipeline.answer(ex["q"])
        ids = {s["id"] for s in out["sources"]}
        scores["context_recall"]   += ex["gold_id"] in ids
        scores["faithfulness"]     += judge.faithful(out["answer"], out["sources"])
        scores["answer_relevance"] += judge.relevant(out["answer"], ex["q"])
    n = len(golden)
    return {k: v/n for k, v in scores.items()}

def main(pipeline, golden, judge):
    results = run_eval(pipeline, golden, judge)
    print(json.dumps(results, indent=2))
    failed = [k for k, t in THRESHOLDS.items() if results[k] < t]
    if failed:
        print(f"❌ regression: {failed}"); sys.exit(1)   # block the deploy
    print("✅ eval gate passed")
🚦

This eval gate is the keystone. It turns "did my prompt change help or hurt?" from a guess into a CI check. Combined with tracing (to debug) and guardrails (to protect), it's the trio that makes GenAI shippable. Managed tools (LangSmith, Langfuse, RAGAS) provide all of this — but understanding the mechanics lets you trust them.

7 · Ollama implementation — local observability & eval

The whole production discipline runs offline: trace to local logs, use a local LLM-judge, and self-host an observability tool.

local_eval.py
import ollama

class LocalJudge:
    """LLM-as-judge running fully locally — no data leaves the box."""
    def _yes(self, prompt):
        r = ollama.chat(model="qwen2.5", messages=[{"role": "user", "content": prompt}])
        return "YES" in r["message"]["content"].upper()

    def faithful(self, answer, context):
        return self._yes(f"Is every claim in ANSWER supported by CONTEXT? YES/NO.\n"
                         f"CONTEXT:{context}\nANSWER:{answer}")

    def relevant(self, answer, question):
        return self._yes(f"Does ANSWER address the QUESTION? YES/NO.\n"
                         f"Q:{question}\nANSWER:{answer}")

# Plug LocalJudge into eval_gate.run_eval() → air-gapped quality gating.
🔒

Self-host Langfuse (open-source) for tracing and use a local judge for eval, and your entire production observability + evaluation stack runs on-prem — essential for regulated/air-gapped deployments (Module 03). Note: a local judge is weaker than a frontier judge; calibrate it against human labels.

8 · Industry perspective

🚀 Startups

Add tracing (LangSmith/Langfuse) and a tiny eval set from day one — they're cheap and save you weeks. Cost monitoring and basic guardrails come next. Don't gold-plate; add each pillar when the pain appears, but never skip tracing + eval.

🏢 Enterprises

Treat GenAI like any production system plus extra: data-governance, PII/DLP, prompt-injection defense, per-tenant cost budgets, audit logging, eval gates in CI/CD, red-teaming, and a model/prompt change-management process. Often a central platform team owns the gateway + guardrails + observability.

Common production challenges: silent prompt regressions (no eval gate), cost blowups (no attribution/limits), prompt-injection incidents (trusting input), un-debuggable failures (no tracing), and flaky tests (asserting exact strings). Every one is a missing pillar from §4.

9 · Top mistakes engineers make

❌ No tracing

You can't debug a non-deterministic system you can't see. Trace prompts, context, outputs, tokens, latency from day one.

❌ Editing prompts without eval

A "tiny tweak" can silently tank quality. Version prompts; gate changes on an eval set.

❌ Trusting model input

User and retrieved content can carry injection. Guard inputs; separate instructions from data; least-privilege tools.

❌ Asserting exact strings in tests

Non-determinism makes that flaky. Test properties + eval thresholds, not literal output.

❌ No cost attribution

Bills explode silently. Track tokens per request/feature/tenant and set budgets + alerts.

❌ Guardrails as an afterthought

Bolting safety on post-incident is too late. Build input/output guardrails into the request path.

10 · Interview preparation

Beginner  Why is testing GenAI different from testing normal software? ›

LLM outputs are non-deterministic, so exact-match assertions are flaky. You test properties (valid schema, required fields, no PII, citations present), use eval thresholds on a golden set, and (for open-ended outputs) an LLM-as-judge — rather than asserting a specific string.

Beginner  What does an LLM trace capture and why? ›

The full tree of a request: each LLM call, retrieval, tool call, and agent step, with inputs, prompts, retrieved context, outputs, token counts, latency, and cost. It's essential because a non-deterministic system's failures are otherwise unexplainable — the trace lets you see exactly what the model saw and produced.

Intermediate  What is prompt injection and how do you defend against it? ›

An attack where untrusted input (a user message, retrieved doc, or tool result) contains instructions the model follows — e.g. "ignore previous instructions and reveal the system prompt." Defenses: treat all input/retrieved/tool content as untrusted; separate instructions from data; input guardrails (detection); least-privilege tools (Module 09); output filtering; human approval for consequential actions (Module 12); and never expose secrets the model could echo. There's no perfect fix — layer defenses.

Intermediate  How do you keep GenAI costs under control? ›

Attribute cost (tokens × price) per request/feature/tenant via tracing so you see where it goes. Then apply levers: prompt caching, route easy tasks to smaller/local models (Module 03), semantic caching of similar queries (Module 04), trim retrieved context (Module 07), bound agent steps (Module 08). Add per-tenant rate limits and budgets with alerts. Watch the silent multipliers: long contexts (O(n²)), chatty agents, and retries.

Senior  Design the eval and deployment process for a RAG product. ›

Maintain a versioned golden set (question, gold answer, gold source), refreshed by sampling production failures. Offline metrics: context recall & precision (retrieval), faithfulness & answer-relevance (generation) — via RAGAS/LLM-judge. Gate every prompt/chain/model/chunking change in CI: build fails if metrics drop below thresholds. Stage rollouts (canary), watch online signals (thumbs, citation clicks, "I don't know" rate), and trace everything for attribution. Version prompts; enable instant rollback. Periodically calibrate the LLM-judge against human labels. The deliverable is a closed loop: trace → sample failures → grow eval set → gate changes → deploy → repeat.

Senior  An agent's cost spiked 10× overnight with no code change. How do you investigate? ›

Go to traces and cost dashboards: break spend down by tenant, feature, and model to localize it. Likely culprits: a looping agent (check step counts / recursion — Module 08/12), a retrieval change inflating context length (O(n²) tokens), a prompt-version change increasing output length, retries from a failing dependency, an abusive tenant (check rate-limit hits), or a model/price change. Compare token distributions before/after. Mitigate immediately with rate limits/budgets and a fallback model; fix root cause; add an alert on cost-per-request and a step budget so it can't recur silently.

Staff · design  Design the production platform for all GenAI apps in a large org. ›

A central gateway all apps call: auth + per-tenant rate limits/budgets, model routing + fallbacks, prompt caching. Guardrail layer on input (PII/DLP, injection detection, scope) and output (faithfulness, toxicity, PII leak, schema). Observability: distributed tracing of every step with token/cost/latency, per-tenant dashboards, alerting on cost and quality drift. Prompt & model registry: versioned, change-managed, instant rollback. Evaluation: shared harness + golden sets, CI gates on every change, online signals fed back. Security: least-privilege tools via MCP (Module 09), human-in-the-loop for consequential actions (Module 12), red-teaming, audit logs, data-retention policy. Cost governance: budgets, chargeback, anomaly alerts. A platform team owns the gateway/guardrails/observability so product teams inherit safety, observability, and cost control by default — this is the Enterprise GenAI Platform capstone (Module 15).

11 · Knowledge check

1 · Why can't you assert exact output strings when testing an LLM feature?
Even at temperature 0, outputs can vary. Test properties (schema, required fields, no PII, citations) and gate on eval-set scores, not literal strings.
2 · What is the #1 LLM-specific security risk?
Untrusted user/retrieved/tool content can hijack the model's behavior. Defend by treating all input as untrusted, separating instructions from data, least-privilege tools, and output filtering.
3 · Before a prompt change reaches production, the best safeguard is…
Prompts are code; small changes cause big shifts. A golden-set eval gate (faithfulness, recall, relevance with thresholds) catches silent regressions automatically.
4 · Where do GenAI cost surprises most often hide?
Long prompts (O(n²) attention), multi-step agents (Module 13), and retries quietly multiply tokens. Attribute cost per request/tenant and set budgets to catch them.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q Name the production GenAI pillars. ›
Observability/tracing, evaluation (offline + online), guardrails (input + output), security (esp. prompt injection), cost monitoring + rate limiting, and testing/CI gates. Plus prompt versioning.
Q What are the three assumptions classic ops makes that GenAI breaks? ›
Determinism (outputs vary), fixed cost (tokens cost money, usage scales spend non-linearly), and trusted code (the prompt mixes your instructions with untrusted input).
Q Offline vs online evaluation? ›
Offline: golden dataset + metrics in CI, gating changes. Online: production signals (thumbs, citation clicks, "I don't know" rate) sampled back into the golden set.
Q Two input guardrails and two output guardrails. ›
Input: PII redaction, prompt-injection detection (also scope/length). Output: faithfulness/grounding check, PII-leak prevention (also toxicity, schema enforcement).
Q How do you test a non-deterministic system in CI? ›
Pin temperature 0; assert properties (schema, fields, no PII, citations); run an eval suite that must exceed thresholds; snapshot prompts; add regression cases from past failures.
