Model Context Protocol (MCP)

In Module 08 you wired tools into an agent by hand — schemas, execution, glue code — bespoke to one app and one framework. MCP is the open standard that makes that integration universal: write a tool server once, and any MCP-capable client (Claude Desktop, IDEs, your own agent) can use it. It's the "USB-C of AI tooling" — one protocol so models and data sources plug together without custom adapters.

⏱ Study + practice · 8–12 h Difficulty · Intermediate Prereq · Module 08, FastAPI basics

1 · Executive summary

The Model Context Protocol (MCP) is an open standard (introduced by Anthropic, now broadly adopted) that defines how applications expose context and capabilities to LLMs. An MCP server publishes three things — tools (actions the model can invoke), resources (data the model can read), and prompts (reusable templates) — over a standard JSON-RPC protocol. An MCP client (inside the host app) discovers and calls them. Build the server once; reuse it everywhere.

❓

Why it exists

Every tool integration was custom per app × per framework — an M×N explosion. MCP collapses it to M+N: tools and clients share one protocol.

🔧

Problem it solves

Standardized, discoverable, reusable tool/data access for LLMs — with a clean security boundary between the model and your systems.

🌐

Where it fits

The interoperability layer for agents (08). LangGraph/CrewAI agents consume MCP servers; your services become MCP servers.

🔑
The one idea to remember

MCP standardizes the agent–tool boundary. Tools = model can do; Resources = model can read; Prompts = reusable templates. Write a server once; every MCP client gets your capabilities for free.

2 · First-principles breakdown

What — a universal adapter for AI capabilities

Before USB, every device had its own connector. Before MCP, every AI app integrated every tool its own way: your ElasticSearch tool for LangChain didn't work in Claude Desktop or in a CrewAI agent — you rewrote it each time. MCP defines a single plug. Expose your database, file system, or API as an MCP server, and any compliant client speaks to it through the same protocol.

🔌
The M×N → M+N insight

With M AI apps and N tools, bespoke integration needs up to M×N connectors. With MCP, each app implements the client once (M) and each tool implements the server once (N) → M+N. That's the same network-effect win that USB, HTTP, and LSP delivered in their domains.

Why — separation of concerns & security boundary

MCP cleanly splits responsibilities: the host (the AI app) manages the model and the conversation; the server owns access to a specific system (your DB, files, API) and decides exactly what it exposes. The model never touches your database directly — it asks the server, which enforces what's allowed. That boundary is a feature: least-privilege by construction.

How — three primitives, one protocol

PrimitiveWhat it isWho controls invocationExample
ToolAn action the model can call (has side effects or computation)Model-controlled — the LLM decides to call itrun_es_query, create_ticket
ResourceRead-only data the model/app can load into contextApp-controlled — host decides what to attachfile://report.pdf, a DB row
PromptA reusable, parameterized prompt templateUser-controlled — surfaced as a command/summarize-incident

4 · Deep technical explanation

The protocol

MCP rides on JSON-RPC 2.0. After a capability negotiation handshake (initialize), the client can call standard methods: tools/list, tools/call, resources/list, resources/read, prompts/list, prompts/get. The server responds with structured results. Because the schema is standardized, discovery is automatic — the host learns what a server offers at runtime.

Transports

TransportHowUse when
stdioServer is a local subprocess; messages over stdin/stdoutLocal tools, desktop apps, dev — simplest & default
Streamable HTTP / SSEServer is an HTTP endpoint; supports remote + streamingRemote/shared servers, multi-user, cloud deployment

Tool registration

A tool needs a name, a clear description (the model reads this to decide when to call it — Module 08), and a typed input schema. The official Python SDK (FastMCP) derives the schema from type hints and the docstring, so a well-typed function is a tool definition.

Resource & prompt registration

  • Resources are addressed by URI (e.g. es://indices, file:///logs/today.log). The host reads them into context — they're data, not actions, so no side effects.
  • Prompts are named, parameterized templates the server offers (e.g. an "investigate incident" workflow). Clients surface them as slash-commands/menu items for the user to trigger.

Security model

The server is the trust boundary. Best practices: least privilege (expose only needed operations — read-only where possible), input validation on every tool, no secrets in tool descriptions, authn/authz for HTTP transports, and guarding against prompt injection via resource/tool content (untrusted data the model reads can carry instructions). Consequential tools should require host-side human approval (Module 08 §6).

Tradeoffs & limitations

Strengths

  • Write once, use in any MCP client
  • Automatic discovery of capabilities
  • Clean security/permission boundary
  • Decouples tools from agent frameworks

Limits

  • An extra layer (process/network) vs. direct calls
  • Young ecosystem; conventions still settling
  • Auth/remote story is evolving
  • Overkill for a single in-process tool

5 · Interactive: discover & call

Step through how a host discovers a server's capabilities and routes a model's tool call.

DEMO Host ↔ ElasticSearch MCP server
🤝 initialize → negotiate capabilities 📋 tools/list → ["list_indices", "search_index", "get_mapping"] 💭 LLM: "user wants errors in app-logs → call search_index" ⚡ tools/call search_index({index:"app-logs", query:"level:ERROR"}) 👁 server runs ES query → returns 23 hits ✅ "Found 23 errors, mostly timeout in service X."

The host never hard-coded these tools — it discovered them via tools/list. Swap in a different server and the same host gains new capabilities with zero code changes.

6 · Practical Python — build an MCP server

Minimal — a FastMCP server with one tool

server_minimal.py
# pip install "mcp[cli]"   — the official Python SDK
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("demo-server")

@mcp.tool()
def add(a: int, b: int) -> int:
    """Add two integers and return the sum."""   # docstring = tool description
    return a + b

@mcp.tool()
def greet(name: str) -> str:
    """Return a friendly greeting for the given name."""
    return f"Hello, {name}!"

if __name__ == "__main__":
    mcp.run()                # stdio transport by default — type hints become the schema

Intermediate — tools + resources + prompts together

server_full.py
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("knowledge-server")
NOTES = {"runbook": "Restart service X via systemctl..."}

@mcp.tool()                                  # ACTION (model-controlled)
def save_note(key: str, text: str) -> str:
    """Persist a note under a key."""
    NOTES[key] = text
    return f"saved {key}"

@mcp.resource("note://{key}")              # DATA (app-controlled, read-only)
def read_note(key: str) -> str:
    """Read a stored note by key."""
    return NOTES.get(key, "(not found)")

@mcp.prompt()                                # TEMPLATE (user-controlled)
def incident_review(service: str) -> str:
    """Reusable incident-review prompt."""
    return (f"Investigate the latest incident for {service}. "
            "List symptoms, likely cause, and remediation steps.")

if __name__ == "__main__":
    mcp.run()

A client that consumes the server

client.py
import asyncio
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def main():
    params = StdioServerParameters(command="python", args=["server_full.py"])
    async with stdio_client(params) as (read, write):
        async with ClientSession(read, write) as s:
            await s.initialize()
            tools = await s.list_tools()           # discovery
            print("tools:", [t.name for t in tools.tools])
            res = await s.call_tool("save_note", {"key": "x", "text": "hi"})
            print(res.content)

asyncio.run(main())

Production — a FastAPI-backed MCP server (your existing API → MCP)

fastapi_mcp.py
# You already build FastAPI services. Two clean ways to expose them via MCP:
# (A) Wrap existing service functions as MCP tools (share business logic).
# (B) Mount MCP as an HTTP transport alongside your REST routes.
from fastapi import FastAPI
from mcp.server.fastmcp import FastMCP
import httpx

# --- shared business logic (used by both REST and MCP) ---
async def _lookup_order(order_id: str) -> dict:
    async with httpx.AsyncClient() as c:
        r = await c.get(f"http://orders-svc/orders/{order_id}")
        return r.json()

# --- REST surface (humans / existing clients) ---
api = FastAPI()
@api.get("/orders/{order_id}")
async def get_order(order_id: str):
    return await _lookup_order(order_id)

# --- MCP surface (LLM agents), HTTP transport for remote use ---
mcp = FastMCP("orders-mcp")

@mcp.tool()
async def lookup_order(order_id: str) -> dict:
    """Look up an order's status and items by its ID."""
    return await _lookup_order(order_id)   # same logic, zero duplication

# Serve MCP over Streamable HTTP (e.g. uvicorn) for remote agents.
app = mcp.streamable_http_app()        # mount alongside or separate from `api`
🧩

The senior pattern: keep business logic in plain functions and expose it through both REST (for humans/services) and MCP (for agents). One implementation, two interfaces — no drift. Your FastAPI + Docker skills transfer directly.

7 · Local MCP development & Ollama

MCP servers run locally over stdio — perfect for offline dev. Pair a local Ollama model (with tool calling, Module 08) as the host's brain and your MCP servers as its hands, for a 100% local agentic stack.

dev.sh
# Inspect/debug a server interactively with the MCP Inspector
mcp dev server_full.py        # opens a UI to list/call tools, read resources

# Register a local server with Claude Desktop (stdio)
# ~/.config/Claude/claude_desktop_config.json:
# { "mcpServers": { "knowledge": {
#     "command": "python", "args": ["/abs/path/server_full.py"] } } }
local_host.py
# Minimal "host": local Ollama model + an MCP server's tools.
import asyncio, json, ollama
from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client

async def run(user_msg):
    p = StdioServerParameters(command="python", args=["server_full.py"])
    async with stdio_client(p) as (r, w), ClientSession(r, w) as s:
        await s.initialize()
        listed = (await s.list_tools()).tools
        # translate MCP tool schemas → Ollama tool format
        tools = [{"type": "function", "function": {
            "name": t.name, "description": t.description,
            "parameters": t.inputSchema}} for t in listed]
        out = ollama.chat(model="qwen2.5",
            messages=[{"role": "user", "content": user_msg}], tools=tools)
        for call in out["message"].get("tool_calls", []):
            res = await s.call_tool(call["function"]["name"],
                                     call["function"]["arguments"])
            print(res.content)

asyncio.run(run("Save a note 'deploy' = 'use blue-green'"))

8 · ElasticSearch MCP server (worked example)

This is exactly the kind of server you'll build at work: wrap ElasticSearch so any agent can investigate logs and indices safely. Note the read-only, least-privilege design.

es_mcp_server.py
from mcp.server.fastmcp import FastMCP
from elasticsearch import Elasticsearch

mcp = FastMCP("elasticsearch")
es = Elasticsearch("http://localhost:9200")

ALLOWED = {"app-logs", "audit", "metrics"}    # allow-list = least privilege

def _check(index: str):
    if index not in ALLOWED:
        raise ValueError(f"index '{index}' not permitted")

@mcp.tool()
def list_indices() -> list[str]:
    """List the ElasticSearch indices available for querying."""
    return sorted(ALLOWED)

@mcp.tool()
def get_mapping(index: str) -> dict:
    """Return the field mapping (schema) of an index."""
    _check(index)
    return es.indices.get_mapping(index=index).body

@mcp.tool()
def search_index(index: str, query: str, size: int = 10) -> dict:
    """Run a Lucene query_string search and return the top hits.

    index: one of the allowed indices.
    query: Lucene syntax, e.g. 'level:ERROR AND service:checkout'.
    size:  max hits to return (capped at 50)."""
    _check(index)
    size = min(size, 50)                          # bound result size (cost/latency)
    res = es.search(index=index, size=size,
        query={"query_string": {"query": query}})
    hits = res["hits"]["hits"]
    return {"total": res["hits"]["total"]["value"],
            "hits": [{"id": h["_id"], "source": h["_source"]} for h in hits]}

@mcp.resource("es://indices")
def indices_resource() -> str:
    """The list of queryable indices, as a readable resource."""
    return "\n".join(sorted(ALLOWED))

@mcp.prompt()
def investigate(symptom: str) -> str:
    """Guided log-investigation prompt."""
    return (f"Investigate '{symptom}'. First list_indices, then get_mapping "
            "for the relevant one, then search_index. Summarize root cause.")

if __name__ == "__main__":
    mcp.run()
🛡️

Notice the safety design: an index allow-list, bounded result size, read-only operations only (no delete/update tools), and rich docstrings so the model queries correctly. This server is the heart of the MCP ElasticSearch Server capstone.

9 · Industry perspective

🚀 Startups

Use existing MCP servers (filesystem, GitHub, Postgres, Slack) to give an assistant capabilities in hours, not weeks. Build one custom server for the proprietary system and you're done.

🏢 Enterprises

Wrap internal systems (data warehouse, ticketing, ES) as governed MCP servers behind auth, with audit logging and least-privilege scopes. One server is reused across every internal AI app — the integration layer of an enterprise AI platform.

Why this matters strategically

MCP turns tool integrations into reusable infrastructure. Instead of each team re-wiring ElasticSearch into LangChain, then again into Claude Desktop, then again into a CrewAI agent, you publish one ES MCP server and every team/app consumes it. That's the difference between N teams × M frameworks of glue and a single maintained capability.

Common production challenges: auth for remote/multi-tenant servers, prompt-injection through resource/tool content, versioning tool schemas without breaking clients, observability across the host↔server boundary, and stopping over-broad tools (always prefer narrow, read-only ones).

10 · Top mistakes engineers make

❌ Exposing over-broad tools

A generic "run any SQL/ES query" or "execute shell" tool is a security hole. Expose narrow, validated, read-only operations.

❌ Weak tool descriptions

The model selects and fills tools from descriptions/schemas. Vague docstrings → wrong calls. Treat them as prompts.

❌ No input validation

The model can pass anything. Validate args and bound result sizes server-side — never trust the caller.

❌ Confusing tools and resources

Actions (side effects) = tools; read-only data = resources. Mixing them muddles the security model.

❌ Ignoring prompt injection

Content the model reads (a log line, a doc) can contain instructions. Treat resource/tool output as untrusted.

❌ Duplicating business logic

Don't reimplement logic in the MCP tool. Share plain functions between REST and MCP surfaces.

11 · Interview preparation

Beginner  What is MCP and what problem does it solve? ›

An open standard for how applications expose context and capabilities (tools, resources, prompts) to LLMs over a common JSON-RPC protocol. It replaces bespoke, per-app-per-tool integrations (M×N) with write-once servers any compliant client can use (M+N) — the "USB-C of AI tooling."

Beginner  Name MCP's three primitives. ›

Tools — actions the model can invoke (model-controlled). Resources — read-only data loaded into context (app-controlled). Prompts — reusable parameterized templates surfaced to the user (user-controlled).

Intermediate  Walk through host, client, and server roles. ›

The host is the AI app (Claude Desktop, IDE, your agent) — it runs the LLM and conversation. It spawns one client per server. Each server wraps a specific system (DB, files, ES) and exposes tools/resources/prompts. Flow: client initializes (capability negotiation) → lists tools → LLM picks one → client calls tools/call → server executes against the real system → result returns. The server is the security boundary.

Intermediate  Tool vs resource — when to use which? ›

Use a tool for actions/computation that the model decides to invoke (side effects: write a record, run a query). Use a resource for read-only data the host attaches to context (a file, a config, a row), addressed by URI with no side effects. The split keeps "what the model can do" separate from "what it can read," which clarifies permissions.

Senior  How do you secure an MCP server exposing ElasticSearch? ›

Least privilege: read-only tools only (no delete/update), an index allow-list, and bounded result sizes. Validate every argument server-side; never expose a raw "run any query" tool — offer constrained operations (search, get_mapping). For HTTP transport, add authn/authz and per-caller scopes. Treat returned log content as untrusted (prompt-injection defense). Log all calls for audit. Require host-side human approval for any consequential action. Keep secrets out of descriptions and out of the model's view.

Senior  Why prefer MCP over hard-coding tools in your agent framework? ›

Decoupling and reuse. Hard-coded tools are tied to one framework and app; an MCP server is consumed by any compliant client (Claude Desktop, IDEs, LangGraph, CrewAI), so one team maintains a capability once and every app benefits. It also enforces a clean security boundary and enables runtime discovery. The cost is an extra layer — justified when a tool is shared across apps; overkill for a single in-process helper.

Staff · design  Design an enterprise MCP gateway exposing 30 internal systems to AI apps. ›

A governed catalog of MCP servers, one per system, behind a central gateway providing: auth (SSO/OAuth, per-app and per-user identity propagated to servers), authorization (least-privilege scopes; the model only sees tools the caller may use), safety (input validation, output redaction/PII, prompt-injection defenses on returned content, human-in-the-loop for consequential tools), observability (trace every host↔server call, per-tool cost/latency, full audit log), schema governance (versioned tool schemas, deprecation policy, contract tests so client upgrades don't break), and reliability (rate limits, timeouts, circuit breakers per backend). Share business logic with existing REST services to avoid drift. The deliverable is reusable, auditable capability infrastructure — the integration spine of the Enterprise GenAI Platform (Module 15).

12 · Knowledge check

1 · What is the core value proposition of MCP?
MCP standardizes the agent–tool boundary. Write a server once and any compliant client can use it, collapsing bespoke per-app-per-tool integrations into a reusable layer.
2 · Which primitive should expose "delete a record"?
Actions with side effects are tools (model-controlled). Resources are read-only data. A destructive tool should additionally require host-side human approval.
3 · How does an MCP client know what a server offers?
After capability negotiation, the client calls standardized list methods to discover tools/resources/prompts at runtime — no hard-coding, so swapping servers needs no client changes.
4 · Best practice for an ElasticSearch MCP tool?
Least privilege: constrained read-only operations, an allow-list of indices, validated args, and capped result sizes. A generic "run anything" tool is a security and cost hazard.
