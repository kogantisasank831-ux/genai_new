Retrieval-Augmented Generation (RAG)

Module 01 left us with a hard limit: an LLM only knows its training data and hallucinates about anything else — your PDFs, your database, last week's events. RAG is the fix that powers most production GenAI today. The idea is almost embarrassingly simple: before the model answers, retrieve the relevant facts and paste them into the prompt. Everything you learned about embeddings (04) and vector DBs (05) now snaps together into one pipeline.

⏱ ~60 min read Difficulty · Intermediate Prereq · Modules 01, 04, 05

1 · Executive summary

RAG grounds an LLM in external knowledge at query time. You split your documents into chunks, embed them into a vector store, and for each user question you retrieve the most relevant chunks, augment the prompt with them, and let the model generate an answer based on that supplied context. No retraining, knowledge stays current, and answers can cite their sources.

❓

Why it exists

Fine-tuning to teach facts is slow, costly, and goes stale. RAG injects fresh, private knowledge at inference with zero retraining.

🔧

Problem it solves

Hallucination and stale/missing knowledge. Ground the model in retrieved evidence and it answers from your data, with citations.

🌐

Where it fits

The dominant pattern for Q&A over documents, support, search, and enterprise knowledge. Advanced techniques layer on in Module 07.

🔑
The one idea to remember

RAG = search + prompt-stuffing. Retrieval decides what the model sees; generation decides how it phrases the answer. Quality is dominated by retrieval — garbage in, garbage out. Fix retrieval first, prompts second.

2 · First-principles breakdown

What — an open-book exam for the model

A closed-book exam tests memory; an open-book exam lets you look things up. A vanilla LLM takes a closed-book exam every time — it answers from memory and confidently bluffs when it doesn't know. RAG turns it into an open-book exam: it hands the model the relevant pages first, so the answer is grounded in evidence rather than recall.

Why — knowledge changes; weights shouldn't have to

Your documents change daily. Retraining a model for every update is infeasible. RAG decouples knowledge (in a vector store you can update instantly) from reasoning (in the frozen model). Add a document → it's searchable immediately. Delete one → it's gone. This is why RAG, not fine-tuning, is the default for factual Q&A.

⚖️
RAG vs fine-tuning — the rule of thumb

RAG teaches the model what (facts, knowledge). Fine-tuning teaches it how (format, style, behavior). Need current/private facts → RAG. Need a consistent tone or output schema → fine-tune. They compose; they're not rivals.

How — the five-step loop

1 · Ingest & chunk

Load documents, split into retrievable chunks (offline, once).

2 · Embed & index

Embed each chunk (Module 04), store vectors + metadata in a vector DB (Module 05). Offline.

3 · Retrieve

Embed the user query, fetch the top-k nearest chunks. Online.

4 · Augment

Insert retrieved chunks into a prompt template with instructions and the question.

5 · Generate

The LLM answers from the supplied context, ideally citing which chunks it used.

4 · Deep technical explanation

Chunking — the most under-rated lever in RAG

Chunking decides what a single retrievable unit is. Too large → the embedding becomes a mushy average and you waste context (Module 04); too small → you lose surrounding context and retrieve fragments. The goal: each chunk is self-contained enough to answer on its own.

StrategyHowBest for
Fixed-size + overlapN tokens per chunk, ~10–20% overlap so ideas aren't cut mid-thoughtThe pragmatic default
Recursive/structuralSplit on paragraphs → sentences, respecting boundariesProse, docs, articles
Document-structure awareSplit by headings/sections (Markdown, HTML)Manuals, wikis, structured docs
SemanticBreak where embedding similarity drops (topic shifts)Mixed-topic long docs
📏

Start with ~300–500 tokens and 50–80 token overlap, then measure. Always attach metadata (source, page, section) to every chunk — it powers citations and the filtering from Module 05. Chunking is task-specific; tune it with an eval set, don't guess once and forget.

Retrieval parameters

  • top-k — how many chunks to fetch. Too few misses evidence; too many adds noise and cost, and risks "lost in the middle" (Module 02). Typical: 3–8.
  • Score threshold — drop chunks below a similarity floor so irrelevant junk doesn't pollute the prompt (and so you can answer "I don't know" when nothing is relevant).
  • Context budget — total retrieved tokens must fit the window with room for the answer (Module 01 token math).

The prompt template — where grounding is enforced

The augmentation prompt is doing real work. It must (1) supply the context, (2) instruct the model to answer only from it, (3) permit "I don't know," and (4) ask for citations.

rag_prompt.txt
You are a precise assistant. Answer the QUESTION using ONLY the
CONTEXT below. If the context does not contain the answer, say
"I don't have enough information to answer that." Cite sources
as [1], [2] matching the context blocks. Do not use outside knowledge.

CONTEXT:
[1] (source: handbook.pdf, p.12) {chunk_1}
[2] (source: handbook.pdf, p.13) {chunk_2}
[3] (source: policy.md, §3)      {chunk_3}

QUESTION: {user_question}

ANSWER (with citations):

Grounding, citations & "I don't know"

Grounding means the answer is traceable to retrieved evidence. Citations make this auditable for users and for your evals (Module 07/14). The explicit "I don't know" escape hatch is what separates a trustworthy RAG system from a confident liar — without it, the model fills gaps with hallucination.

Tradeoffs & limitations

Strengths

  • Fresh, private knowledge with no retraining
  • Auditable, citable answers
  • Cheaper & faster to update than fine-tuning

Limits (→ Module 07 fixes)

  • Naive vector search misses exact terms (→ hybrid)
  • Top-k ordering imperfect (→ reranking)
  • Vague queries retrieve poorly (→ query expansion)
  • Multi-hop questions need orchestration (→ agentic RAG)

6 · Practical Python

Minimal — RAG in ~30 lines

rag_minimal.py
import numpy as np
from sentence_transformers import SentenceTransformer
from openai import OpenAI

emb = SentenceTransformer("all-MiniLM-L6-v2")
llm = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

docs = ["Returns are accepted within 30 days of purchase.",
        "Shipping is free on orders over $50.",
        "Warranty covers manufacturing defects for 1 year."]
vecs = emb.encode(docs, normalize_embeddings=True)   # index

def rag(question, k=2):
    q = emb.encode(question, normalize_embeddings=True)
    top = np.argsort(-(vecs @ q))[:k]                # retrieve
    context = "\n".join(f"[{i+1}] {docs[d]}" for i, d in enumerate(top))
    prompt = (f"Answer using ONLY this context, cite [n], or say you don't know.\n"
              f"CONTEXT:\n{context}\n\nQUESTION: {question}\nANSWER:")   # augment
    r = llm.chat.completions.create(model="llama3.2", temperature=0,
        messages=[{"role": "user", "content": prompt}])    # generate
    return r.choices[0].message.content

print(rag("How long do I have to return something?"))
# "You have 30 days to return a purchase. [1]"

Intermediate — chunking + a vector store + threshold

rag_chunked.py
from langchain_text_splitters import RecursiveCharacterTextSplitter

def chunk_document(text, source):
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=500, chunk_overlap=60,            # tokens-ish; tune!
        separators=["\n\n", "\n", ". ", " "])
    return [{"text": c, "source": source, "chunk": i}
            for i, c in enumerate(splitter.split_text(text))]

def retrieve(qvec, vecs, chunks, k=5, min_score=0.3):
    scores = vecs @ qvec
    order = scores.argsort()[::-1][:k]
    # threshold: drop weak matches so we can say "I don't know"
    return [(chunks[i], float(scores[i])) for i in order if scores[i] >= min_score]

Production — a RAG service with citations & "no answer" handling

rag_service.py
from dataclasses import dataclass
from fastapi import FastAPI
from pydantic import BaseModel

@dataclass
class Chunk:
    text: str; source: str; page: int; score: float

SYSTEM = ("Answer ONLY from the provided context. Cite sources as [n]. "
          "If the context is insufficient, reply exactly: NO_ANSWER.")

class RAGPipeline:
    def __init__(self, embedder, store, llm):
        self.embedder, self.store, self.llm = embedder, store, llm

    def answer(self, question: str, tenant: str, k=5):
        qvec = self.embedder.embed_query(question)
        hits = self.store.search(qvec, tenant=tenant, k=k)      # Module 05 repo
        hits = [h for h in hits if h.score >= 0.3]
        if not hits:
            return {"answer": "I don't have enough information.", "sources": []}
        ctx = "\n".join(f"[{i+1}] (src:{h.source} p{h.page}) {h.text}"
                        for i, h in enumerate(hits))
        out = self.llm.complete(f"{SYSTEM}\n\nCONTEXT:\n{ctx}\n\nQ: {question}\nA:",
                                temperature=0)
        if "NO_ANSWER" in out:
            return {"answer": "I don't have enough information.", "sources": []}
        return {"answer": out,
                "sources": [{"source": h.source, "page": h.page, "score": h.score} for h in hits]}

app = FastAPI()
class Q(BaseModel): question: str; tenant: str = "default"

@app.post("/ask")
def ask(q: Q):
    return pipeline.answer(q.question, q.tenant)

7 · Ollama implementation — fully offline RAG

Local embedding + local vector DB + local LLM = a RAG system where confidential documents never leave the machine.

offline_rag.py
import ollama, numpy as np

emb = lambda t: np.array(ollama.embeddings(model="nomic-embed-text", prompt=t)["embedding"])

def offline_rag(question, chunks, k=3):
    cvecs = np.stack([emb(c) for c in chunks])
    cvecs /= np.linalg.norm(cvecs, axis=1, keepdims=True)
    q = emb(question); q /= np.linalg.norm(q)
    top = np.argsort(-(cvecs @ q))[:k]
    ctx = "\n".join(f"[{i+1}] {chunks[d]}" for i, d in enumerate(top))
    out = ollama.chat(model="qwen2.5", messages=[{"role": "user",
        "content": f"Use ONLY this context, cite [n], else say you don't know.\n{ctx}\n\nQ: {question}"}])
    return out["message"]["content"]
🔒

Swap the in-memory NumPy index for local Qdrant/pgvector (Module 05) and you have a production-shaped, air-gapped RAG service — the core of the PDF Q&A and on-prem capstones.

8 · Industry perspective

🚀 Startups

RAG is usually the first GenAI feature: a "chat with your docs/data" product. Ship naive RAG fast, then iterate on chunking and retrieval where users complain. Eval-driven, not architecture-astronaut.

🏢 Enterprises

RAG over wikis, tickets, contracts, and policies — with strict access control (per-user/tenant pre-filtering, Module 05), audit trails via citations, and PII guardrails (Module 14). Retrieval quality is a measured KPI, not a hope.

The most common production architecture

Ingestion pipelineload · chunk · embed · upsert (scheduled)
Vector store + metadataaccess control, freshness
RAG serviceretrieve → prompt → generate → cite
Eval + observabilityfaithfulness, recall, cost (Module 14)

Common production challenges: stale indexes (no re-ingestion schedule), retrieval recall too low (bad chunking), hallucination despite context (weak prompt/no "I don't know"), access-control leaks (missing tenant filter), and no way to measure quality (no eval set).

9 · Top mistakes engineers make

❌ Blaming the LLM for retrieval failures

Most "the model is dumb" bugs are bad retrieval. Inspect the retrieved chunks first — if the answer isn't in them, no prompt can save you.

❌ Chunking once and forgetting

Chunk size/overlap massively affects quality. Tune it against an eval set per corpus.

❌ No "I don't know" path

Without it, the model fabricates when retrieval comes up empty. Always provide and test the escape hatch.

❌ Stuffing top-50 chunks

More context isn't better — noise + lost-in-the-middle + cost. Retrieve precisely; rerank later (Module 07).

❌ No citations

Unverifiable answers can't be trusted or evaluated. Cite chunks and surface sources in the UI.

❌ Forgetting to re-ingest

An index frozen at launch slowly rots. Schedule ingestion; handle updates and deletes.

10 · Interview preparation

Beginner  What is RAG and what problem does it solve? ›

Retrieval-Augmented Generation retrieves relevant documents and inserts them into the LLM's prompt before it answers. It solves hallucination and stale/missing knowledge by grounding the model in fresh, private, external data — without retraining — and enables source citations.

Beginner  When would you choose RAG over fine-tuning? ›

Use RAG when you need current or private facts/knowledge that change over time. Use fine-tuning to teach behavior/format/style. RAG updates instantly (add/remove docs) and is cheaper; fine-tuning bakes knowledge into weights and goes stale. They can be combined.

Intermediate  Walk through the RAG pipeline end to end. ›

Offline: load documents → chunk (size + overlap) → embed each chunk → store vectors + metadata in a vector DB. Online: embed the query → retrieve top-k nearest chunks (with metadata filters and a score threshold) → build a prompt that supplies the context and instructs grounding + citations → LLM generates the answer → return answer with sources.

Intermediate  How does chunk size affect retrieval quality? ›

Large chunks dilute the embedding (averaging many topics → poor matching) and waste context tokens. Small chunks lose surrounding context and retrieve fragments. The sweet spot makes each chunk self-contained (~300–500 tokens with overlap). It's corpus-specific — tune with an eval set and respect document structure.

Senior  Your RAG gives correct-looking but wrong answers. How do you debug systematically? ›

Split the pipeline. Retrieval check: log the retrieved chunks — does the answer actually appear in them? If not, it's a retrieval problem (chunking, embedding model, top-k, threshold, missing hybrid/rerank). Generation check: if the evidence is present but the answer is wrong, it's a prompt problem (not enforcing grounding, no "I don't know", lost-in-the-middle from too many chunks). Build an eval set with known answers and measure retrieval recall and answer faithfulness separately so you fix the right stage.

Staff · design  Design a RAG platform over 10M documents with per-user access control and freshness SLAs. ›

Ingestion: a scheduled/event-driven pipeline (connectors → parse → chunk → embed → upsert) with idempotent updates and deletes; track model_version for re-embed migrations. Storage: sharded vector DB with per-document ACL metadata; enforce access via pre-filtering on the user's permissions (never post-filter security). Serving: retrieve → (hybrid + rerank, Module 07) → grounded prompt with citations → generate; cache hot queries; "I don't know" guardrail. Freshness: SLA-driven ingestion cadence + a staleness metric per source. Quality: continuous eval (retrieval recall@k, faithfulness, answer-relevance) gating deploys; observability and cost tracking (Module 14). Treat retrieval quality and access control as the two non-negotiables.

11 · Knowledge check

1 · In RAG, which stage most determines answer quality?
RAG is garbage-in-garbage-out: the model can only answer from what it's given. Debug retrieval (chunks actually contain the answer?) before blaming generation.
2 · You need the system to admit when it doesn't know. What's essential?
Drop weak matches via a threshold and explicitly allow a "no answer" reply. Otherwise the model fabricates to fill the gap.
3 · Why is RAG usually preferred over fine-tuning for factual Q&A?
RAG decouples knowledge (updatable vector store) from reasoning (frozen model). Add/remove docs instantly, cite sources, no costly retraining. Fine-tuning is for behavior/style.
4 · Retrieving 50 chunks instead of 5 most likely…
More context dilutes signal with irrelevant chunks, costs more tokens, and models attend less reliably to the middle of long contexts. Retrieve precisely, then rerank (Module 07).
