LangGraph

LCEL chains (Module 10) are acyclic — they flow one way, start to finish. But real agents (Module 08) loop: reason, act, observe, and come back. They branch, run steps in parallel, pause for human approval, and need to remember where they were if they crash. LangGraph models your agent as a state machine — a graph of nodes and edges over shared state — giving you the control, durability, and human-in-the-loop that production agents demand.

⏱ Study + labs · 16–24 h Difficulty · Advanced Prereq · Modules 08, 10

1 · Executive summary

LangGraph is a library for building stateful, cyclic LLM applications as graphs. You define a shared State (a typed object), Nodes (functions that read state and return updates), and Edges (which node runs next — fixed or conditional). The graph can contain cycles (the agent loop), run nodes in parallel, persist progress via checkpointing, and pause for human input. It's the production-grade orchestration layer for everything in Module 08.

❓

Why it exists

Agent loops, branching, retries, parallelism, and human approval can't be expressed as a DAG. LangGraph adds cycles, state, and durability.

🔧

Problem it solves

Reliable control flow for agents: explicit state, conditional routing, checkpoint/resume, and human-in-the-loop — instead of an opaque while-loop.

🌐

Where it fits

The orchestration engine. Wraps LLM/tool nodes (08), consumes retrievers (10/11), and underpins multi-agent systems (13).

🔑
The one idea to remember

An agent is a state machine: State + Nodes + Edges. Nodes mutate shared state; edges decide the path; cycles enable loops; checkpoints make it durable and pausable. You design the control flow explicitly instead of hoping a while-loop behaves.

2 · First-principles breakdown

What — a flowchart your agent actually executes

You've drawn agent flowcharts: "call the model → if it wants a tool, run it → loop back → else finish." LangGraph lets you build that exact flowchart in code and run it. Boxes are nodes (do something), arrows are edges (go somewhere next), diamonds are conditional edges (branch on a condition). The data passed along the arrows is the state.

📦
State is the spine

Every node receives the current State and returns a partial update. LangGraph merges updates into the state via reducers (e.g. "append to the messages list" vs "overwrite this field"). The whole run is just: start state → node updates state → edge picks next node → repeat until END.

Why — control, durability, and trust

A hand-rolled agent loop (Module 08) works but is opaque: hard to add branching, impossible to pause/resume, and a crash loses everything. LangGraph makes the control flow explicit (you can see and test every path), durable (checkpoints let it resume after a crash or a long-running tool), and safe (it can stop at a node and wait for a human to approve before continuing). Those three are exactly what separates a demo agent from a production one.

How — the build recipe

1 · Define State

A TypedDict (often with reducers) describing everything the graph tracks.

2 · Write nodes

Functions (state) → partial state update. Each does one thing: call the LLM, run a tool, retrieve, etc.

3 · Wire edges

Fixed edges (A → B) and conditional edges (a router function picks the next node).

4 · Compile

graph.compile(checkpointer=...) → a runnable, durable app.

5 · Invoke/stream

Run with a thread id; state persists per thread for memory and resume.

4 · Deep technical explanation

State & reducers

State is a typed schema (TypedDict). By default a node's return overwrites matching keys. For accumulating values (like a message list), you attach a reducer — e.g. Annotated[list, add_messages] means "append, don't replace." Reducers are also what merge parallel branches deterministically.

Nodes & edges

ConceptWhat it is
NodeA function (state) → dict update. The unit of work (LLM call, tool, retrieve).
Normal edgeadd_edge("a", "b") — always go a → b.
Conditional edgeadd_conditional_edges("a", router_fn, {...}) — a function returns the next node's name based on state.
START / ENDSentinel nodes marking entry and termination.
CycleAn edge back to an earlier node — the agent loop. Bounded by a recursion_limit.

Conditional routing — the heart of agent control

A router is just a function of state returning a string (the next node). "Did the LLM request a tool?" → "tools" else END. "Is the answer good enough?" → END else "refine" (reflection, Module 08). This is where you encode all the agent's decision logic explicitly and testably.

Checkpointing & durability

Compile with a checkpointer (in-memory for dev; SQLite/Postgres for prod) and every step's state is saved per thread_id. Consequences: (1) memory — resume a conversation by reusing the thread; (2) durability — crash mid-run and continue from the last checkpoint; (3) time-travel — inspect or fork from any past state; (4) human-in-the-loop — pause and resume hinge on persisted state.

Human-in-the-loop

Use interrupt() inside a node (or compile with interrupt_before=["tools"]) to pause the graph at a decision point. The state is checkpointed; your app surfaces the proposed action to a human; on approval you resume the graph. This is the production mechanism for the "unsafe actions need approval" guard from Module 08.

Tradeoffs & limitations

Strengths

  • Explicit, testable control flow
  • Cycles, branching, parallelism
  • Durable checkpoint/resume + memory
  • First-class human-in-the-loop
  • Streaming of state/events

Limits

  • More boilerplate than a quick chain
  • Steeper learning curve (state/reducers)
  • Overkill for simple acyclic pipelines (use LCEL)
  • You must still bound loops (recursion_limit)

Minimal — state, nodes, a conditional edge

graph_minimal.py
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]   # reducer = append, not overwrite

def agent(state: State):
    last = state["messages"][-1].content
    reply = f"Echo: {last}"                    # pretend LLM call
    return {"messages": [{"role": "assistant", "content": reply}]}

def route(state: State):              # conditional edge fn → next node name
    return END if "bye" in state["messages"][-1].content.lower() else "agent"

g = StateGraph(State)
g.add_node("agent", agent)
g.add_edge(START, "agent")
g.add_conditional_edges("agent", route, {"agent": "agent", END: END})
app = g.compile()
print(app.invoke({"messages": [{"role": "user", "content": "hi"}]}))

Intermediate — a real ReAct agent with tools & memory

react_graph.py
from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode
from langgraph.checkpoint.memory import MemorySaver
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """Current weather for a city."""
    return f"{city}: 22°C clear"

tools = [get_weather]
model = ChatOpenAI(model="gpt-4o-mini", temperature=0).bind_tools(tools)

class State(TypedDict):
    messages: Annotated[list, add_messages]

def agent(state): return {"messages": [model.invoke(state["messages"])]}

def should_continue(state):          # conditional routing
    return "tools" if state["messages"][-1].tool_calls else END

g = StateGraph(State)
g.add_node("agent", agent)
g.add_node("tools", ToolNode(tools))      # prebuilt tool executor
g.add_edge(START, "agent")
g.add_conditional_edges("agent", should_continue, ["tools", END])
g.add_edge("tools", "agent")            # the cycle!

app = g.compile(checkpointer=MemorySaver())   # durable + memory
cfg = {"configurable": {"thread_id": "u1"}}    # per-thread state
for ev in app.stream({"messages": [("user", "weather in Oslo?")]}, cfg):
    print(ev)

Intermediate+ — human-in-the-loop & parallel fan-out

hitl_parallel.py
# --- Human-in-the-loop: pause before the tools node ---
app = g.compile(checkpointer=MemorySaver(), interrupt_before=["tools"])
cfg = {"configurable": {"thread_id": "t1"}}

app.invoke({"messages": [("user", "delete record 42")]}, cfg)   # runs, then PAUSES
state = app.get_state(cfg)
print("proposed:", state.values["messages"][-1].tool_calls)   # show human
# ... human approves ...
app.invoke(None, cfg)                       # resume from checkpoint → executes

# --- Parallel fan-out: 3 edges from "split" run concurrently ---
g2 = StateGraph(State)
for n in ["split", "web", "docs", "db", "merge"]:
    g2.add_node(n, globals()[n] if n in globals() else (lambda s: s))
g2.add_edge(START, "split")
for b in ["web", "docs", "db"]:
    g2.add_edge("split", b)            # fan-out (parallel)
    g2.add_edge(b, "merge")            # fan-in (reducer merges)
g2.add_edge("merge", END)

Production — durable graph with Postgres checkpointing & recursion guard

prod_graph.py
from langgraph.checkpoint.postgres import PostgresSaver
from fastapi import FastAPI
from pydantic import BaseModel

# Durable checkpointer → survives restarts, enables resume + audit.
checkpointer = PostgresSaver.from_conn_string("postgresql://localhost/agents")
checkpointer.setup()
app_graph = g.compile(checkpointer=checkpointer,
                       interrupt_before=["tools"])     # HITL for risky tools

api = FastAPI()
class Msg(BaseModel): text: str; thread: str

@api.post("/chat")
def chat(m: Msg):
    cfg = {"configurable": {"thread_id": m.thread},
           "recursion_limit": 25}             # bound the cycle (Module 08 guard)
    result = app_graph.invoke({"messages": [("user", m.text)]}, cfg)
    snap = app_graph.get_state(cfg)
    pending = snap.next                              # non-empty ⇒ paused for approval
    return {"messages": result["messages"][-1].content,
            "awaiting_approval": bool(pending)}

@api.post("/approve")
def approve(thread: str):
    cfg = {"configurable": {"thread_id": thread}}
    return {"resumed": app_graph.invoke(None, cfg)}   # continue from checkpoint
🛡️

Every Module 08 agent guard has a clean home here: recursion_limit bounds loops; interrupt_before + Postgres checkpoints give durable human approval and audit; conditional edges make every decision explicit and testable. LangGraph turns the opaque while-loop into a system you can reason about.

7 · Ollama implementation

LangGraph orchestrates any model — point the agent node at a local model for a fully offline stateful agent.

local_graph.py
from langchain_ollama import ChatOllama
from langgraph.prebuilt import create_react_agent
from langchain_core.tools import tool

@tool
def inventory(item: str) -> str:
    """Stock count for an item."""
    return f"{item}: 42 units"

# Prebuilt ReAct agent graph backed by a LOCAL model — offline + stateful
model = ChatOllama(model="qwen2.5", temperature=0)
agent = create_react_agent(model, tools=[inventory])

for ev in agent.stream({"messages": [("user", "how many pumps in stock?")]}):
    print(ev)
⚙️

create_react_agent is the prebuilt version of the graph you wired by hand in §6 — great for getting started. Build the graph explicitly when you need custom routing, parallelism, or HITL. Local models flail more (Module 08), so keep recursion_limit tight and tools well-described.

8 · Industry perspective

🚀 Startups

Start with create_react_agent for speed; graduate to an explicit graph when you need approval gates, branching, or parallel retrieval. The checkpointer gives you durable chat memory almost for free.

🏢 Enterprises

Value the explicit, auditable control flow, Postgres-checkpointed durability, and human-in-the-loop for consequential actions. The graph is the spec — reviewable, testable, and compliant. LangGraph Platform adds deployment/persistence at scale.

9 · Top mistakes engineers make

❌ No recursion limit

A cyclic graph can loop forever. Always set recursion_limit (and add reflection/escalation).

❌ Wrong reducer

Overwriting where you meant to append (or vice versa) corrupts state — especially with parallel branches. Choose reducers deliberately.

❌ Using LangGraph for a simple chain

If there's no loop/branch/state, LCEL is simpler. Don't add a state machine you don't need.

❌ In-memory checkpointer in prod

State vanishes on restart. Use SQLite/Postgres for durability, resume, and audit.

❌ Skipping human-in-the-loop for risky tools

interrupt_before on destructive nodes is the safety mechanism. Use it.

❌ Hiding logic in nodes instead of edges

Routing decisions belong in conditional edges (explicit, testable), not buried inside node side effects.

10 · Interview preparation

Beginner  What is LangGraph and why not just use LCEL? ›

LangGraph models LLM apps as stateful graphs (state + nodes + edges) that support cycles, branching, parallelism, durability, and human-in-the-loop. LCEL composes acyclic pipelines and can't express loops or pause/resume. Agents inherently loop and branch, so they need a graph, not a DAG.

Beginner  What are nodes and edges? ›

A node is a function that takes the current state and returns a partial update (the unit of work — an LLM call, a tool, a retrieval). An edge connects nodes: a normal edge always goes A→B; a conditional edge runs a router function that returns the next node's name based on state. START and END are sentinels.

Intermediate  What is a reducer and why does it matter? ›

A reducer defines how a node's returned value merges into existing state. By default keys are overwritten; for accumulating fields (like a message list) you use a reducer such as add_messages to append. Reducers are also essential for deterministically merging the outputs of parallel branches — without the right reducer, concurrent updates clobber each other.

Intermediate  How does checkpointing enable memory and human-in-the-loop? ›

A checkpointer persists state after each step, keyed by thread_id. Memory: reuse a thread to continue a conversation with full prior state. Human-in-the-loop: interrupt_before a node pauses the graph and checkpoints state; your app shows the proposed action to a human; calling invoke with None on the same thread resumes from that exact checkpoint. Same mechanism enables crash-resume and time-travel debugging.

Senior  Design an agent that must get human approval before any write action. ›

Separate read tools from write tools into different nodes (or route by tool risk). Compile with a durable checkpointer and interrupt_before=["write_tools"]. Flow: agent proposes → conditional edge routes a write to the write node → graph pauses and checkpoints → your UI surfaces the proposed call (with diff/dry-run) → human approves/edits/rejects → resume from checkpoint to execute, or inject a rejection message and loop back to the agent. Bound the loop with recursion_limit, log every state transition for audit, and store checkpoints in Postgres so approvals can happen asynchronously (minutes/hours later) and survive restarts.

Senior  How do you run retrieval from three sources concurrently and merge? ›

From a single "split" node, add edges to three retrieval nodes (web/docs/db); LangGraph runs nodes with no dependency between them concurrently. Each writes to a state field with an appropriate reducer (e.g. a list that accumulates results) so their outputs merge deterministically at a downstream "merge" node, which all three edge into. Wall-clock becomes the slowest branch, not the sum. Guard each branch with its own error handling so one failure doesn't sink the merge.

Staff · design  Design a durable, multi-step research agent platform on LangGraph. ›

Model each workflow as an explicit graph: planner node → parallel retrieval/tool nodes (fan-out with reducers) → synthesis node → reflection node (conditional edge loops back if quality is low). Compile with a Postgres checkpointer for durability, resume, and audit; key runs by thread_id for memory and multi-session continuation. Add interrupt_before on consequential nodes for human approval; bound cycles with recursion_limit and step/token budgets (Module 08). Wrap external tools as MCP servers (Module 09) and retrievers as LlamaIndex query engines (Module 11). Add tracing/observability (Module 14) over the event stream, an eval suite over graph trajectories gating deploys, and per-tenant state isolation. Stream state/events to the UI for transparency. The graph is the auditable spec; durability + HITL + explicit routing are what make it production-grade.

11 · Knowledge check

1 · Why can't an LCEL chain implement a ReAct agent loop?
The ReAct loop revisits the agent node after each tool call — a cycle. LCEL pipelines are one-directional DAGs, so cyclic control flow needs LangGraph.
2 · A node returns {"messages": [new_msg]} and the list grows instead of being replaced. Why?
Reducers define merge behavior. Annotated[list, add_messages] appends returns rather than overwriting — essential for conversation state and parallel merges.
3 · What does interrupt_before=["tools"] do?
It's the human-in-the-loop mechanism: the graph halts before the tools node, persists state, and resumes from that checkpoint once approved.
4 · Which is essential to prevent an infinite agent loop?
Cyclic graphs can loop forever. recursion_limit bounds the number of steps; combine with reflection or escalation for graceful termination.

12 · Active recall

Answer first, then reveal. Re-test tomorrow and in a week.

Q Name the three core LangGraph building blocks. ›
State (typed shared data), Nodes (functions that update state), Edges (which node runs next — normal or conditional). Cycles + checkpointing build on these.
Q What is a conditional edge? ›
An edge whose target is decided at runtime by a router function of the current state, returning the next node's name (e.g. "tools" vs END). It's where agent decision logic lives explicitly.
Q List the four things checkpointing enables. ›
Memory (resume a thread), durability (crash-resume), time-travel (inspect/fork past states), and human-in-the-loop (pause/resume).
Q How do you run nodes in parallel and merge them safely? ›
Add multiple edges from one node (fan-out → concurrent execution); have branches write to a state field with an appropriate reducer; converge their edges on a merge node (fan-in).
Q When do you use LangGraph instead of LCEL? ›
When you need cycles (agent loops), conditional branching, parallelism, durable state/resume, or human-in-the-loop. For one-way acyclic pipelines, LCEL is simpler.
